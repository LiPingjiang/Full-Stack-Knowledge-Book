# 2.4 前端工程化：你早就有的 Maven / Gradle 直觉

> 后端工程师的偏见：「前端不就是几个 HTML/JS 文件丢服务器上吗，要什么工程化？」
> 真相：现代前端的工程化复杂度**不亚于后端**，而且和你熟悉的 Maven/Gradle 几乎一一对应。
> 这一章帮你把后端的构建直觉，平移到前端工具链上。

---

## 一、为什么前端需要工程化

先回答那个偏见。早期前端确实是「几个文件丢上去」，但现代前端要面对：

- **模块化**：成千上万个 JS 文件，需要管理依赖关系（像你后端的 package 与 import）
- **依赖管理**：要引入海量第三方库（像 Maven 中央仓库）
- **编译转换**：TS → JS、JSX → JS、新语法 → 老浏览器能懂的语法（像 Java 编译成字节码）
- **打包优化**：把一堆文件压缩、合并、按需拆分，让用户加载得快
- **质量保障**：代码规范检查、自动化测试（像你后端的 Checkstyle、JUnit）

这一整套，就是**前端工程化**。它要解决的问题，和你后端用 Maven/Gradle 解决的问题**高度同构**。

---

## 二、核心对照：前端工具链 vs Maven / Gradle

直接上对照表，这是本章最该记住的东西：

| 前端 | 后端（Maven/Gradle） | 作用 |
|------|---------------------|------|
| `package.json` | `pom.xml` / `build.gradle` | 项目描述 + 依赖声明 + 脚本 |
| npm / pnpm / yarn | Maven / Gradle | 包管理器，拉依赖、跑脚本 |
| npm 仓库（npmjs.com） | Maven 中央仓库 | 第三方依赖的来源 |
| `node_modules/` | `~/.m2/repository` + classpath | 下载下来的依赖存放处 |
| `package-lock.json` | `gradle.lockfile` | 锁定精确版本，保证可复现构建 |
| Vite / Webpack | Maven 编译 + 打包插件 | 编译 + 打包成可部署产物 |
| Babel / tsc / SWC | javac | 把源码编译成目标环境能跑的代码 |
| ESLint | Checkstyle / SpotBugs | 静态代码检查 |
| Prettier | 代码格式化插件 | 统一代码风格 |
| Vitest / Jest | JUnit | 单元测试框架 |

看完这张表，你应该松一口气：**前端工程化没有任何新概念，全是你后端玩剩下的，只是换了名字。**

## 三、`package.json`：前端的 `pom.xml`

打开任意前端项目，第一个看的就是 `package.json`。它和 `pom.xml` 一一对应：

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "test": "vitest",
    "lint": "eslint ."
  },
  "dependencies": {
    "react": "^18.2.0",
    "zustand": "^4.5.0"
  },
  "devDependencies": {
    "vite": "^5.0.0",
    "typescript": "^5.3.0"
  }
}
```

逐段对应你熟悉的 `pom.xml`：

- `dependencies` = `<dependencies>` 里 `scope=compile` 的依赖（运行时要用）
- `devDependencies` = `scope=provided/test` 的依赖（只在开发/构建时用，不进生产包）
- `scripts` = Maven 的 `<plugin>` goal 或 Gradle 的 task（如 `mvn test` ≈ `npm run test`）
- `^18.2.0` 这个版本号 = 语义化版本，`^` 类似 Maven 的版本范围（允许小版本升级）

跑命令也一一对应：

```bash
npm install          # ≈ mvn install（拉所有依赖）
npm run dev          # 启动开发服务器
npm run build        # ≈ mvn package（打生产包）
npm run test         # ≈ mvn test
```

## 四、模块化：前端的 import / package

后端的模块化你早就精通：用 `package` 组织、用 `import` 引入。前端走过一段混乱的历史，但现在统一到了 **ES Modules（ESM）** 标准，写法和你的直觉完全一致：

```javascript
// math.js —— 导出，像 public 方法对外暴露
export function add(a, b) {
  return a + b;
}
export const PI = 3.14;

// main.js —— 导入，像 import 一个类的方法
import { add, PI } from './math.js';
console.log(add(1, 2));
```

| 前端 ESM | Java | 说明 |
|---------|------|------|
| `export` | `public` | 对外暴露 |
| 不 export | `private` | 模块内私有 |
| `import { x } from './m'` | `import com.x.M` | 引入依赖 |
| 一个文件 = 一个模块 | 一个 `.java` 文件 | 组织单元 |

> 历史包袱提醒：你可能会在老代码里看到 `require()` / `module.exports`（CommonJS 规范），那是 Node 早期的模块系统。现在新项目统一用 `import/export`（ESM），知道有这回事即可。

## 五、构建工具：Vite 是怎么回事

现代前端构建的事实标准是 **Vite**（读音「veet」）。你只需理解它做两件事：

**开发时**：起一个本地服务器，你改代码它**毫秒级热更新**到浏览器（类比 Spring Boot DevTools 的热重载，但快得多）。

**构建时**：把你的 TS/JSX/多文件，编译、打包、压缩成几个优化过的静态文件（HTML/CSS/JS），可以直接丢到任何静态服务器或 CDN。这一步就是前端的 `mvn package` ——产出可部署产物。

```bash
npm run build
# 产物在 dist/ 目录，类似 Maven 的 target/ 目录
# dist/ 里就是最终要部署的 HTML/CSS/JS
```

老项目里你可能还会遇到 **Webpack**（Vite 之前的王者，配置更繁琐）。原理类似，知道它是「同类工具的上一代」即可，新项目首选 Vite。

## 六、质量保障：ESLint + Prettier + 测试

这一层和你后端的代码质量工具完全对应，思想无需重新学：

- **ESLint** = Checkstyle / SpotBugs，静态扫描代码里的潜在问题和不规范写法。
- **Prettier** = 代码格式化工具，统一缩进、引号、分号等风格（团队不用再为格式吵架）。
- **Vitest / Jest** = JUnit，写单元测试：

```javascript
import { describe, it, expect } from 'vitest';
import { add } from './math.js';

describe('add 函数', () => {
  it('1 + 2 应该等于 3', () => {
    expect(add(1, 2)).toBe(3);     // 像 JUnit 的 assertEquals
  });
});
```

`expect(...).toBe(...)` 就是断言，对应 JUnit 的 `assertEquals`。整套测试理念（AAA：Arrange-Act-Assert）和你后端一模一样。

## 七、给后端大脑的「翻译词典」（速查总表）

| 你想做的事 | 后端怎么做 | 前端怎么做 |
|----------|----------|----------|
| 声明项目和依赖 | `pom.xml` | `package.json` |
| 拉取依赖 | `mvn install` | `npm install` |
| 打生产包 | `mvn package` | `npm run build` |
| 跑测试 | `mvn test` | `npm run test` |
| 静态检查 | Checkstyle | ESLint |
| 编译源码 | javac | tsc / Babel / SWC |
| 模块导出 | `public` | `export` |
| 模块导入 | `import` | `import` |
| 产物目录 | `target/` | `dist/` |

## 本章小结

- 现代前端工程化的复杂度不亚于后端，但**没有任何新概念**——全是 Maven/Gradle 那套思想换了名字。
- `package.json` = `pom.xml`，npm/pnpm = Maven/Gradle，npm 仓库 = Maven 中央仓库。
- 模块化已统一到 **ESM**（`import/export`），写法和你的 `import` 直觉一致；老代码里的 CommonJS（`require`）知道即可。
- 构建工具首选 **Vite**：开发热更新 + 构建打包（前端的 `mvn package`），产物丢 `dist/`。
- 质量保障三件套：ESLint（≈Checkstyle）、Prettier（格式化）、Vitest/Jest（≈JUnit），理念与后端一致。

至此第二章完结。你现在拥有了**前端的整套知识地图**：三件套（界面）、框架（组件 + 声明式）、状态管理（数据治理）、工程化（构建工具链）。**这些知识没有一个是孤立的新东西，全都能在你的后端经验里找到锚点。** 接下来第三章，我们回到你的主场——把 Java 的深度知识彻底夯实，为后面的多语言对比打好地基。
