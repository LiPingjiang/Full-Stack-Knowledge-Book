# 2.2 现代前端框架：React / Vue 的组件化与声明式渲染

> 后端工程师常问：「不就是改个网页吗，为什么要搞这么重的框架？」
> 答案和你当年「为什么不用裸 Servlet 而要上 Spring」是同一个逻辑——**框架在替你解决重复、易错的脏活**。
> 这一章讲清楚前端框架到底替你解决了什么。

---

## 一、先理解「没有框架」有多痛

回顾 [1.2 从无状态到有状态](../part1-mindset-shift/02-从无状态到有状态.md) 里的命令式 UI：每次状态变了，你都要手动跑去把界面对应的地方一处处改掉。我们把那个痛点放大来看。

假设一个页面要显示「购物车有 3 件商品」，这个数字出现在：顶部角标、购物车标题、结算按钮文案。用原生 JS（无框架）：

```javascript
let count = 3;

function updateCart(newCount) {
  count = newCount;
  // 灾难开始：你得手动同步每一处
  document.getElementById('badge').textContent = count;
  document.getElementById('cart-title').textContent = `购物车（${count}）`;
  document.getElementById('checkout-btn').textContent = `结算 ${count} 件`;
  // 漏掉任何一处，界面就和数据不一致 = bug
  // 页面越复杂，要同步的地方越多，越容易漏
}
```

这就像你后端**不用 ORM，每次都手写 JDBC**：能跑，但样板代码多、易出错、难维护。前端框架要解决的核心痛点，就是**消灭这种「手动同步状态到界面」的脏活**。

---

## 二、框架的第一个核心：声明式渲染（`UI = f(state)`）

框架的第一个杀手锏，就是把命令式变成**声明式**。你不再写「怎么一步步改界面」，而只声明「界面长什么样取决于状态」，状态一变，框架自动重新渲染。

同样的购物车，用 React：

```jsx
function Cart() {
  const [count, setCount] = useState(3);

  // 我只"声明"了三处都用 count，至于怎么更新到屏幕，框架包办
  return (
    <>
      <span className="badge">{count}</span>
      <h2>购物车（{count}）</h2>
      <button>结算 {count} 件</button>
    </>
  );
}

// 改数据只需一行，三处界面自动同步，绝不会漏
setCount(5);
```

对比一下你熟悉的东西：

- 命令式 = 手写 JDBC，每处 `UPDATE` 都要自己写、自己保证一致。
- 声明式 = 用 ORM / 数据库视图，你只声明映射关系，底层数据变了，查出来自动是新的。

框架内部靠 **虚拟 DOM diff**（React）或 **响应式依赖追踪**（Vue）来实现「高效地只更新变化的部分」，但这是它的内部实现，你享受结果即可——就像你用 Hibernate 不需要关心它内部怎么生成 SQL。

<details>
<summary><b>展开：「响应式」到底是什么意思？和后端的 Reactive 是一回事吗？</b></summary>

前端说的「响应式」（Reactivity）一句话就能讲清：**你改了数据，界面自动更新——不需要你手动去找到界面上用到这个数据的地方逐个刷新。**

**用 Excel 类比最好懂**：A1 格写了 `100`，B1 格写了 `=A1*2`，C1 格写了 `=A1+B1`。你改 A1 为 `200`，B1 和 C1 自动变——你不需要手动刷新它们。Vue 的响应式系统就是这个意思：`ref(100)` 相当于 A1 格，模板里用到它的地方相当于 B1、C1，你改了 ref 的值，所有用到它的地方自动更新。

**底层怎么做到的？** Vue 3 用 JavaScript 的 `Proxy`（代理对象）拦截你对数据的读和写：

| 时机 | Proxy 做了什么 | 类比 |
|------|---------------|------|
| 模板渲染时**读**了某个数据 | 记下「这块模板依赖这个数据」（**依赖收集**） | 数据库视图记住了自己依赖哪些表 |
| 你**改**了这个数据 | set 拦截器触发 → 查依赖表 → 只重渲染依赖了它的那部分（**精准更新**） | 触发器：数据一改，关联操作自动执行 |

**React 和 Vue 的区别**：React 没有 Proxy 自动追踪，它靠你手动调 `setState` 通知框架「数据变了」，然后重新执行整个组件函数、做虚拟 DOM diff 找差异。所以 React 是「手动通知 + diff 找差异」，Vue 是「自动追踪 + 精准更新」——**结果都是「数据变了界面跟着变」，但机制不同**。

**和后端的 Reactive（WebFlux / RxJava）完全不是一回事。** 后端的 Reactive 是「响应式编程」，解决的是异步非阻塞 IO 和流式数据处理（Mono/Flux，见 [3.6 网络 IO 模型](../part3-java-deep/06-网络IO模型.md) 中 Spring WebFlux 的介绍）；前端的 Reactive 是「响应式数据绑定」，解决的是数据变化自动同步到界面。**名字一样，问题域完全不同，别混淆。**

</details>

<details>
<summary><b>展开：后端为什么需要知道 DOM？虚拟 DOM 又是什么？</b></summary>

**DOM（Document Object Model）= 浏览器把 HTML 解析后生成的一棵树形对象。** 你可以把它理解成「浏览器内存里的一张表」——HTML 是建表 DDL，DOM 是跑起来之后内存里那张表的实例。每个 HTML 标签（`<div>`、`<span>`、`<button>`）就是树里的一个节点对象，`document.getElementById('xxx')` 拿到的就是树上的一个节点，改它的属性 = 改了页面显示。

**后端程序员为什么需要知道它？** 因为前端的一切交互，本质都是在操作这棵树。你理解 DOM，才能理解：

| 你会遇到的场景 | 和 DOM 的关系 |
|--------------|--------------|
| 组件渲染性能优化 | 操作真实 DOM 很慢（触发布局重算+重绘），所以才有虚拟 DOM |
| React 列表为什么要加 `key` | diff 算法靠 key 判断「是新增节点还是移动了」 |
| SSR（服务端渲染）为什么能加速首屏 | 服务端直接吐 HTML → 浏览器不用等 JS 执行后才构建 DOM |
| SEO 优化 | 搜索引擎爬虫只看 HTML/DOM，看不到 JS 动态生成的内容 |

**虚拟 DOM** 是 React 的核心优化手段：先在 JS 里用普通对象模拟一棵「虚拟树」（轻量、操作快），每次状态变化时生成新虚拟树 → 和旧虚拟树做 diff → 算出最小差异 → 一次性批量打到真实 DOM 上。

类比：**批量 UPDATE 合并成一条 SQL 提交，而不是一行行 UPDATE 触发一行行 flush。** 或者像 Git diff——不是把整个文件删了重写，而是算出哪几行变了只改那几行。

</details>

<details>
<summary><b>展开：setCount 是函数内部的变量，为什么能在事件回调里调用？——闭包</b></summary>

你的直觉完全对——从 Java 视角看，`const [count, setCount] = useState(3)` 确实是函数内部的局部变量。上面示例中 `setCount(5)` 写在函数外面是**为了演示效果的简化写法**，实际场景里它只在组件内部或其事件回调里调用：

```jsx
function Cart() {
  const [count, setCount] = useState(3);

  // setCount 在事件回调里被调用——这才是真实写法
  return <button onClick={() => setCount(count + 1)}>加一件</button>;
}
```

但 JS 确实允许「内部定义的东西被内部的另一个函数引用并带出去」——这个机制叫**闭包（Closure）**。

**用 Java 来类比**：闭包最接近 Lambda 表达式捕获外部变量。

```java
// Java Lambda 也能「记住」外部变量
int threshold = 10;
Predicate<Integer> isAbove = n -> n > threshold; // Lambda 捕获了 threshold
```

区别在于：Java 要求被捕获的变量是 `effectively final`（不能改），而 JS 的闭包没有这个限制——`setCount` 本身就是一个函数引用，被 `onClick` 回调「记住」了。每次 React 重新执行 `Cart()` 函数时，都会创建新的 `count` 和 `setCount`，但事件回调拿到的永远是当次渲染时的闭包快照。

**一句话总结**：JS 函数是一等公民，可以被传来传去；闭包让它「记住」自己出生时的环境。这是 React Hooks 能工作的底层前提。

</details>

## 三、框架的第二个核心：组件化

框架的第二个杀手锏是**组件化**——把界面拆成可复用、可组合的积木。这一点你在 [1.5 UI 建模](../part1-mindset-shift/05-从数据建模到UI建模.md) 已经预习过，这里看它在代码里长什么样。

一个组件，本质是一个**「接收数据（props）、返回界面」的函数**。你可以完全把它类比成后端的一个方法或一个可复用的 Service：

```jsx
// 定义一个可复用组件，像定义一个方法
function UserCard({ name, avatar, role }) {   // props 就是入参
  return (
    <div className="user-card">
      <img src={avatar} />
      <span>{name}</span>
      <small>{role}</small>
    </div>
  );
}

// 到处复用它，像调用方法，传不同的参数
<UserCard name="李先生" avatar="/a.png" role="架构师" />
<UserCard name="王女士" avatar="/b.png" role="工程师" />
```

组件化带来的好处，和你后端「把逻辑抽成可复用方法/服务」完全一致：

| 组件化的好处 | 后端的对应实践 |
|------------|--------------|
| 一处定义，到处复用 | 抽公共方法 / 工具类 |
| 各组件职责单一、独立 | 单一职责的 Service |
| 组件可独立测试 | 单元测试一个方法 |
| 组合小组件成大页面 | 组合小方法成复杂业务 |

<details>
<summary><b>展开：组件不就是个函数/类吗？为什么单独命名「组件」？前端没有类的概念吗？</b></summary>

前端当然有类（ES6 `class`），React 早期就是用 class 写组件的。但这个问题的关键在于：**组件不只是一个函数，它是一个「自带状态 + 生命周期 + 被框架调度 + 可组合成树」的 UI 单元。** 和普通函数/方法相比，它多出了几个框架赋予的维度：

| 维度 | 普通函数/方法 | 组件 |
|------|-------------|------|
| 有自己的状态 | ❌ 纯计算、无状态 | ✅ `useState` 让每个实例有独立状态 |
| 有生命周期 | ❌ | ✅ 挂载/更新/销毁 |
| 由框架调度执行 | ❌ 你自己决定什么时候调用 | ✅ 状态变了框架自动重新执行它 |
| 嵌套组合成树 | 调用链（线性） | 组件树（和 DOM 树对应） |
| 每个实例独立 | 取决于你怎么写 | ✅ 同一组件用两次，状态互不干扰 |

后端最接近的类比，其实不是「一个方法」，而是**一个有状态的微服务**——它有自己的数据（state）、有初始化和销毁逻辑（生命周期）、暴露接口（props）给外部调用、多个实例各自独立运行。

**为什么要起「组件」这个名字？** 因为它和框架之间有一个**契约**：你把函数写成特定形状（接收 props、返回 JSX），框架就承诺在状态变化时自动重新执行它、高效更新 DOM。普通函数没有这层契约——你调了它就跑，没人帮你管状态和重渲染。「组件」这个词强调的就是这层**框架与你之间的合作协议**。

所以：组件 = 函数 + 状态 + 生命周期 + 框架调度契约。它当然底层是函数（或类），但「组件」这个名字标记了它参与的那套框架协议。

</details>

## 四、React vs Vue：选哪个，以及它们的本质差异

这是后端工程师最纠结的选择题。先给结论：**对你而言，先学哪个都行，两者思想相通，学会一个再上手另一个只需几天。** 下面讲清它们的真实差异，帮你选。

| 维度 | React | Vue |
|------|-------|-----|
| 心智模型 | 「一切皆 JS」，用 JS 表达一切（JSX） | 「模板 + 增强」，HTML 模板里加指令 |
| 上手门槛 | 略陡（需理解函数式、Hooks 心智） | 较平缓（更接近传统 HTML 直觉） |
| 灵活度 | 极高，但「自由」也意味着要自己做选择 | 约定更多，开箱即用的东西更全 |
| 生态/招聘 | 全球最大，大厂多用 | 国内（尤其中小厂）流行度高 |
| 状态更新 | 手动调 `setState`，重新执行函数 | 响应式，改数据自动追踪更新 |

一个直观的代码气质对比，同样是「点击计数」：

```jsx
// React：一切皆 JS
function Counter() {
  const [n, setN] = useState(0);
  return <button onClick={() => setN(n + 1)}>点了 {n} 次</button>;
}
```

```vue
<!-- Vue：模板 + 响应式 -->
<script setup>
import { ref } from 'vue';
const n = ref(0);
</script>
<template>
  <button @click="n++">点了 {{ n }} 次</button>
</template>
```

**给你的选型建议**：如果你追求生态、招聘面、长期主流，且不排斥「一切皆 JS」的思路，选 **React**（也是 AI 工具支持最好的）。如果你更喜欢「贴近 HTML、约定多、上手快」，选 **Vue**。在 AI Coding 时代，这个选择的成本被进一步拉低了——AI 两个都熟，你换框架时它能帮你快速翻译。

## 五、组件的「生命周期」：又一个状态机

组件不是静态的，它有**生命周期**：被创建（挂载到页面）→ 更新（状态变化重渲染）→ 销毁（从页面移除）。这又是一个你熟悉的**状态机**概念，也呼应 [1.1](../part1-mindset-shift/01-从请求响应到用户交互.md) 讲的「前端程序常驻、需手动清理」。

React 用 `useEffect` 统一处理生命周期相关的「副作用」：

```jsx
function Timer() {
  useEffect(() => {
    // 组件挂载时执行：开一个定时器
    const id = setInterval(() => console.log('tick'), 1000);

    // 返回的函数在组件销毁时执行：清理定时器（防止 [1.1] 说的内存泄漏）
    return () => clearInterval(id);
  }, []);  // 空依赖数组 = 只在挂载/销毁时跑

  return <div>计时中...</div>;
}
```

这个「挂载时启动、销毁时清理」的模式，对应你后端的 `@PostConstruct` / `@PreDestroy`，或 `try-with-resources` 的资源管理。本质都是**「在生命周期的特定时点做对应的初始化和收尾」**。

<details>
<summary><b>展开：React Hooks 全家福——后端视角速查表</b></summary>

`useEffect` 只是 React Hooks 中的一个。Hooks 是 React 给函数组件「赋能」的一组工具函数——**每个 Hook 解决一类特定问题**。你不需要一上来全学，但认个脸、知道什么时候用哪个就够了：

| Hook | 一句话说清 | 后端类比 | 重要度 |
|------|----------|---------|--------|
| **`useState`** | 给组件加状态变量，值变了界面自动更新 | 类似一个会触发视图刷新的 setter | ⭐⭐⭐ 必会 |
| **`useEffect`** | 处理副作用：发请求、操作 DOM、启停定时器 | `@PostConstruct` + `@PreDestroy` | ⭐⭐⭐ 必会 |
| **`useContext`** | 跨层级共享数据，不用一层层传 props | 类似 Spring 的依赖注入：子组件直接「注入」祖先提供的值，不用逐层传递 | ⭐⭐ 常用 |
| **`useRef`** | 保存一个「不触发重渲染」的可变引用 | 类似一个实例变量（字段）：值变了不会触发 UI 更新，常用于拿到 DOM 节点或保存定时器 ID | ⭐⭐ 常用 |
| **`useMemo`** | 缓存一个计算结果，依赖不变就不重算 | 类似 `@Cacheable`：输入没变就返回缓存值，避免每次渲染都重复昂贵计算 | ⭐ 优化用 |
| **`useCallback`** | `useMemo` 的特化版：缓存一个函数引用 | 避免每次渲染都创建新函数导致子组件白重渲染 | ⭐ 优化用 |
| **自定义 Hook** | 把可复用的状态逻辑抽成函数，以 `use` 开头命名 | 就像抽一个公共 Service / 工具方法：`useAuth()`、`useFetch(url)` | ⭐⭐ 工程必备 |

**核心心智模型**：React 的官方理念是 **「UI 是带状态的函数」**——组件就是一个函数，Hooks 就是给这个函数「挂载」各种能力（状态、副作用、缓存、上下文……）的工具。理解了这句话，所有 Hook 都是自然推导出来的：函数本身没有状态，所以需要 `useState`；函数本身没有副作用，所以需要 `useEffect`；函数每次调用都是全新的，所以需要 `useMemo` 缓存——以此类推。

**`useContext` 稍微展开一句**：React 组件是树形嵌套的，数据默认只能通过 props 从父组件一层层传给子组件。如果「当前登录用户」这个数据要从最顶层传到第 5 层深的组件，中间每一层都得加一个 props 透传——这叫 **prop drilling**（属性逐层穿透），非常烦。`useContext` 的作用就是让祖先组件把数据「广播」出去，任何后代组件直接用 `useContext` 就能拿到，跳过中间层。这跟 Spring 里你不需要手动 new 依赖再一层层传进去、而是直接 `@Autowired` 注入，是同一个思路。

</details>

## 六、给后端大脑的「翻译词典」

| 前端概念 | 后端类比 | 说明 |
|---------|---------|------|
| 不用框架（原生 JS） | 裸 Servlet / 手写 JDBC | 能跑但样板多、易错 |
| 框架 | Spring / Hibernate | 替你解决重复脏活 |
| 声明式渲染 | ORM / 数据库视图 | 声明映射，变化自动传播 |
| 组件 | 可复用方法 / Service | 接收入参（props）返回结果（界面） |
| 组件组合 | 方法组合成业务 | 小积木拼大页面 |
| 组件生命周期 | `@PostConstruct`/`@PreDestroy` | 特定时点做初始化/清理 |
| `useEffect` 清理函数 | `try-with-resources` | 防止资源泄漏 |

---

## 本章小结

- 前端框架存在的理由，和你后端「上 Spring 而非裸 Servlet」一样：**替你消灭重复、易错的脏活**。
- 框架两大核心：**声明式渲染**（`UI = f(state)`，类比 ORM/视图）和**组件化**（可复用积木，类比可复用方法）。
- React（一切皆 JS、生态最大、AI 支持最好）vs Vue（模板 + 响应式、上手快），**思想相通，先学哪个都行**。
- 组件**生命周期**是又一个状态机，`useEffect` 的「挂载启动 / 销毁清理」对应后端的资源生命周期管理。

记住一句话：**前端框架不是给前端「炫技」用的，它和 Spring 是同一种东西——把工程师从重复劳动里解放出来，让你专注业务逻辑。** 你早就懂这种价值。
