# 2.3 前端状态管理：从「方法传参」到「全局状态」

> 后端工程师的困惑：「数据不就放数据库 / 缓存吗，前端搞什么状态管理？」
> 关键认知：前端的「状态」是**内驻在用户这一次会话内存里的数据**（回顾 [1.2](../part1-mindset-shift/02-从无状态到有状态.md)）。
> 当状态需要在很多组件间共享时，怎么传、放哪、谁来改——这就是状态管理要解决的问题。

---

## 一、问题的起点：状态要在组件间共享

[2.2](./02-现代框架.md) 讲了组件像「方法」，靠 props（入参）接收数据。最简单的传值方式就是**父组件把状态当 props 传给子组件**，像方法调用层层传参。

但麻烦来了。考虑一个典型场景：用户登录信息（用户名、头像、权限）。这个状态：

- 顶部导航栏要用（显示头像）
- 侧边菜单要用（按权限显示菜单项）
- 页面深处的某个按钮要用（判断能否点击）

如果只靠 props 一层层往下传，就会出现 **prop drilling（属性钻取）** ——数据要穿过十几层根本用不到它的中间组件，只为送到最深处那个组件。这就像你后端**一个参数为了传到最底层方法，被迫加到沿途每一个方法的签名里**，又脏又难维护。

```jsx
// prop drilling 的痛：user 穿过一堆根本不关心它的中间层
<App user={user}>
  <Layout user={user}>           {/* Layout 不用 user，只是过路 */}
    <Sidebar user={user}>        {/* Sidebar 不用 user，只是过路 */}
      <Menu user={user}>         {/* 终于到了真正要用的地方 */}
```

---

## 二、状态的分层：先想清楚「这个状态归谁管」

在学具体工具前，先建立**最重要的认知**——前端状态不是只有一种，要按「作用范围」分层管理。这和你后端「数据该放局部变量、还是成员变量、还是缓存、还是数据库」的分层直觉一模一样。

| 状态类型 | 作用范围 | 后端类比 | 用什么管 |
|---------|---------|---------|---------|
| 局部状态 | 单个组件内部 | 方法内局部变量 | 组件自带的 `useState` |
| 跨组件共享状态 | 多个组件共享 | 单例 Bean / 应用级缓存 | Context / 状态库 |
| 服务端状态 | 来自后端 API 的数据 | 数据库数据的本地副本 | React Query / SWR |
| URL 状态 | 反映在地址栏 | 请求参数 | 路由库 |

**新手最常犯的错，就是把所有状态一股脑塞进一个全局状态库**——相当于你后端把所有变量都写成全局静态变量。正确做法是：**能局部就局部，确实需要共享的才往上提。**

---

## 三、轻量方案：Context（框架内置的「依赖注入」）

对于「跨组件共享但不频繁变化」的状态（如当前用户、主题色、语言），React 内置的 **Context** 就够了。它的本质，就是你无比熟悉的**依赖注入**：

```jsx
// 1. 创建一个 Context，像声明一个可被注入的容器
const UserContext = createContext();

// 2. 在顶层"提供"这个状态（像 Spring 把 Bean 放进容器）
function App() {
  const [user, setUser] = useState({ name: '李先生', role: 'admin' });
  return (
    <UserContext.Provider value={user}>
      <Layout />   {/* 中间层不用再传 user 了！ */}
    </UserContext.Provider>
  );
}

// 3. 任何深层组件直接"注入取用"，无需层层传递
function DeepButton() {
  const user = useContext(UserContext);   // 像 @Autowired 直接拿
  return <button disabled={user.role !== 'admin'}>管理操作</button>;
}
```

对比一下，这就是 Spring 的依赖注入思想：**不用手动一层层传递依赖，需要的地方直接声明取用，由容器负责供给。** 你早就精通这套思路。

---

## 四、重量方案：状态库（Redux / Zustand / Pinia）

当应用变大，共享状态变多、变化频繁、来源复杂时，光靠 Context 会力不从心（性能与组织都跟不上）。这时引入专门的**状态库**。

状态库的核心思想是建立一个**集中的、单一数据源（single source of truth）**，所有共享状态放进去统一管理。这非常像你后端的**集中式缓存 / 状态服务**。

以现代轻量库 **Zustand** 为例（比老牌 Redux 简洁得多）：

```javascript
import { create } from 'zustand';

// 定义一个"全局 store"，像定义一个全局的状态服务
const useCartStore = create((set) => ({
  items: [],                                          // 状态字段
  addItem: (item) => set((s) => ({ items: [...s.items, item] })),  // 修改方法
  clear: () => set({ items: [] }),
}));

// 任何组件都能读、能改这个全局状态
function CartBadge() {
  const count = useCartStore((s) => s.items.length);  // 只订阅 items 长度
  return <span>{count}</span>;
}

function AddButton({ product }) {
  const addItem = useCartStore((s) => s.addItem);
  return <button onClick={() => addItem(product)}>加入购物车</button>;
}
```

Redux 引入了更严格的**单向数据流约束**（action → reducer → 新 state），目的是让状态变化**可追踪、可预测、可调试**——这和你后端推崇的「**状态变更要有明确入口、可审计**」是同一个工程哲学。代价是样板代码多，所以现代项目越来越多用 Zustand（React）或 Pinia（Vue）这类轻量方案。

Vue 阵营对应的官方状态库是 **Pinia**，思想一致，写法更贴合 Vue 的响应式风格。

## 五、最容易被后端忽略的：服务端状态

这是后端工程师**最该重视、却最容易忽视**的一类状态。

你后端的数据都在数据库，是「真相」。但前端从 API 拿到数据后，会在浏览器内存里存一份**副本**。问题随之而来：

- 这份副本会**过期**（后端数据变了，前端还显示旧的）
- 多个组件可能**重复请求**同一份数据
- 需要处理**加载中 / 失败 / 重试 / 缓存**等一堆状态

把服务端数据当普通状态硬塞进 Redux，是经典反模式。正确做法是用专门的**服务端状态库**，如 **React Query（TanStack Query）** 或 **SWR**：

```jsx
import { useQuery } from '@tanstack/react-query';

function UserProfile({ id }) {
  // 一行声明：要这个数据，库自动帮你处理缓存、过期、重试、加载态
  const { data, isLoading, error } = useQuery({
    queryKey: ['user', id],
    queryFn: () => fetch(`/api/users/${id}`).then((r) => r.json()),
  });

  if (isLoading) return <div>加载中...</div>;
  if (error) return <div>出错了</div>;
  return <div>{data.name}</div>;
}
```

你可以把它理解成前端的**「带自动失效策略的本地缓存层」**——就像你后端用 Caffeine / Redis 缓存数据库结果，并配置过期与刷新策略。React Query 把这套缓存哲学搬到了前端，替你管理「数据库数据在前端的副本」。

---

## 六、给后端大脑的「翻译词典」

| 前端概念 | 后端类比 | 说明 |
|---------|---------|------|
| prop drilling | 参数被迫穿过沿途所有方法签名 | 该避免的反模式 |
| 局部状态 `useState` | 方法内局部变量 | 能局部就局部 |
| Context | 依赖注入（`@Autowired`） | 深层直接取用，无需层层传 |
| 状态库（Redux/Zustand） | 集中式状态服务 / 缓存 | 单一数据源 |
| Redux 单向数据流 | 状态变更统一入口、可审计 | 可追踪可调试 |
| 服务端状态（React Query） | Caffeine/Redis 缓存 + 失效策略 | 管理「DB 数据在前端的副本」 |

---

## 本章小结

- 状态管理要解决的核心问题：**状态如何在组件间共享**，避免 prop drilling 这种「参数穿肠」的脏活。
- 最重要的认知是**状态分层**：局部状态、共享状态、服务端状态、URL 状态各归其位，**别一股脑塞全局**（对应你「别滥用全局静态变量」的直觉）。
- 共享状态：轻量用 **Context**（本质是依赖注入），重量用**状态库**（本质是集中式状态服务 + 单向数据流约束）。
- 服务端状态最该被后端重视：用 **React Query / SWR** 当作「前端的缓存层」，自动管理过期、重试、加载态——就像你后端配 Redis 缓存策略。

一句话记住：**前端状态管理 = 给「内驻在会话内存里的数据」做分层治理。** 你后端怎么决定数据放局部变量、成员变量、缓存还是数据库，前端就怎么决定状态放哪一层。这是同一种工程判断力。
