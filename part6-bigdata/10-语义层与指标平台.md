# 6.10 语义层与指标平台

> **一句话定位**：语义层是数据仓库和业务用户之间的「翻译层」——它把物理表结构抽象为业务可理解的维度、度量和指标，让不同消费端（BI 工具、API、Agent）用统一口径查询数据，避免「同一个指标，十个人算出十个数」。

---

## 一、为什么需要语义层

### 1.1 没有语义层的痛点

```
业务方 A: "GMV 是含退款的还是不含退款的？"
业务方 B: "我的报表和你的报表数字对不上"
数据开发: "又来一个取数需求，又要写一遍 SQL..."
BI 工具:  "换个 BI 工具，所有指标定义要重新配一遍"
```

**根本问题**：指标的业务逻辑散落在各处（SQL 脚本、BI 报表、代码里），没有统一的「单一事实来源」（Single Source of Truth）。

### 1.2 语义层解决什么

| 问题 | 语义层的解决方式 |
|------|----------------|
| 指标口径不一致 | 统一定义，所有消费端引用同一份定义 |
| 重复开发 | 定义一次，多处复用（BI/API/Agent/报表） |
| 权限难管控 | 在语义层统一做行列级权限过滤 |
| 换 BI 工具成本高 | 语义层与 BI 解耦（Headless BI） |
| 业务理解门槛高 | 用业务术语暴露数据，屏蔽物理表复杂性 |

## 二、核心概念模型

### 2.1 四层抽象

```
┌─────────────────────────────────────────┐
│  指标（Metric）                          │
│  "GMV = SUM(order_amount) WHERE status≠退款" │
├─────────────────────────────────────────┤
│  度量（Measure）                         │
│  "order_amount 的 SUM / COUNT / AVG"     │
├─────────────────────────────────────────┤
│  维度（Dimension）                       │
│  "城市、日期、品类、渠道"                  │
├─────────────────────────────────────────┤
│  数据集 / 数据模型（Dataset / Model）     │
│  "订单事实表 JOIN 商品维表 JOIN 城市维表"   │
└─────────────────────────────────────────┘
```

### 2.2 关键术语

| 术语 | 定义 | 示例 |
|------|------|------|
| **维度（Dimension）** | 分析的切面/角度 | 城市、日期、品类 |
| **度量（Measure）** | 可聚合的数值字段 + 聚合方式 | SUM(amount)、COUNT(order_id) |
| **指标（Metric）** | 度量 + 业务过滤条件 + 时间粒度 | 日 GMV = SUM(amount) WHERE status='paid' |
| **技术口径** | 指标的 SQL 实现细节 | 哪张表、哪个字段、什么过滤条件 |
| **业务口径** | 指标的业务含义描述 | "GMV 指已支付且未退款的订单金额总和" |
| **派生指标** | 基于原子指标 + 修饰词 + 时间 | "北京地区近7天 GMV" |
| **复合指标** | 多个指标的运算 | "客单价 = GMV / 订单数" |

### 2.3 指标体系分层

```
原子指标（Atomic Metric）
  └── 一个度量 + 一个聚合方式
  └── 例: SUM(order_amount)

派生指标（Derived Metric）
  └── 原子指标 + 维度限定 + 时间限定
  └── 例: 北京地区_近7天_GMV

复合指标（Composite Metric）
  └── 多个原子/派生指标的四则运算
  └── 例: 客单价 = GMV / 支付订单数
```

### 2.4 指标体系设计方法论

指标体系不是"想到什么指标就加什么指标"，而是需要系统化的方法论来保证**完整性**（不遗漏关键指标）和**结构性**（指标之间有逻辑关系）。

#### A. OSM 模型（Object-Strategy-Measure）——指导思想

OSM 是指标体系建设的顶层框架，回答三个问题：

```
O（Object 目标）：业务的核心 KPI 是什么？
  └── 例: 提升 GMV

S（Strategy 策略）：为达成目标，有哪些行动路径？
  └── 例: 提升新客转化率、提升老客复购率、提升客单价

M（Measure 度量）：如何量化评估每个策略的效果？
  └── 例: 新客首单转化率、7日复购率、平均客单价
```

**OSM 的价值**：让指标体系"自顶向下"与业务目标对齐，避免"有数据但不知道看什么"。

#### B. AARRR 海盗指标法——从产品生命周期拆解

AARRR 从产品角度，按用户生命周期的五个阶段拆解指标：

| 阶段 | 英文 | 核心问题 | 典型指标 |
|------|------|---------|---------|
| **获取** | Acquisition | 用户从哪来？ | 渠道 UV、注册转化率、CPA |
| **激活** | Activation | 用户有没有体验到核心价值？ | 首单率、关键行为完成率 |
| **留存** | Retention | 用户会不会回来？ | 次日/7日/30日留存率 |
| **变现** | Revenue | 用户贡献了多少收入？ | ARPU、LTV、付费率 |
| **推荐** | Referral | 用户会不会带来新用户？ | 分享率、邀请转化率、K-factor |

#### C. UJM 用户旅程地图（User Journey Map）——从用户行为路径拆解

UJM 从用户角度出发，描述用户完成目标的完整路径，在每个节点设置指标：

```
用户旅程示例（电商）：

  曝光 → 点击 → 浏览 → 加购 → 下单 → 支付 → 收货 → 评价
   │      │      │      │      │      │      │      │
   ↓      ↓      ↓      ↓      ↓      ↓      ↓      ↓
  曝光UV  CTR  浏览深度 加购率 下单率 支付率 妥投率 好评率

每相邻两个节点之间就是一个"转化率"，哪个环节转化率低就是业务瓶颈
```

**AARRR vs UJM**：AARRR 是宏观的产品生命周期视角，UJM 是微观的单次用户行为路径视角。两者互补。

#### D. MECE 原则——指标分级治理

MECE（Mutually Exclusive, Collectively Exhaustive）= 完全独立、相互穷尽。

在对核心指标进行下钻拆解时，保证子指标之间不重叠、不遗漏：

```
GMV 的 MECE 拆解示例：

GMV = 成交用户数 × 平均客单价                    ← 一级拆解
    = (点击UV × 访购率) × 平均客单价              ← 二级拆解
    = (曝光UV × CTR × 访购率) × 平均客单价        ← 三级拆解

每一级拆解都是乘法/加法关系，子项之间完全独立、合起来等于父项
```

#### E. 指标分级 T1/T2/T3——纵向分层

| 级别 | 定位 | 使用者 | 示例 |
|------|------|--------|------|
| **T1** | 公司战略层指标 | CEO/VP | GMV、DAU、毛利率 |
| **T2** | 业务策略层指标 | 业务负责人 | 新客 GMV、老客复购率、各渠道 ROI |
| **T3** | 业务执行层指标 | 一线运营/产品 | 某活动点击率、某页面跳出率、某渠道 CPA |

```
T1 → T2 → T3 的关系：

T1: GMV（公司看这个）
  ├── T2: 新客 GMV（增长团队看这个）
  │     ├── T3: 各渠道新客注册数
  │     ├── T3: 新客首单转化率
  │     └── T3: 新客平均客单价
  └── T2: 老客 GMV（用户运营看这个）
        ├── T3: 7日复购率
        ├── T3: 老客平均客单价
        └── T3: 流失召回率
```

#### F. 场景化搭建——人、货、场

互联网业务常用"人货场"三个维度交叉构建指标体系：

| 维度 | 含义 | 指标示例 |
|------|------|---------|
| **人**（用户） | 谁在用？ | 用户画像、活跃度、留存、LTV |
| **货**（产品/商品） | 用什么？ | 商品动销率、SKU 覆盖率、缺货率 |
| **场**（场景/渠道） | 在哪用？ | 各渠道转化率、各页面停留时长、搜索/推荐占比 |

#### G. 指标生命周期管理

指标不是定义完就结束，需要全生命周期管理：

```
定义 → 生产 → 消费 → 下线

定义：明确业务口径、技术口径、负责人、数据源
生产：ETL 加工、数据质量校验、产出 SLA
消费：BI 报表、API 接口、Agent 查询
下线：指标不再使用时，走下线流程，避免僵尸指标占用资源
```

#### H. 价值导向 + 降维思路

指标体系建设本质是对复杂系统的"降维"——从繁杂的数据中抽取最能反映系统状态的几个关键维度（类比 PCA 主成分分析）：

```
设计原则：
  ① 能描绘宏观概况 — 用简化的几个数描绘系统全貌
  ② 能描绘阶段节点 — 看到某部分完成到什么程度
  ③ 能描绘变化趋势 — 哪些部分增加、哪些减少、推进节奏如何

实操方法：
  ① 定性：列举、挖掘、归纳所有价值点
  ② 定量：针对每类价值找到可量化的指标（抓主因）
  ③ 统一量纲：不同体系的资源可能量纲不同，要么统一，要么分类比较
```

### 3.1 整体架构

```
┌──────────────────────────────────────────────────────┐

├──────────────────────────────────────────────────────┤
```

### 3.2 核心模块

#### 3.2.1 语义模型管理

```yaml
# 示例：指标定义（类 dbt metrics / Cube.js schema）
metrics:
  - name: gmv
    label: "GMV（成交总额）"
    description: "已支付且未退款的订单金额总和"

    owners: ["数据团队"]

```

#### 3.2.2 SQL 生成引擎

语义层的核心能力——把「指标 + 维度 + 过滤条件」翻译为可执行 SQL。

##### A. 传统程序化语义层（确定性生成）

**核心思路**：不是一步到位生成 SQL，而是分阶段——先将请求解析为结构化中间表示（IR），再从 IR 确定性地生成 SQL。

```
阶段一：语义解析 → 结构化查询意图（IR）
  - 查指标注册表 → 获取聚合表达式 + 口径过滤条件
  - 查维度定义 → 获取物理列名
  - 确定数据集（物理表）和 JOIN 路径
  - 注入权限过滤条件

阶段二：IR → 可执行 SQL
  - 根据目标引擎方言（Doris/Presto/ClickHouse）生成最终 SQL
```

**为什么不能一步到位**：从「指标名」到「可执行 SQL」之间隔着大量需要查表才能确定的信息——指标的聚合表达式、口径过滤条件、物理表名、JOIN 关系、行级权限等。一步到位意味着把这些逻辑全耦合在一起，不可维护。

**主流开源实现对比**（基于源码分析）：

| 项目 | 中间表示 | 生成方式 | 核心设计 |
|------|---------|---------|---------|
| **Cube.js** | JSON Query → AST → SQL | 三阶段 | Schema 驱动，measure/dimension 定义展开，多数据库方言适配 |
| **dbt MetricFlow** | Metric Query → Dataflow DAG → SQL | 两阶段 | 构建查询 DAG（类似数据库逻辑执行计划），自动推导 JOIN 路径 |
| **Looker (LookML)** | Explore → SQL Model → SQL | 三阶段 | 模型即代码，基于 explore 中定义的 relationship 自动 JOIN 推导 |
| **大厂自研** | 请求 → 指标展开 → 查询计划 → SQL | 三阶段+路由 | 预计算命中 + 引擎路由 + 权限注入 |

Cube.js Schema 定义示例：
```javascript
cube('Orders', {
  sql: `SELECT * FROM public.orders`,
  measures: {
    gmv: {
      type: 'sum',
      sql: 'amount',
      filters: [{ sql: `${CUBE}.status = 'paid'` }]
    }
  },
  dimensions: {
    city: { sql: 'city', type: 'string' },
    category: { sql: 'category', type: 'string' }
  }
});
```

##### B. LLM 驱动的 SQL 生成（当前主流方案）

随着大模型能力提升，SQL 生成出现了三种 LLM 模式。以下基于对开源项目源码的实际分析：

---

**模式一：纯 LLM 直接生成 SQL（RAG + Text-to-SQL）**

代表项目：**Vanna.ai**（GitHub 12k+ stars，v2.0，283 个 Python 文件）

```
用户问题
  → 向量检索（RAG）：从 ChromaDB 中检索相关 DDL + 文档 + 历史 Q&A
  → 组装 Prompt：system prompt + 检索到的上下文 + 用户问题
  → LLM 直接生成 SQL
  → 正则提取 SQL（从 markdown code block 中抽取）
  → 执行 SQL 返回结果
```

**源码实证**（`vanna/legacy/base/base.py` → `generate_sql` 方法）：
```python
def generate_sql(self, question: str, allow_llm_to_see_data=False):
    # Step 1: RAG 检索
    question_sql_list = self.get_similar_question_sql(question)  # 相似 Q&A 对
    ddl_list = self.get_related_ddl(question)                    # 相关表结构
    doc_list = self.get_related_documentation(question)          # 相关文档

    # Step 2: 组装 Prompt
    prompt = self.get_sql_prompt(
        question=question,
        question_sql_list=question_sql_list,
        ddl_list=ddl_list,
        doc_list=doc_list,
    )

    # Step 3: LLM 生成
    llm_response = self.submit_prompt(prompt)

    # Step 4: 正则提取 SQL
    return self.extract_sql(llm_response)
```

**Prompt 结构**（源码中的 `get_sql_prompt`）：
```
System: "You are a {dialect} expert. Generate a valid SQL query..."
  + "===Tables\n" + DDL 列表
  + "===Additional Context\n" + 文档列表
  + "===Question-SQL Pairs\n" + 历史相似问答（few-shot）
  + "===Response Guidelines\n"
    1. 上下文充分 → 直接生成 SQL
    2. 需要探查某列的值 → 生成 intermediate_sql 先查
    3. 上下文不足 → 解释为什么无法生成
User: 用户问题
```

**完成度评估**：⭐⭐⭐⭐ — 生产可用。RAG 检索 + Prompt 工程完整，支持 intermediate SQL（先探查数据再生成最终 SQL），支持 ChromaDB/Pinecone/Qdrant 等多种向量库。**但没有语义层**——不保证口径一致性，同一个问题可能生成不同 SQL。

**模式二：LLM Agent + 工具调用（ReAct 多步推理）**

代表项目：**Dataherald**（GitHub 3k+ stars，102 个 Python 文件）

```
用户问题
  → Agent 规划（LLM 决定调用哪些工具、什么顺序）
  → Tool 1: DbTablesWithRelevanceScores — 用 embedding 相似度找相关表
  → Tool 2: DbRelevantTablesSchema — 获取相关表的 DDL
  → Tool 3: DbRelevantColumnsInfo — 获取列的描述和样本值
  → Tool 4: DbColumnEntityChecker — 验证字符串值是否存在
  → Tool 5: FewshotExamplesRetriever — 检索相似 Q&A 对
  → Tool 6: SqlDbQuery — 执行生成的 SQL 验证正确性
  → 如果报错 → 自动修复重试
  → 返回最终 SQL
```

**源码实证**（`dataherald/sql_generator/dataherald_sqlagent.py`）：

Agent 的工具集（`SQLDatabaseToolkit.get_tools()`）：
```python
def get_tools(self) -> List[BaseTool]:
    tools = []
    tools.append(QuerySQLDataBaseTool(...))        # 执行 SQL
    tools.append(GetUserInstructions(...))          # 获取管理员指令
    tools.append(SystemTime(...))                   # 获取当前时间
    tools.append(TablesSQLDatabaseTool(...))        # embedding 找相关表
    tools.append(SchemaSQLDatabaseTool(...))        # 获取表 Schema
    tools.append(InfoRelevantColumns(...))          # 列信息
    tools.append(ColumnEntityChecker(...))          # 验证实体值
    tools.append(GetFewShotExamples(...))           # few-shot 示例
    return tools
```

Agent 的执行计划（源码中的 `PLAN_WITH_FEWSHOT_EXAMPLES_AND_INSTRUCTIONS`）：
```
1) 检索相似 Q&A 对
2) 获取管理员指令（业务规则约束）
3) 用 embedding 找相关表
4) 获取表 Schema
5) 获取列详细信息
6) [可选] 获取当前时间
7) 验证字符串值是否存在于列中
8) 生成 SQL 并执行验证
```

**关键设计**：表的相关性排序用 embedding 余弦相似度（`TablesSQLDatabaseTool`），不是简单的关键词匹配：
```python
def _run(self, user_question: str) -> str:
    question_embedding = self.get_embedding(user_question)
    # 每张表用 "表名 + 列名 + 描述" 拼接作为表示
    df["table_embedding"] = self.get_docs_embedding(df.table_representation)
    df["similarities"] = df.table_embedding.apply(
        lambda x: self.cosine_similarity(x, question_embedding)
    )
    df = df.sort_values(by="similarities", ascending=False).head(TOP_TABLES)
```

**完成度评估**：⭐⭐⭐⭐ — 生产可用。工具链完整（7 个专用工具），有 few-shot 检索、管理员指令注入、自动执行验证和错误重试。**但最后一次 commit 是 2024-07，项目已停止维护**。

**模式三：LangChain SQL Agent（通用框架）**

代表项目：**langchain-community** `create_sql_agent`

```
用户问题
  → Agent（ReAct / Tool-Calling）
  → Tool 1: ListSQLDatabaseTool — 列出所有表
  → Tool 2: InfoSQLDatabaseTool — 获取表 Schema + 样本行
  → Tool 3: QuerySQLDatabaseTool — 执行 SQL
  → Tool 4: QuerySQLCheckerTool — LLM 检查 SQL 语法
  → 返回结果
```

**源码实证**（`langchain_community/agent_toolkits/sql/toolkit.py`）：
```python
class SQLDatabaseToolkit(BaseToolkit):
    def get_tools(self) -> List[BaseTool]:
        return [
            QuerySQLDatabaseTool(db=self.db, ...),   # 执行 SQL
            InfoSQLDatabaseTool(db=self.db, ...),    # 获取 Schema
            ListSQLDatabaseTool(db=self.db, ...),    # 列出表
            QuerySQLCheckerTool(db=self.db, llm=self.llm, ...),  # SQL 检查
        ]
```

System Prompt（`sql/prompt.py`）：
```
You are an agent designed to interact with a SQL database.
Given an input question, create a syntactically correct {dialect} query...
Never query for all columns from a specific table...
You MUST double check your query before executing it...
DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.)
```

**完成度评估**：⭐⭐⭐ — 框架级，非开箱即用。只有 4 个基础工具，没有 embedding 相关性排序、没有 few-shot 检索、没有管理员指令。适合作为起点二次开发，不适合直接用于生产。官方已标记为 legacy，推荐迁移到 LangGraph。

**模式四：SQL Chat（最简实现）**

代表项目：**SQL Chat**（GitHub 4k+ stars，67 个 TS 文件）

```python
# 完整的 SQL 生成逻辑（全部代码）：
getPrompt: (engine, schema) => {
    basicPrompt = [`You are a ${engine} db and SQL expert.`]
    if (schema) finalPrompt.push(`This is my db schema:\n\n${schema}`)
    return finalPrompt.join("\n\n")
}
```

**完成度评估**：⭐⭐ — Demo 级。纯 Prompt 拼接，没有 RAG、没有工具调用、没有验证。本质上就是把 Schema 塞进 system prompt 让 LLM 直接生成。

##### C. 生产级架构：LLM + 语义层混合（当前最佳实践）

上述纯 LLM 方案的共同问题：**无法保证口径一致性**。同一个问题问两次可能生成不同 SQL。

生产级方案（Databricks Genie / ThoughtSpot Sage / 大厂自研）的做法是 **LLM 做意图识别，语义层做 SQL 生成**：

```
用户: "上周北京各品类的GMV排名"
  │
  ▼
Step 1: LLM 意图解析 → 结构化 JSON（不是 SQL）
  {
    "metric": "gmv",
    "dimensions": ["category"],
    "filters": [{"city": "北京"}],
    "time": "last_week",
    "sort": {"metric": "gmv", "order": "desc"}
  }
  │
  ▼
Step 2: 语义层接管（确定性，非 LLM）
  - 查指标定义 → SUM(order_amount) WHERE status='paid' AND refund_status='none'
  - 确定物理表 → dws_order_detail
  - 注入权限过滤
  - 生成 SQL（确定性的，保证口径一致）
  │
  ▼
Step 3: 执行 + LLM 生成自然语言总结
```

**为什么这样设计**：
- LLM 擅长：理解自然语言、识别意图、消歧、多轮对话
- LLM 不擅长：保证 SQL 语法正确、保证口径一致、处理权限
- 让 LLM 做它擅长的（理解），让程序做它擅长的（生成正确 SQL）

**实际路由策略**：
```
用户提问
  → LLM 意图识别
    → 简单指标查询（80%）→ 走语义层（确定性 SQL，口径一致）
    → 复杂归因/探索（20%）→ 走 Agent 多步（LLM 生成 SQL + 自动验证修复）
```

##### D. 各模式对比总结

| 维度 | 传统语义层 | 纯 LLM (Vanna) | Agent (Dataherald) | LLM+语义层混合 |
|------|-----------|---------------|-------------------|--------------|
| **SQL 正确性** | ⭐⭐⭐⭐⭐ 100% | ⭐⭐⭐ 70-85% | ⭐⭐⭐⭐ 80-90% | ⭐⭐⭐⭐⭐ ~99% |
| **口径一致性** | ⭐⭐⭐⭐⭐ 保证 | ❌ 无法保证 | ❌ 无法保证 | ⭐⭐⭐⭐⭐ 语义层保证 |
| **自然语言理解** | ❌ 不支持 | ⭐⭐⭐⭐ 好 | ⭐⭐⭐⭐ 好 | ⭐⭐⭐⭐⭐ 最好 |
| **多步分析** | ❌ 不支持 | ❌ 单次 | ⭐⭐⭐⭐ 自动多步 | ⭐⭐⭐⭐ 复杂问题走 Agent |
| **部署复杂度** | 中（需建模） | 低 | 中 | 高（需语义层+LLM） |
| **适用场景** | 固定报表/BI | 探索性查询 | 探索性分析 | 生产级智能 BI |

**传统语义层的 SQL 生成示例**（程序化确定性生成）：

```
输入: 查询 "北京地区按品类的近7天GMV"
  → 指标: gmv
  → 维度: category
  → 过滤: city = '北京', dt >= '2024-01-15'

SQL 生成:
  SELECT category, SUM(order_amount) AS gmv
  FROM dws_order_detail
  WHERE order_status = 'paid'
    AND refund_status = 'none'
    AND city = '北京'
    AND dt >= '2024-01-15'
  GROUP BY category
  ORDER BY gmv DESC
```

#### 3.2.3 查询下推与路由

```
查询请求
  → 判断是否命中预计算（物化视图/ADS 表）
    → 命中 → 直接查 Doris/ClickHouse（毫秒级）
    → 未命中 → 判断数据量级
      → 小数据量 → Presto/Trino 交互式查询（秒级）
      → 大数据量 → Spark SQL 离线查询（分钟级）
```

**查询下推（Pushdown）**：将操作从计算引擎层「下推」到存储/数据源层执行，减少跨层数据传输：
- 谓词下推（Predicate Pushdown）— 将 WHERE 过滤条件推到数据源侧执行，源侧只返回满足条件的行
  - 分区裁剪（Partition Pruning）是谓词下推的典型效果：当谓词命中分区键时，直接跳过整个不相关的分区目录
  - 文件级 Data Skipping 也是谓词下推的效果：利用文件/行组的 min/max 统计信息跳过不相关的数据块
- 投影下推（Projection Pushdown）— 将「只需要哪些列」的信息推到数据源，源侧只读取 SELECT 涉及的列（对列式存储效果极好）
- 聚合下推（Aggregation Pushdown）— 将 GROUP BY + 聚合函数推到数据源侧预聚合，只返回聚合结果而非明细行

#### 3.2.4 权限过滤

| 权限类型 | 实现方式 | 示例 |
|---------|---------|------|
| **行级权限** | SQL 自动注入 WHERE 条件 | 区域经理只能看自己区域的数据 |
| **列级权限** | 查询时脱敏或隐藏字段 | 普通用户看不到手机号明文 |
| **指标权限** | 控制谁能查哪些指标 | 财务指标仅财务部门可见 |
| **数据集权限** | 控制谁能访问哪些数据集 | 敏感业务数据集需审批 |

```sql
-- 行级权限注入示例
-- 原始 SQL:
SELECT city, SUM(amount) FROM orders GROUP BY city;

-- 注入权限后:
SELECT city, SUM(amount) FROM orders
WHERE city IN (SELECT city FROM user_data_scope WHERE user_id = 'current_user')
GROUP BY city;
```


## 五、指标口径管理

### 5.1 口径不一致的常见原因

| 原因 | 示例 |
|------|------|
| 过滤条件不同 | A 算 GMV 含退款，B 不含 |
| 时间口径不同 | A 按下单时间，B 按支付时间 |
| 数据源不同 | A 从实时表取，B 从离线表取 |
| 聚合粒度不同 | A 按自然日，B 按 24 小时滚动 |
| 去重逻辑不同 | A 按订单去重，B 按用户去重 |

### 5.2 口径管理最佳实践

1. **指标注册制** — 所有指标必须在指标平台注册，未注册的不允许上报表
2. **口径文档化** — 每个指标必须有业务口径 + 技术口径 + 负责人
3. **指标认证** — 核心指标需要数据团队认证后才能对外使用
4. **变更审计** — 指标定义变更需要审批，变更历史可追溯
5. **一致性校验** — 定期对比不同出口的同一指标数值，发现偏差告警

### 5.3 指标分级

| 级别 | 定义 | 管理要求 |
|------|------|---------|
| L1 核心指标 | 公司级经营指标（GMV/DAU/利润） | 最严格，变更需 VP 审批 |
| L2 业务指标 | 业务线级指标（转化率/复购率） | 业务负责人审批 |
| L3 分析指标 | 分析师自定义的探索性指标 | 自助创建，定期清理 |

## 六、与 AgentBI 的关系

语义层是 AgentBI（自然语言问数）的**基础设施**：

```
用户: "上周北京的 GMV 是多少？"
  → AgentBI 意图识别: 查询指标
  → 语义解析: 指标=GMV, 维度=城市, 过滤=北京, 时间=上周
  → 语义层: 查找 GMV 指标定义 → 生成 SQL
  → 执行引擎: 路由到 Doris 执行
  → 返回结果 + 可视化
```

**没有语义层的 NL2SQL**：LLM 直接面对物理表，容易生成错误 SQL、口径不一致。
**有语义层的 NL2SQL**：LLM 只需要识别「哪个指标 + 哪些维度 + 什么过滤」，SQL 由语义层保证正确性。

## 七、面试深度剖析

### Q1: 语义层和数仓 ADS 层有什么区别？

**答**：ADS 层是物理表，已经把指标算好存下来了（预计算）；语义层是逻辑层，定义了指标的计算规则，可以动态生成 SQL 查任何层的数据。ADS 是语义层的一种加速手段（预计算命中），但语义层的能力不限于 ADS。

### Q2: 查询下推是怎么实现的？

**答**：语义层生成逻辑查询计划后，根据数据源的能力做改写。比如 Doris 支持聚合下推，就把 GROUP BY 推到 Doris 侧执行；如果数据源是 MySQL 且不支持某些函数，就只下推过滤条件，聚合在上层做。本质是「能力协商」——知道数据源能做什么，就尽量让它做。

### Q3: 行级权限怎么保证不影响查询性能？

**答**：① 权限条件尽量走索引（如 city IN (...)）；② 预计算时按权限维度预分区；③ 权限范围大的用户直接查全量表，范围小的走过滤；④ 缓存权限解析结果，避免每次查询都查权限表。

### Q4: 指标口径变更了怎么办？

**答**：① 版本化管理，保留历史口径；② 变更时评估影响范围（哪些报表/API 在用）；③ 提供新旧口径对比数据；④ 设置过渡期，旧口径标记为 deprecated；⑤ 通知所有消费方。

### Q5: 如何设计一个指标平台的技术架构？

**答**：
- **元数据存储**：MySQL/PostgreSQL 存指标定义、维度、数据集关系
- **SQL 引擎**：基于指标定义动态生成 SQL（模板引擎 + AST 改写）
- **查询路由**：根据数据量/时效性路由到不同引擎（Doris/Presto/Spark）
- **缓存层**：Redis 缓存热点查询结果 + 预计算结果
- **权限服务**：集成公司权限系统，查询时注入过滤条件
- **API 层**：RESTful API 对外暴露，支持 BI/Agent/自定义应用调用
