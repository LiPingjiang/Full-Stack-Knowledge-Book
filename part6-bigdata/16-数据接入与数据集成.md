# 6.16 数据接入与数据集成

> **一句话定位**：数据接入是数据平台的「入口」——它负责将散落在业务数据库、日志系统、消息队列、第三方 API、文件系统中的数据，以批量或实时的方式统一汇聚到数据平台（数仓/数据湖），是所有后续计算和分析的前提。

---

## 一、为什么需要统一数据接入

### 1.1 数据源的多样性

```
一个典型企业的数据源：
  - MySQL / PostgreSQL / Oracle（业务数据库）
  - MongoDB / HBase（NoSQL）
  - Kafka / RocketMQ（消息队列）
  - Nginx / 应用日志（日志文件）
  - 第三方 API（广告平台、支付平台）
  - Excel / CSV（人工上传）
  - IoT 设备（传感器数据）
```

### 1.2 没有统一接入层的痛点

| 问题 | 表现 |
|------|------|
| **烟囱式开发** | 每个数据源写一套接入脚本，重复造轮子 |
| **Schema 不一致** | 上游改了字段，下游全链路崩溃 |
| **延迟不可控** | 有的 T+1，有的实时，没有统一 SLA |
| **无法追溯** | 数据什么时候接入的、接了多少、有没有丢 |
| **运维困难** | 几百个接入任务，故障排查靠人肉 |

## 二、数据接入方式全景

### 2.1 批量接入 vs 实时接入

| 维度 | 批量接入 | 实时接入（CDC） |
|------|---------|---------------|
| **延迟** | T+1 或小时级 | 秒~分钟级 |
| **方式** | 全量/增量 SELECT | 监听 Binlog/WAL |
| **对源库影响** | 有（查询压力） | 极小（读日志） |
| **适用场景** | 历史数据迁移、冷数据同步 | 实时数仓、实时入湖 |
| **典型工具** | DataX / SeaTunnel / Sqoop | Flink CDC / Debezium / Canal |

### 2.2 接入方式分类

```
┌─────────────────────────────────────────────────────────┐
│                    数据接入方式                            │
├──────────────────────┬──────────────────────────────────┤
│      批量接入         │          实时接入                 │
├──────────────────────┼──────────────────────────────────┤
│ • 全量同步            │ • CDC（Change Data Capture）     │
│ • 增量同步（时间戳）   │ • 消息队列消费                   │
│ • 文件导入            │ • 日志实时采集                    │
│ • API 拉取           │ • 数据库触发器（不推荐）           │
├──────────────────────┼──────────────────────────────────┤
│ DataX / SeaTunnel    │ Flink CDC / Debezium / Canal     │
│ Sqoop / Airbyte      │ Flume / Filebeat / Logstash      │
└──────────────────────┴──────────────────────────────────┘
```

## 三、CDC（Change Data Capture）

### 3.1 什么是 CDC

CDC 是通过监听数据库的变更日志（MySQL Binlog / PostgreSQL WAL / Oracle Redo Log），实时捕获数据的增删改操作，将变更事件流式传输到下游系统。

```
MySQL Binlog
  → CDC 工具解析 Binlog
    → 生成变更事件（INSERT/UPDATE/DELETE）
      → 写入 Kafka / 直接写入湖表
```

### 3.2 CDC 方案对比

| 方案 | 架构 | 特点 | 适用场景 |
|------|------|------|---------|
| **Flink CDC** | Flink 作业直连数据库 | 全增量一体、无需 Kafka 中转、Exactly-Once | ⭐ 实时入湖首选 |
| **Debezium** | Kafka Connect 插件 | 生态丰富、支持多种数据库、社区活跃 | Kafka 生态、需要消息缓冲 |
| **Canal** | 独立服务 | 阿里开源、轻量、专注 MySQL | MySQL 单一场景 |
| **Maxwell** | 独立服务 | 轻量、JSON 输出 | 简单 MySQL CDC |
| **Oracle GoldenGate** | 商业产品 | Oracle 官方、企业级 | Oracle 数据库 |

### 3.3 Flink CDC 详解

Flink CDC 是当前最主流的实时数据接入方案：

```sql
-- Flink SQL: 创建 MySQL CDC 源表
CREATE TABLE orders_cdc (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(18,2),
    status STRING,
    create_time TIMESTAMP(3),
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'connector' = 'mysql-cdc',
    'hostname' = '10.0.0.1',
    'port' = '3306',
    'username' = 'cdc_user',
    'password' = '***',
    'database-name' = 'order_db',
    'table-name' = 'orders',
    'scan.startup.mode' = 'initial'  -- 先全量再增量
);

-- 直接写入 Iceberg 湖表
INSERT INTO iceberg_orders
SELECT * FROM orders_cdc;
```

**Flink CDC 核心优势**：
| 特性 | 说明 |
|------|------|
| **全增量一体** | `initial` 模式先做全量快照，再无缝切换到增量 Binlog |
| **无需 Kafka** | 直连数据库，减少中间环节 |
| **Exactly-Once** | 基于 Flink Checkpoint 保证端到端一致性 |
| **Schema Evolution** | 支持上游 DDL 变更自动传播 |
| **多表合并** | 一个作业同步多张表，减少连接数 |
| **无锁快照** | 全量阶段不锁表（基于 MVCC） |

### 3.4 Debezium 架构

```
┌──────────┐     ┌──────────────────┐     ┌──────────┐
│  MySQL   │────→│ Kafka Connect    │────→│  Kafka   │
│ (Binlog) │     │ + Debezium Plugin│     │ (Topic)  │
└──────────┘     └──────────────────┘     └────┬─────┘
                                               │
                                    ┌──────────▼──────────┐
                                    │  下游消费者           │
                                    │  Flink / Spark /     │
                                    │  ClickHouse / ES     │
                                    └─────────────────────┘
```

**Debezium 变更事件格式**：
```json
{
  "before": {"order_id": 1001, "status": "pending", "amount": 99.00},
  "after":  {"order_id": 1001, "status": "paid", "amount": 99.00},
  "op": "u",  // c=create, u=update, d=delete, r=read(snapshot)
  "ts_ms": 1705900800000,
  "source": {"db": "order_db", "table": "orders", "pos": 12345}
}
```

### 3.5 Canal 简介

Canal 是阿里开源的 MySQL Binlog 解析工具：

```
MySQL Master → Canal Server（伪装为 Slave 拉取 Binlog）→ Canal Client → 下游
```

**特点**：轻量、专注 MySQL、国内使用广泛。但功能不如 Debezium 丰富（不支持 PostgreSQL/MongoDB 等）。

## 四、批量接入工具

### 4.1 DataX

阿里开源的离线数据同步工具，采用 Framework + Plugin 架构：

```
┌──────────┐     ┌──────────────────────────┐     ┌──────────┐
│  Reader  │────→│  Framework（调度+缓冲）   │────→│  Writer  │
│ (MySQL)  │     │  Channel（内存队列）       │     │ (HDFS)   │
└──────────┘     └──────────────────────────┘     └──────────┘
```

**配置示例**：
```json
{
  "job": {
    "content": [{
      "reader": {
        "name": "mysqlreader",
        "parameter": {
          "connection": [{"jdbcUrl": ["jdbc:mysql://10.0.0.1:3306/db"], "table": ["orders"]}],
          "column": ["order_id", "user_id", "amount", "create_time"],
          "where": "create_time >= '${bizdate}'"
        }
      },
      "writer": {
        "name": "hdfswriter",
        "parameter": {
          "path": "/data/ods/orders/dt=${bizdate}",
          "fileType": "orc",
          "column": [{"name": "order_id", "type": "bigint"}, ...]
        }
      }
    }],
    "setting": {"speed": {"channel": 5}}
  }
}
```

### 4.2 Apache SeaTunnel（原 Waterdrop）

新一代数据集成平台，支持批流一体：

| 特性 | DataX | SeaTunnel |
|------|-------|-----------|
| **架构** | 单机 | 分布式（Flink/Spark 引擎） |
| **流式** | ❌ 仅批量 | ✅ 批流一体 |
| **CDC** | ❌ | ✅ 内置 CDC Connector |
| **扩展性** | 单机瓶颈 | 水平扩展 |
| **社区** | 维护较少 | Apache 孵化，活跃 |
| **Connector** | 丰富 | 丰富且持续增长 |

### 4.3 其他批量工具

| 工具 | 定位 | 特点 |
|------|------|------|
| **Sqoop** | Hadoop 生态批量导入 | 老牌工具，基于 MapReduce，已停止维护 |
| **Airbyte** | 开源 ELT 平台 | SaaS 化、300+ Connector、UI 友好 |

## 五、日志采集

### 5.1 日志采集架构

```
应用服务器（日志文件）
  → Filebeat / Fluentd（轻量采集 Agent）
    → Kafka（缓冲层）
      → Flink / Logstash（清洗转换）
        → Elasticsearch（检索）/ HDFS（分析）/ ClickHouse（聚合）
```

### 5.2 采集工具对比

| 工具 | 定位 | 资源占用 | 特点 |
|------|------|---------|------|
| **Filebeat** | 轻量日志采集 | 极低（Go） | Elastic 生态、简单可靠 |
| **Fluentd** | 统一日志层 | 中（Ruby/C） | 插件丰富、CNCF 项目 |
| **Fluent Bit** | 超轻量采集 | 极低（C） | 容器/IoT 场景 |
| **Logstash** | 日志处理管道 | 高（JVM） | 功能强大但重 |
| **Flume** | Hadoop 日志采集 | 中（Java） | 老牌、Hadoop 生态 |

## 六、Schema Registry 与 Schema Evolution

### 6.1 为什么需要 Schema Registry

数据接入最大的痛点之一：**上游改了表结构，下游全崩**。

```
场景：
  业务方在 orders 表加了一列 coupon_amount
  → CDC 开始传输新字段
  → 下游 Flink 作业不认识新字段 → 报错
  → 下游 Hive 表 Schema 不匹配 → 数据丢失
```

Schema Registry 是「Schema 的注册中心」——所有数据的 Schema 在这里注册、版本化管理、兼容性检查。

### 6.2 Schema 兼容性规则

| 兼容性级别 | 允许的变更 | 不允许的变更 |
|-----------|-----------|-------------|
| **BACKWARD** | 删列、加可选列 | 加必填列 |
| **FORWARD** | 加列、删可选列 | 删必填列 |
| **FULL** | 加/删可选列 | 加/删必填列 |
| **NONE** | 任意变更 | — |

### 6.3 主流方案

| 方案 | 生态 | 特点 |
|------|------|------|
| **Confluent Schema Registry** | Kafka | 最成熟，支持 Avro/Protobuf/JSON Schema |
| **AWS Glue Schema Registry** | AWS | 云原生，与 Glue Catalog 集成 |
| **Iceberg Schema Evolution** | 湖仓 | 表格式内置，基于列 ID |
| **Hive Metastore** | Hadoop | 传统方案，能力有限 |

### 6.4 Schema 变更传播

```
上游 DDL 变更（ALTER TABLE ADD COLUMN）
  → CDC 捕获 DDL 事件
  → Schema Registry 注册新版本
  → 兼容性检查（是否向后兼容？）
    → 兼容 → 自动传播到下游（Iceberg 自动加列）
    → 不兼容 → 告警 + 人工介入
```

### 7.2 异构数据源适配设计

```java
// 数据源适配器接口
public interface DataSourceAdapter {
    // 连接测试
    boolean testConnection(DataSourceConfig config);
    
    // 获取元数据（库/表/列）
    List<TableMeta> getMetadata(DataSourceConfig config);
    
    // 获取 Schema
    Schema getSchema(String database, String table);
    
    // 创建读取器（批量）
    DataReader createReader(ReadConfig config);
    
    // 创建 CDC 监听器（实时）
    CDCListener createCDCListener(CDCConfig config);
}

// 每种数据源实现一个 Adapter
public class MySQLAdapter implements DataSourceAdapter { ... }
public class PostgreSQLAdapter implements DataSourceAdapter { ... }
public class MongoDBAdapter implements DataSourceAdapter { ... }
```

### 7.3 接入任务生命周期

```
创建 → 配置（源/目标/映射/调度）→ 测试 → 上线 → 运行 → 监控 → 维护
```

**关键设计点**：
| 设计点 | 说明 |
|--------|------|
| **幂等性** | 重复执行不产生重复数据（基于主键去重或 Upsert） |
| **断点续传** | 失败后从上次位点继续，不需要全量重跑 |
| **限速** | 控制对源库的查询压力（QPS 限制、时间窗口） |
| **数据校验** | 接入后对比源和目标的行数/Checksum |
| **告警** | 延迟告警、失败告警、数据量异常告警 |

## 八、面试深度剖析

### Q1: CDC 和定时全量同步有什么区别？各自适用什么场景？

**答**：CDC 监听数据库变更日志，实时捕获增删改，延迟秒级，对源库几乎无压力；全量同步定时 SELECT 全表或增量数据，延迟小时/天级，对源库有查询压力。CDC 适合实时数仓、实时入湖；全量同步适合历史数据迁移、对时效性要求不高的场景。生产中通常 CDC 做实时层，全量同步做离线层的初始化和校验。

### Q2: Flink CDC 的全增量一体是怎么实现的？

**答**：`initial` 模式下，Flink CDC 先对表做一次无锁快照（基于 MVCC 的一致性读），记录快照时的 Binlog 位点；快照完成后，从该位点开始消费增量 Binlog。全量和增量之间无缝衔接，不丢不重。无锁快照的关键是利用数据库的 MVCC——在一个一致性事务中分 chunk 读取，不阻塞业务写入。

### Q3: 上游 Schema 变更了怎么办？

**答**：分三层处理——① 检测：CDC 工具捕获 DDL 事件或 Schema Registry 检测到新版本；② 兼容性判断：加列（向后兼容）可自动处理，改类型/删列（不兼容）需人工介入；③ 传播：兼容变更自动传播到下游（Iceberg 自动加列、Flink 作业自动适配）。生产中建议所有 Schema 变更走审批流程，并在数据契约中约定兼容性规则。

### Q4: 如何保证数据接入不丢不重？

**答**：① 不丢：CDC 基于 Binlog 位点 + Flink Checkpoint 保证 Exactly-Once；批量同步基于增量标记（时间戳/自增 ID）+ 断点续传；② 不重：目标端用 Upsert 语义（基于主键去重），或用 Iceberg/Hudi 的 MERGE 能力；③ 校验：定期对比源和目标的行数、Checksum，发现差异自动修复。

### Q5: DataX 和 SeaTunnel 怎么选？

**答**：DataX 是单机架构，适合中小规模批量同步，配置简单、稳定可靠，但无法水平扩展；SeaTunnel 是分布式架构（基于 Flink/Spark），支持批流一体、内置 CDC、可水平扩展，适合大规模数据集成。新项目推荐 SeaTunnel；已有 DataX 的团队如果规模不大可以继续用。
