# 6.17 多租户与权限体系

> **一句话定位**：数据平台的权限体系是「谁能看什么数据、做什么操作」的完整管控——从平台级的多租户隔离，到表级/行级/列级的细粒度权限控制，再到数据脱敏和审计追踪，确保数据安全合规的同时不影响数据使用效率。

---

## 一、为什么数据平台需要专门的权限体系

### 1.1 与后端服务权限的区别

| 维度 | 后端服务权限 | 数据平台权限 |
|------|------------|------------|
| **控制对象** | API 接口、功能按钮 | 数据库/表/行/列/指标 |
| **粒度** | 接口级（粗） | 行列级（细） |
| **数据量** | 单次请求几条记录 | 一次查询可能扫描亿级数据 |
| **消费方式** | 应用调用 API | SQL 查询、BI 报表、API、Agent |
| **合规要求** | 一般 | 严格（GDPR/个保法/数据分类分级） |

### 1.2 数据平台权限的核心挑战

| 挑战 | 说明 |
|------|------|
| **粒度细** | 不只是「能不能访问这张表」，还要控制「能看哪些行、哪些列」 |
| **消费端多** | 同一份数据被 BI、SQL 查询、API、Agent 等多种方式消费 |
| **动态性强** | 人员变动、组织调整、数据分类变更频繁 |
| **性能敏感** | 权限过滤不能显著影响查询性能 |
| **审计要求** | 谁在什么时间查了什么数据，必须可追溯 |

## 二、多租户隔离

### 2.1 什么是数据平台的多租户

多租户是指多个团队/业务线共享同一套数据平台基础设施，但彼此之间数据和资源互相隔离。

### 2.2 隔离层级

```
┌─────────────────────────────────────────────────────┐
│                  平台层隔离                           │
│  租户 A（外卖事业部）│ 租户 B（到店事业部）│ ...     │
├─────────────────────────────────────────────────────┤
│                  资源层隔离                           │
│  计算队列隔离 │ 存储空间隔离 │ 网络隔离              │
├─────────────────────────────────────────────────────┤
│                  数据层隔离                           │
│  库/Schema 隔离 │ 表级权限 │ 行列级权限             │
├─────────────────────────────────────────────────────┤
│                  元数据层隔离                         │
│  Catalog 隔离 │ 指标隔离 │ 血缘隔离                 │
└─────────────────────────────────────────────────────┘
```

### 2.3 隔离方案对比

| 方案 | 隔离程度 | 成本 | 适用场景 |
|------|---------|------|---------|
| **独立集群** | 最强（物理隔离） | 最高 | 强合规要求（金融/政务） |
| **共享集群 + 队列隔离** | 强（资源隔离） | 中 | 大厂内部多业务线 |
| **共享集群 + 逻辑隔离** | 中（权限隔离） | 低 | 中小规模、信任度高 |
| **Schema/Database 隔离** | 中（命名空间隔离） | 低 | 同一业务的多环境 |

### 2.4 资源隔离实现

```yaml
# YARN 队列配置示例
queues:
  - name: root.delivery      # 外卖事业部
    capacity: 40%
    max-capacity: 60%
    users: [delivery_team]
    
  - name: root.hotel          # 到店事业部
    capacity: 30%
    max-capacity: 50%
    users: [hotel_team]
    
  - name: root.platform       # 平台公共
    capacity: 30%
    max-capacity: 40%
    users: [platform_team]
```

**Kubernetes 资源隔离**：
```yaml
# K8s ResourceQuota
apiVersion: v1
kind: ResourceQuota
metadata:
  name: delivery-quota
  namespace: delivery
spec:
  hard:
    requests.cpu: "100"
    requests.memory: "200Gi"
    limits.cpu: "200"
    limits.memory: "400Gi"
```

## 三、权限模型

### 3.1 RBAC（基于角色的访问控制）

```
用户 → 角色 → 权限

示例：
  用户「张三」→ 角色「数据分析师-外卖」→ 权限集合：
    - 可读 dws_delivery_* 所有表
    - 可读 ads_delivery_* 所有表
    - 不可访问 ods 层原始数据
    - 不可看用户手机号（列级脱敏）
```

### 3.2 ABAC（基于属性的访问控制）

比 RBAC 更灵活——基于用户属性、资源属性、环境属性动态决策：

```
规则: IF user.department = '华北区' 
      AND resource.type = '订单表'
      AND time.hour BETWEEN 9 AND 18
      THEN allow READ WHERE city IN ('北京','天津','石家庄')
```

### 3.3 数据平台权限层级

| 层级 | 控制内容 | 示例 |
|------|---------|------|
| **Catalog 级** | 能否访问某个数据源 | 只能访问 Hive Catalog，不能访问生产 MySQL |
| **库/Schema 级** | 能否访问某个数据库 | 只能访问 dws 库，不能访问 ods 库 |
| **表级** | 能否访问某张表 | 可读 dws_order_daily，不可读 dws_user_profile |
| **列级** | 能否看某些列 | phone_number 列脱敏显示 |
| **行级** | 能否看某些行 | 只能看 city='北京' 的数据 |
| **操作级** | 能做什么操作 | 只能 SELECT，不能 INSERT/UPDATE/DROP |

## 四、行列级权限实现

### 4.1 行级权限（Row-Level Security）

**原理**：在用户查询时自动注入 WHERE 条件，限制可见数据范围。

```sql
-- 用户原始查询
SELECT city, SUM(amount) AS gmv FROM dws_order_daily GROUP BY city;

-- 权限系统自动改写（该用户只能看华北区）
SELECT city, SUM(amount) AS gmv FROM dws_order_daily
WHERE city IN ('北京', '天津', '石家庄', '太原')  -- 自动注入
GROUP BY city;
```

**实现方式**：

| 方式 | 原理 | 优劣 |
|------|------|------|
| **SQL 改写** | 在查询引擎层拦截 SQL，注入 WHERE 条件 | 通用，但需要 SQL 解析能力 |
| **视图** | 为每个角色创建带过滤条件的视图 | 简单，但视图数量爆炸 |
| **引擎原生** | 数据库/引擎内置 RLS（如 PostgreSQL） | 性能好，但依赖引擎支持 |
| **Ranger Policy** | Apache Ranger 策略引擎 | 统一管控，生态集成好 |

### 4.2 列级权限与数据脱敏

| 脱敏方式 | 示例 | 适用场景 |
|---------|------|---------|
| **完全隐藏** | 查询结果中不返回该列 | 完全无权限 |
| **部分遮盖** | `138****5678` | 需要知道格式但不需要明文 |
| **哈希替换** | `a3f2b8c1...` | 需要做关联但不需要明文 |
| **区间化** | 年龄 25 → `20-30` | 统计分析场景 |
| **置空** | `NULL` | 简单处理 |

```sql
-- Ranger 脱敏策略效果
-- 管理员看到：
SELECT user_id, phone, name FROM users;
-- | 1001 | 13812345678 | 张三 |

-- 普通分析师看到（phone 脱敏）：
SELECT user_id, phone, name FROM users;
-- | 1001 | 138****5678 | 张三 |

-- 外部人员看到（phone 隐藏、name 脱敏）：
SELECT user_id, phone, name FROM users;
-- | 1001 | [无权限] | 张* |
```

### 4.3 动态脱敏 vs 静态脱敏

| 方式 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **动态脱敏** | 查询时实时脱敏 | 一份数据多种视图、灵活 | 有性能开销 |
| **静态脱敏** | 提前生成脱敏副本 | 查询无额外开销 | 数据冗余、同步成本 |

**生产建议**：在线查询用动态脱敏（Ranger/语义层注入）；数据导出/测试环境用静态脱敏（生成脱敏副本）。

## 五、Apache Ranger

### 5.1 Ranger 是什么

Apache Ranger 是 Hadoop 生态的统一权限管理框架，支持对 HDFS、Hive、HBase、Kafka、Spark、Presto 等组件做细粒度权限控制。

### 5.2 Ranger 架构

```
┌─────────────────────────────────────────────────────┐
│                  Ranger Admin                        │
│  策略管理 │ 用户/组管理 │ 审计查看 │ 报表             │
├─────────────────────────────────────────────────────┤
│                  策略存储（MySQL/PostgreSQL）          │
├─────────────────────────────────────────────────────┤
│                  Ranger Plugin（嵌入各组件）          │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐      │
│  │Hive    │ │HDFS    │ │HBase   │ │Kafka   │      │
│  │Plugin  │ │Plugin  │ │Plugin  │ │Plugin  │      │
│  └────────┘ └────────┘ └────────┘ └────────┘      │
│  ┌────────┐ ┌────────┐ ┌────────┐                  │
│  │Presto  │ │Spark   │ │ES      │                  │
│  │Plugin  │ │Plugin  │ │Plugin  │                  │
│  └────────┘ └────────┘ └────────┘                  │
├─────────────────────────────────────────────────────┤
│                  审计日志（Solr / ES / HDFS）         │
└─────────────────────────────────────────────────────┘
```

### 5.3 Ranger 策略示例

```json
{
  "policyName": "delivery_analyst_policy",
  "resources": {
    "database": {"values": ["dws"]},
    "table": {"values": ["dws_order_*"]},
    "column": {"values": ["*"], "excludes": ["phone", "id_card"]}
  },
  "policyItems": [{
    "users": ["delivery_analyst_group"],
    "accesses": [{"type": "select"}],
    "rowFilterPolicyItems": [{
      "rowFilterInfo": {"filterExpr": "business_line = 'delivery'"}
    }],
    "dataMaskPolicyItems": [{
      "dataMaskInfo": {"dataMaskType": "MASK_SHOW_LAST_4"},
      "columns": ["phone"]
    }]
  }]
}
```

### 5.4 Ranger vs 其他方案

| 方案 | 定位 | 生态 | 特点 |
|------|------|------|------|
| **Apache Ranger** | Hadoop 生态统一权限 | Hive/HDFS/HBase/Kafka/Spark | 最成熟、Plugin 机制 |
| **Apache Sentry** | Hadoop 权限（已退役） | Hive/HDFS | 已合并到 Ranger |
| **Unity Catalog** | Databricks 统一治理 | Delta Lake/Spark/ML | 云原生、与 Databricks 深度集成 |
| **Privacera** | 数据治理平台（商业） | 多云多引擎 | 基于 Ranger 增强 |
| **自研权限服务** | 完全定制 | 任意 | 贴合业务但开发成本高 |

## 六、审计与合规

### 6.1 审计日志

记录所有数据访问行为，满足合规和安全审计要求：

```json
{
  "timestamp": "2024-01-20T10:30:00Z",
  "user": "zhangsan",
  "action": "SELECT",
  "resource": "dws.dws_order_daily",
  "columns_accessed": ["city", "amount", "user_id"],
  "row_filter_applied": "city IN ('北京')",
  "mask_applied": {"phone": "MASK_LAST_4"},
  "client_ip": "10.0.1.100",
  "query": "SELECT city, SUM(amount) FROM dws_order_daily GROUP BY city",
  "result_rows": 5,
  "duration_ms": 230,
  "access_result": "ALLOWED"
}
```

### 6.2 数据分类分级

| 级别 | 定义 | 示例 | 管控要求 |
|------|------|------|---------|
| **L1 公开** | 可对外公开的数据 | 公司简介、公开报告 | 无特殊要求 |
| **L2 内部** | 仅内部使用 | 内部报表、运营数据 | 内部人员可访问 |
| **L3 敏感** | 涉及业务敏感信息 | 营收数据、竞争分析 | 需审批、有时效 |
| **L4 机密** | 涉及个人隐私/核心商业秘密 | 手机号、身份证、薪资 | 严格管控、脱敏、审计 |

### 6.3 合规要求映射

| 法规 | 核心要求 | 数据平台实现 |
|------|---------|------------|
| **个人信息保护法** | 最小必要、知情同意、可删除 | 列级权限、脱敏、数据生命周期 |
| **GDPR** | 数据最小化、被遗忘权、可移植 | 行级过滤、数据删除能力、导出 |
| **数据安全法** | 分类分级、安全评估 | 数据分级标签、审计日志 |

## 七、权限与数据平台各层的集成

### 7.1 集成全景

```
┌─────────────────────────────────────────────────────┐
│                  统一权限服务                         │
│  用户/角色管理 │ 策略管理 │ 审计 │ 脱敏规则          │
├─────────────────────────────────────────────────────┤
│                  各层集成点                           │
│                                                     │
│  语义层 ──→ 指标级权限 + 行级过滤注入               │
│  查询引擎 ──→ Ranger Plugin 拦截 SQL                │
│  Catalog ──→ 表/库级可见性控制                      │
│  调度系统 ──→ 任务执行身份 + 数据访问范围            │
│  数据接入 ──→ 源端权限验证 + 目标端写入权限          │
│  AgentBI ──→ 用户身份透传 + 权限过滤注入            │
│  数据服务API ──→ Token 认证 + 数据范围限制           │
└─────────────────────────────────────────────────────┘
```

### 7.2 权限在查询链路中的执行时机

```
用户发起查询
  → 身份认证（谁？）
  → 资源鉴权（能访问这张表吗？）
  → 列级权限（哪些列需要脱敏/隐藏？）
  → 行级权限（注入 WHERE 过滤条件）
  → SQL 改写（应用所有权限规则）
  → 查询执行
  → 结果脱敏（动态脱敏处理）
  → 审计记录（记录本次访问）
  → 返回结果
```

## 八、面试深度剖析

### Q1: 行级权限怎么实现？会不会影响查询性能？

**答**：主要通过 SQL 改写实现——在查询执行前，根据用户的权限策略自动注入 WHERE 条件。性能影响取决于过滤条件是否能走索引：① 如果过滤列有索引/分区（如 city），影响很小；② 如果过滤条件复杂（如子查询），可能有较大影响。优化方案：权限维度尽量对齐分区键、预计算权限范围缓存到 Redis、对大范围权限用户直接跳过过滤。

### Q2: 多租户的资源隔离怎么做？

**答**：分三层——① 计算隔离：YARN 队列/K8s Namespace + ResourceQuota，限制每个租户的 CPU/内存上限；② 存储隔离：不同租户的数据放不同目录/Bucket，通过 HDFS ACL 或 S3 Policy 控制；③ 网络隔离：不同租户的服务在不同网段（强隔离场景）。大多数场景用「共享集群 + 队列隔离 + 权限控制」即可，不需要物理隔离。

### Q3: Ranger 的 Plugin 机制是怎么工作的？

**答**：Ranger Plugin 以 JAR 包形式嵌入到各组件中（如 Hive Server、Presto Coordinator）。当用户发起请求时，Plugin 拦截请求，向 Ranger Admin 查询权限策略（策略会缓存在本地，定期同步），根据策略决定允许/拒绝/脱敏。Plugin 还负责将访问记录异步写入审计日志。这种设计的好处是权限决策在数据面本地完成，不需要每次都调用远程服务。

### Q4: 数据脱敏有哪些方案？动态和静态怎么选？

**答**：动态脱敏在查询时实时处理（Ranger DataMask Policy / 语义层注入），优点是一份数据多种视图、灵活；缺点是有性能开销。静态脱敏提前生成脱敏副本（ETL 时处理），优点是查询无额外开销；缺点是数据冗余、同步成本。生产建议：在线分析用动态脱敏，数据导出/测试环境/外部共享用静态脱敏。

### Q5: 如何设计一个数据平台的统一权限体系？

**答**：四层设计——① 身份层：对接公司统一认证（SSO/LDAP），用户→角色→权限组；② 策略层：RBAC + ABAC 混合，支持库/表/行/列四级粒度；③ 执行层：各引擎通过 Plugin 或 SQL 改写执行权限策略；④ 审计层：所有访问记录入审计日志，支持合规审查。关键原则：最小权限、默认拒绝、权限可继承（库级权限自动继承到表）、变更可审计。
