# 6.15 平台可观测性

> **一句话定位**：数据平台可观测性是对整个数据链路（采集→存储→计算→服务）的「全景监控」——不仅要知道任务跑没跑完，还要知道跑得快不快、资源用了多少、数据延迟多久、哪里是瓶颈，让平台团队从「救火」转向「预防」。

---

## 一、为什么数据平台需要专门的可观测性

### 1.1 与后端服务可观测性的区别

| 维度 | 后端服务（微服务） | 数据平台 |
|------|-------------------|---------|
| **关注对象** | 请求、接口、服务实例 | 任务、数据流、数据表 |
| **核心指标** | QPS、RT、错误率 | 任务耗时、数据延迟、资源利用率 |
| **时间尺度** | 毫秒~秒 | 分钟~小时 |
| **故障模式** | 服务不可用、超时 | 任务失败、数据延迟、数据错误、资源不足 |
| **链路追踪** | 请求在服务间的调用链 | 数据在表/任务间的流转链（血缘） |
| **SLA 定义** | 可用性 99.99% | 数据产出时间 < 08:00 |

### 1.2 可观测性三支柱在数据平台的映射

| 支柱 | 后端服务 | 数据平台 |
|------|---------|---------|
| **Metrics（指标）** | QPS/RT/错误率 | 任务耗时/资源用量/数据量/延迟 |
| **Logs（日志）** | 应用日志/访问日志 | 任务日志/引擎日志/审计日志 |
| **Traces（链路）** | 分布式调用链 | 数据血缘/任务依赖链/端到端延迟 |

## 二、核心监控维度

### 2.1 任务层监控

| 指标 | 说明 | 告警阈值示例 |
|------|------|-------------|
| **任务成功率** | 成功任务数 / 总任务数 | < 95% 告警 |
| **任务耗时** | 单个任务的执行时长 | 超过历史均值 3 倍告警 |
| **任务排队时间** | 从调度到实际开始执行的等待时间 | > 30 分钟告警 |
| **任务重试次数** | 重试频率反映系统稳定性 | 重试率 > 10% 告警 |
| **SLA 达成率** | 关键任务是否按时完成 | 未达成立即告警 |
| **任务积压** | 待执行任务队列长度 | 持续增长告警 |

### 2.2 计算引擎监控

#### Spark 监控

| 指标 | 说明 | 关注点 |
|------|------|--------|
| **Executor 内存使用** | 堆内/堆外内存占用 | OOM 预警 |
| **Shuffle 数据量** | Stage 间数据传输量 | 数据倾斜信号 |
| **GC 时间占比** | GC 耗时 / 总耗时 | > 10% 需优化 |
| **Task 倾斜度** | 最慢 Task / 中位数 Task 耗时 | > 5x 有数据倾斜 |
| **Spill 到磁盘** | 内存不足溢写磁盘的数据量 | 需要增加内存 |

#### Flink 监控

| 指标 | 说明 | 关注点 |
|------|------|--------|
| **Checkpoint 耗时** | 快照完成时间 | 持续增长说明 State 过大 |
| **Checkpoint 失败率** | 失败次数 / 总次数 | > 0 需排查 |
| **反压（Backpressure）** | 算子处理速度跟不上输入 | 持续反压需扩容或优化 |
| **Watermark 延迟** | 当前 Watermark 与实际时间的差 | 延迟过大影响窗口计算 |
| **消费延迟（Lag）** | Kafka 消费位点落后量 | 持续增长需扩容 |
| **State 大小** | RocksDB State 占用 | 持续增长需清理或优化 |

#### Presto/Trino 监控

| 指标 | 说明 | 关注点 |
|------|------|--------|
| **查询并发数** | 同时运行的查询数 | 接近上限需限流 |
| **查询排队数** | 等待资源的查询数 | 持续积压需扩容 |
| **内存使用率** | 集群总内存占用 | > 80% 有 OOM 风险 |
| **查询失败率** | OOM/超时导致的失败 | > 5% 需优化 |
| **P99 查询耗时** | 99 分位查询延迟 | 反映用户体验 |

### 2.3 数据链路监控

| 指标 | 说明 | 告警场景 |
|------|------|---------|
| **端到端延迟** | 数据从产生到可查询的总延迟 | 实时链路延迟 > 5 分钟 |
| **数据新鲜度** | 表/分区的最新数据时间 | 超过预期产出时间 |
| **数据量趋势** | 每日/每小时数据量变化 | 突增/突降 |
| **Schema 变更** | 上游表结构变化 | 可能导致下游任务失败 |
| **数据倾斜** | 分区/Key 的数据分布 | 某个分区数据量异常大 |

### 2.4 资源层监控

| 指标 | 说明 | 关注点 |
|------|------|--------|
| **集群 CPU 利用率** | 计算资源使用情况 | 持续 > 80% 需扩容 |
| **集群内存利用率** | 内存资源使用情况 | 接近上限有 OOM 风险 |
| **磁盘使用率** | 存储空间占用 | > 80% 需清理或扩容 |
| **网络 IO** | 节点间数据传输量 | Shuffle 密集时的瓶颈 |
| **YARN/K8s 队列** | 资源队列使用率和等待情况 | 队列满导致任务排队 |
| **HDFS/S3 吞吐** | 存储层读写速度 | 影响任务执行速度 |

## 四、关键场景实践

### 4.1 端到端数据延迟监控

```
数据产生（业务DB）
  → CDC 采集延迟（Debezium/Flink CDC）
    → Kafka 传输延迟
      → Flink 处理延迟
        → 写入湖表延迟（Checkpoint 间隔）
          → 查询可见延迟（Catalog 刷新）

端到端延迟 = Σ 各环节延迟
```

**监控方式**：
- 在数据中埋入时间戳（event_time）
- 在每个环节记录处理时间
- 计算 `当前时间 - event_time` = 端到端延迟
- 按分位数（P50/P95/P99）展示

### 4.2 任务 SLA 监控

```yaml
# SLA 监控配置
sla_monitors:
  - name: "核心报表产出"
    tasks:
      - dag: daily_report_pipeline
        final_task: ads_report_refresh
    deadline: "08:00"
    
    # 预警机制
    checkpoints:
      - time: "06:00"
        expected_progress: "DWS 层完成"
        alert_if_not: warning
      - time: "07:00"
        expected_progress: "ADS 层完成"
        alert_if_not: critical
      - time: "07:45"
        expected_progress: "报表刷新完成"
        alert_if_not: critical_escalate
```

### 4.3 资源利用率分析

```sql
-- 示例：分析各团队的 Spark 资源使用效率
SELECT
    team,
    SUM(executor_hours) AS total_executor_hours,
    SUM(actual_compute_time) / SUM(executor_hours) AS utilization_rate,
    AVG(shuffle_bytes / input_bytes) AS shuffle_ratio,
    COUNT(CASE WHEN status = 'FAILED' THEN 1 END) / COUNT(*) AS failure_rate
FROM spark_job_metrics
WHERE dt = '${yesterday}'
GROUP BY team
ORDER BY total_executor_hours DESC;
```

**资源优化建议**：
| 现象 | 可能原因 | 优化方向 |
|------|---------|---------|
| 利用率 < 30% | 资源申请过多 | 减少 Executor 数或内存 |
| Shuffle 比例 > 50% | 数据倾斜或 JOIN 策略不当 | 优化 SQL、调整并行度 |
| GC 时间 > 10% | 内存不足 | 增加内存或减少缓存数据 |
| 排队时间 > 执行时间 | 集群资源不足 | 扩容或优化调度优先级 |

### 4.4 故障根因分析

```
告警：ADS 报表未按时产出
  → 查看任务依赖链（哪个上游任务延迟了？）
    → DWS 汇总任务耗时异常（平时 30 分钟，今天 2 小时）
      → 查看 Spark UI（哪个 Stage 慢？）
        → Shuffle Stage 有数据倾斜（某个 Task 处理了 90% 的数据）
          → 查看数据分布（某个 user_id 有异常大量数据）
            → 根因：爬虫用户产生了大量异常数据
              → 修复：过滤异常用户 + 加盐打散
```

**根因分析工具链**：
1. **告警** → 发现问题
2. **任务血缘** → 定位延迟环节
3. **引擎监控** → 定位性能瓶颈
4. **日志检索** → 查看具体错误
5. **数据分析** → 确认数据层面的根因

### 5.2 引擎专项大盘

每个计算引擎一个专项大盘，包含：
- **Spark**：活跃 Application 数、Executor 内存/CPU、Shuffle 量、GC 时间、失败任务
- **Flink**：运行中 Job 数、Checkpoint 耗时/失败、反压算子、消费 Lag、State 大小
- **Presto**：并发查询数、排队数、内存使用、P99 延迟、失败查询
- **Kafka**：Topic 消费 Lag、Broker 磁盘、分区 Leader 分布

## 六、面试深度剖析

### Q1: 数据平台的可观测性和后端微服务有什么区别？

**答**：核心区别在关注对象和时间尺度——后端关注请求级别的延迟和可用性（毫秒级），数据平台关注任务级别的耗时和数据产出时效（分钟~小时级）。后端的链路追踪是请求在服务间的调用链，数据平台的「链路」是数据在表/任务间的流转链（血缘）。此外数据平台还需要关注数据质量维度（准确性/完整性），这是后端不需要的。

### Q2: 如何监控端到端数据延迟？

**答**：在数据源埋入 event_time，在每个处理环节记录处理时间戳，最终用 `当前时间 - event_time` 计算端到端延迟。关键是要覆盖全链路：CDC 采集延迟 + Kafka 传输延迟 + 计算引擎处理延迟 + 写入延迟 + 查询可见延迟。按 P50/P95/P99 分位数展示，设置阈值告警。

### Q3: Flink 任务反压怎么排查？

**答**：① 看 Flink Web UI 的 Backpressure 面板，找到反压源头算子；② 看该算子的 Metrics（处理速率、忙碌率）；③ 常见原因：下游 Sink 慢（如写 Kafka/DB 慢）、State 访问慢（RocksDB 性能）、数据倾斜（某个 Key 数据量过大）、GC 频繁；④ 解决：扩并行度、优化 State 访问、加盐打散、异步 IO。

### Q4: 如何设计数据平台的告警体系？

**答**：分四层——① 基础设施层（CPU/内存/磁盘/网络）；② 引擎层（Spark/Flink/Presto 的核心指标）；③ 任务层（成功率/耗时/SLA）；④ 数据层（质量/延迟/量级）。告警需要分级（Critical/Warning/Info）、收敛（同一问题不重复告）、关联（上游告警时抑制下游）、升级（未处理自动升级）。

### Q5: 如何做资源成本优化？

**答**：① 建立资源使用度量（按团队/任务统计 CPU·时/内存·时）；② 识别低效任务（利用率 < 30%、Shuffle 比例过高、GC 过多）；③ 推动优化（减少资源申请、优化 SQL、调整并行度）；④ 建立成本分摊机制（按团队计费，推动自主优化）；⑤ 弹性伸缩（K8s 按需扩缩，避免资源闲置）。
