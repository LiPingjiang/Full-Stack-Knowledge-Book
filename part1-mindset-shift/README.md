# 第一部分 · 后端转前端的思维模式转变

> 转前端最难的，从来不是 `useState` 怎么写、CSS 的 `flex` 怎么用。
> 那些查文档、问 AI 半小时就会了。真正卡住资深后端工程师的，是**心智模型的错位**。

---

## 为什么思维转变比语法更重要

你写 Java 十年，脑子里有一套根深蒂固的「世界观」：

- 程序是**自上而下**执行的，一个请求进来，跑一遍逻辑，返回结果，结束。
- 状态是**短暂**的，方法栈上的局部变量用完即弃，长期状态都在数据库里。
- 类型是**契约**，编译器是你的安全网，编译不过就是写错了。
- 程序运行在**你掌控的服务器**上，环境可预测，资源可规划。

这套世界观在后端无往不利。但当你打开浏览器写前端时，**每一条都会被打脸**：

- 前端代码不是「跑一遍就结束」，而是**常驻内存、等待用户随时触发事件**。
- 状态不在数据库里，而是**散落在内存中的一堆变量**，还要时刻和界面保持同步。
- 你面对的是动态类型的 JavaScript，编译器（曾经）帮不了你。
- 你的代码运行在**用户的设备**上——可能是十年前的安卓机，也可能是断网的地铁里。

如果你不重装这套世界观，硬用后端思维写前端，结果就是：代码能跑，但处处别扭，到处是你自己都说不清的 bug。**这一部分，就是带你完成这次世界观的重装。**

## 本部分章节

每一章对应一组「后端直觉 → 前端现实」的认知错位，我们逐个击破。

### [1.1 从「请求-响应」到「用户交互」](./01-从请求响应到用户交互.md)
后端的同步线性思维 → 前端的事件驱动思维。这是最根本的一跳。

### [1.2 从「无状态」到「有状态」](./02-从无状态到有状态.md)
后端把「无状态」奉为信条（方便水平扩展），前端却天生有状态，而且状态管理是前端工程的核心难题。

### [1.3 从「强类型」到「类型光谱」](./03-从强类型到类型光谱.md)
Java 的强类型不是非黑即白，类型其实是一道光谱。理解 TypeScript 如何在动态世界里重建你熟悉的安全网。
> 延伸阅读：[Java → JS/TS 的语言迁移](../part4-multilang-compare/02-Java到JS-TS.md)

### [1.4 从「单一运行时」到「多运行时」](./04-从单一运行时到多运行时.md)
你只伺候过一个 JVM。前端世界里同一段 JS 要在浏览器、Node、Deno、甚至边缘函数里跑，运行时差异会反复咬你。

### [1.5 从「数据建模」到「UI 建模」](./05-从数据建模到UI建模.md)
你擅长用 ER 图和表结构建模数据。前端要求你用「组件树 + 状态」建模界面——这是一种你尚未练过的建模肌肉。

## 一句话总览

| 维度 | 后端直觉 | 前端现实 |
|------|---------|---------|
| 执行模型 | 请求来了跑一遍就结束 | 常驻内存，等事件触发 |
| 状态归属 | 状态在数据库，进程无状态 | 状态在内存，且要和 UI 同步 |
| 类型系统 | 强类型，编译器兜底 | 动态类型起家，TS 重建安全网 |
| 运行环境 | 我掌控的服务器，可预测 | 用户的设备，千差万别 |
| 建模对象 | 数据（表 / 实体 / 关系） | 界面（组件 / 状态 / 交互） |

读完这一部分，你不会立刻变成前端高手，但你会**停止用后端的方式误解前端**——这本身就能帮你少走 80% 的弯路。
