# 并发模型库 · Java 线程与虚拟线程

> 本篇深挖 Java 并发的底层机制，是 [3.1 Java 并发体系](../part3-java-deep/01-并发体系.md) 和 [4.1 高并发对比](../part4-multilang-compare/01-高并发HTTP服务对比.md) 的「钩子」目标。

---

## 一、平台线程：OS 线程的 1:1 映射

JDK 21 之前的 `Thread`（现在称为**平台线程，platform thread**），与操作系统内核线程是 **1:1 映射**——一个 Java 线程对应一个 OS 线程。

```
   Java 平台线程        OS 内核线程
   ┌──────────┐  1:1  ┌──────────┐
   │ Thread A  │ ────> │ kernel-1 │
   │ Thread B  │ ────> │ kernel-2 │
   └──────────┘       └──────────┘   ← OS 调度器负责切换
```

这个 1:1 模型决定了平台线程的全部特性：

- **栈固定**：默认每线程约 1MB 栈（`-Xss` 可调），且在创建时就分配。
- **调度在内核**：线程切换是内核态的上下文切换，要陷入内核、保存/恢复寄存器、可能刷 TLB，开销约几微秒。
- **数量受限**：受内存（每个 1MB）和内核调度能力限制，几千个就吃力，上万个基本不可行。
- **阻塞 = 浪费**：线程执行阻塞 IO 时被 OS 挂起，那 1MB 栈和内核资源就被占着干等。

## 二、虚拟线程：M:N 调度的革命

JDK 21 的**虚拟线程（virtual thread）** 引入了 **M:N 调度**——M 个虚拟线程被复用到 N 个平台线程（称为**载体线程，carrier thread**）上，N 通常等于 CPU 核数。

```
   M 个虚拟线程 (海量)         N 个载体线程 (少量, ≈核数)      OS 线程
   ┌─────────┐
   │ VT-1     │ ─┐
   │ VT-2     │  ├──挂载──> ┌──────────┐  1:1  ┌────────┐
   │ VT-3     │  │          │ carrier-1 │ ────> │ kernel │
   │ ...      │ ─┘          └──────────┘       └────────┘
   │ VT-100万 │ ──挂载──>   ┌──────────┐  1:1  ┌────────┐
   └─────────┘             │ carrier-2 │ ────> │ kernel │
                           └──────────┘       └────────┘
```

关键机制：

- **栈在堆上、按需增长**：虚拟线程的栈（称为 stack chunk）存在 JVM 堆里，几百字节起步，按需增长。所以能开几百万个。
- **JVM 用户态调度**：虚拟线程的调度由 JVM（一个 ForkJoinPool）在**用户态**完成，不陷入内核，切换成本极低。

> **先纠正一个最常见的误解**：很多人把虚拟线程理解成「一个 OS 线程之上封装了若干虚拟线程」——**数量上是 M:N（少量载体线程承载海量虚拟线程）没错，但「在 OS 线程之上封装」这个方向说反了**。真正「封装 OS 线程」的是**平台线程**（`new Thread()` 背后就是一个内核线程的薄壳，调度权在 OS 手里）。虚拟线程恰恰相反，它本质是一个**堆上的 Java 对象**（保存着自己的栈和执行进度），平时**不绑定任何 OS 线程**；只有要干活的那一刻才临时挂载到载体线程上跑，一阻塞就被卸下来。所以准确的心智模型是：**平台线程 = 「OS 线程 + 薄壳」；虚拟线程 = 「堆对象 + 临时借工位」**。它不是封装 OS 线程，而是**把调度权从 OS 抢回到了 JVM 用户态**。一个虚拟线程一生可能先后被挂到 A、B、C 多个载体线程上，跟载体线程没有固定归属。

## 三、为什么虚拟线程这么难做？为什么 JDK 21 才有？

M:N 调度的「想法」并不新——早期 Java 1.1 时代就有用户态的 **Green Threads**（也是 M:N），但因当时利用不了多核、调度做不好，1.2 之后被砍掉、全面转向 1:1 的 OS 线程。**所以问题从来不是「想不想做」，而是「能不能做对」。** 虚拟线程真正的拦路虎是：怎么在**不让程序员改一行代码、不污染整个生态**的前提下，做到「阻塞时把执行状态完整存下、之后精确恢复」。拆成三个硬骨头：

| 难点 | 要解决什么 | 为什么难 |
|------|-----------|---------|
| **① Continuation（续体）** | 虚拟线程跑到阻塞点时，要把**整个调用栈**（执行到第几行、每层局部变量）冻结存进堆，之后从断点那条字节码精确恢复 | 这要求 JVM 有能力「任意暂停/恢复一段 Java 执行流」，是对 JVM 栈管理的**深度改造**，也是 Loom 最核心、最难啃的部分 |
| **② 阻塞点要「自动」让出** | 老代码、老库**一行不改**就能享受红利，意味着 `java.net`/`java.io`/JUC 里**所有会阻塞的地方**都要重写成「触发卸载-挂载」而非真阻塞 | 这是贯穿整个标准库的**海量工程改造**，牵一发动全身 |
| **③ 跟现有线程模型无缝兼容** | 虚拟线程必须长得跟普通 `Thread` 一样（它就是 `Thread` 子类），`ThreadLocal`、调试器、各种框架对线程的假设都要继续能用 | 「在飞行中换引擎，还要让所有仪表盘照常工作」，兼容性约束极苛刻 |

> 所以**为什么 JDK 21 才有**：① 早年有意识地选了 1:1（务实利用多核）；② 真正把 1:1 逼到墙角的是近十年的超高并发（C10K/C100K），需求到了投入才划算；③ Continuation 依赖 JVM 多年在栈管理、JIT、GC 上的演进逐步攒齐。Project Loom 从 **2017 年立项**，磨了 **6 年多**、反复推翻设计跑预览版，到 **JDK 21（2023）** 才正式转正。**它慢，不是简单东西拖到现在，而是因为它本就是『无侵入兼容 + JVM 深度改造』必须同时做到的硬核工程。**

## 四、核心魔法：挂载（mount）与卸载（unmount）

虚拟线程「阻塞自动让出」的秘密，在于 **mount/unmount 机制**：

1. 虚拟线程要运行时，**挂载（mount）** 到一个载体线程上执行。
2. 当它执行到**阻塞操作**（如阻塞 IO、`Thread.sleep`、加锁等待）时，JVM 把它的栈**保存到堆里**，并将它从载体线程**卸载（unmount）**。
3. 载体线程立刻空出来，去跑别的虚拟线程。
4. 等阻塞操作完成（IO 就绪），这个虚拟线程被重新调度，**挂载**回某个载体线程，从断点继续。

```
虚拟线程 VT 执行中...
     │
     ▼ 遇到阻塞 IO (如 socket.read)
  [unmount] ── VT 的栈存入堆，载体线程释放 ──> 载体线程去跑 VT-other
     │
     ▼ IO 完成
  [mount]   ── VT 重新挂载到某个载体线程，从断点继续 ──> 继续执行
```

**这就是「写起来像阻塞、跑起来像异步」的原理**：你写的 `socket.read()` 看似阻塞，但 JVM 在底层把「阻塞」转换成了「卸载-等待-重新挂载」，载体线程从不空转。这与 [Go goroutine](./go-goroutine-csp.md) 的调度思想高度一致。

<details>
<summary><b>展开：那「同步阻塞」在虚拟线程下到底阻塞了没有？——它在两层上同时是两个东西</b></summary>

这是虚拟线程最烧脑、也最精髓的一点。答案是：**阻塞了，但只阻塞了「便宜的那个」，没阻塞「昂贵的那个」。** 必须分三个视角看：

| 站在哪一层看 | 是同步阻塞还是异步非阻塞？ | 解释 |
|------------|--------------------------|------|
| **业务代码 / 虚拟线程** 这一层 | **同步阻塞**（真的，不是装的） | `read()` 真的会让**这个虚拟线程**停下、等数据、再顺着往下走。写法、语义、调试看到的调用栈，全是同步阻塞 |
| **载体线程 / OS 内核** 这一层 | **异步非阻塞** | 真实 OS 线程从没为这次 IO 干等——虚拟线程一阻塞就被卸载，载体线程立刻去跑别的。内核眼里就是标准 epoll 非阻塞 IO |
| 中间的 **JVM 运行时** | 干「翻译」的活 | 把上层的「同步阻塞调用」翻译成底层的「非阻塞 IO + 卸载/挂载」 |

所以最准确的说法是：**它在语义层是同步阻塞，在执行层是异步非阻塞。** 「同步阻塞」描述**代码的写法与语义**，「异步非阻塞」描述**底层真实发生的事**。虚拟线程的全部价值就是**把这两层解耦**——写法层享受同步的简单，执行层拿到异步的性能。

> 打个比方：你打电话订位「在线等」（对你是同步阻塞，你就等回话）；但接线员没捏着话筒陪你干等，而是把你挂起、先接别的电话，有结果再回拨你（对系统是异步非阻塞）。**你的体验是「同步等」，系统的真相是「异步调度」**——虚拟线程就是那个聪明的接线员。

</details>

## 五、坑：Pinning（钉住）

虚拟线程有一个要注意的陷阱——**pinning（钉住）**。在某些情况下，虚拟线程**无法从载体线程卸载**，即使它阻塞了也会一直占着载体线程：

主要发生在两种情况：

1. **在 `synchronized` 块/方法内阻塞**：JDK 21 中，持有 `synchronized` 监视器时阻塞会导致 pinning（这会削弱虚拟线程的优势）。
2. **调用 native 方法（JNI）时阻塞**。

```java
// JDK 21 中可能 pinning 的写法
synchronized (lock) {
    blockingIO();   // 在 synchronized 内阻塞 → 虚拟线程被钉住，载体线程被占
}

// 推荐改用 ReentrantLock，它对虚拟线程友好，阻塞时能正常卸载
lock.lock();
try {
    blockingIO();   // 不会 pinning
} finally {
    lock.unlock();
}
```

> 实践建议（JDK 21）：在虚拟线程密集的代码里，**用 `ReentrantLock` 替代 `synchronized`** 来避免 pinning。注意：JDK 24+（JEP 491）已解决了 `synchronized` 的 pinning 问题，但在过渡期了解这个坑很重要。

## 六、对照其他模型

| 维度 | Java 平台线程 | Java 虚拟线程 | 对照 |
|------|--------------|--------------|------|
| 映射 | 1:1 (OS 线程) | M:N (用户态调度) | 似 [Go goroutine](./go-goroutine-csp.md) |
| 栈 | 固定 1MB | 堆上按需增长 | 似 goroutine 的可增长栈 |
| 阻塞处理 | OS 挂起，浪费 | mount/unmount 自动让出 | 似 goroutine 让出 |
| 写法 | 同步阻塞 | 同步阻塞（不变！） | vs [Rust async](./rust-async-tokio.md) 的显式 await |

**最重要的对照**：虚拟线程和 [Go goroutine](./go-goroutine-csp.md) 在机制上几乎是同一套东西（用户态 M:N 调度 + 阻塞让出）。而与 [Rust/Node/Python 的 async](./rust-async-tokio.md) 相比，虚拟线程的最大优势是**不需要 `async/await` 关键字污染代码**——你的同步代码原封不动就能享受高并发，没有「函数染色」问题。

<details>
<summary><b>展开：什么是「函数染色（function coloring）」？为什么它是 async 派的原罪</b></summary>

这个概念来自 Bob Nystrom 的著名文章《What Color is Your Function?》。核心规则就一句话：

> **在 `async/await` 体系里，函数被分成两种「颜色」：普通函数（同步色）和 `async` 函数（异步色）。而 `async` 这个颜色会沿调用链向上传染。**

传染规则非常恶心：

| 规则 | 后果 |
|------|------|
| 想 `await` 一个 async 函数 | 调用它的函数自己也必须是 `async` |
| 一个函数变成 `async` | 它的所有调用者**也得跟着变 async** |
| 链条一路传染到 `main` | 整条调用栈被「染成」异步色，无一幸免 |

```js
// 一旦最底层的 db.query 是 async……
async function getUser(id) { return await db.query(...); }   // 被迫 async
async function handler(req) { return await getUser(req.id); } // 被迫 async
async function route() { return await handler(...); }         // 被迫 async
// ……异步色一路向上传染，谁也躲不掉
```

**为什么是原罪**：它把「这个函数会不会做 IO」这种实现细节，强行暴露成了**函数签名的一部分**，破坏了封装；同步代码和异步代码无法自由组合（同步函数里没法直接 `await`）；老的同步库要接入异步世界，往往要整条链路重写。Rust async、JS/Node、Python asyncio 全都有这个病。

**而虚拟线程和 [Go goroutine](./go-goroutine-csp.md) 是「无色」的**：你照常写同步函数、同步调用，没有 `async` 关键字、没有传染、没有签名污染——IO 是否阻塞由运行时在底层处理，对代码完全透明。这就是「隐藏异步」流派相对「暴露异步」流派最大的工程价值。

</details>

## 七、和 JDK 8 极致异步写法相比，性能差多少？

一个常见且关键的疑问：既然底层都是 epoll，那 **JDK 21 虚拟线程** 和 **JDK 8 把 NIO/Netty 写到极致** 的异步代码，性能到底差多少？

**结论：在 IO 密集型、且双方都写到极致的前提下，峰值吞吐基本是同一量级——因为底层踩的是同一套 epoll。** 虚拟线程并没有发明新的、更快的 IO 原语。

所以虚拟线程的核心价值，**不是单纯让性能更高，而是用「同步写法」拿到「异步性能」，从而消除异步的全部隐性代价**：

| 异步（NIO/Netty/CompletableFuture）的隐性代价 | 虚拟线程怎么解决 |
|------------------------------------------|----------------|
| 难写：回调地狱、链式拼接、控制流被打碎 | 同步顺序写，if/for/try-catch 照常用 |
| 难调：调用栈断裂，异常栈看不出业务路径 | 调用栈完整，断点/异常栈一目了然 |
| 难维护：业务逻辑被异步骨架淹没 | 业务逻辑就是主线，无骨架污染 |
| 生态割裂：同步库不能直接用，得找异步版 | 现有同步库（JDBC 等）原地受益 |
| 函数染色：async 一路向上传染 | 无 async 关键字，零传染 |

所以「核心是方便写代码」这个理解**主干是对的，但不止于「方便」**——它把可维护性、可调试性、生态兼容性一起救了回来。

**两个重要例外，别记错：**

- **CPU 密集型任务**：虚拟线程、平台线程、NIO 三者性能**几乎一样**。因为瓶颈在算力不在 IO，没有 IO 等待可以「让出」，调度优化无从发挥。
- **pinning 退化**：一旦发生 [pinning](#五坑pinning钉住)（如 `synchronized` 内阻塞），虚拟线程退化成占着载体线程干等，吞吐会明显劣于精心写的 NIO。这是虚拟线程**可能反而更慢**的主要场景。

## 本篇小结

- 平台线程 = OS 线程 1:1 映射，1MB 栈、内核调度、阻塞浪费。
- 虚拟线程 = M:N 用户态调度，堆上按需栈、JVM 调度、阻塞自动让出（mount/unmount）。
- 虚拟线程不是「OS 线程之上的封装」，而是**把调度权从 OS 抢回 JVM 用户态**；难做（Continuation/标准库重写/兼容），所以 JDK 21 才落地。
- 「同步阻塞」在**语义层**是真阻塞、在**执行层**是异步非阻塞，JVM 运行时做翻译。
- 性能上与 JDK 8 极致异步同量级（底层同为 epoll），核心价值是**用同步写法消除异步的难写/难调/难维护/生态割裂/函数染色**；CPU 密集型三者持平，pinning 是退化例外。
- 虚拟线程与 [Go goroutine](./go-goroutine-csp.md) 同源，且无 [async](./rust-async-tokio.md) 的函数染色问题。
