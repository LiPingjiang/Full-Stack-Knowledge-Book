# 9.2 基础架构全景：从车端到云端

在深入具体领域知识（数据闭环、仿真、通信协议、数据格式）之前，先建立自动驾驶基础架构的全景认知。这一节回答一个问题：支撑自动驾驶系统从研发到量产运营的技术底座长什么样？

自动驾驶基础架构不是单一系统，而是车端、云端、网络与边缘三个层面的协同体系。核心逻辑是**数据飞轮**——数据从车端采集，到云端处理、标注、训练，新模型 OTA 推回车端，车端用新模型跑出更好的对比数据，又回传云端。整个基础设施的优化目标就是让飞轮转得更快。

## 1. 车端基础架构（Vehicle Edge）

车端是自动驾驶最前端，硬件和软件紧耦合在一起。

### 计算平台

自动驾驶算法（感知、规划、控制）需要极强的算力，不能跑在普通 CPU 上。量产车通常用 SoC（System on Chip）方案——NVIDIA Orin（254 TOPS）、特斯拉自研 FSD 芯片（144 TOPS）、地平线征程 5（128 TOPS）。SoC 内部集成 CPU（ARM 架构，跑规划和控制逻辑）+ GPU/NPU/AI 加速器（跑感知模型推理）+ ISP（图像信号处理）+ 视频编码器。算力不是唯一的，内存带宽同样关键——模型推理需要大量数据搬运，HBM（高带宽内存）是瓶颈。

车规级要求——AEC-Q100 芯片认证（-40°C 到 105°C 工作温度、15 年寿命）、ISO 26262 功能安全等级 ASIL-D、电磁兼容性。这些约束让车端算力比同代云端 GPU 落后 2-3 年。

### 传感器系统

多传感器融合是标配——Camera（6-12 个，覆盖 360°，800 万像素）、LiDAR（1-3 个，128 线，300 米探测距离）、Radar（5 个，前向长距+四角短距）、超声波雷达（12 个，泊车用）、IMU/GPS/RTK（定位）。传感器的数据通过车载以太网（10/100 Gbps）或 LVDS/FPD-Link 连接到计算平台。时间同步——所有传感器数据必须打同一个时间戳，用 PTP（IEEE 1588）协议做亚微秒级硬件级时钟同步（参见 9.4 第 2 节仿真时序同步）。

### 车载通信中间件

传感器数据、算法输出、控制指令在车端各模块间流转需要通信中间件。ROS（机器人社区标准）或 CyberRT（百度 Apollo 自研）是主流选择。核心设计目标不是高吞吐而是低延迟+确定性——感知到控制的端到端延迟要求 < 100ms，99.99 分位也必须满足，否则安全无法保证。CyberRT 用共享内存零拷贝、协程优先级调度、CPU 绑核来保证这一点（详见 9.4 第 3 节）。

### 车载操作系统

不是 Android 不是 Linux 桌面，而是实时操作系统（RTOS）或实时 Linux。QNX（BlackBerry 的 RTOS，汽车行业市场份额最高，微内核架构 ASIL-D 认证）、VxWorks（风河，航空航天出身）、RT-Linux（Linux + PREEMPT_RT 实时补丁，开源）。操作系统要保证——中断响应延迟 < 100 微秒、高优先级任务能抢占低优先级、内存隔离防止一个模块崩溃影响其他模块。

有些方案用 Hypervisor 做虚拟化——一个 SoC 上跑两个 OS，ASIL-D 级的 RTOS 跑安全关键控制逻辑，ASIL-B 级的 Linux 跑信息娱乐和感知，两者隔离。这跟第三章 A8 Linux 操作系统基础中的 namespace + cgroup 隔离思路类似，但车端用的是更底层的硬件级虚拟化。

### 数据采集与存储

测试车需要把行驶数据录制下来上传到云端。车端用 SSD（1-4TB NVMe）做本地缓存，用 Rosbag 或 cyber_record 格式录制（详见 9.4 第 4 节数据格式）。采集策略——全量录制（开发期，每天 TB 级）或触发式录制（量产期，只在检测到 Corner Case 时录制几十秒）。车端到云端的回传用 4G/5G，但带宽不够（4G 上行 50Mbps，一天 TB 级数据传不完），所以高价值数据才回传，全量数据靠回场后用千兆以太网有线回传。

## 2. 云端基础架构（Cloud Platform）

云端是自动驾驶的大脑和数据仓库，承担数据存储、模型训练、仿真测试、OTA 分发四大职能。

### 数据存储层

对象存储（S3/MinIO）存原始大文件——Rosbag、点云、图像、模型权重，PB 级容量，纠删码冗余（参见 6.2 分布式存储两种范式）。结构化数据用 MySQL/PostgreSQL 存（车辆信息、任务元数据、评测报告），时序数据用 InfluxDB/TDengine 存（传感器遥测、系统监控指标），标注数据用图数据库或文档数据库存（标注框、类别、属性）。文件检索——不直接查文件内容，而是查元数据索引（车辆 ID + 时间范围 + 场景类型）定位文件 URI，再从对象存储拉取。

### 计算集群层

训练集群——大规模 GPU 集群（数百到数千张 A100/H100），用 K8s + NVIDIA Device Plugin 管理，分布式训练用 NCCL 做 GPU 间通信（InfiniBand 或 NVLink）。仿真集群——GPU 服务器做闭环仿真渲染，CPU 服务器做数据回放推理，通过 K8s Job 调度，按任务类型分资源池（参见 9.4 第 2 节仿真技术架构）。推理集群——Auto-Labeling 和数据处理用的 GPU 推理服务，用 Triton Inference Server 或自研推理框架，支持模型热更新。

### 数据闭环管线

这是云端的核心——采集 → 清洗 → 标注 → 训练 → 仿真 → 评测 → 部署的完整闭环。由 9 个微服务协同完成（详见 9.4 第 6 节微服务架构）。数据闭环的关键指标是迭代速度——从发现一个 Bad Case 到新模型部署到测试车的时间，业界领先公司能做到 1-2 周，落后公司要 1-2 个月。

### 仿真平台

仿真平台是云端区别于普通大数据平台的核心特征。包含场景库（OpenSCENARIO/OpenDRIVE 定义）、仿真器（CARLA/自研）、调度器（K8s Job 批量调度 GPU）、Auto-Grading（自动评分）。大规模仿真——Waymo 每天仿真数百万英里，特斯拉每天跑数千万帧影子模式数据回放（详见 9.4 第 2 节仿真体系）。

### OTA 分发

新模型训练完成后需要推送到车队。OTA（Over-The-Air）系统负责——模型打包（Docker 镜像 + 模型权重 + 配置）、版本管理（灰度发布——先推 1% 车队验证，再逐步全量）、断点续传（车端网络不稳定）、回滚机制（新版本有问题自动回退到上一版本）。OTA 的安全性要求极高——如果模型被篡改或传输损坏，直接影响行车安全，所以需要数字签名 + 完整性校验。

## 3. 网络与边缘基础设施

车端和云端之间不是直连的，中间有边缘层和通信网络。

### V2X（Vehicle to Everything）

车-车（V2V）、车-路（V2I）、车-云（V2N）通信。V2V 让车知道视野外的情况（前车的前方堵车），V2I 让车接收红绿灯相位和路侧传感器数据。通信标准——DSRC（IEEE 802.11p，专用短程通信）或 C-V2X（基于 5G 的蜂窝车联网）。V2X 的延迟要求比普通通信高——安全关键消息的端到端延迟要求 < 20ms。

### 边缘计算

不所有数据都传到云端——部分处理在边缘节点完成。路侧边缘计算单元（MEC，Mobile Edge Computing）部署在路口或基站，处理路侧摄像头和 LiDAR 数据，把结果（目标位置、红绿灯状态）低延迟传给车。区域数据中心做数据预处理——测试车回到车场后，通过有线网络上传到区域 DC，区域 DC 做数据清洗和压缩后传到中心云。边缘计算减少云端带宽压力和回传延迟。

### 影子模式

特斯拉开创的模式——量产车在后台跑自动驾驶算法（不影响人类驾驶），把算法输出和人类驾驶员的实际操作做对比，发现差异场景（算法和人决策不一致）就自动录制上传。这是一种大规模被动数据采集——数百万辆车每天贡献数十亿帧对比数据，比测试车队的数据量高几个数量级。但只适用于已量产的公司。

## 4. 研发基础设施

除了车端和云端的生产系统，自动驾驶研发还需要一套 DevOps 基础设施。

### CI/CD 流水线

代码提交 → 编译 → 单元测试 → 集成测试 → 模型训练 → 仿真回归 → 构建镜像 → 部署到测试车。整个流水线自动化，每次代码提交都触发一轮回归仿真——用几百个核心场景验证新代码有没有引入回归 Bug。CI/CD 的速度决定了团队迭代效率，大厂能做到"提交到部署测试车 < 2 小时"。这跟第三章 A7 开发工具链中的 CI/CD 概念一致，但自动驾驶的 CI/CD 多了"模型训练"和"仿真回归"两个独有环节。

### 数据版本管理

数据集像代码一样有版本——训练集 V1、V2、V3，标注格式 v1.0、v2.0。用 DVC（Data Version Control）或自研工具管理大文件版本，用 Git 管理 metadata（哪个版本包含哪些文件、标注历史）。模型版本管理——用 MLflow 或自研平台记录每次训练的 commit、数据集版本、超参数、训练指标、仿真评分。这跟 9.4 第 5 节数据血缘直接关联——版本管理是血缘追踪的前提。

### 监控与可观测性

车端运行时监控——CPU/GPU 利用率、内存、温度、算法延迟（感知推理耗时、规划计算耗时）、传感器健康状态（Camera 是否黑屏、LiDAR 是否点云稀疏）。这些数据通过 4G/5G 回传到云端，用 Prometheus + Grafana 做可视化，异常自动告警。云端服务监控——第三章 3.14 中可观测性三支柱（Metrics + Logs + Traces）直接适用。

### 安全与合规基础设施

数据合规——各国法规对自动驾驶数据有不同要求（中国《个人信息保护法》要求人脸和车牌脱敏，欧盟 GDPR 要求用户数据可删除）。数据安全——传输加密（TLS）、存储加密（KMS 管理密钥）、访问控制（RBAC + 审计日志）。功能安全——ISO 26262 流程贯穿研发全生命周期，从需求到测试到运维都有安全要求。

## 5. 架构选型的核心逻辑

自动驾驶基础架构的选型有一个核心逻辑贯穿始终——**安全关键系统要求确定性，而非单纯的高性能。**

这解释了几个看似奇怪的选择：为什么车端用 RTOS 而不是普通 Linux（确定性调度，保证控制循环不抖动）；为什么用 CyberRT 而不是 Kafka（延迟 < 10ms 且抖动极小，Kafka 做不到）；为什么仿真要保真度和速度两级（低保真跑量覆盖，高保真精跑验证安全关键场景）；为什么 OTA 要灰度发布（新模型可能有问题，必须渐进验证）。

从架构层面看，自动驾驶基础架构本质上是**一个数据飞轮**——数据从车端采集，到云端处理和标注，训练出更好的模型，OTA 推回车端，车端用新模型跑出更好的影子模式对比数据，又回传云端。整个基础设施的优化目标就是让这个飞轮转得更快——数据采集更快、标注更快、训练更快、仿真更快、部署更快。飞轮转速快的公司迭代速度快，模型迭代次数多，最终算法质量高。
