# 9.3 通用知识缺口：K8s 调度、对象存储、可观测性、工作流引擎

本节梳理 JD 暴露的通用技术知识缺口，分析每个知识点在自动驾驶数据平台中的作用，并标注应该补充到本书哪个章节。

## 1. Kubernetes 与容器编排

### 为什么需要

JD 要求"精通 K8s/Docker 生态，有 Operator 开发或 K8s 调度器深度定制经验"。自动驾驶仿真平台的核心是"在云端批量调度数千个 Docker 容器，每个容器跑一个仿真任务"，这超出了传统微服务部署（Deployment + Service）的范畴，需要 Job 批量调度、自定义调度器、Operator 自动化运维。

### 核心知识点

**K8s 调度框架（Scheduler Framework）：** 默认调度器通过 Filter（过滤不满足条件的节点）→ Score（对剩余节点打分）→ Bind（绑定 Pod 到节点）三阶段完成调度。自动驾驶场景需要自定义调度策略——比如 GPU 亲和性调度（优先把需要 GPU 的仿真任务分配到有 GPU 的节点）、数据本地性调度（把任务调度到数据所在节点，减少大文件传输）、拓扑约束（感知和规划算法不能跑在同一节点上，避免资源竞争）。

**K8s Job 批量调度：** 仿真任务是短生命周期批处理任务，用 Job 而非 Deployment。关键参数：`completions`（完成多少个 Pod 算成功）、`parallelism`（最大并发数）、`backoffLimit`（失败重试次数）。自动驾驶场景的实践：设置 `parallelism` 控制并发仿真数（受 GPU 资源限制），设置 `backoffLimit` 处理偶发失败（仿真器崩溃），用 `ttlSecondsAfterFinished` 自动清理已完成 Job 的 Pod。

**Operator 模式（CRD + Controller）：** Operator 是用代码封装运维知识的模式——定义自定义资源（CRD，如 `SimulationTask`），编写 Controller 监听 CRD 变化，自动执行创建 Pod、管理镜像、收集结果等操作。自动驾驶平台用 Operator 实现"提交一个仿真请求 → 自动拉起 Docker 容器 → 注入场景数据 → 执行仿真 → 收集评分结果 → 销毁容器"的全流程自动化。Operator 的核心机制是 Informer（监听 API Server 变更）+ Reconcile Loop（将实际状态向期望状态收敛）。

**HPA/VPA 弹性伸缩：** HPA（Horizontal Pod Autoscaler）根据 CPU/GPU 利用率或自定义指标水平扩缩容 Pod 数量；VPA（Vertical Pod Autoscaler）调整 Pod 的 CPU/内存 requests/limits。自动驾驶数据平台中，标注服务用 HPA（标注任务多时自动扩容）、数据预处理用 VPA（大文件处理需要更多内存）。

### 补充位置

补充到第三章，作为 3.14 微服务与分布式锁的延伸——微服务部署依赖 K8s，K8s 是微服务架构的运行时基础设施。新增附录 A10 或直接扩展 3.14。

## 2. 对象存储 S3/MinIO

### 为什么需要

JD 要求"有对象存储（S3/MinIO）的大文件处理经验"。自动驾驶的传感器数据（LiDAR 点云、摄像头图像序列、Rosbag 录制）是海量大文件，不适合存 MySQL 或 HBase，适合对象存储——扁平命名空间、无限容量、高吞吐读写。

### 核心知识点

**S3 API 协议：** S3 已成为对象存储的事实标准 API。核心操作：PutObject（上传）、GetObject（下载）、Multipart Upload（分片上传）、ListObjects（列桶内对象）。Multipart Upload 是处理大文件的关键——把 10GB 的 Rosbag 分成 100 个 100MB 的分片并行上传，支持断点续传（某片失败只需重传该片），上传完成后 SendComplete 合并。

**MinIO 架构：** MinIO 是开源的 S3 兼容对象存储，用 Go 实现。核心特性：纠删码（Erasure Coding）存储——数据分成 N 个数据块 + M 个校验块，任一节点故障只要剩余数据块 + 校验块 ≥ N 就能恢复，比三副本（HDFS）节省约 50% 存储空间。分布式模式：每个节点既存数据又做网关（无中心元数据服务器，去中心化架构）。

**与 HDFS 的对比：** HDFS 是文件系统语义（目录树 + 文件路径），适合 Hadoop/Spark 生态的大数据分析；S3/MinIO 是对象存储语义（bucket + key，扁平命名空间），适合大文件的存储和检索。自动驾驶场景两者都用——S3/MinIO 存原始传感器数据（Rosbag/PCD），HDFS 存处理后的结构化数据（标注结果、特征向量、训练数据集）。

### 补充位置

补充到第六章，作为 6.2 HDFS 的对照——分布式存储的两种范式：文件系统（HDFS）vs 对象存储（S3/MinIO）。

## 3. HBase 与 LSM Tree

### 为什么需要

JD 要求"熟练掌握 NoSQL（MongoDB/HBase）"。自动驾驶场景下，HBase 适合存时序传感器元数据——每条记录是"车辆ID + 时间戳 + 传感器类型 + 数据文件路径"，按 RowKey 范围扫描查询某辆车某段时间的所有数据。HBase 的 LSM Tree 存储引擎天然适合海量写入（顺序写 + 异步合并），比 B+ 树的随机写性能高一个数量级。

### 核心知识点

**LSM Tree（Log-Structured Merge-Tree）：** 写入流程是顺序写 WAL（Write-Ahead Log）→ 写 MemTable（内存中的有序结构）→ MemTable 满了刷盘成 SSTable（Sorted String Table，磁盘上的有序文件）。读取时需要合并查询 MemTable + 多层 SSTable（用[布隆过滤器](../part3-java-deep/A1-核心数据结构原理.md#三布隆过滤器bloom-filter用-1-的误判换-99-的内存节省)加速）。后台 Compaction 定期合并多层 SSTable，清理已删除数据。核心 trade-off：牺牲读性能（需要合并多层数据）换取写性能（纯顺序写）。

**HBase RowKey 设计：** RowKey 是 HBase 唯一的索引，设计好坏决定查询性能。自动驾驶场景的 RowKey 设计：`vehicle_id + reverse_timestamp + sensor_type`——按车辆分区，同一车辆的数据按时间倒序排列（最新数据在最前面），通过 RowKey 前缀扫描快速查询某辆车的数据。避免热点问题：如果只用时间戳做 RowKey，所有写入都打到同一个 RegionServer（因为时间戳递增），需要加随机前缀打散。

### 补充位置

补充到第六章，作为 6.2 HDFS 或 6.3 Hive 的对照——不同存储引擎适用于不同场景：HDFS（大文件分析）、Hive（SQL-on-HDFS）、HBase（海量时序 KV）、S3/MinIO（大文件对象存储）。

## 4. Pulsar 消息队列

### 为什么需要

JD 要求"熟悉 Kafka/Pulsar 消息队列"。Kafka 已在第三章 3.12 覆盖，Pulsar 需要补充。Pulsar 在自动驾驶场景中的优势是存储计算分离——Broker 只负责消息转发，存储由 BookKeeper 负责，可以独立扩展计算和存储。对于仿真任务的海量事件流（任务提交、状态变更、结果通知），Pulsar 的分层架构比 Kafka 更灵活。

### 核心知识点

**Pulsar vs Kafka 架构差异：** Kafka 的 Broker 既负责消息路由又负责存储（Partition 物理在 Broker 上），扩容需要数据迁移（Rebalance Partition）。Pulsar 的分层架构——Broker 只负责消息路由和协议处理，存储层 BookKeeper 负责持久化，计算和存储可以独立扩缩容。Pulsar 支持 Topic 级别的扩容（Partition 可以跨 Broker 分散），而 Kafka 是 Partition 级别。

**Pulsar 的订阅模式：** Exclusive（独占，同一时刻只有一个 Consumer 消费）、Shared（共享，多个 Consumer 轮询消费）、Failover（故障转移，主备模式）。Kafka 只有 Consumer Group 内的 Partition 分配，没有这么灵活的订阅模式。

### 补充位置

补充到第三章 3.12 消息队列，作为 Kafka 的对比——消息队列的两种架构：计算存储耦合（Kafka）vs 计算存储分离（Pulsar）。

## 5. 可观测性三支柱

### 为什么需要

JD 要求"建设平台的可观测性体系（Monitoring/Logging/Tracing）"。第三章 3.14 有链路追踪简介，但只是一个维度。自动驾驶仿真平台有数千个并发任务，每个任务跨多个服务（调度器 → 容器管理 → 仿真器 → 评分器 → 结果存储），需要三支柱协同才能定位问题。

### 核心知识点

**Metrics（指标监控）：** 用 Prometheus + Grafana。核心指标分四类——Traffic（请求量/QPS/任务吞吐量）、Latency（P50/P99 延迟/任务执行时长）、Errors（错误率/仿真失败率/任务重试率）、Saturation（CPU/GPU/内存利用率/队列积压深度）。Prometheus 的 Pull 模型（主动抓取 Exporter 暴露的指标），适合容器环境（每个 Pod 暴露 /metrics 端点）。

**Logging（日志聚合）：** 用 ELK（Elasticsearch + Logstash + Kibana）或 Loki。结构化日志（JSON 格式，包含 trace_id、task_id、service_name），通过 trace_id 关联到 Tracing，通过 task_id 关联到具体仿真任务。核心实践：日志分级（DEBUG/INFO/WARN/ERROR）、采样策略（高流量服务采样 10% DEBUG 日志）、日志聚合查询（按 task_id 聚合一个任务跨所有服务的日志）。

**Tracing（分布式追踪）：** 用 Jaeger 或 OpenTelemetry。一个仿真任务从提交到完成的完整调用链——提交请求 → 调度器分配资源 → 拉起 Docker 容器 → 仿真器加载数据 → 算法执行 → 评分器计算结果 → 存储结果。每个 Span 记录服务名、操作名、开始时间、持续时间、tags（task_id、vehicle_id、scenario_id）。核心价值：定位"任务执行慢"到底卡在哪个环节。

## 6. 工作流引擎设计模式

### 为什么需要

JD 要求"设计灵活的工作流引擎（Workflow），串联预处理-标注-质检-训练的自动化生产管线"。这是 DAG 任务调度的工程实现，跟我们第八章 Notebook 的 marimo DirectedGraph 有概念相通之处——节点是任务，边是依赖关系，拓扑排序决定执行顺序。但 Notebook 的 DAG 是进程内 Cell 级别的，这里是分布式任务级别的。

### 核心知识点

**DAG 任务编排：** 任务定义为节点，依赖关系为边。拓扑排序保证被依赖的任务先执行。循环检测防止死循环。失败重试——某节点失败后，根据策略重试该节点或整个子图。断点续跑——记录每个节点的执行状态，恢复时从最后未完成的节点继续。

**状态机设计：** 每个任务节点的状态：Pending（等待依赖完成）→ Ready（依赖已满足，等待资源）→ Running（正在执行）→ Success（执行成功）→ Failed（执行失败）→ Cancelled（被取消）。状态转移需要持久化（存 MySQL/PostgreSQL），保证服务重启后能恢复。

**并发控制：** 资源池限制（同时执行的任务数不超过 GPU 数量）、优先级队列（紧急任务插队）、反压机制（队列积压时降低提交速率）。参考 Airflow 的 Pool 概念或 K8s 的 ResourceQuota。

**业界方案对比：** Airflow（Python DAG 定义，重编排轻运行，适合批处理）、Argo Workflows（K8s 原生，DAG 定义在 YAML，每个任务是 K8s Pod，适合容器化场景）、Tekton（K8s 原生 CI/CD Pipeline，Step 概念更细粒度）、Dagster（数据资产为中心，DAG 节点关联数据资产，适合数据管线）。自动驾驶场景最合适的是 Argo Workflows——K8s 原生、YAML 声明式、每个任务是独立 Pod、支持 DAG 和 Steps 两种编排模式。
