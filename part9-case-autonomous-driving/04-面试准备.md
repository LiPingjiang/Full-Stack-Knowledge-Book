# 9.4 面试准备路径与高频问题

## 准备优先级

根据 JD 权重和面试考察概率，按以下优先级准备：

**第一优先级（必考，占面试 40%）：分布式系统 + K8s 调度**

这是岗位的技术核心。你需要能完整回答"如何设计一个支持数千并发仿真任务的调度系统"。准备路径：先复习第三章 3.10（CAP/BASE/Raft）、3.14（分布式锁/链路追踪/服务发现）、3.7（限流/熔断/降级/幂等），然后补充 K8s 调度框架、Job 批量调度、Operator 模式、HPA/VPA。

**第二优先级（高概率，占面试 25%）：存储 + 消息队列**

面试官会问"海量传感器数据如何存储和检索"。准备路径：复习第三章 3.12（Kafka 可靠投递/幂等消费）、第三章 3.9（MySQL 索引/事务/分库分表）、第六章 6.2（HDFS），然后补充对象存储 S3/MinIO、HBase/LSM Tree、Pulsar。

**第三优先级（中概率，占面试 20%）：工作流引擎 + 系统设计**

面试官会要求你白板设计标注管线。准备路径：理解 DAG 任务编排原理（第八章 8.4 marimo 的 DirectedGraph 提供了概念基础），研究 Argo Workflows 的 DAG 模型，准备状态机设计和并发控制方案。

**第四优先级（领域知识，占面试 15%）：自动驾驶概念**

不是所有面试官都深入追问领域知识，但会考察你"是否理解业务"。准备路径：理解数据闭环概念、ROS/CyberRT 通信模型、Rosbag/PCD 格式、仿真调度流程、Auto-Grading 评分体系。重点准备"如何把业务概念转化为技术架构"。

## 高频面试问题与参考回答

### 问题 1：如何设计支持数千并发仿真任务的调度系统？

**考察点：** K8s 调度、分布式系统设计、资源隔离

**参考回答框架：**

整体架构分四层。接入层用 REST API + Kafka 接收仿真请求，请求入 Kafka 队列缓冲，避免突发流量压垮调度器。调度层从 Kafka 消费请求，创建 K8s Job，每个 Job 是一个独立 Pod 运行仿真器 + 算法 Docker。调度策略用自定义 Scheduler——Filter 阶段过滤掉没有 GPU 的节点，Score 阶段优先选择数据本地性强的节点（场景数据在哪台机器上就优先调度到那台，减少大文件传输）。资源隔离用 K8s ResourceQuota 做团队配额限制，用 PriorityClass 做任务优先级（紧急 Bug 验证 > 回归测试 > 批量验证）。运维层用 Operator 管理任务全生命周期——监听 SimulationTask CRD，自动创建 Job、注入场景数据、收集评分结果、清理容器。

并发控制用 Job 的 parallelism 参数限制最大并发数（等于集群 GPU 总数），用 K8s 的 limit ranges 限制每个 Pod 的 CPU/内存。失败重试用 backoffLimit 处理偶发失败，超过重试次数标记任务失败并通知。镜像分发用 Dragonfly P2P 加速，分层缓存减少拉取量。

关键设计决策：为什么用 K8s Job 而不是自己实现任务调度器——因为 K8s 已经处理了节点故障、Pod 重启、资源调度、健康检查等基础设施问题，自己实现成本高且容易出 bug。自定义 Scheduler 只做策略增强，不做底层调度。

### 问题 2：海量 Rosbag 文件如何存储和高效检索？

**考察点：** 对象存储、HBase、存储架构选型

**参考回答框架：**

双存储架构。原始文件存 S3/MinIO 对象存储——Rosbag 文件 10-100GB 大文件，对象存储天然适合大文件存储，Multipart Upload 分片并行上传支持断点续传，纠删码比三副本节省 50% 空间。元数据存 HBase——每条记录的 RowKey 是 `vehicle_id + reverse_timestamp + sensor_type`，包含 Rosbag 在 S3 的 URI、时间范围、传感器配置、采集地点、天气条件。查询"车辆 A 在某时段的 LiDAR 数据"时，先通过 HBase RowKey 前缀扫描拿到文件 URI 列表，再从 S3 下载对应 Rosbag，用 rosbag 库解析提取 LiDAR 帧。

为什么不用 HDFS——HDFS 适合 Hadoop/Spark 生态的批处理分析（Parquet 列存、SQL 查询），但 Rosbag 是自定义二进制格式，不能用 SQL 查询内部内容，只能按文件级别存储和检索。HDFS 的小文件问题和 NameNode 内存瓶颈在大规模 Rosbag 场景更严重。HDFS 适合存处理后的结构化数据（标注结果、特征向量）。

为什么 HBase 而不是 MySQL——元数据量级是数十亿条（数百辆车 × 每天数千个 Rosbag × 多种传感器），写入吞吐量要求高（每秒数千条元数据记录），HBase 的 LSM Tree 顺序写性能远高于 MySQL 的 B+ Tree 随机写。HBase 的 RowKey 前缀扫描天然适合"按车辆 + 时间范围"查询。

### 问题 3：工作流引擎如何设计？如何串联标注管线？

**考察点：** DAG 编排、状态机、系统设计

**参考回答框架：**

工作流定义用 YAML 声明式 DAG——每个节点是一个任务（预处理、Auto-Labeling、人工标注、质检、训练），边表示依赖关系。用户定义 DAG 拓扑，引擎负责拓扑排序和调度。

执行模型参考 Argo Workflows——每个任务节点对应一个 K8s Pod，用容器镜像隔离运行环境。预处理任务用 Spark 镜像（处理大规模数据），Auto-Labeling 任务用模型推理镜像（GPU Pod），人工标注任务不启动 Pod（标注员在 Web 平台操作，任务状态变更为"等待标注"，超时自动降级）。

状态机设计：Pending（等待依赖完成）→ Ready（依赖满足，等待资源分配）→ Running（Pod 已启动执行中）→ Success/Failed（执行结果）→ Cancelled（被取消）。状态持久化到 PostgreSQL，服务重启后从未完成节点恢复。失败重试策略——可配置重试次数和超时时间，超过重试次数标记 Failed 并触发告警。

并发控制用资源池——GPU 资源池限制 Auto-Labeling 并发数，人工标注并发数不受 GPU 限制但受标注员数量限制（从标注员管理系统获取在线人数）。优先级队列支持紧急任务插队。

断点续跑——记录每个节点的 checkpoint（输入数据 URI、输出数据 URI、执行状态），恢复时从最后成功的节点继续。数据资产跟踪——每个节点的输出注册到数据血缘系统，记录"这个标注结果来自哪个 Rosbag、经过哪些处理步骤"。

### 问题 4：仿真器和算法之间的通信中间件如何设计？

**考察点：** 通信协议、时序同步、实时性

**参考回答框架：**

核心挑战是两个——时序同步和低延迟。仿真器按帧产生虚拟传感器数据（LiDAR 10Hz、Camera 30Hz、Radar 20Hz），算法需要按时间戳对齐多传感器数据后才能处理。

通信方式选择：同容器内用共享内存（mmap 零拷贝，延迟 < 1ms，适合仿真器和算法在同一 Pod 的情况）；跨容器用 ZeroMQ（参考第三章 3.12，延迟 < 5ms，适合仿真器和算法分开部署的情况）。不用 Kafka——Kafka 的设计目标是高吞吐而非低延迟，消息经过磁盘持久化和 ISR 复制后延迟 > 10ms，不满足实时仿真要求。

时序同步方案——仿真器为每帧数据打时间戳（仿真世界的时间，不是系统墙钟时间），算法侧用缓冲区按时间戳对齐多传感器数据：维护三个队列（LiDAR/Camera/Radar），当三个队列都有同一时间戳附近的数据时触发处理。对齐窗口设为 50ms（LiDAR 10Hz 一帧 100ms，Camera 30Hz 一帧 33ms，窗口需要覆盖最慢的传感器一帧时间）。

如果面试官追问 ROS/CyberRT——CyberRT 用共享内存 + Protobuf + DPDK 实现了 < 10ms 的稳定延迟和协程优先级调度，是自动驾驶专用通信中间件。但仿真场景中我们不需要那么极致的实时性（不是真车在跑），用 ZeroMQ 或共享内存就足够了。

### 问题 5：如何建设可观测性体系？

**考察点：** Monitoring/Logging/Tracing 实践

**参考回答框架：**

三支柱协同。Metrics 用 Prometheus + Grafana——核心指标分四类：Traffic（任务提交 QPS、队列积压深度）、Latency（任务等待时间 P50/P99、任务执行时长 P50/P99）、Errors（仿真失败率、任务重试率、Pod 异常率）、Saturation（GPU 利用率、CPU 利用率、内存利用率）。Prometheus 用 Pull 模型抓取每个 Pod 的 /metrics 端点，Grafana 做可视化看板和告警规则。

Logging 用 Loki（轻量级，比 ELK 省资源）或 ELK。结构化 JSON 日志，关键字段 trace_id（关联调用链）、task_id（关联仿真任务）、service_name（服务名）。日志采集用 Fluent Bit（DaemonSet 方式在每个节点收集容器日志）。查询场景："查 task_id=T123 跨所有服务的日志"——通过 task_id 聚合一个仿真任务从提交到完成的所有日志。

Tracing 用 Jaeger。一个仿真任务的完整调用链：API Gateway → 调度器 → K8s API Server → Pod 创建 → 仿真器启动 → 数据加载 → 仿真执行 → 评分计算 → 结果存储。每个 Span 记录服务名、操作名、起止时间、tags（task_id、vehicle_id、scenario_id、gpu_id）。定位"任务执行慢"时，看 Tracing 找到耗时最长的 Span，再查该 Span 对应服务的 Metrics 和 Logs。

告警策略——GPU 利用率 < 50% 持续 10 分钟告警（可能调度异常）、任务失败率 > 5% 告警（可能镜像问题或数据问题）、队列积压 > 1000 告警（扩容信号）。
