# 9.3 领域知识：自动驾驶数据闭环与仿真体系

本节分析 JD 中的自动驾驶领域专属知识——这些不是通用技术，而是行业壁垒。理解这些概念能让你在面试中展现"懂业务"而不是"只会写代码"。

## 1. 数据闭环（Data Loop）

### 概念

自动驾驶的数据闭环是一个迭代式数据飞轮：采集真实道路数据 → 清洗和标注 → 用标注数据训练模型 → 部署模型到测试车 → 测试中发现 Bad Case（模型表现差的场景）→ 定向采集更多类似场景数据 → 标注和训练 → 模型迭代。每一轮迭代都让模型更强，数据资产不断积累。

### 技术映射

数据闭环的每个环节都对应一个平台模块：

采集环节——测试车产生的原始传感器数据（Rosbag 格式）上传到对象存储（S3/MinIO），元数据（车辆ID、时间、地点、传感器配置）存入 HBase。技术挑战是上传带宽（一辆车每天 TB 级数据）和数据去重（相同场景不重复采集）。

清洗环节——自动过滤无效数据（传感器故障、数据损坏）、去重（基于场景相似度，用 SimHash 或 MinHash，参见第三章 A1 布隆过滤器和 MinHash）、脱敏（人脸/车牌模糊处理）。工作流引擎编排清洗管线。

标注环节——Auto-Labeling（模型自动生成预标注）+ 人工标注修正。Auto-Labeling 是模型推理服务化（第七章 7.3 模型部署基础），用当前最好的模型推理生成预标注，人工标注员在预标注基础上修正，效率比纯人工高 5-10 倍。

训练环节——标注数据按训练集/验证集/测试集划分，送入模型训练流水线。训练完成后自动评测模型质量，合格的模型进入仿真验证。

## 2. 仿真体系（Simulation）

### 概念

虚拟里程测试是用仿真环境替代物理道路测试。核心价值：安全性（不用真车冒险测试危险场景）、效率（数百个仿真任务并行跑，一天相当于物理测试数月）、可复现（相同场景可重复验证）、覆盖度（可以生成现实中罕见的 Corner Case）。

### 技术架构

仿真平台的技术架构分四层：

场景层——定义测试场景（地图、天气、交通流、障碍物位置）。场景可以是手工编写的（OpenSCENARIO 格式）或自动生成的（从真实数据中提取 Corner Case 并泛化放大）。场景库管理是数据平台的一部分——按场景类型索引、按难度分级、按覆盖度排序。

调度层——K8s Job 批量调度。每个仿真任务是一个 Docker 容器，包含仿真器（CARLA/LGSVL/自研）+ 算法镜像（Apollo/自研感知规划算法）。调度器需要考虑 GPU 亲和性（仿真渲染需要 GPU）、数据本地性（场景数据在哪个节点）、并发限制（GPU 数量限制并发数）。

执行层——仿真器和算法在容器内通过通信中间件（ROS Bridge 或 CyberRT）交换数据。仿真器按帧产生虚拟传感器数据（Camera 图像、LiDAR 点云、Radar 数据），算法接收数据并输出控制指令（方向盘角度、油门、刹车），仿真器执行指令更新虚拟世界状态，循环往复。

评测层——Auto-Grading 自动评分。核心指标：是否碰撞（Collision Rate）、是否偏离车道（Lane Departure）、是否闯红灯（Red Light Violation）、舒适度指标（急刹/急转弯次数）、交通规则遵守率。评分结果存入数据库，支持按场景、算法版本、指标维度聚合查询。

### 关键技术挑战

并发调度——数百到数千个仿真任务并发，每个任务独占一个 GPU。调度器需要支持优先级（紧急 Bug 验证任务插队）、公平调度（不同团队配额隔离）、抢占式调度（低优先级任务被高优先级抢占时需要安全退出并释放 GPU）。

镜像管理——算法镜像可能 10-20GB（包含模型权重和依赖库），每次算法更新产生新镜像。镜像分发到数百个节点是瓶颈，需要 P2P 分发（Dragonfly/Kraken）或镜像分层缓存（只有变化的层需要拉取）。

结果收集——每个仿真任务产生 GB 级的结果数据（日志、传感器录制、评分报告）。需要异步收集到对象存储，不能阻塞调度器主循环。

## 3. ROS / CyberRT 通信机制

### ROS（Robot Operating System）

ROS 不是操作系统，是机器人通信中间件。核心概念：

Node——独立进程，封装一个功能模块（如 LiDAR 驱动、感知算法、规划算法）。

Topic——发布订阅模式的通信通道。一个 Node 向 Topic 发布消息（如 LiDAR Node 向 `/lidar_points` Topic 发布点云数据），其他 Node 订阅该 Topic 接收数据。类似 Kafka 的 Topic 但运行在同一台机器的进程间。

Service——请求响应模式。一个 Node 提供服务（如 `/get_map`），其他 Node 发送请求获取响应。类似 RPC。

Action——长耗时异步请求，有持续反馈。比如"导航到目标点"可能需要数秒，期间持续反馈当前位置和进度。

### CyberRT

CyberRT 是百度 Apollo 对 ROS 的改进重写。核心优化：

实时性——ROS 基于 Linux 进程和 TCP/UDP 通信，延迟不稳定（可能 > 100ms）。CyberRT 用共享内存通信（同一台机器的 Node 间通过 mmap 共享数据，零拷贝）和 DPDK 网络栈（用户态网络，绕过内核协议栈），延迟稳定在 < 10ms。

确定性调度——ROS 没有调度优先级概念，所有 Node 平等竞争 CPU。CyberRT 支持协程优先级和 CPU 绑核，保证感知算法（安全关键）优先于日志记录（非关键）获得 CPU 时间。

数据格式——ROS 用自定义的 msg 格式，CyberRT 用 Protobuf，序列化/反序列化性能更好。

### 面试要点

面试官不会要求你精通 ROS/CyberRT 的 API 细节，但会考察你是否理解"自动驾驶系统为什么需要专用的通信中间件"——核心答案是实时性和确定性。通用的消息队列（Kafka/RabbitMQ）是为高吞吐设计的，不是为低延迟设计的。自动驾驶的安全关键场景要求控制循环延迟 < 10ms 且抖动极小（99.99 分位延迟也 < 10ms），这超出了通用中间件的能力。

## 4. Rosbag / PCD 数据格式

### Rosbag

Rosbag 是 ROS 的数据录制格式，本质是一个序列化的消息流文件。一辆测试车一个 Rosbag 文件包含一次行驶的所有传感器数据——按时间戳排列的 LiDAR 点云帧、摄像头图像帧、雷达数据帧、GPS/IMU 航迹、CAN 总线数据（车速、方向盘角度）。单个 Rosbag 文件通常 10-100GB。

解析处理——Rosbag 是自定义二进制格式（不是标准 JSON/Protobuf），需要用 rosbag 库（Python/C++）读取。处理流程：打开 Rosbag → 按时间戳遍历消息 → 按 Topic 分流（LiDAR 数据送点云处理管线，Camera 数据送图像处理管线）→ 按时间戳对齐多传感器数据。

### PCD（Point Cloud Data）

PCD 是点云数据格式（PCL 库定义）。LiDAR 每秒扫描产生数百万个 3D 点（x, y, z, intensity），单帧 PCD 文件约 5-200MB。点云处理是自动驾驶感知的基础——目标检测（识别车辆、行人、障碍物）、地面分割（区分可行驶区域和障碍物）、目标跟踪（跨帧追踪同一目标）。

### 面试要点

理解"非结构化数据"的含义——传统数据平台处理的是结构化数据（表、行、列），自动驾驶平台处理的是二进制大文件（Rosbag/PCD/图像）。这导致存储架构、IO 模式、处理方式都不同——对象存储而非数据库，流式读取而非 SQL 查询，GPU 加速处理而非 CPU 聚合。

## 5. 数据血缘（Data Lineage）

### 概念

JD 要求理解"数据血缘"。在自动驾驶数据平台中，数据血缘比传统数仓更复杂——不仅要追踪"哪个 ETL 任务读取哪个表写入哪个表"，还要追踪"这个标注结果来自哪段 Rosbag 数据的哪一帧、由哪个模型生成的预标注、由哪个标注员修正的、最终被哪个训练任务使用了"。

### 技术实现

数据血缘在自动驾驶场景的三层追踪：

存储层血缘——对象存储中的 Rosbag 文件 → 清洗后的 Rosbag 文件（去重/脱敏后）→ 提取出的点云帧序列 → 标注结果文件 → 训练数据集。每一层都记录上下游文件的 URI 和转换关系。

任务层血缘——采集任务 → 清洗任务 → 标注任务 → 训练任务。工作流引擎记录每个任务的输入输出。跟第八章 Notebook 09 中数据血缘追踪（AST 分析 + Spark Listener + 血缘图）的技术路线类似。

模型层血缘——模型 V2 用训练集 A（来自 Rosbag #1-#500）训练 → 模型 V2 的仿真评分结果 → 发现 Bad Case 在场景 #234 → 场景 #234 加入训练集 B → 模型 V3 用训练集 B 训练。这条链路决定了模型迭代的可追溯性——出了问题可以回溯到训练数据。
