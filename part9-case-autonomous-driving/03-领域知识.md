# 9.3 领域知识：自动驾驶数据闭环与仿真体系

本节分析 JD 中的自动驾驶领域专属知识——这些不是通用技术，而是行业壁垒。理解这些概念能让你在面试中展现"懂业务"而不是"只会写代码"。

## 1. 数据闭环（Data Loop）

### 概念

自动驾驶的数据闭环是一个迭代式数据飞轮：采集真实道路数据 → 清洗和标注 → 用标注数据训练模型 → 部署模型到测试车 → 测试中发现 Bad Case（模型表现差的场景）→ 定向采集更多类似场景数据 → 标注和训练 → 模型迭代。每一轮迭代都让模型更强，数据资产不断积累。

### 技术映射

数据闭环的每个环节都对应一个平台模块：

采集环节——测试车产生的原始传感器数据（Rosbag 格式）上传到对象存储（S3/MinIO），元数据（车辆ID、时间、地点、传感器配置）存入 HBase。技术挑战是上传带宽（一辆车每天 TB 级数据）和数据去重（相同场景不重复采集）。

清洗环节——自动过滤无效数据（传感器故障、数据损坏）、去重（基于场景相似度，用 SimHash 或 MinHash，参见第三章 A1 布隆过滤器和 MinHash）、脱敏（人脸/车牌模糊处理）。工作流引擎编排清洗管线。

标注环节——Auto-Labeling（模型自动生成预标注）+ 人工标注修正。Auto-Labeling 是模型推理服务化（第七章 7.3 模型部署基础），用当前最好的模型推理生成预标注，人工标注员在预标注基础上修正，效率比纯人工高 5-10 倍。

训练环节——标注数据按训练集/验证集/测试集划分，送入模型训练流水线。训练完成后自动评测模型质量，合格的模型进入仿真验证。

### 标注的类型

自动驾驶的标注对象涵盖点云和图像两大类，标注类型远不止"画框赋名"：

3D 标注框是最核心的类型——在 LiDAR 点云中画 3D 立方体框住目标物体，标注类别（轿车、卡车、行人、自行车、锥桶）和属性（车辆是静止还是运动、行人是站立还是行走）。这个框是三维的（长宽高 + 中心点坐标 + 偏航角 yaw），标注员在点云可视化工具中旋转视角，调整框的位置、大小和朝向，确保框紧贴物体轮廓。

2D 标注框是在 Camera 图像上画矩形框，标注同样的类别。还有遮挡标注（物体被前方车辆挡住一部分，需要标注遮挡比例）、截断标注（物体在图像边缘被截断）、群体标注（远处一堆行人密集到无法逐个标注，标注一个区域为"人群"）。

语义分割是对图像中每个像素或点云中每个点标注类别——整张图像的每个像素打标签（道路、天空、建筑物、车辆、行人、植被），或者整帧点云的每个点打标签。不是画框而是用画笔工具刷区域，工作量比画框大得多。

车道线标注是在图像或点云上标注车道分界线、道路边缘线、停止线，属性包括线类型（实线、虚线、双线）、颜色（白线、黄线）、车道编号。

多目标跟踪标注是把同一个物体在连续帧中关联起来——第 100 帧的车辆 A 和第 101 帧的车辆 A 是同一个目标，需要给同一个 track ID。一个 30 分钟的视频有 54000 帧（30FPS），每个目标都要跨帧追踪，工作量极大。

### 自动化标注的闭环问题

自动化标注形成的正反馈循环（模型 V5 预标注 → 人工修正 → 训练 V6 → V6 预标注更准 → 修正更少）在实践中需要考虑以下问题：

**预标注置信度阈值设定。** 模型对每个预测目标输出置信度（confidence score），平台需要设定阈值决定哪些预标注直接采纳、哪些需要人工审核。阈值太高（如 0.9）则大部分框都需要人工审核，自动化效果不明显；阈值太低（如 0.3）则低质量预标注混入标注数据，训练出的模型可能学到错误标注。实际工程中通常分三档——高置信度（> 0.9）自动采纳，中置信度（0.5-0.9）人工抽检，低置信度（< 0.5）丢弃或强制人工标注。阈值不是固定的，随着模型版本迭代需要动态调整。

**标注一致性。** 不同标注员对同一个目标的标注可能不一致——A 把远处的物体标为"行人"，B 标为"骑自行车的人"；A 画的框紧贴物体轮廓，B 画的框偏大一圈。这种不一致会导致训练数据噪声，模型学到矛盾的标注。解决方案是制定标注规范文档（标注 Guideline），定期做标注员一致性测试（同一段数据分给多个标注员标注，计算 IoU 一致性分数），低一致性标注员需要重新培训。

**模型偏见累积。** 正反馈循环有一个隐患——模型 V5 在某些场景上表现差（比如夜间场景），预标注质量差，标注员修正后训练 V6，但 V6 因为夜间训练数据少仍然表现差，V6 的夜间预标注仍然差，循环往复模型始终学不好夜间场景。这叫"模型偏见累积"——越擅长的场景预标注越好、训练数据越多、模型越强；越弱的场景预标注越差、训练数据积累越慢、模型始终弱。解决方案是对模型弱项场景做定向采集和强制人工标注，不依赖模型预标注。

**数据版本管理。** 每次模型迭代都产生新的标注数据集版本——V5 用训练集 A（10 万帧），V6 用训练集 B（15 万帧，包含 A 的修正版 + 5 万帧新数据）。需要版本管理——标注数据集有版本号，记录每帧的标注历史（谁标的、什么时候标的、基于哪个模型版本的预标注、修正了什么）。训练时需要确保用的是正确版本的数据，防止混用不同版本的标注导致训练不稳定。

**质检流程设计。** 标注完成后需要质检（Quality Assurance）。质检不是全量检查（成本太高），而是抽样检查——通常抽 10-20% 的标注帧由高级标注员或质检员复核。质检维度包括：框位置精度（IoU > 0.7 算合格）、类别正确率、漏检率（标注员漏标了多少目标）、误检率（标注员多标了多少不存在的目标）。质检不合格的批次退回重新标注。质检结果还用于评估标注员质量——连续多次质检不合格的标注员需要重新培训或淘汰。

**标注工具的要求。** 专业标注工具（如 CVAT、Scale AI、Labelbox 或自研平台）需要支持 3D 点云可视化（俯视图、侧视图、透视图同步显示）、2D-3D 联动标注（在图像上画 2D 框，工具自动在点云中生成对应 3D 框）、多帧跟踪辅助（标注员标好第 1 帧的目标 ID，工具用跟踪算法自动传播到后续帧，标注员只修正偏差帧）、批量操作（批量删除低置信度框、批量修改类别）。工具效率直接决定标注产能。

## 2. 仿真体系（Simulation）

### 概念

虚拟里程测试是用仿真环境替代物理道路测试。核心价值：安全性（不用真车冒险测试危险场景）、效率（数百个仿真任务并行跑，一天相当于物理测试数月）、可复现（相同场景可重复验证）、覆盖度（可以生成现实中罕见的 Corner Case）。

### 仿真任务类型

仿真任务不是单一形式，实际工程中有三种类型，资源需求和测试目标各不相同：

**数据回放（Log Replay）。** 拿一段历史采集的 Rosbag 数据（真实传感器录制），不做任何虚拟世界渲染，直接把数据"喂"给不同版本的算法镜像，比较它们的输出差异。比如 Rosbag #234 里有一个行人横穿马路的场景，用算法 V5 跑一遍检测到行人的置信度是 0.72，用 V6 跑一遍是 0.89，说明 V6 在这个场景上改进了。这种任务不需要仿真器（没有虚拟世界），不需要实时闭环（数据是录制好的，算法的输出不影响传感器数据），本质上是"离线推理 + 结果比对"。计算量相对小，可以大批量并行——1000 段 Rosbag × 3 个算法版本 = 3000 个任务同时跑。但局限是只能测感知算法（感知只看输入数据），不能测规划和控制（规划和控制需要根据感知结果做决策，决策会影响车辆运动，但回放数据的车辆轨迹是固定的不会变）。

**闭环仿真（Closed-loop Simulation）。** 仿真器实时生成虚拟传感器数据，算法处理数据后输出控制指令（方向盘、油门、刹车），仿真器执行控制指令更新虚拟车辆的位置和姿态，再根据新位置渲染下一帧传感器数据。这是一个实时的、闭环的、交互式的仿真——算法的行为会影响后续的传感器输入。比如测试"前方车辆急刹时自车能否及时避让"——仿真器生成前方车辆急刹的场景，自车算法（感知+规划+控制）需要检测到前方车辆、规划避让轨迹、输出刹车和转向指令，仿真器执行这些指令让虚拟车辆减速变道。如果算法有 bug（感知漏检前方车辆），虚拟车辆就会撞上去——Auto-Grading 系统记录"碰撞"这个结果，测试失败。这种任务需要仿真器（CARLA/Apollo Simulation），需要 GPU 做渲染，需要实时性（一帧延迟超时仿真就不连续），计算量大，并发受 GPU 数量限制——一台 8 卡 GPU 服务器最多同时跑 8 个闭环仿真。

**混合仿真（Hybrid Simulation）。** 把真实 Rosbag 数据的前半段回放给算法，到某个时间点切换到仿真模式——仿真器从回放数据最后一帧的车辆位置和场景状态出发，接续生成后续的虚拟传感器数据。既利用了真实数据的逼真性（前半段是真实采集的），又能测试算法在闭环场景下的表现（后半段是仿真器根据算法行为动态生成的）。常用于"事故重演 + 假设推演"——拿一段真实事故的 Rosbag，回放到事故发生前，然后在仿真中尝试不同的算法策略看能否避免事故。

### 仿真器（Simulator）

仿真器是一个软件系统，在计算机里构造虚拟的 3D 世界，模拟真实的道路、交通、天气和传感器，让自动驾驶算法在里面运行。它做四件事：

构造虚拟世界——用游戏引擎（CARLA 用 Unreal Engine，LGSVL 用 Unity，NVIDIA DRIVE Sim 用 Omniverse 实时光线追踪）搭建 3D 城市环境（道路、车道线、红绿灯、建筑物、树木、行人、其他车辆），控制天气（晴/雨/雪/雾）、光照（白天/黄昏/夜晚）、时间（模拟太阳角度变化）。道路网络通常用 OpenDRIVE 格式描述（基于 XML，定义车道宽度、坡度、曲率、交叉路口拓扑）。

模拟传感器——从虚拟世界中生成跟真实传感器一样的数据。虚拟摄像头根据参数（焦距、视场角、分辨率）渲染图像；虚拟 LiDAR 从虚拟位置发射虚拟激光射线，跟 3D 模型求交，计算反射距离和反射强度，生成点云；虚拟 Radar 模拟电磁波反射计算距离和相对速度。这些虚拟传感器数据在格式上跟真实数据完全一致，算法端不需要区分"仿真的还是真实的"。

模拟交通参与者——虚拟世界中的其他车辆、行人、自行车按交通规则或脚本运动。可以注入对抗性行为（一辆车突然闯红灯）测试算法的紧急避让能力。

模拟车辆动力学——算法输出的控制指令（方向盘角度、油门开度、刹车压力）通过物理引擎计算车辆运动（轮胎模型、悬挂系统、质量分布）。低精度仿真用简化自行车模型（只考虑运动学），高精度仿真用多体动力学模型（考虑轮胎滑移、侧倾、载荷转移）。

仿真保真度与速度的取舍——高保真仿真（光线追踪渲染、精确物理引擎）接近真实但慢（一帧渲染 200ms，跑一秒仿真需要 2 秒），适合验证算法正确性。低保真仿真（简化渲染、近似物理）快但可能不够真实，适合大规模回归测试。大厂通常两级——低保真跑量验证覆盖率，高保真精跑验证关键场景。

业界主流仿真器——CARLA（开源，Unreal Engine 4，学术界用最多）、LGSVL（开源，Unity，已停止维护）、NVIDIA DRIVE Sim（商业，Omniverse 光追渲染质量最高）、Apollo Simulation（百度自研）、Waymo Simulation City（自研，每天仿真数百万英里，不开源）。

### 仿真器与算法栈的通信

仿真器和算法栈是两个独立的软件系统，中间需要通信机制把两者连起来形成闭环。

从仿真器到算法栈传的是虚拟传感器数据——一帧包含 6 个 Camera 图像（约 6MB/帧未压缩 RGB）+ 3 个 LiDAR 点云（约 2.4MB/帧），合计约 43MB/帧，10-30Hz 帧率意味着每秒传输 400MB-1.3GB。从算法栈到仿真器传的是控制指令——方向盘转角、油门开度、刹车压力，数据量很小（几十字节）但延迟要求高。

通信方式三种：ROS Bridge（仿真器和算法都走 ROS Topic，简单但 TCP 延迟不稳定 1-100ms）；共享内存（同一台机器通过 mmap 零拷贝，延迟 < 1ms，CyberRT 的方案）；ZeroMQ 或 gRPC（跨容器跨节点，同机走 IPC 获得低延迟，跨机走 TCP 牺牲延迟换取部署灵活性）。

三个核心问题——数据转化（仿真器内部数据结构如 Unreal FVector 转成算法栈期望的 ROS 消息或 Protobuf，注意 BGR vs RGB 通道顺序、左手 vs 右手坐标系差异）；时序同步（所有同帧传感器数据标注同一虚拟时间戳，按时间戳顺序投递不乱序）；通信延迟（往返延迟必须小于一帧时间，LiDAR 10Hz 一帧 100ms，通信中间件延迟预算 5-10ms，剩余留给算法处理）。

### 时序同步的设计与实现

**仿真场景——用虚拟时钟。** 仿真器维护一个虚拟时钟变量 `sim_time`，每次仿真步进时递增（步长 10-100ms）。一个仿真步内同时渲染所有传感器数据，标注同一个 `sim_time`。但不同传感器帧率不同（LiDAR 10Hz 每 100ms 一帧，Camera 30Hz 每 33ms 一帧），LiDAR 是每 3 个 Camera 帧才生成一次。算法栈用同步缓冲区对齐——维护多个按时间戳排序的队列，当 LiDAR 帧到达时去 Camera 队列找时间戳最近的帧，找到后一起送给感知算法。同步窗口（等待其他传感器数据到达的超时时间）通常设为 25-50ms，必须小于最快传感器的帧间隔（Camera 33ms），否则同一帧可能匹配到两个 LiDAR 帧。

**真实场景——比仿真难得多。** 真实传感器有独立硬件时钟，存在偏差和漂移（LiDAR 可能比系统时钟快 2ms/小时，跑一天差几十毫秒）。三种方案：PTP（IEEE 1588 精度时间协议，通过以太网物理层时间戳做亚微秒级时钟同步，高端系统标配）、GPS PPS（每秒一个硬件脉冲校准时钟，微秒级精度，成本中等）、软件时间戳（驱动收到数据时打系统时间，毫秒级精度，只适合功能验证）。

需要考虑的问题——帧率不匹配（LiDAR 10Hz × Camera 30Hz，一个 LiDAR 帧对应 3 个 Camera 帧，选最近的还是全部用或做时间插值）；帧丢失（某帧没收到，用上一帧代替还是跳过还是报错，需定义超时策略和降级策略）；乱序到达（网络延迟抖动导致第 6 帧先于第 5 帧到达，用按时间戳排序的优先队列重排，超过缓冲深度丢弃并告警）；坐标系对齐（时间对齐后还要用标定参数——外参矩阵——把不同传感器的数据变换到同一坐标系）。

仿真和真实的最大区别——仿真器的 `sim_time` 精确、确定、可复现（跑 100 次时间戳完全一致）。真实场景时间有抖动、漂移、丢失，不可完全复现。仿真测试通过的算法不一定在真实场景通过，时序鲁棒性需要额外验证。

### 技术架构

仿真平台的技术架构分四层：

场景层——定义测试场景（地图、天气、交通流、障碍物位置）。场景可以是手工编写的（OpenSCENARIO 格式）或自动生成的（从真实数据中提取 Corner Case 并泛化放大）。场景库管理是数据平台的一部分——按场景类型索引、按难度分级、按覆盖度排序。

调度层——K8s Job 批量调度。每个仿真任务是一个 Docker 容器，包含仿真器（CARLA/LGSVL/自研）+ 算法镜像（Apollo/自研感知规划算法）。调度器需要考虑 GPU 亲和性（仿真渲染需要 GPU）、数据本地性（场景数据在哪个节点）、并发限制（GPU 数量限制并发数）。

执行层——仿真器和算法在容器内通过通信中间件（ROS Bridge 或 CyberRT）交换数据。仿真器按帧产生虚拟传感器数据（Camera 图像、LiDAR 点云、Radar 数据），算法接收数据并输出控制指令（方向盘角度、油门、刹车），仿真器执行指令更新虚拟世界状态，循环往复。

评测层——Auto-Grading 自动评分。核心指标：是否碰撞（Collision Rate）、是否偏离车道（Lane Departure）、是否闯红灯（Red Light Violation）、舒适度指标（急刹/急转弯次数）、交通规则遵守率。评分结果存入数据库，支持按场景、算法版本、指标维度聚合查询。

### 自动化评测体系（Auto-Grading）

JD 要求"结果自动化评测（Auto-Grading）服务"。自动化评测是仿真闭环的最后一环——仿真跑完后，自动计算算法在这个场景上的表现分数，判定通过还是失败，不需要人工看回放视频逐帧检查。

**评测的对象分两类。**

感知评测——针对感知算法（数据回放场景）。把 Rosbag 数据喂给感知算法，算法输出检测结果（哪些目标、什么类别、什么位置），跟 Ground Truth（人工标注的真值）比对，计算精度指标。核心指标：precision（精确率——检测出来的目标中有多少是对的，误检率 = 1 - precision）、recall（召回率——真实目标中有多少被检测出来了，漏检率 = 1 - recall）、AP（Average Precision——精度-召回曲线下面积，综合指标）、IoU（Intersection over Union——检测框和真值框的交并比，> 0.5 算合格检测，> 0.7 算优秀）。AP 是感知评测最核心的单一指标，业界标准是 mAP（mean AP，所有类别 AP 的平均值）。

行为评测——针对规划和控制算法（闭环仿真场景）。算法在仿真器里"开车"，评测它开得好不好。核心指标：安全性（是否碰撞——一票否决，碰撞即失败）、合规性（是否闯红灯、是否逆行、是否违规变道）、舒适性（急刹次数——减速度 > 3m/s² 算急刹、急转弯次数——横向加速度 > 2m/s² 算急转弯、 jerk——加加速度即加速度的变化率，越大越不舒服）、效率性（到达目的地的时间、平均车速、跟车距离是否合理）。

**感知评测怎么做。**

真值来源——Ground Truth 来自人工标注。Rosbag 数据在被选为测试集之前，标注团队把每帧的所有目标人工标注（3D 框+类别+属性），标注结果作为真值。真值质量直接决定评测可信度——如果真值本身有错（标注员漏标了一个行人），算法检测到了但跟真值不匹配，会被误判为"误检"。所以测试集的标注质量要求最高——通常双重标注+交叉检查。

匹配算法——检测结果和真值的匹配是核心计算。对于 3D 框检测，用匈牙利匹配算法（Hungarian Algorithm）做最优匹配——在检测结果集合和真值集合之间找一对一的最优匹配，最大化总 IoU。未匹配的检测结果算误检（false positive），未匹配的真值算漏检（false negative）。匹配阈值通常设 IoU > 0.5 算成功匹配。

难度分级——nuScenes 评测体系把目标按检测难度分三档：Easy（距离近、无遮挡、目标完整）、Medium（距离中等、部分遮挡）、Hard（距离远、严重遮挡、目标截断）。分别计算每个难度级别的 AP，再取平均。这样能看到算法在哪些场景上表现好、哪些场景差。

**行为评测怎么做。**

碰撞检测——仿真器物理引擎自带碰撞检测，当虚拟车辆的车身包围盒跟其他物体的包围盒相交时触发碰撞事件，记录碰撞时间、碰撞对象、碰撞位置、碰撞速度。一票否决——任何碰撞即场景失败。

车道偏离检测——持续检查虚拟车辆的中心点是否在当前车道边界内。车道边界用 OpenDRIVE 中的车道宽度定义。偏离超过 0.3 米（半个车宽）记为偏离。

舒适度量化——用 CAN 总线数据（或仿真器输出的虚拟 CAN 数据）计算纵向加速度、横向加速度、加加速度。舒适性指标有明确阈值参考——正常驾驶纵向加速度 < 2m/s²，急刹 > 3m/s²；正常横向加速度 < 1.5m/s²，急转弯 > 2m/s²；jerk（加加速度）正常 < 2m/s³，过大 > 5m/s³。舒适度分数 = 各指标加权扣分。

**评测报告生成。**

每个仿真任务结束后生成一份评测报告，包含：场景信息（场景 ID、天气、时间段、交通流密度）、算法版本（V6 commit hash）、逐帧指标（每帧的检测 AP、碰撞事件、偏离事件）、聚合指标（整体 mAP、碰撞次数、舒适度总分）、通过/失败判定（碰撞=失败，mAP < 0.8=失败，其他指标按权重计算总分 < 60 分=失败）。

评测报告存入数据库（MySQL/PostgreSQL），支持多维度查询——"V6 在夜间场景的 mAP 是多少"、"V6 vs V5 在雨天场景的碰撞率对比"、"场景 #234 在所有算法版本上的表现趋势"。这些查询结果驱动数据闭环——发现 V6 在夜间 mAP 低 → 夜间场景标注数据不足 → 定向采集夜间数据 → 下一轮迭代改进。

### 关键技术挑战

**并发控制。** 不是所有任务一起跑，而是受资源限制排队跑。不同任务类型资源需求不同——闭环仿真每个任务需要一个 GPU（做渲染），一台 8 卡 GPU 服务器最多同时跑 8 个；数据回放不需要 GPU（只需要 CPU 做推理），并发受 CPU 核数限制。调度器需要区分任务类型，分配不同资源池——GPU 池给闭环仿真，CPU 池给数据回放。优先级方面，紧急任务（线上发现 Bug 要紧急验证）需要插队——暂停低优先级任务释放 GPU，跑完紧急任务再恢复。公平调度方面，不同团队用 K8s ResourceQuota 做配额隔离——A 团队最多用 4 张 GPU，B 团队最多用 4 张，防止单个团队占满集群。抢占式调度——低优先级任务被高优先级抢占时需要安全退出（保存当前帧状态、释放 GPU 显存、清理临时文件）。

**资源隔离。** 每个仿真任务跑在独立的 Docker 容器里，容器之间用 cgroup 做 CPU/内存/GPU 隔离。一个任务不能占掉所有 GPU 显存导致其他任务 OOM。一个任务崩溃了（算法段错误）不影响其他任务——容器隔离保证故障不传播。K8s 的 limit ranges 限制每个 Pod 的资源上下限，PriorityClass 定义优先级等级。

**镜像管理。** 每个算法版本是一个 Docker 镜像（10-20GB，包含模型权重、依赖库、Apollo 框架）。算法每周迭代一次，就产生一个新镜像。问题一：镜像太大——拉取 20GB 镜像到计算节点需要几分钟，100 个任务同时启动要拉同一个镜像，网络带宽打满。解法是镜像分层缓存——基础层（CUDA、Ubuntu、Apollo 框架）不变不用重复拉，只拉变化的层（模型权重约 500MB-1GB）。问题二：镜像分发——更进一步用 P2P 分发（Dragonfly/Kraken），第一个节点从镜像仓库拉取，后续节点从已拉取的节点 P2P 拉取，而不是都打镜像仓库。问题三：镜像版本管理——V5 镜像和 V6 镜像共存，回放测试要同时拉两个镜像做对比，需要镜像 tag 管理策略。

**结果收集。** 每个仿真任务产生 GB 级的结果数据（日志、传感器录制、评分报告）。需要异步收集到对象存储，不能阻塞调度器主循环。

## 3. ROS / CyberRT 通信机制

### ROS（Robot Operating System）

ROS 不是操作系统，是机器人通信中间件。核心概念：

Node——独立进程，封装一个功能模块（如 LiDAR 驱动、感知算法、规划算法）。

Topic——发布订阅模式的通信通道。一个 Node 向 Topic 发布消息（如 LiDAR Node 向 `/lidar_points` Topic 发布点云数据），其他 Node 订阅该 Topic 接收数据。类似 Kafka 的 Topic 但运行在同一台机器的进程间。

Service——请求响应模式。一个 Node 提供服务（如 `/get_map`），其他 Node 发送请求获取响应。类似 RPC。

Action——长耗时异步请求，有持续反馈。比如"导航到目标点"可能需要数秒，期间持续反馈当前位置和进度。

### CyberRT

CyberRT 是百度 Apollo 对 ROS 的改进重写。核心优化：

实时性——ROS 基于 Linux 进程和 TCP/UDP 通信，延迟不稳定（可能 > 100ms）。CyberRT 用共享内存通信（同一台机器的 Node 间通过 mmap 共享数据，零拷贝）和 DPDK 网络栈（用户态网络，绕过内核协议栈），延迟稳定在 < 10ms。

确定性调度——ROS 没有调度优先级概念，所有 Node 平等竞争 CPU。CyberRT 支持协程优先级和 CPU 绑核，保证感知算法（安全关键）优先于日志记录（非关键）获得 CPU 时间。

数据格式——ROS 用自定义的 msg 格式，CyberRT 用 Protobuf，序列化/反序列化性能更好。

### 面试要点

面试官不会要求你精通 ROS/CyberRT 的 API 细节，但会考察你是否理解"自动驾驶系统为什么需要专用的通信中间件"——核心答案是实时性和确定性。通用的消息队列（Kafka/RabbitMQ）是为高吞吐设计的，不是为低延迟设计的。自动驾驶的安全关键场景要求控制循环延迟 < 10ms 且抖动极小（99.99 分位延迟也 < 10ms），这超出了通用中间件的能力。

## 4. 自动驾驶数据格式全景

自动驾驶数据平台处理的数据格式远不止 Rosbag 和 PCD，涵盖点云、图像、CAN 总线、GPS/IMU、业界开放数据集等多类格式。理解这些格式的内部结构和数据量级，是设计存储和处理管线的基础。

### 4.1 Rosbag——多传感器数据的容器格式

Rosbag 本身不定义具体的数据编码方式，它是一个"消息流容器"——把自动驾驶各模块通过 ROS 通信产生的消息按时间戳顺序录制到一个文件里。文件内部结构是二进制的，分三层。最外层是 Bag Header（版本号、chunk 数量、索引偏移量），中间是 Chunk（数据块，每个 chunk 是一段时间内的消息打包），最内层是 Connection 和 Chunk Info（元数据索引，记录每个 Topic 的消息类型和位置偏移，用于快速 seek 定位）。

每个消息的内部格式取决于该 Topic 的消息类型。LiDAR 的 `sensor_msgs/PointCloud2` 消息用 ROS 自定义的二进制序列化（不是 JSON 也不是 Protobuf），包含点云的 fields 描述（每个维度名、偏移量、数据类型）、点数、行数、步长，以及裸点云数据（每个点的 x/y/z/intensity 按固定字节对齐排列）。Camera 的 `sensor_msgs/Image` 消息包含编码格式（rgb8/bayer_bggr8/mono16 等）、宽高、步长和裸像素数据。CAN 总线的 `can_msgs/Frame` 消息包含 CAN ID、数据长度和最多 8 字节的数据。这些不同类型的消息都被序列化成二进制塞进同一个 Rosbag 文件，按时间戳排列。

解析处理——Rosbag 是自定义二进制格式，需要用 rosbag 库（Python/C++）读取。处理流程：打开 Rosbag → 按时间戳遍历消息 → 按 Topic 分流（LiDAR 数据送点云处理管线，Camera 数据送图像处理管线）→ 按时间戳对齐多传感器数据。

一个典型的 30 分钟城市道路测试 Rosbag 文件可能 20-80GB，包含 6 个 Camera 流（30FPS × 1920×1080 RGB）、3 个 LiDAR 流（10FPS × 10 万点）、5 个 Radar 流、IMU 流（100Hz）、GPS 流（10Hz）、CAN 总线流（50Hz）。单辆测试车每天产生 TB 级数据。

### 4.2 PCD——单帧点云的点列表格式

PCD 是 PCL（Point Cloud Library）定义的格式，描述某一时刻 LiDAR 扫描到的所有 3D 点。有两种存储方式。ASCII 格式可读但体积大——每行一个点的数据，如 `1.2 3.4 5.6 0.8` 表示 x=1.2 y=3.4 z=5.6 intensity=0.8。Binary 格式紧凑——头部是文本声明（VERSION、FIELDS、SIZE、TYPE、COUNT、WIDTH、HEIGHT、VIEWPOINT、POINTS、DATA），之后是裸二进制数据（每个点的字段值按 float32 或 float64 连续排列）。

点云数据的维度不止 x/y/z。LiDAR 扫描的每个点除了三维坐标，通常还有 intensity（反射强度，反映物体表面材质）、ring（扫描线编号，标识点来自第几条激光线）、time（相对时间戳，用于运动补偿）。Velodyne 64 线 LiDAR 扫一圈产生约 12 万个点，每个点 5 个 float32 字段（20 字节），单帧 PCD 约 2.4MB。一秒 10 帧，一分钟就是 1.4GB。

除了 PCD，点云还有其他格式。LAS 是国际测绘行业标准（美国摄影测量与遥感学会定义），支持更丰富的元数据（分类标记、RGB 颜色、多次回波），广泛用于测绘领域但自动驾驶用得少。Velodyne 原始格式是 LiDAR 厂商的私有二进制格式，包含 packet 头 + 点数据，通常在录制阶段用厂商格式，后续处理时转换成 PCD。

### 4.3 Camera 图像格式——原始和压缩并存

自动驾驶的摄像头数据有两种存储方式。第一种是 Raw 格式——直接存 Bayer 矩阵（传感器原始输出，每个像素只有 R/G/B 中的一个通道，2×2 矩阵排列 BGGR/GBRG 等），数据量 1920×1080×1 字节约 2MB/帧，需要后端 ISP（Image Signal Processor）做去马赛克、白平衡、曝光校正。存 Raw 是因为自动驾驶算法有时需要自己控制 ISP 参数，或者用 RAW 数据训练感知模型效果更好。第二种是压缩格式——PNG（无损压缩，适合标注场景）或 JPEG（有损压缩，存储省空间），或者 YUV420（视频编码常用的色彩空间，NV12 是常见的打包方式）。

在 Rosbag 中，Camera 数据通常以 `sensor_msgs/Image` 存储（未压缩的裸像素数据，包含 encoding 字段标识格式），或以 `sensor_msgs/CompressedImage` 存储（JPEG/PNG 压缩后的数据）。30FPS 的 1080p 摄像头一秒约 60MB（压缩前）或 10MB（JPEG 压缩后）。一辆车 6 个摄像头，一秒就是 60-360MB。

### 4.4 CAN 总线数据格式——车辆内部状态数据

CAN（Controller Area Network）是汽车的内部通信总线，自动驾驶需要读取车速、方向盘角度、油门开度、刹车状态、转向灯状态等车辆状态。CAN 数据本身是极简的二进制帧——每帧只有 11 位或 29 位 CAN ID + 最多 8 字节数据 + CRC。但 8 字节里怎么编码车速、转速等信号，每个车型不同，需要 DBC（Database CAN）文件描述映射关系——DBC 定义了"CAN ID 0x158 的第 0-15 位表示车速，单位 km/h，scale=0.01，offset=0"。

在 Rosbag 中 CAN 数据以 `can_msgs/Frame` 消息存储，包含 ID、数据长度、数据和时间戳。处理时需要配合 DBC 文件解析出具体信号值。一秒约 50-100 条 CAN 消息，数据量极小（每条几十字节），但频率高、实时性强。

### 4.5 GPS/IMU 定位数据格式

GPS 数据传统用 NMEA 0183 格式——文本协议，每行以 `$` 开头，如 `$GPGGA,092750.000,5321.6802,N,00630.3372,W,1,8,1.03,61.7,M,55.3,M` 表示定位、经纬度、卫星数、海拔。自动驾驶用 RTK-GPS（厘米级精度），精度从普通 GPS 的 3-10 米提升到 2-5 厘米。

IMU（惯性测量单元）数据包含三轴加速度和三轴角速度，通常用二进制格式，频率 100-200Hz。INS（惯导系统）数据是 GPS + IMU 融合后的输出——位置、速度、姿态（roll/pitch/yaw），在 Rosbag 中以 `sensor_msgs/NavSatFix`（GPS）和 `sensor_msgs/Imu`（IMU）或 `nav_msgs/Odometry`（融合后位姿）存储。

### 4.6 Radar 雷达数据格式

Radar 数据通常是厂商私有的二进制格式，常见的表示方式是 RVL（Range-Velocity-Level）——以距离（Range）、相对速度（Velocity）、功率水平（Level）三个维度组织数据。跟 LiDAR 点云的区别是 Radar 能直接测量相对速度（多普勒效应），但角分辨率差（LiDAR 0.1° vs Radar 1-5°），适合远距离测距测速不适合精细形状识别。在 Rosbag 中通常以厂商自定义消息类型存储。

### 4.7 业界开放数据集格式

各自动驾驶公司发布的公开数据集有自己的格式。KITTI 格式——最简单的目录结构，`.bin` 文件存二进制点云（纯 float32 数组，无头部），`.png` 存图像，`.txt` 存标注框。NuScenes 格式——JSON 描述的标注数据库，包含 sample、sample_data、annotation 等关系表，支持多传感器时序对齐。Waymo Open Dataset——用 TensorFlow 的 TFRecord 格式（Protobuf 序列化），集成在 TF 生态中。Apollo 自有格式——CyberRT 用 `cyber_record` 格式替代 Rosbag，类似但用 Protobuf 序列化消息，性能更好。

### 4.8 格式多样性的工程挑战

这些格式的多样性本身就是数据平台的一个挑战——采集用 Rosbag/厂商格式，存储用 PCD/PNG/DBC，标注用 NuScenes/KITTI 格式，训练用 TFRecord/HDF5/Numpy 格式。各环节格式不同，需要大量的格式转换和数据管道，这正是 JD 中"工作流引擎串联预处理-标注-质检-训练"要解决的问题。

### 面试要点

理解"非结构化数据"的含义——传统数据平台处理的是结构化数据（表、行、列），自动驾驶平台处理的是二进制大文件（Rosbag/PCD/图像/CAN）。这导致存储架构、IO 模式、处理方式都不同——对象存储而非数据库，流式读取而非 SQL 查询，GPU 加速处理而非 CPU 聚合。

## 5. 数据血缘（Data Lineage）

### 概念

JD 要求理解"数据血缘"。在自动驾驶数据平台中，数据血缘比传统数仓更复杂——不仅要追踪"哪个 ETL 任务读取哪个表写入哪个表"，还要追踪"这个标注结果来自哪段 Rosbag 数据的哪一帧、由哪个模型生成的预标注、由哪个标注员修正的、最终被哪个训练任务使用了"。

### 技术实现

数据血缘在自动驾驶场景的三层追踪：

存储层血缘——对象存储中的 Rosbag 文件 → 清洗后的 Rosbag 文件（去重/脱敏后）→ 提取出的点云帧序列 → 标注结果文件 → 训练数据集。每一层都记录上下游文件的 URI 和转换关系。

任务层血缘——采集任务 → 清洗任务 → 标注任务 → 训练任务。工作流引擎记录每个任务的输入输出。跟第八章 Notebook 09 中数据血缘追踪（AST 分析 + Spark Listener + 血缘图）的技术路线类似。

模型层血缘——模型 V2 用训练集 A（来自 Rosbag #1-#500）训练 → 模型 V2 的仿真评分结果 → 发现 Bad Case 在场景 #234 → 场景 #234 加入训练集 B → 模型 V3 用训练集 B 训练。这条链路决定了模型迭代的可追溯性——出了问题可以回溯到训练数据。
