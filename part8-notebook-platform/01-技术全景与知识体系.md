# 8.1 Notebook 技术全景与知识体系

## 1. 什么是 Notebook？为什么它值得全栈研究？

Notebook 是一种交互式计算环境，将代码、输出、可视化、文档组织在一个个"单元格（Cell）"中，支持顺序执行和即时反馈。

为什么 Notebook 是全栈开发的绝佳研究案例？因为它"麻雀虽小五脏俱全"：

- 前端有代码编辑器（CodeMirror/Monaco）、富文本渲染（Markdown/LaTeX/MIME）、数据可视化（图表/表格）、布局系统
- 后端有进程管理（Kernel 生命周期）、通信协议（WebSocket/ZeroMQ/gRPC）、多租户隔离
- 数据层有 Spark/Hive/Flink 集成、大数据 SQL 直查、DataFrame 可视化
- AI 层有 LLM 代码补全、NL2SQL、Agent 化单元格、上下文 RAG 注入

一个 Notebook 系统几乎覆盖了全栈开发的所有核心问题。

## 2. Notebook 的三种编程模型

理解不同 Notebook 方案的"编程模型"是面试中的高区分度知识点。当前存在三种模型：

### 模型一：命令式 REPL（Jupyter 的模型）

Jupyter 的执行模型是 **REPL（Read-Eval-Print Loop）**，即命令式顺序执行。用户手动决定运行哪个 Cell、何时运行，Kernel 维护一个全局命名空间，Cell 的执行顺序完全由用户行为决定。

这不是"非响应式"，而是"手动命令式"——更准确的说法是 Jupyter 采用的是 **imperative sequential execution model（命令式顺序执行模型）**。用户就是"指挥官"，每一条 execute_request 就是一条命令，Kernel 忠实地执行但不主动追踪依赖关系。

这种模型的优点是灵活性极高——你可以跳过某些 Cell、以任意顺序执行、甚至执行 Cell 3 后再回过头执行 Cell 1。但缺点也很明显：隐藏状态问题（Cell 3 可以引用 Cell 5 定义的变量，只要 Cell 5 已被执行过）、代码和输出不一致（修改了 Cell 1 的代码但忘了重新运行依赖它的 Cell 3）。

### 模型二：响应式数据流（marimo / Pluto.jl / Observable 的模型）

响应式 Notebook 通过 AST 静态分析构建 Cell 间的依赖图（DAG），当某个 Cell 被修改时，所有依赖它的下游 Cell 自动重新执行或被标记为 stale。这就是 **reactive dataflow execution model（响应式数据流执行模型）**。

marimo 的 `DirectedGraph` 组合了 `MutableGraphTopology`（拓扑结构）、`DefinitionRegistry`（变量注册表）、`CycleTracker`（循环检测），通过 `transitive_closure` 计算传递性依赖。Pluto.jl 用 Julia 的宏系统做类似的依赖分析。Observable Runtime 用 JavaScript 的 getter/setter 机制实现浏览器内的响应式。

响应式模型消除了隐藏状态——删除 Cell 时变量自动清除，修改 Cell 时下游自动更新。代价是失去了"以任意顺序执行"的灵活性。

### 模型三：混合式（ipyflow 的模型）

ipyflow 是一个有趣的中间路线——它是一个 **reactive Python kernel for Jupyter**，在 Jupyter 的 REPL 基础上叠加了数据流追踪。ipyflow 在执行过程中动态追踪符号和 Cell 间的数据流关系，当检测到某个 Cell 的输入发生变化时，提示用户（但不强制）重新执行依赖的 Cell。

这是 **bolt-on reactivity（附加式响应）**——保留 Jupyter 的灵活性，同时提供数据流感知能力。ipyflow 的目标是 `ipykernel` 的 drop-in replacement（直接替换品），提供严格超集的功能。

### 三种模型对比

| 维度 | 命令式 REPL（Jupyter） | 响应式数据流（marimo/Pluto） | 混合式（ipyflow） |
|---|---|---|---|
| 执行触发 | 用户手动 | 自动（依赖变更触发） | 手动 + 提示 |
| 依赖追踪 | 无 | AST 静态分析 | 运行时动态追踪 |
| 隐藏状态 | 有（主要痛点） | 无（自动清理） | 有（但可感知） |
| 执行顺序 | 任意（用户决定） | 拓扑排序（DAG 决定） | 任意（带建议） |
| 可复现性 | 差 | 好 | 中等 |
| 灵活性 | 最高 | 受限（不能乱序） | 高 |
| 典型代表 | Jupyter Notebook/Lab | marimo, Pluto.jl, Observable | ipyflow |

面试要点：不要说"Jupyter 是非响应式的"，而要说"Jupyter 采用命令式顺序执行模型，用户手动控制执行顺序；marimo 采用响应式数据流执行模型，通过 AST 分析构建 DAG 自动管理依赖；ipyflow 是混合路线，在 Jupyter 基础上叠加数据流追踪"。

## 3. Notebook 开发需要掌握的知识图谱

**协议与通信层：** Jupyter Message Protocol（ZeroMQ 5 通道：shell/iopub/stdin/control/heartbeat）、WebSocket 通信、Kernel 接口规范、ipynb JSON 格式。marimo 用 ZeroMQ 6 通道（PUSH/PULL）+ MessagePack 序列化。Zeppelin 用 Thrift RPC。Polynote 用 gRPC + Protobuf。ZeroMQ 的 Socket 模式（REQ-REP/PUB-SUB/PUSH-PULL/ROUTER-DEALER）是理解 Kernel 通信的基础，详见 [3.12 消息队列 — ZeroMQ 小节](../part3-java-deep/12-消息队列.md#九zeromq--不是消息队列的消息队列)。协议全景对比详见 [8.2 通信协议全景对比](./02-通信协议全景对比.md)。

**服务端框架：** Jupyter Server（Tornado）、JupyterHub（多用户管理）、Enterprise Gateway（远程 Kernel 集群）。marimo 用 Starlette/ASGI。大厂自研网关（如腾讯 KernelProxy，Go 实现）。

**前端框架：** JupyterLab 组件体系（Lumino Widget，面向对象 Widget 树）、Notebook v7（JupyterLab 单文档封装，详见 [8.3 Jupyter 源码架构分析](./03-Jupyter源码架构分析.md)）、Module Federation 构建体系。marimo 用 React + CodeMirror 6。nteract 用 React + Electron。大厂自研前端通常基于 Monaco Editor。

**执行引擎：** IPython Kernel（标准 Python Kernel，比原生 Python REPL 多出魔法命令、富显示、对象自省等能力）、PySpark Kernel（比 spark-shell 多出富显示、Magic 命令、Cell 级执行等）、IRKernel（R 语言）、xeus-sql（SQL Kernel）。IPython Kernel 与 PySpark Kernel 的深度剖析详见 [8.5 执行引擎深度剖析](./05-执行引擎深度剖析.md)。注意区分"执行引擎 PySpark Kernel"和"数据生态 Spark/Hive/Flink 集成"——前者是代码如何跑，后者是平台如何跟数据系统打通（多引擎路由、元数据注入、Cell2DAG 调度转换、血缘追踪等）。

**数据生态：** Spark/Hive/Flink 集成（多引擎统一接入、表元数据注入、Cell2DAG 调度转换）、DataFrame 可视化、大数据 SQL 直查、数据血缘追踪（非开源 Notebook 原生能力，需平台层自建，详见 [8.9 业务场景 — 数据血缘追踪](./09-业务场景与最佳实践.md)）。

**AI 集成：** LLM 代码补全、NL2SQL、NL2Cell、Agent 化单元格、上下文 RAG 注入。
