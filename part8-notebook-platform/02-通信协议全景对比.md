# 8.2 通信协议全景对比

Notebook 系统的通信协议是面试中容易被深挖的方向。不同 Notebook 方案在通信协议上的选择差异很大，理解这些差异背后的设计权衡是体现技术深度的关键。

### 1. Jupyter Message Protocol（Jupyter 生态标准）

Jupyter 的通信架构是两层式：前端 ↔ 服务器走 WebSocket，服务器 ↔ Kernel 走 ZeroMQ。

> **ZeroMQ 基础知识**：ZeroMQ 不是传统消息队列（Kafka/RabbitMQ），而是 Brokerless 的并发通信库，提供 REQ-REP / PUB-SUB / PUSH-PULL / ROUTER-DEALER 等 Socket 模式。详见 [3.12 消息队列 — 第九节 ZeroMQ](../part3-java-deep/12-消息队列.md#九zeromq--不是消息队列的消息队列)。

**服务器↔Kernel 层（ZeroMQ 5 通道）：**

shell 通道处理代码执行请求和回复（execute_request/execute_reply），是双向的 ROUTER/DEALER 模式。iopub 通道广播执行输出和状态变更（stream/display_data/execute_result/error/status），是 PUB/SUB 模式，所有连接的客户端都能收到。stdin 通道处理 input() 调用，是 ROUTER/DEALER 模式。control 通道处理中断和 Kernel 信息查询（interrupt_request/kernel_info_request），优先级高于 shell。heartbeat 通道做存活检测（纯 PING/PONG，无消息格式），是 REP/REQ 模式。

每条消息是 JSON 序列化的字典，包含 header（msg_id/msg_type/session/username）、parent_header（触发消息的 header）、metadata、content 四个部分。消息类型由 header 中的 msg_type 字段区分。

**前端↔服务器层（WebSocket）：**

Jupyter Server 将 ZMQ 消息转译为 WebSocket 帧推送给前端。前端发送的 execute_request 等消息也通过 WebSocket 传到服务器，再转发到 ZMQ shell 通道。

**设计特点：** ZeroMQ 提供低延迟、高吞吐的进程间通信，适合 Kernel 作为独立进程运行的场景。5 通道分离关注点——heartbeat 独立通道避免执行阻塞影响存活检测，iopub 的 PUB/SUB 模式天然支持多客户端订阅。缺点是 JSON 序列化开销大（大数据传输时明显），5 个 socket 的管理复杂度高。

### 2. marimo 自定义协议（双协议架构）

从源码分析（`marimo/_ipc/connection.py`、`marimo/_server/api/endpoints/ws_endpoint.py`、`marimo/_messaging/serde.py`），marimo 也是两层式架构，但两层都和 Jupyter 不同。

**服务器↔Kernel 层（ZeroMQ 6 通道，PUSH/PULL 模式）：**

从 `marimo/_ipc/connection.py` 的 `Connection` 类可以看到 6 个通道：

control 通道传输命令消息（CommandMessage），对应 Jupyter 的 shell。ui_element 通道传输 UI 组件值更新（BatchableCommand），这是 marimo 独有的——因为 marimo 的响应式 UI 组件（Slider、Dropdown 等）需要高频值更新，单独走一个通道避免和代码执行请求混在一起。completion 通道传输代码补全请求（CodeCompletionCommand），Jupyter 是走 shell 通道的，marimo 独立出来避免补全请求被长执行阻塞。input 通道传输 stdin 输入。stream 通道传输 Kernel 输出流（KernelMessage），对应 Jupyter 的 iopub。win32_interrupt 通道仅在 Windows 平台使用，传输中断信号。

所有通道都是 PUSH/PULL 模式（单向），不是 Jupyter 的 ROUTER/DEALER（双向）。序列化用 `msgspec.msgpack`（二进制 MessagePack），不是 JSON。每个 `Channel` 有类型安全的 decoder，如 `Channel.Pull(context, msg_type=CommandMessage)`。

另外，marimo 还有 AppHost ZMQ 连接（`marimo/_session/app_host/connection.py`），用于多进程沙箱场景，用 4 个 socket（mgmt、response、cmd、stream），其中 cmd 和 stream 用多帧 multipart 传输，序列化用 pickle（注释中写了 TODO 要替换掉 pickle）。

**前端↔服务器层（WebSocket + REST 双通道）：**

WebSocket 端点 `/ws` 处理实时通信——Kernel 输出推送、Cell 状态更新等。消息格式从 `ws_formatter.py` 可以看到是 `{"op": "cell-op", "data": {...}}`，`op` 是操作类型。`Notification` 基类用 `msgspec.Struct` 的 `tag_field="op"` 实现标记联合（tagged union），比 Jupyter 的 msg_type 字段更类型安全。

REST API 处理同步操作——从 `execution.py` 可见 `/execute`、`/run`、`/interrupt`、`/restart_session`、`/shutdown`、`/set_ui_element_value`、`/function_call` 等端点。

实时协作用的独立 WebSocket 端点 `/ws_sync`，基于 Loro CRDT（不是 Jupyter 生态的 Yjs）。

**与 Jupyter 的关键差异：** marimo 没有 heartbeat 通道——通过 WebSocket 连接状态 + TTL 超时（`session.ttl_seconds`）管理 Kernel 生命周期。PUSH/PULL 单向通道比 Jupyter 的 ROUTER/DEALER 简单，但需要更多通道来补偿（6 vs 5）。MessagePack 二进制序列化比 JSON 更高效。REST + WebSocket 双通道比 Jupyter 的纯 WebSocket 更灵活。

### 3. Apache Zeppelin（自研 Interpreter Protocol）

Zeppelin 完全不使用 ZeroMQ，也不使用 Jupyter 协议。

**通信架构：** 前端 ↔ Zeppelin Server 走 WebSocket + REST，Zeppelin Server ↔ Interpreter 走 Thrift RPC。

Zeppelin 的 Interpreter 是独立的 JVM 进程（支持 Scala/Python/R/SQL 等），通过 Apache Thrift 定义的接口通信。Thrift 提供了跨语言的 IDL（接口定义语言）和二进制序列化，比 JSON 更高效，比 gRPC 更老牌。

前端通过 WebSocket 接收 Paragraph（Zeppelin 的 Cell 叫法）的执行结果和状态。同步操作（如创建 Notebook、列出解释器）走 REST API。

**与 Jupyter 的关键差异：** 无 ZeroMQ 依赖，部署更简单。Thrift RPC 提供强类型接口，但不如 Jupyter 协议灵活。Interpreter 进程是长驻的，支持跨 Paragraph 的会话状态共享。

### 4. Polynote（gRPC + Protocol Buffers）

Netflix 的 Polynote 是唯一使用 gRPC 的主流 Notebook 方案。

**通信架构：** 前端 ↔ Server 走 WebSocket（streaming） + REST，Server ↔ Kernel 走 gRPC。

Polynote 的设计特色是强类型——所有消息用 Protocol Buffers 定义，编译生成多语言客户端。这天然支持 Polynote 的核心特性：混合语言 Cell（Python Cell 调用 Scala Cell 的变量）。跨语言的数据序列化需要严格的类型定义，gRPC + Protobuf 完美匹配。

gRPC 基于 HTTP/2，天然支持双向流（bidirectional streaming），适合 Notebook 的实时输出推送。相比 ZeroMQ，gRPC 有更好的负载均衡、服务发现和追踪支持，更适合云原生环境。

**与 Jupyter 的关键差异：** Protobuf 提供编译时类型检查，Jupyter 的 JSON 消息是运行时解析。gRPC 的 HTTP/2 多路复用比 ZeroMQ 的多 socket 更节省连接。但 gRPC 的学习曲线和调试难度比 ZeroMQ 高。

### 5. Observable Runtime（纯浏览器，无服务端）

Observable 是完全不同的范式——没有服务端 Kernel，JavaScript 直接在浏览器中执行。

**通信架构：** 无 WebSocket，无 ZeroMQ。数据通过 ES Module import（HTTP）加载，代码在浏览器中通过 Observable Runtime 执行。

Observable Runtime 实现了响应式数据流（类似 marimo 的 DAG），但完全在浏览器中运行。Cell 间的依赖关系通过 JavaScript 的 getter/setter 和 Observable 的 Runtime API 管理。当一个 Cell 的值变化，依赖它的 Cell 自动重新执行。

**与 Jupyter 的关键差异：** 零服务端部署成本，但只能运行 JavaScript（不能运行 Python/R）。无状态管理问题（每次刷新页面就是全新执行）。无网络延迟（代码在本地执行），但受浏览器内存限制。

### 6. 闭源方案简述

**Google Colab** 不使用 Jupyter 协议（尽管兼容 ipynb 格式）。前端使用 gRPC 与后端通信，Kernel 运行在 Google 的 Borg 集群上。Colab 能支持 Jupyter 的 magic 命令是因为模拟了 IPython 执行环境，但传输层完全自研。不开源。

**Databricks** 自研 WebSocket + REST 协议。Notebook 直接与 Spark Driver 通信，不经过 Jupyter Server。有自研的 Command API 和 Result API，支持多语言 Kernel（Scala/Python/R/SQL）通过同一协议统一管理。不开源。

**Deepnote / Hex** 云原生 Notebook，自研 WebSocket 协议，内置实时协作（CRDT/OT）。具体协议不公开。

### 8. 协议选型的设计权衡（面试要点）

**为什么 Jupyter 用 ZeroMQ？** ZeroMQ 提供低延迟的进程间通信，brokerless 架构不需要中间件。5 通道分离设计让 heartbeat 不受执行阻塞影响。ZeroMQ 的 PUB/SUB 模式天然支持多客户端订阅同一 Kernel 的输出。缺点是 5 个 socket 的管理复杂度高，且 ZeroMQ 不是云原生友好的（不支持 HTTP/2 多路复用、不适合跨网络通信）。

**为什么 marimo 也用 ZeroMQ 但通道结构不同？** marimo 独立出 ui_element 通道是因为响应式 UI 组件需要高频值更新。独立出 completion 通道是因为补全请求不应被长执行阻塞。用 PUSH/PULL 替代 ROUTER/DEALER 是简化设计——marimo 不需要 Jupyter 的多客户端同时执行场景。用 MessagePack 替代 JSON 是性能优化——二进制序列化在大数据传输时优势明显。

**为什么 Polynote 用 gRPC？** gRPC + Protobuf 提供编译时类型安全，Polynote 的混合语言 Cell 特性需要严格的跨语言类型定义。gRPC 基于 HTTP/2，天然支持双向流和云原生部署。但 gRPC 的调试工具不如 ZeroMQ 成熟，学习曲线更高。

**为什么 Zeppelin 用 Thrift？** Thrift 是 Apache 生态的老牌 RPC 框架，Zeppelin 作为 Apache 顶级项目选择 Thrift 是生态一致性。Thrift 的 IDL 支持多语言生成，适合 Zeppelin 的多语言 Interpreter 架构。但 Thrift 的社区活跃度不如 gRPC。

**面试建议回答思路：** 协议选型需要考虑四个因素——传输延迟（ZeroMQ 最低）、类型安全（Protobuf/Thrift 最强）、部署复杂度（WebSocket+REST 最简单）、云原生友好度（gRPC 最优）。没有银弹，Jupyter 选 ZeroMQ 是因为 Kernel 作为独立进程的场景最适合，marimo 沿用 ZeroMQ 但优化了通道结构和序列化方式，Polynote 选 gRPC 是因为跨语言需求，Zeppelin 选 Thrift 是因为生态一致性。
