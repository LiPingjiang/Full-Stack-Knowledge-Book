# 8.10 面试高频问题与参考回答

### Q1：Jupyter Notebook 的 Kernel 通信机制是什么？

Jupyter 基于 ZeroMQ 实现 5 个 socket 通道：shell 通道处理代码执行请求/回复，iopub 通道广播输出和状态，stdin 通道处理 input() 调用，control 通道处理中断，heartbeat 通道做存活检测。前端通过 WebSocket 连接到 Jupyter Server，Server 再通过 ZMQ 与 Kernel 通信。每条消息是 JSON 格式，包含 header、parent_header、metadata、content 四部分。

### Q2：Jupyter Notebook v7 和 JupyterLab 有什么区别？

Notebook v7 基于 JupyterLab 组件构建，但使用 `NotebookShell` 替代 `LabShell`。核心区别是 NotebookShell 的 main 区域只允许一个 Widget（单文档模式），而 LabShell 支持多 Tab 并排。Notebook v7 是多页面应用（每个页面独立 HTML），JupyterLab 是单页应用（SPA）。Notebook v7 的代码量不大（约 4000 行 TypeScript），核心能力来自 `@jupyterlab/*` 依赖。

### Q3：marimo 的响应式执行是如何工作的？

marimo 通过 `ScopedVisitor` 对每个 Cell 做 AST 分析，提取变量定义和引用。然后通过 `DirectedGraph` 构建有向无环图，节点是 Cell，边是变量依赖。当 Cell 被修改时，通过 `transitive_closure` 计算所有下游 Cell，按拓扑排序自动执行或标记为 stale。`DirectedGraph` 组合了 `MutableGraphTopology`（拓扑结构）、`DefinitionRegistry`（变量注册表）、`CycleTracker`（循环检测），通过 `threading.Lock` 保证线程安全。

### Q4：大厂为什么需要自研 Notebook 网关？

原生 JupyterHub 是单机架构，无法支撑十万级租户。自研网关解决五个问题：多租户 Kernel 路由和资源隔离、弹性伸缩（动态扩缩容 Kernel 实例）、元数据注入（执行前自动鉴权和注入数据上下文）、安全审计（记录所有操作）、跨集群调度（Kernel 可以运行在不同计算集群上）。

### Q5：Notebook 中如何处理与 Spark 的交互？

三种模式：Client Mode（PySpark 直连，Driver 在 Kernel 内，简单但资源受限）、Cluster Mode（通过 Livy/Enterprise Gateway 提交到集群，Driver 在集群内，生产推荐）、Cell2DAG（将 Cell 编译为调度任务，从交互转为生产）。关键挑战：序列化（Py4J 桥接 Python 和 JVM）、生命周期管理（Spark Session 与 Kernel 的绑定）、长任务超时（WebSocket 超时与 Spark 作业运行时间的冲突）、结果展示（大数据量结果的分页和内存控制）。

### Q6：如何保证 Notebook 的可复现性？

Jupyter 的可复现性差，因为执行顺序由用户控制。marimo 通过 DAG 保证确定性执行顺序。生产环境的最佳实践：Restart Kernel & Run All 验证、Notebook 版本控制（ipynb 去 output 后提交 Git 或使用 marimo 的 .py 格式）、参数化执行（Papermill 或 marimo script 模式）、调度系统执行时强制 Run All。

### Q7：Notebook 前端如何处理大量 Cell 的渲染性能？

虚拟滚动只渲染可视区域的 Cell；输出区域高度限制（超 100 行折叠）；CodeMirror 6 增量解析；大输出截断和分页；Web Worker 处理 CPU 密集任务。Jupyter Notebook v7 的 `scrollOutput` 插件实现了自动折叠长输出的逻辑。

### Q8：如何设计一个支持多租户的 Notebook 系统？

四层架构：协议层兼容 Jupyter Message Protocol；网关层自研 KernelProxy 做多租户路由和资源隔离（Kernel 运行在 K8s Pod 中，通过 cgroup/namespace 隔离）；前端层根据场景选择 JupyterLab 改造或自研；生态层与数据平台/AI 平台/权限体系打通。关键设计：Kernel 池化减少冷启动、空闲超时回收、崩溃自动恢复、全链路审计。
