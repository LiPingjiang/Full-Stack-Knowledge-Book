# 8.8 核心难点与解决方案

### 1. 多租户 Kernel 管理

**问题：** 原生 JupyterHub 是单机架构，上千用户就会崩溃。生产环境需要支持十万级租户，每个用户有独立的 Kernel 运行环境，资源隔离、弹性伸缩、故障恢复都是挑战。

**解决方案：** 自研 Kernel 网关（如腾讯 KernelProxy）。核心设计：一个 Go/Java 进程代理所有用户的 WebSocket 连接到后端 Kernel 进程；Kernel 运行在 K8s Pod 或 YARN Container 中，通过 cgroup/namespace 做资源隔离；根据负载自动扩缩容 Kernel 实例；Kernel 崩溃后自动迁移和恢复。

面试要点：理解 JupyterHub 的 Hub + Proxy + Spawner + Authenticator 四组件模型、Enterprise Gateway 的远程 Kernel 管理、K8s 上的 Kernel 调度策略。

### 2. Notebook 与 Spark/Hive 交互

**问题：** 数据分析师在 Notebook 中写 PySpark 代码处理海量数据，但 Spark Driver 运行在 YARN/K8s 集群上，Notebook Kernel 运行在单独服务器上，两者之间的序列化、网络传输、资源管理是核心难点。

**解决方案（三种模式）：**

模式一 — Client Mode（PySpark 直连）：Spark Driver 运行在 Notebook Kernel 进程内，直接通过 PySpark API 提交任务到 YARN 集群。优点是简单直接，缺点是 Driver 资源占用 Notebook 服务器、大数据量时 Kernel 容易 OOM。

模式二 — Cluster Mode（Livy/Enterprise Gateway）：通过 Apache Livy 或 Jupyter Enterprise Gateway 将 Spark 作业提交到集群，Driver 运行在集群内部。Notebook 只负责发送代码和接收结果，不承担计算压力。大厂普遍采用这种方式。

模式三 — Cell2DAG 编译（腾讯/阿里特色）：静态分析 Cell 代码的 AST，自动识别表依赖和计算逻辑，将交互式 Cell 编译成调度系统的 DAG 节点。用户在 Notebook 里调试好代码后，一键转为定时调度任务，无需重新编写。

面试要点：理解 PySpark 的序列化机制（Py4J 桥接）、Spark Magic 的工作原理（%%spark magic 调用 Livy REST API）、Kernel 与 Spark Session 的生命周期管理、如何处理长时间运行的 Spark 作业与 WebSocket 超时的冲突。

### 3. Kernel 生命周期管理

**问题：** Kernel 是有状态进程，需要管理启动、空闲检测、超时回收、崩溃恢复、资源清理。生产环境中 Kernel 泄漏（进程没被正确清理）是常见问题。

**解决方案：** 通过 heartbeat 通道检测 Kernel 存活（定期 PING/PONG）；空闲超时自动回收（如 30 分钟无活动 cgroup kill 进程）；崩溃后自动重启并恢复 Notebook 状态（从 checkpoints 恢复 Cell 代码和执行状态）；资源清理通过 cgroup/namespace 确保进程树完全回收。

### 4. 前端 Cell 渲染性能

**问题：** Notebook 有大量 Cell（几百个），每个 Cell 包含代码编辑器、输出区域、工具栏，同时渲染会导致性能问题。长输出（如打印百万行日志）会卡死浏览器。

**解决方案：** 虚拟滚动（只渲染可视区域的 Cell）；输出区域高度限制（Jupyter Notebook v7 的 `scrollOutput` 插件超过 100 行自动折叠）；CodeMirror 6 的增量解析（仅解析变更部分）；Web Worker 处理语法高亮等 CPU 密集任务。

### 5. 安全与权限

**问题：** Notebook 允许执行任意代码，生产环境中必须防止恶意代码逃逸、控制数据访问权限、审计所有操作。

**解决方案：** Kernel 运行在隔离容器中（Docker/K8s Pod），通过 cgroup/namespace 限制资源；代码执行前做 AST 分析和权限注入（如腾讯 MetaInjector，自动注入表权限、数据血缘）；所有 execute_request 记录审计日志；支持行级脱敏和数据水印；Notebook 文件本身的信任机制（Jupyter 的 trusted/untrusted 机制）。

### 6. Module Federation 构建复杂度

**问题：** Jupyter 生态有大量第三方扩展，需要动态加载而不重新构建主应用。Module Federation 的 shared 配置出错会导致运行时找不到模块或版本冲突。

**解决方案：** `createShared()` 函数统一管理三种来源的共享配置（package.json resolutions、扩展包 dependencies、jupyterlab.sharedPackages 自定义配置）；严格版本对齐策略；`singleton: true` 确保全局单例；`import: false` 控制不共享的包。
