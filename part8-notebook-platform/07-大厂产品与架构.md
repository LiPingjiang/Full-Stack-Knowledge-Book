# 8.7 大厂 Notebook 产品与架构

## 大厂 Notebook 产品全景

### 1. 腾讯四条产品线

**TEG 数据中台部 — TBDS Notebook / 数平 Notebook（内部大数据底座）：** 集团内部自用大数据基建，全公司数据分析师/算法/数仓唯一标准工作台。重度魔改 JupyterHub，原生打通内部元数据 MetaLake，Hive/StarRocks/Flink/Spark 零拷贝直读内部数仓，支持"交互式 Kernel + 一键把 cell 转换成定时调度任务"。

**CSIG 大数据产品部 — WeData Notebook（对外商业化）：** 对外政企客户 DataOps + MLOps 一体化开发平台。和内部 TBDS 90% 同源，2024 年 8 月官方发布，基于腾讯云 Cloud Studio 的 Jupyter Notebook 构建，支持 AI 原生能力（预装 Scikit-learn/Pandas、对接 TI 平台）、Serverless 弹性资源。2025 年 2 月对接 DeepSeek。

**CSIG 混元 AI 平台部 — TI-ONE / HAI Notebook（AI 训练专用）：** 两条子产品线，TI-ONE Notebook 面向传统机器学习建模，HAI 面向大模型微调/预训练/LoRA。不侧重大数据，侧重异构算力调度：GPU/TPU/分布式训练集群统一 Kernel 调度，支持分布式 Kernel（一个单元格自动切多机多卡），原生支持 DeepSpeed、Megatron-LM。

**PCG 云原生研发部 — Cloud Studio Notebook（代码原生，集团统一前端标准）：** 前端完全自研，抛弃 JupyterLab 前端，自研组件化编辑器内核（基于 Monaco，VSCode 内核）。多模态单元格（代码/Markdown/PDF/表格/图表/Agent 指令块），所有元宝、WorkBuddy、ima 全部复用这套 Cell 执行引擎。战略最高优先级：全集团统一 Notebook 前端引擎。

### 2. 阿里、字节、百度、华为

**阿里：** MaxCompute Notebook（大数据计算，PyODPS/MaxFrame）、PAI-DSW（AI 训练，JupyterLab + WebIDE + Terminal）、DataWorks Notebook（数据开发治理，与 MaxCompute 深度集成，Data+AI 一体化 Pipeline）。底层调度由自研飞天系统支撑，DataWorks 可以把 Notebook 单元格直接转成工作流节点。

**字节：** DataLeap Notebook（一站式数据中台），基于 JupyterHub + JupyterLab + Enterprise Gateway 深度修改，资源调度基于 YARN 和 TOS。

**百度：** BML Notebook（企业级 AI 开发平台，内置 ERNIE）、飞桨 AI Studio（面向开发者/学生，免费 GPU 算力）。

**华为：** ModelArts Notebook（基于 JupyterLab，昇腾 GPU 加速，与芯片深度绑定）、开发者空间 Notebook（2025 年上线，融合交互式编程与自动化工作流）。

### 4. 行业共识与趋势

"Jupyter 协议 + 自研运行时"已成行业共识：所有大厂在通信协议层都严格兼容 Jupyter Message Protocol + ZeroMQ + ipynb 格式，真正的护城河在调度层和生态层。前端正在分化为两条路线——腾讯彻底抛弃 JupyterLab 自研 Monaco 内核，阿里/字节/华为基于 JupyterLab 深度改造。"Notebook = 湖仓交互入口"是共同终局，Agent 化单元格是下一个竞争焦点。

## Notebook 四层技术架构

### 1. 通信协议层（标准层，100% Jupyter 兼容）

所有大厂在这一层都严格实现 Jupyter 消息协议。面试中需要深入理解以下内容：

**Jupyter Message Protocol** 基于 ZeroMQ 的 5 个 socket 通道工作。shell 通道处理代码执行请求和回复（execute_request/execute_reply），iopub 通道广播执行输出和状态变更（stream/display_data/execute_result/error/status），stdin 通道处理 input() 调用，control 通道处理中断和 Kernel 信息查询，heartbeat 通道做存活检测（PING/PONG）。

每条消息是一个 JSON 序列化的字典，包含 header（msg_id/msg_type/session/username）、parent_header（触发消息的 header）、metadata、content 四个部分。理解消息流转是排查 Notebook 执行问题的基本功。

**ipynb 文件格式** 是 JSON，顶层结构包含 nbformat（版本号）、metadata（Kernel 信息、语言信息）、cells（单元格数组）。每个 Cell 有 cell_type（code/markdown/raw）、source（代码字符串数组）、outputs（输出数组，仅 code cell）、execution_count、metadata。outputs 中的每项有 output_type（stream/display_data/execute_result/error），display_data 和 execute_result 包含 data 字段（MIME 类型到内容的映射）。

### 2. 服务网关层（大厂自研分水岭，90% 工作量）

JupyterHub 只做多用户登录和 Spawner 调度，大厂在这一层全部重写。以腾讯为例：

**KernelProxy 网关服务**（Go 实现的自研网关）：多租户 Kernel 路由、弹性伸缩、跨集群代理。原生 JupyterHub 是单机架构，上万用户就会崩溃，腾讯通过自研网关支撑十万级租户。核心设计是一个 Go 进程代理所有用户的 WebSocket 连接到后端 Kernel 进程，支持动态扩缩容和跨集群调度。

**统一元数据注入服务 MetaInjector：** 单元格执行前自动注入表权限、数据血缘、存储上下文。用户在 Cell 里直接写 hive 表名不需要写完整路径，网关自动鉴权。这需要深度定制 Kernel 的 execute_request 处理逻辑，在代码执行前做 AST 分析和权限注入。

**Cell2DAG 编译器：** 静态解析每个单元格代码的 AST，自动识别表依赖关系，把交互式单元格自动编译成调度 DAG，一键上线定时任务。这是 DataOps 的核心能力，开源生态没有任何实现。

面试要点：理解 JupyterHub 的架构（Hub + Proxy + Spawner + Authenticator 四组件）、Enterprise Gateway 的远程 Kernel 管理、以及为什么大厂必须自研网关（多租户隔离、弹性伸缩、安全审计、元数据注入）。

### 3. 前端渲染层

两条技术路线：路线 A（腾讯 Cloud Studio）基于 Monaco Editor 自研组件化 Cell 引擎，支持多模态单元格，向 AI 原生 Notebook 演进。路线 B（阿里/字节/华为）基于 JupyterLab 深度改造，保留 Lumino Widget 体系，叠加自研插件。

Jupyter Notebook v7 使用的是 JupyterLab 组件的子集，通过 NotebookShell 实现单文档界面（详见第五章源码分析）。marimo 则使用 React + CodeMirror 6 全新实现（详见第六章源码分析）。

### 4. 上层生态粘合层

和自家湖仓、调度平台、大模型、权限体系打通。内置代码模型、AST 代码解析、自动血缘生成、NL2Cell、上下文 RAG 注入。这一层是大厂 Notebook 的差异化竞争力所在。
