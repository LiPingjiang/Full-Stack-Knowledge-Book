# 8.9 业务场景关注点与最佳实践

## 业务场景关注点

### 场景 1：大数据分析平台（如腾讯 TBDS / 阿里 DataWorks）

**核心关注：** Notebook 与湖仓引擎的深度集成。

具体要点：PySpark/PyODPS 与 Kernel 的交互模式（Client Mode vs Cluster Mode）；表权限自动注入和鉴权（用户在 Cell 里写表名，网关自动鉴权）；SQL Cell 与 Python Cell 的混合执行；Cell2DAG 将交互式分析转为调度任务；大数据量查询结果的分页展示和内存控制（不能把 GB 级结果全部加载到浏览器）；数据血缘自动追踪（AST 分析 Cell 代码中的表引用）。

面试可能问：用户在 Notebook 中执行 `spark.sql("SELECT * FROM huge_table")` 返回百万行数据，如何防止 Kernel OOM 和浏览器卡死？答案要点：Spark 端做 limit 下推、Kernel 端做结果分页、前端做虚拟滚动、提供 DataFrame 交互式查看器（支持搜索/过滤/排序而不重新查询）。

#### 数据血缘追踪（数据治理方向必考点）

数据血缘**不是任何开源 Notebook 原生支持的**。Jupyter、marimo、Zeppelin、Polynote 都没有内置血缘追踪能力。但这个能力对数据方向的 Notebook 平台至关重要，大厂都是在平台层自建，核心思路是两层采集：

**代码执行前的 AST 分析。** 用户写 `spark.sql("INSERT INTO table_b SELECT * FROM table_a")`，平台在 execute_request 到达 Kernel 之前，先对 Cell 代码做 AST 解析，提取出读操作（table_a）和写操作（table_b），记录到血缘系统。这需要自研一个代码分析层——不是简单的正则匹配，而是要理解 SQL 语法和 Python 上下文（比如 `table_name = "user_table"` 然后 `spark.sql(f"SELECT * FROM {table_name}")` 这种动态引用）。

**执行后的结果采集。** 有些血缘需要从 Spark 的执行计划中获取（Spark Catalyst optimizer 解析后的 logical plan 包含完整的表引用），平台通过 Spark Listener 或 Hook 采集执行计划，解析出真实的表依赖。

**血缘图谱构建。** 将多次执行的读写关系汇总，构建出表级别的血缘 DAG。这个图可以用于影响分析（改了 table_a 会影响哪些下游表）、根因分析（table_c 的数据异常是哪个 Cell 引入的）、数据治理（哪些表被大量引用、哪些是冷表）。

腾讯的 MetaInjector、阿里的 DataWorks、字节的数据血缘系统都是这种平台层自建能力。如果面试聊到数据治理方向，这是一个很好的切入点——说明你理解 Notebook 不只是"代码执行器"，更是"数据资产的入口"。

### 场景 2：AI 训练平台（如腾讯 TI-ONE / 阿里 PAI-DSW）

**核心关注：** 异构算力调度和分布式训练。

具体要点：GPU/TPU 资源的分配和回收（Kernel 启动时分配 GPU、关闭时释放）；分布式 Kernel（一个 Notebook 对应多机多卡的训练集群）；与训练框架集成（PyTorch DDP、DeepSpeed、Megatron-LM）；Checkpoint 托管和模型版本管理；训练过程中的实时监控（GPU 利用率、loss 曲线实时展示在 Notebook 中）；长时训练任务的断点续传。

面试可能问：用户在 Notebook 中启动一个需要 8 张 A100 的分布式训练任务，如何管理 Kernel 与 GPU 资源的生命周期？答案要点：Kernel 启动时通过调度系统申请 GPU 资源、训练代码通过环境变量感知分布式拓扑、Kernel 异常退出时自动释放 GPU、训练日志通过 iopub 通道实时推送到前端。

### 场景 3：生产环境 DataOps（调度上线）

**核心关注：** 从交互式分析到生产调度的无缝转换。

具体要点：Cell2DAG 编译器将 Cell 代码静态分析为调度依赖图；参数化执行（Notebook 作为模板，接受运行时参数）；调度系统的容错和重试机制；生产环境与开发环境的隔离（开发用交互式 Kernel，生产用批处理 Kernel）；执行结果的监控和告警。

面试可能问：用户在 Notebook 中调试好一段数据处理代码，如何一键转为每天定时执行的生产任务？答案要点：AST 分析提取 Cell 间的依赖关系构建 DAG、自动识别外部表依赖（上游表就绪后触发）、参数化（硬编码路径替换为调度参数）、生成调度系统可执行的 Job 定义、监控执行状态和告警异常。

### 场景 4：多人协作与实时编辑

**核心关注：** 多用户同时编辑同一 Notebook 的冲突处理。

具体要点：CRDT（Conflict-free Replicated Data Type）或 OT（Operational Transformation）算法解决并发编辑冲突；实时同步 Cell 的代码变更和执行状态；用户光标和选区共享；执行锁（防止多人同时运行同一 Cell 导致状态混乱）。marimo 通过 `_server/rtc/` 模块提供实时协作能力，Jupyter 生态有 jupyterlab-collaboration 扩展（基于 Yjs CRDT）。

### 场景 5：安全合规环境（政企客户）

**核心关注：** 数据安全和审计合规。

具体要点：所有代码执行审计日志；数据访问的行级脱敏和列级权限控制；Notebook 文件的版本管理和审批流程；代码安全扫描（防止恶意代码执行，如 `os.system("rm -rf /")`）；网络隔离（Kernel 运行在 VPC 内，不能访问公网）；数据导出控制（防止敏感数据通过 Notebook 泄露）。

### 场景 6：AI 原生 Notebook（未来方向）

**核心关注：** LLM 与 Notebook 的深度融合。

具体要点：AI 代码补全（理解当前 Notebook 的变量上下文，提供精准补全）；NL2Cell（自然语言生成代码）；Agent 化单元格（一个 Cell 就是一个智能体，自动拆分任务、调用上下游数据资产、递归执行多轮推理）；上下文 RAG 注入（将相关文档和代码片段注入 LLM 上下文）；AI 辅助调试（分析错误输出，自动建议修复方案）。

面试可能问：如何让 AI 代码补全理解当前 Notebook 的变量上下文？答案要点：在补全请求中注入当前 Notebook 的变量名、类型、值摘要（不是完整值，控制 token 数量）；利用 AST 分析提取 Cell 间的依赖关系作为上下文；使用 Embedding 检索相关文档和历史代码片段。

## Notebook 开发最佳实践

### 1. 架构设计最佳实践

始终兼容 Jupyter 协议层——即使前端和调度层全部自研，也要保持 Jupyter Message Protocol 和 ipynb 格式的兼容性，这样可以复用整个 Jupyter 生态的 Kernel 和工具链。前端层根据团队能力选择：有强前端团队的可以像腾讯一样自研 Monaco 内核，否则基于 JupyterLab 深度改造更稳妥。网关层必须自研，这是大厂 Notebook 的核心竞争力。

### 2. Kernel 管理最佳实践

使用容器化部署 Kernel（Docker/K8s），通过 cgroup 做资源隔离；实现 Kernel 池化（预启动一批 Kernel 减少冷启动延迟）；空闲超时自动回收（默认 30 分钟）；崩溃自动恢复并从 checkpoint 恢复状态；所有 Kernel 操作记录审计日志。

### 3. 前端性能最佳实践

虚拟滚动处理大量 Cell；输出区域高度限制和懒加载；CodeMirror 6 增量解析；大输出截断和分页（服务端控制单次返回的数据量）；Web Worker 处理 CPU 密集任务（语法高亮、Markdown 渲染）。

### 4. 安全最佳实践

Kernel 运行在隔离容器中；代码执行前做 AST 分析和权限注入；所有 execute_request 记录审计日志；数据访问行级脱敏；网络隔离（Kernel 不能访问公网）；Notebook 文件信任机制。

### 5. 可扩展性最佳实践

插件系统采用依赖注入 + Token 模式（JupyterLab 的 `JupyterFrontEndPlugin`），支持第三方扩展动态加载；API 设计遵循 RESTful + WebSocket 双通道（REST 管理，WebSocket 实时通信）；配置系统支持热更新（不重启服务修改配置）。
