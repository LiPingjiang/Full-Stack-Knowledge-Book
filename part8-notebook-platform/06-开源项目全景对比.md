## 2. 技术路线选择矩阵

**如果你要做一个交互式数据分析工具：** Jupyter 生态（JupyterLab/Notebook v7 + JupyterHub）是最成熟的起点，生态丰富，Kernel 多语言支持好。

**如果你要做一个可复现的研究工具：** marimo 是最佳选择，响应式 DAG 保证一致性，纯 Python 存储 Git 友好，可直接作为脚本执行。

**如果你要做一个大数据开发平台：** Zeppelin 的 Interpreter 架构天然适合多语言大数据场景，Thrift RPC 对 JVM 生态友好。或者基于 Jupyter + Enterprise Gateway + SparkMagic 组合。

**如果你要做一个嵌入式 Notebook 组件：** Starboard 或 Observable Runtime，纯浏览器执行无需服务端，可以嵌入到任何 Web 应用中。

**如果你要在 Jupyter 上增加响应式能力：** ipyflow，它是 ipykernel 的直接替换品，不需要改前端，安装即用。

**如果你要做桌面端 Notebook：** nteract，Electron + React，原生桌面体验，支持双击 .ipynb 打开。

## 3. 源码研究路径建议

按以下顺序阅读源码，从易到难：

第一步 — Observable Runtime（最小，~4K 行 JS）：理解浏览器内响应式执行的基本原理，不涉及通信协议和后端。

第二步 — Starboard（~15K 行 TS）：理解纯浏览器 Notebook 的完整架构，包括 Cell 模型、插件系统、可嵌入设计。

第三步 — Jupyter Notebook v7（~4K 行 TS + Python 薄壳）：理解 Jupyter 生态的 Shell/Plugin 架构、Module Federation 构建、与 JupyterLab 的关系。

第四步 — Pluto.jl（~20K 行 Julia）：理解响应式 Notebook 的依赖分析实现（Julia 宏 vs Python AST）、Notebook 状态管理、WebSocket 通信。

第五步 — marimo（~80K 行 Python + TS）：理解完整的响应式 Notebook 系统设计——DirectedGraph 数据流、ZeroMQ 6 通道、React 前端、AI 集成。

第六步 — ipyflow（~30K 行 Python）：理解如何在 Jupyter Kernel 层面叠加数据流追踪，而不改前端。

第七步 — Polynote（~50K 行 Scala）：理解 gRPC + Protobuf 的 Notebook 通信、混合语言 Cell 的跨语言序列化。

第八步 — Apache Zeppelin（~200K 行 Java/Scala）：理解企业级 Notebook 的完整实现——Interpreter 生命周期、Thrift RPC、权限管理、调度集成。
