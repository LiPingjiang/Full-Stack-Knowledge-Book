# 4.4 Java → Rust：所有权革命与零成本抽象

> Rust 是这五门语言里**最难、也最颠覆认知**的一门。它用一套全新的「所有权」机制，在**没有 GC** 的前提下保证内存安全。
> 这一节兑现 [3.3 JVM 运行时](../part3-java-deep/03-JVM运行时.md) 和 [3.4 类型系统](../part3-java-deep/04-类型系统.md) 的钩子。准备好被「重新教育」内存观。

---

## 一、Rust 想解决的问题：既要安全，又要极致性能

先理解 Rust 的定位。在 Rust 之前，编程语言面临一个「**二选一**」的困境：

- **要内存安全** → 用 GC（Java/Go/Python），代价是 STW 停顿、内存开销、运行时不可控。
- **要极致性能和控制** → 用手动管理（C/C++），代价是容易写出内存泄漏、悬垂指针、数据竞争等致命 bug。

**Rust 的野心：两个都要。** 既要 C/C++ 级别的性能和控制（无 GC、无运行时开销），又要内存安全（不靠程序员自觉，靠**编译器在编译期强制保证**）。它实现这个目标的核心武器，就是**所有权（Ownership）**。

---

## 二、所有权：兑现 [3.3] 的钩子，无 GC 的内存管理

回顾 [3.3 JVM 运行时](../part3-java-deep/03-JVM运行时.md)：Java 对象在堆上，由 GC 回收。Rust 完全不同——它用**所有权 + 作用域**，让内存在**编译期就确定何时回收**，运行时**没有 GC**。

**三条所有权规则**（Rust 内存模型的全部基础）：

1. 每个值有且只有一个**所有者（owner）**。
2. 同一时刻只能有一个所有者。
3. 当所有者**离开作用域**，值被**立即释放**（确定性回收，不等 GC）。

```rust
{
    let s = String::from("hello");   // s 是这个字符串的所有者
    // ... 使用 s
}   // s 离开作用域，字符串内存【立即】被释放（无需 GC）
```

这就是兑现 [3.3](../part3-java-deep/03-JVM运行时.md) 钩子的核心：**Rust 没有 GC、没有 STW、内存占用完全可预测**，因为「何时释放」在编译期就由作用域确定了。

**所有权转移（move）** 是最反 Java 直觉的地方：

```rust
let s1 = String::from("hello");
let s2 = s1;              // 所有权从 s1 "移动"给 s2
println!("{}", s1);      // ❌ 编译错误！s1 的所有权已经转移，不能再用
```

在 Java 里 `s2 = s1` 只是复制引用，两个变量都能用。在 Rust 里这是**所有权转移**，`s1` 之后失效。这防止了「两个变量都以为自己拥有、都去释放」的双重释放 bug。

## 三、借用与生命周期：在不转移所有权下使用

总不能每次用个值都转移所有权。Rust 用**借用（borrow）**——临时「租用」而不夺走所有权，用 `&` 表示：

```rust
fn calculate_length(s: &String) -> usize {   // 借用，不夺走所有权
    s.len()
}

let s = String::from("hello");
let len = calculate_length(&s);   // 借给函数用
println!("{} 长度 {}", s, len);    // ✅ s 还能用，因为只是借出去又还回来了
```

**借用规则（编译器铁律，这是 Rust 安全的关键）**：

> 在任意时刻，要么有**任意多个不可变借用**（`&T`，只读），要么有**唯一一个可变借用**（`&mut T`，可写），**两者不能同时存在**。

```rust
let mut s = String::from("hello");
let r1 = &s;        // 不可变借用
let r2 = &s;        // 再来一个不可变借用，OK（多个只读没问题）
let r3 = &mut s;    // ❌ 编译错误！已有不可变借用时，不能再可变借用
```

**这条规则直接兑现了 [3.2 JMM](../part3-java-deep/02-内存模型JMM.md) 的钩子！** 回顾 JMM 那一整套 volatile、happens-before 的复杂性，根源是「多个线程同时读写共享可变内存」。Rust 的借用规则在**编译期**就禁止了「同时存在可写和其他访问」——**数据竞争在 Rust 里根本无法编译通过**。Java 靠程序员小心 + JMM 规则在运行时保证，Rust 靠编译器在编译期强制保证。这是「fearless concurrency（无畏并发）」的来源。

> **生命周期（lifetime）** 是借用的延伸——编译器要确保「借用不会比被借用的值活得更久」（避免悬垂引用）。这是 Rust 学习曲线最陡的部分，初期会被借用检查器「教育」无数次，但过了这关，你写的并发代码就有了编译期的安全保证。

## 四、类型系统：兑现 [3.4] 的钩子，单态化泛型

回顾 [3.4 类型系统](../part3-java-deep/04-类型系统.md)：Java 泛型用类型擦除（运行时信息丢失，有一堆坑）。Rust 泛型走**单态化（monomorphization）**——完全相反的路：

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {   // 泛型函数
    let mut largest = &list[0];
    for item in list {
        if item > largest { largest = item; }
    }
    largest
}
```

编译器看到你用 `largest::<i32>` 和 `largest::<f64>`，会**为每个具体类型生成一份专门的机器码**（就像你手写了两个版本）。这意味着：

- **运行时类型信息完整保留**，没有 Java 那种擦除的坑（能 new 泛型、能精确分发）。
- **零成本抽象**——泛型代码运行时和手写具体类型代码一样快，没有装箱/类型检查开销。
- 代价：编译产物更大、编译更慢（因为生成了多份代码）。

这就是 Rust 标志性的「**零成本抽象（zero-cost abstraction）**」哲学：**高级抽象不该有运行时代价**。你用泛型、迭代器、闭包写出优雅代码，编译后和手写底层代码一样高效。

## 五、错误处理：兑现 [3.5] 的钩子，Result 把异常做对了

回顾 [3.5 异常体系](../part3-java-deep/05-异常体系.md)：Java 受检异常「想做对却做砸了」（强制处理但样板泛滥）。**Rust 的 `Result` 可以说是把受检异常的理想真正实现好了。**

Rust 没有异常，用 `Result<T, E>` 枚举把错误编进**类型系统**：

```rust
enum Result<T, E> {
    Ok(T),    // 成功，带成功值
    Err(E),   // 失败，带错误值
}

fn read_user(id: &str) -> Result<User, String> {
    if id.is_empty() {
        return Err("id 不能为空".to_string());
    }
    Ok(User { id: id.to_string() })
}
```

调用者**必须处理 Result**（编译器强制，否则警告），但 Rust 用 **`?` 运算符**让「显式传播错误」变得优雅，彻底解决了 Java 受检异常的「样板泛滥」：

```rust
fn process() -> Result<User, String> {
    let user = read_user("123")?;   // ? = 如果 Err 就直接 return Err，否则取出 Ok 值
    // 等价于 Java 一长串 try-catch，但只用一个 ? 字符
    Ok(user)
}
```

对比一下三种哲学（也呼应 [4.3 Go](./03-Java到Go.md)）：

- **Java 受检异常**：编译器强制处理，但 try-catch 样板多、会传染。
- **Go error**：错误是普通值，`if err != nil` 显式但啰嗦。
- **Rust Result + `?`**：错误进类型系统（像受检异常那样强制），但 `?` 让传播只需一个字符——**既强制又简洁**。

很多人认为 Rust 的 `Result + ?` 是错误处理设计的「集大成者」——它实现了 Java 受检异常的初心，又避开了它的坑。详见 [4.6 错误处理对比](./06-错误处理对比.md)。

## 六、给后端大脑的「翻译词典」与上手建议

| Rust 概念 | Java 类比 | 关键差异 |
|----------|----------|---------|
| 所有权 + 作用域 | GC | 编译期确定回收，无 GC、无 STW |
| move（所有权转移） | 引用复制 | 转移后原变量失效 |
| 借用 `&` / `&mut` | 引用 | 编译期借用检查，禁止数据竞争 |
| 生命周期 | — | 编译器保证借用不悬垂 |
| 单态化泛型 | 擦除泛型 | 运行时类型保留、零成本、产物更大 |
| `Result<T,E>` + `?` | 受检异常 | 错误进类型系统，但更简洁 |
| trait | interface | 类似，但更强大（关联类型等） |
| `Option<T>` | `Optional<T>` | 编译期强制处理 null（无 NPE） |

**给 Java 工程师的上手心法**：
1. **接受「被编译器教育」**。前两周你会和借用检查器「打架」，这是正常的。它在帮你提前消灭运行时才会发现的 bug。
2. **先用 `clone()` 过渡**。搞不定所有权时，先 `.clone()` 复制一份让它编译通过，跑通了再优化——这是合理的学习策略。
3. **`Option<T>` 让你告别 NPE**。Rust 没有 null，可能为空的值用 `Option`，编译器强制你处理「空」的情况，[3.5](../part3-java-deep/05-异常体系.md) 提到的 NPE 在 Rust 里编译期就被消灭了。

## 本章小结

- Rust 的野心是「既要内存安全，又要极致性能」，核心武器是**所有权**。
- **所有权 + 作用域**实现**无 GC** 的确定性内存回收（兑现 [3.3](../part3-java-deep/03-JVM运行时.md) 钩子），无 STW、内存可预测。
- **借用规则**（多个只读 XOR 一个可写）在**编译期**禁止数据竞争（兑现 [3.2 JMM](../part3-java-deep/02-内存模型JMM.md) 钩子），实现「无畏并发」。
- **单态化泛型**与 Java 类型擦除相反（兑现 [3.4](../part3-java-deep/04-类型系统.md) 钩子），运行时类型完整 + 零成本抽象。
- **`Result + ?`** 把 [3.5 受检异常](../part3-java-deep/05-异常体系.md) 的理想真正实现好——既强制又简洁。
- 学习曲线最陡，但带来编译期的安全保证。适合系统编程、高性能服务、对延迟/内存极敏感的场景。
