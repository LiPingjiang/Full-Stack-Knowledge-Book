# 4.6 错误处理对比：同一个场景，四种哲学

> 兑现 [3.5 异常体系](../part3-java-deep/05-异常体系.md) 的核心钩子。
> 我们用**同一个具体场景**——「读一个文件并把内容解析成数字」——让 Java、Go、Rust、JS/TS 各实现一遍，把四种错误处理哲学放在一起对比。

---

## 统一场景

> 读取一个文件，把里面的内容解析成整数，返回这个数。可能失败的点：(1) 文件不存在/读取失败；(2) 内容不是合法数字。

这个场景有**两个独立的失败点**，足以暴露每种语言错误处理的特点。下面看四种实现。

---

## 一、Java：异常（try-catch）

```java
import java.nio.file.*;

public class Parser {
    // 用受检异常声明可能的失败
    public int readNumber(String path) throws IOException, NumberFormatException {
        String content = Files.readString(Path.of(path));   // 可能抛 IOException
        return Integer.parseInt(content.trim());             // 可能抛 NumberFormatException
    }

    // 调用者用 try-catch 处理
    public void caller() {
        try {
            int n = readNumber("data.txt");
            System.out.println("读到: " + n);
        } catch (IOException e) {
            System.err.println("文件读取失败: " + e.getMessage());
        } catch (NumberFormatException e) {
            System.err.println("内容不是合法数字: " + e.getMessage());
        }
    }
}
```

**特点**（回顾 [3.5](../part3-java-deep/05-异常体系.md)）：

- 错误是**「跳出正常流程」的事件**——正常逻辑直接顺着写，错误在 catch 里集中处理。
- 优点：正常路径代码干净（不被错误检查打断）；异常自动沿调用栈向上冒泡。
- 缺点：受检异常带来 `throws` 传染；错误处理和发生点分离，控制流「隐式跳转」难追踪;空 catch 容易吞异常。

---

## 二、Go：error 返回值

```go
package main

import (
    "os"
    "strconv"
    "strings"
)

// 错误作为普通返回值（约定放最后）
func readNumber(path string) (int, error) {
    content, err := os.ReadFile(path)
    if err != nil {
        return 0, err   // 文件读取失败，显式返回错误
    }
    n, err := strconv.Atoi(strings.TrimSpace(string(content)))
    if err != nil {
        return 0, err   // 解析失败，显式返回错误
    }
    return n, nil   // 成功
}

func caller() {
    n, err := readNumber("data.txt")
    if err != nil {
        // 处理错误（注意：Go 不细分是哪种错误，需要时用 errors.Is/As 判断）
        println("出错:", err.Error())
        return
    }
    println("读到:", n)
}
```

**特点**（回顾 [4.3](./03-Java到Go.md)）：

- 错误是**普通返回值**，每一步都要 `if err != nil` 显式检查。
- 优点：错误路径**完全显式可见**，没有隐藏的控制流跳转，代码「所见即所得」。
- 缺点：啰嗦——你会写无数个 `if err != nil`，正常逻辑被错误检查反复打断。

## 三、Rust：Result + `?`

```rust
use std::fs;

// 返回 Result，错误进类型系统
fn read_number(path: &str) -> Result<i32, Box<dyn std::error::Error>> {
    let content = fs::read_to_string(path)?;   // ? : 失败就直接 return Err
    let n: i32 = content.trim().parse()?;      // ? : 解析失败也直接 return Err
    Ok(n)                                       // 成功
}

fn caller() {
    match read_number("data.txt") {
        Ok(n) => println!("读到: {}", n),
        Err(e) => eprintln!("出错: {}", e),    // 编译器强制你处理 Err 分支
    }
}
```

**特点**（回顾 [4.4](./04-Java到Rust.md)）：

- 错误编进**类型系统**（`Result<T, E>`），编译器**强制**调用者处理（不处理会警告）。
- **`?` 运算符**让「失败就向上传播」只需一个字符——既有 Go 的显式（你能看到每个 `?`），又没有 Go 的啰嗦（不用写 `if err != nil`）。
- `match` 强制你处理 `Ok` 和 `Err` 两种情况，编译期保证不漏掉错误分支。
- 这就是 [3.5](../part3-java-deep/05-异常体系.md) 说的：**Rust 把 Java 受检异常「强制处理」的理想真正实现好了，又避开了它的样板与传染。**

## 四、JS/TS：try-catch（无受检）

```typescript
import { readFile } from 'node:fs/promises';

// 没有受检异常，函数签名不声明会抛什么
async function readNumber(path: string): Promise<number> {
  const content = await readFile(path, 'utf-8');   // 可能 reject（抛错）
  const n = parseInt(content.trim(), 10);
  if (isNaN(n)) {
    throw new Error('内容不是合法数字');             // 手动抛
  }
  return n;
}

async function caller() {
  try {
    const n = await readNumber('data.txt');
    console.log('读到:', n);
  } catch (e) {
    // 注意：catch 到的 e 类型是 unknown，不知道具体是什么错误
    console.error('出错:', (e as Error).message);
  }
}
```

**特点**（回顾 [4.2](./02-Java到JS-TS.md)）：

- 像 Java 的 try-catch，但**没有受检异常**——函数签名不声明会抛什么，调用者**无从得知该处理什么**（全靠文档或经验）。
- 异步场景错误通过 Promise reject 传播，`async/await` 下用 try-catch 捕获。
- 缺点：`catch` 到的错误类型是 `unknown`，要自己判断；很容易忘记处理某个错误路径（编译器不提醒）。

## 五、四种哲学横向对比

| 维度 | Java 异常 | Go error | Rust Result+? | JS/TS try-catch |
|------|----------|----------|---------------|-----------------|
| 错误的本质 | 跳出流程的事件 | 普通返回值 | 类型系统的一部分 | 跳出流程的事件 |
| 是否强制处理 | 受检强制/非受检不强制 | 约定强制（易忽略） | **编译器强制** | 不强制 |
| 显式程度 | 隐式冒泡 | 完全显式 | 显式(`?`)但简洁 | 隐式冒泡 |
| 正常路径代码 | 干净 | 被 `if err` 打断 | 较干净(`?`) | 干净 |
| 漏处理的后果 | 受检编译错/非受检运行时炸 | 编译能过但逻辑错 | **编译期就拦住** | 运行时炸 |
| 代码啰嗦度 | 中 | 高 | 低 | 中 |

## 六、给后端大脑的洞察

把这四种放在一起，你能看到一条清晰的**设计演进脉络**：

1. **Java 受检异常**首先尝试「强制处理错误」，理念正确，但用 try-catch 实现导致样板和传染（[3.5](../part3-java-deep/05-异常体系.md) 的争议）。
2. **Go** 走向另一极端——干脆把错误降成普通值，用啰嗦换「完全显式、无隐藏控制流」。
3. **Rust** 集大成——把错误放进类型系统（恢复了「强制处理」的理想），又用 `?` 运算符消除啰嗦。**既强制又简洁，编译期就拦住漏处理。**
4. **JS/TS** 保留了 Java 式 try-catch 但去掉受检，灵活但最容易漏处理错误。

**这个对比的最大价值，不是记住语法，而是理解「错误处理是一个有权衡的设计问题」**：要不要强制处理?要显式还是简洁?要编译期保证还是运行时灵活?每门语言给出了不同答案，没有绝对正确，只有场景适配。

理解了这一层，你就能在 AI Coding 时代**判断 AI 生成的错误处理代码是否地道**——它有没有按目标语言的哲学来处理错误（Go 该 `if err != nil`、Rust 该用 `?`、Java 别吞异常）。这种判断力正是第五章要讲的核心。

---

## 本章小结

- 同一个「读文件解析数字」场景，四种语言展现四种错误处理哲学。
- **Java 异常**（事件式、受检争议）→ **Go error**（普通值、显式啰嗦）→ **Rust Result+?**（进类型系统、既强制又简洁，集大成）→ **JS/TS try-catch**（无受检、最易漏处理）。
- 核心洞察：错误处理是**有权衡的设计问题**（强制 vs 灵活、显式 vs 简洁、编译期 vs 运行时），Rust 的 `Result + ?` 被视为把 Java 受检异常理想真正实现好的方案。
- 这种「理解设计权衡」的能力，是 AI Coding 时代判断代码地道与否的关键。
