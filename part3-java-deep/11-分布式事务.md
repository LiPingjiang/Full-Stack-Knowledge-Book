# 3.11 分布式事务：跨服务的数据一致性怎么保证

> **一句话定位**：微服务拆完之后，一个业务动作横跨多个服务、多个数据库，本地事务 ACID 的"免费午餐"没了。你必须用**分布式事务方案**来保证跨服务的数据一致性——但不同方案在**一致性强度、性能、复杂度**之间的取舍完全不同。面试考的核心就一件事：**给你一个场景，你选哪种方案，为什么？**

---

## 一、为什么需要分布式事务？——从单体拆微服务说起

### 1.1 单体时代：一切很简单

单体应用时代，所有业务逻辑共享同一个数据库。一个"下单"操作：

```sql
BEGIN;
INSERT INTO orders (...) VALUES (...);         -- 创建订单
UPDATE inventory SET stock = stock - 1 WHERE ...; -- 扣库存
INSERT INTO payment (...) VALUES (...);        -- 创建支付单
COMMIT;
```

一个本地事务搞定，数据库帮你兜底 ACID，你什么都不用操心。

### 1.2 微服务时代：每个服务有自己的库

拆完微服务后，同一个"下单"变成了：

| 服务 | 数据库 | 操作 |
|------|--------|------|
| 订单服务 | order_db | 创建订单 |
| 库存服务 | inventory_db | 扣减库存 |
| 支付服务 | payment_db | 冻结资金 |

三个服务、三个数据库——你没法用一个 `BEGIN...COMMIT` 把它们包起来了。

### 1.3 跨库的 ACID 不再免费

订单创建成功但库存扣减失败怎么回滚？订单状态“已支付”但支付服务没扣钱怎么办？这就是分布式事务要解决的问题——**在不可靠的网络环境下，让多个独立服务的操作要么一起成功，要么一起失败（或最终达到一致状态）**。

## 二、2PC（两阶段提交）

### 2.1 核心思想

引入一个**协调者（Coordinator）**，分两阶段完成事务：

```
阶段一：Prepare（投票）
  协调者 → 所有参与者："你们准备好了吗？"
  参与者：执行事务但不提交，锁住资源，回复 YES / NO

阶段二：Commit / Rollback（执行）
  如果所有人回复 YES → 协调者发 Commit，参与者提交事务
  如果有人回复 NO   → 协调者发 Rollback，参与者回滚事务
```

### 2.2 优缺点

| 维度 | 说明 |
|------|------|
| **优点** | 强一致性；实现相对简单（数据库 XA 协议原生支持） |
| **缺点1：同步阻塞** | Prepare 到 Commit 之间，参与者持有锁不释放，其他事务被阻塞 |
| **缺点2：单点问题** | 协调者挂了，参与者不知道该提交还是回滚，一直锁着资源 |
| **缺点3：数据不一致** | 网络分区时，部分参与者收到 Commit、部分没收到 → 脑裂 |

典型实现：MySQL XA 事务、Java JTA 规范、Seata XA 模式。

<details>
<summary><b>展开：面试追问——协调者 Prepare 后挂了怎么办？</b></summary>

**场景**：协调者发完 Prepare 收到所有 YES，准备发 Commit，这时候挂了。

**问题**：
- 参与者已经执行了事务、锁住了资源，但不知道该提交还是回滚
- 参与者互相之间不通信，谁也不知道别人的投票结果

**后果**：资源被无限期锁定，系统卡死。

**缓解手段**（不是完美解决）：
1. **超时机制**：参与者等待超时后自行回滚（但如果其他参与者已经提交，数据就不一致了）
2. **协调者持久化日志**：协调者恢复后根据日志继续流程
3. **选举新协调者**：通过一致性协议选出新协调者接管（但这本身又是一个分布式问题）

这正是 2PC 的致命伤——**协调者是单点，一旦在关键时刻挂了，整个协议陷入不确定状态**。

</details>

## 三、3PC（三阶段提交）

### 3.1 在 2PC 基础上加了什么？

3PC 把 2PC 的两阶段拆成三阶段，并加入**超时机制**：

| 阶段 | 动作 | 关键改进 |
|------|------|---------|
| **CanCommit** | 协调者问参与者"你能提交吗？"（只问能力，不执行） | 早期发现不可用节点，减少无谓的资源锁定 |
| **PreCommit** | 参与者执行事务但不提交（等价于 2PC 的 Prepare） | 参与者有超时机制：超时后默认**中止** |
| **DoCommit** | 协调者通知提交 | 参与者有超时机制：超时后默认**提交** |

### 3.2 相比 2PC 的改进

- **参与者有超时决策权**：不会永远阻塞
- **降低了阻塞窗口**：CanCommit 阶段不锁资源，只有 PreCommit 后才开始占用

### 3.3 为什么工程中很少用？

1. **复杂度显著增高**：多了一轮网络交互，实现难度大
2. **仍有不一致窗口**：PreCommit 后如果网络分区，部分参与者超时提交、部分超时中止 → 数据不一致
3. **性能更差**：多了一轮 RTT
4. **没有主流框架原生支持**

> 面试结论：3PC 是 2PC 的理论改进，面试知道它的存在和改进点即可，工程实践中直接跳过。

## 四、TCC（Try-Confirm-Cancel）

### 4.1 核心思想

TCC 是业务层面的分布式事务方案——把每个服务的操作拆成三个接口：

| 阶段 | 职责 | 举例（转账扣款） |
|------|------|-----------------|
| **Try** | 资源预留/检查，不做真正业务变更 | 冻结转出方 100 元（available -= 100, frozen += 100） |
| **Confirm** | 真正提交，使用 Try 预留的资源 | 扣除冻结金额（frozen -= 100） |
| **Cancel** | 释放 Try 预留的资源 | 解冻（frozen -= 100, available += 100） |

### 4.2 流程伪代码

```java
// === Try 阶段 ===
// 订单服务
orderService.tryCreate(orderId);        // 创建订单，状态=TRYING

// 库存服务
inventoryService.tryDeduct(skuId, 1);   // 冻结库存：available-1, frozen+1

// 支付服务
paymentService.tryFreeze(userId, 100);  // 冻结余额

// === 如果所有 Try 成功 → Confirm ===
orderService.confirm(orderId);          // 状态 → CONFIRMED
inventoryService.confirm(skuId, 1);     // frozen-1, sold+1
paymentService.confirm(userId, 100);    // frozen-100

// === 如果任何 Try 失败 → Cancel ===
orderService.cancel(orderId);           // 状态 → CANCELLED
inventoryService.cancel(skuId, 1);      // frozen-1, available+1
paymentService.cancel(userId, 100);     // frozen-100, available+100
```

### 4.3 优缺点

| 维度 | 说明 |
|------|------|
| **优点1** | 无全局锁，性能高（Try 阶段只是"冻结"，不阻塞其他事务） |
| **优点2** | 每个阶段都是本地事务，天然支持高可用 |
| **优点3** | 灵活，可以自定义补偿逻辑 |
| **缺点1** | 业务侵入性强——每个服务要实现 Try/Confirm/Cancel 三个接口 |
| **缺点2** | 开发成本高，要处理幂等、空回滚、悬挂三大难题 |

### 4.4 适用场景

- 资金类业务（账户冻结→扣款→解冻）
- 库存预扣
- 对一致性要求高、对性能也有要求的场景

<details>
<summary><b>展开：面试必追问——空回滚、悬挂、幂等</b></summary>

**1. 空回滚（Empty Rollback）**

**场景**：Try 请求因网络超时没有到达参与者，协调者判定 Try 失败 → 直接调 Cancel。参与者收到 Cancel，但它压根没执行过 Try。

**问题**：Cancel 不能报错，否则协调者一直重试。

**解决**：Cancel 时检查是否有对应的 Try 记录，没有 → 直接返回成功（空回滚），并记录一条"已回滚"标记。

---

**2. 悬挂（Suspend）**

**场景**：Try 请求网络延迟很大，协调者超时 → 先调了 Cancel（空回滚成功）。过了一会儿，迟到的 Try 请求才到达参与者。

**问题**：如果 Try 正常执行了冻结，但 Cancel 已经结束了，这笔冻结的资源永远不会被释放。

**解决**：Try 时检查是否已有"已回滚"标记，如果有 → 拒绝执行 Try。

---

**3. 幂等**

**场景**：Confirm/Cancel 可能因超时被重试多次。

**解决**：Confirm 和 Cancel 必须幂等——重复调用结果不变。通常用**事务状态表**记录每笔事务的当前状态，已 Confirm/Cancel 的直接返回成功。

---

**总结口诀**：
- 空回滚 → Cancel 前查有没有 Try
- 悬挂 → Try 前查有没有 Cancel
- 幂等 → 状态表 + 唯一事务ID

</details>

## 五、Saga 模式

### 5.1 核心思想

把一个长事务拆成一系列**本地事务**（T1, T2, T3...），每个本地事务都有对应的**补偿事务**（C1, C2, C3...）：

- 正常流程：T1 → T2 → T3 → 完成
- 某步失败：T1 → T2 → T3(失败) → C2 → C1 → 回滚完成

### 5.2 两种实现方式

| 方式 | 别名 | 原理 | 优缺点 |
|------|------|------|--------|
| **编排式（Orchestration）** | 集中式 | 有一个中心「编排器」按顺序调用各服务，失败时按反序调补偿 | 流程清晰可控；编排器是单点 |
| **协调式（Choreography）** | 事件驱动式 | 各服务通过事件互相触发，没有中心节点 | 去中心化、松耦合；流程难追踪、调试困难 |

### 5.3 编排式示例

编排器定义：step1 create(补偿 cancel) → step2 deduct(补偿 revert) → step3 pay(补偿 refund)。如果 step3 失败，反序执行 revert → cancel。

### 5.4 优缺点

| 维度 | 说明 |
|------|------|
| **优点1** | 无全局锁，各步骤都是本地事务提交后再进入下一步 |
| **优点2** | 适合长流程（几分钟甚至几天的业务流程） |
| **优点3** | 各服务只需暴露正向+补偿接口，不像 TCC 要三个 |
| **缺点1** | **无隔离性**：T1 提交后、T2 还没开始时，T1 的数据对外可见 |
| **缺点2** | 补偿逻辑复杂（有些操作不可逆：短信发出去了怎么补偿？） |
| **缺点3** | 如果补偿也失败了，需要人工介入 |

### 5.5 Saga vs TCC 对比

| 维度 | TCC | Saga |
|------|-----|------|
| 阶段数 | 3（Try/Confirm/Cancel） | 2（正向/补偿） |
| 隔离性 | 有（Try 阶段冻结资源，外部看不到中间态） | 无（每步提交后数据可见） |
| 业务侵入 | 高（每个服务实现三个接口） | 中（正向+补偿） |
| 适合事务时长 | 短事务（毫秒~秒级） | 长事务（秒~天级） |
| 适用场景 | 资金、库存等需要隔离的场景 | 电商下单全流程、旅行预订、审批流 |
| 典型框架 | Seata TCC、Hmily | Seata Saga、Temporal、Camunda |

## 六、本地消息表

### 6.1 核心思想

**业务操作 + 写消息记录在同一个本地事务中**，保证"业务做了 = 消息一定会发出去"。

### 6.2 完整流程

1. 订单服务在**同一个本地事务**中：`INSERT orders` + `INSERT local_message_table (status=PENDING)` → COMMIT
2. 后台定时任务扫描 PENDING 消息 → 投递到 MQ
3. 库存服务消费消息 → 扣减库存 → 回传确认
4. 订单服务收到确认 → 标记消息为 DONE

### 6.3 关键伪代码

```java
// === 生产者端：业务+消息在同一本地事务 ===
@Transactional
public void createOrder(OrderDTO dto) {
    orderMapper.insert(dto);
    localMessageMapper.insert(new LocalMessage(
        UUID.randomUUID().toString(), "ORDER_CREATED",
        JSON.toJSONString(dto), "PENDING"));
}

// === 定时扫描投递 ===
@Scheduled(fixedDelay = 5000)
public void scanAndSend() {
    List<LocalMessage> pending = localMessageMapper.selectByStatus("PENDING");
    for (LocalMessage msg : pending) {
        mqProducer.send(msg.getTopic(), msg.getBody());
        localMessageMapper.updateStatus(msg.getId(), "SENT");
    }
}

// === 消费者端：幂等消费 ===
@MQListener(topic = "ORDER_CREATED")
public void onOrderCreated(String body) {
    OrderDTO dto = JSON.parseObject(body, OrderDTO.class);
    if (inventoryService.alreadyProcessed(dto.getOrderId())) return;
    inventoryService.deduct(dto.getSkuId(), dto.getQuantity());
}
```

### 6.4 优缺点

| 维度 | 说明 |
|------|------|
| **优点1** | 简单可靠，利用已有的数据库，不依赖外部中间件的事务能力 |
| **优点2** | 天然保证"业务和消息"的原子性（同一个本地事务） |
| **优点3** | 下游消费幂等即可，对下游侵入小 |
| **缺点1** | 消息表会膨胀——需要定期清理已完成的记录 |
| **缺点2** | 定时扫描有延迟（秒级~分钟级），非实时 |
| **缺点3** | 消息表和业务表耦合在同一个库，分库时比较麻烦 |

## 七、最大努力通知

### 7.1 核心思想

发送方**尽最大努力**将通知送达接收方，但**不保证一定成功**。如果多次重试仍然失败，就不再重试——接收方需要提供一个**主动查询接口**，自己来确认状态。

### 7.2 适用场景

- **跨公司/跨平台**通知：支付宝通知商户支付结果、微信回调
- 一致性要求低：允许短时不一致，最终人工或定时对账兜底

### 7.3 重试策略

递增间隔重试（如 5s → 30s → 5min → 30min），最多重试 N 次后放弃，接收方需提供主动查询接口。

### 7.4 与本地消息表的区别

| 维度 | 本地消息表 | 最大努力通知 |
|------|-----------|-------------|
| 可靠性 | 高——必须保证消息投递成功 | 低——只是尽力，可能丢失 |
| 一致性保证 | 最终一致 | 不保证一致，靠对账兜底 |
| 适用范围 | 内部服务间 | 跨公司/跨平台 |
| 主动方 | 发送方不断重试直到成功 | 发送方有限重试后放弃，接收方主动查询 |
| 典型场景 | 订单→库存扣减 | 支付回调通知商户 |

## 八、方案选型——面试万能对比表

### 8.1 全维度对比

| 方案 | 一致性 | 性能 | 业务侵入 | 实现复杂度 | 适用场景 |
|------|--------|------|---------|-----------|---------|
| **2PC/XA** | 强一致 | 低（同步阻塞+全局锁） | 低（数据库/框架透明支持） | 低 | 短事务、对一致性要求极高、能接受性能损失 |
| **TCC** | 强一致 | 高（无全局锁） | 高（每服务3个接口） | 高 | 资金类、库存类、高性能+强一致 |
| **Saga** | 最终一致 | 高 | 中（正向+补偿） | 中 | 长流程、跨多服务、可接受短暂不一致 |
| **本地消息表** | 最终一致 | 中（定时扫描有延迟） | 低 | 低 | 内部服务间异步解耦 |
| **最大努力通知** | 弱一致 | 高 | 低 | 低 | 跨公司/外部回调、对账兜底 |

### 8.2 面试回答模板

> 面试官问：「你们的分布式事务是怎么做的？」

答题模板：

1. **先说场景**："我们的下单流程涉及订单、库存、支付三个服务..."
2. **再说选型理由**："因为是资金相关，需要强一致但又不能阻塞太久，所以选了 TCC..."
3. **讲关键设计**："Try 阶段做冻结、Confirm 做扣款、Cancel 做解冻，通过事务状态表保证幂等..."
4. **提一下踩过的坑**："上线初期遇到过空回滚问题，后来加了 Try 执行标记解决的..."

<details>
<summary><b>展开：面试高频追问集锦</b></summary>

**Q1：2PC 和 TCC 有什么区别？**
- 2PC 是**资源层**方案（数据库协议级别），对业务透明
- TCC 是**业务层**方案（需要业务代码实现 Try/Confirm/Cancel）
- 2PC 有全局锁、性能差；TCC 无锁、性能好但开发成本高

**Q2：Saga 和 TCC 什么时候选哪个？**
- 需要隔离性（不让外部看到中间态）→ TCC
- 事务时间长（跨天/跨小时）→ Saga
- 操作不可逆（如发短信）→ Saga + 人工补偿

**Q3：分布式事务能不能完全避免？**
- 能！很多时候通过**合理的服务拆分**避免跨服务事务
- 或者通过**冗余数据 + 对账**代替实时一致性
- 架构设计的最高境界：让业务流程天然不需要分布式事务

**Q4：CAP 和分布式事务的关系？**
- 2PC/TCC → 偏 CP（追求一致性，牺牲部分可用性/性能）
- Saga/本地消息表 → 偏 AP + 最终一致（BASE 思想）
- 没有银弹，只有适合场景的 trade-off

</details>

## 九、常见框架

### 9.1 Seata

Seata 是阿里开源的分布式事务框架，**一个框架支持四种模式**：

| 模式 | 原理 | 侵入性 | 性能 | 适用场景 |
|------|------|--------|------|---------|
| **AT** | 自动拦截 SQL，生成 undo_log，失败自动回滚 | 极低（加注解即可） | 中 | 大多数 CRUD 场景 |
| **TCC** | 业务实现 Try/Confirm/Cancel 接口 | 高 | 高 | 资金、库存等需要精细控制的场景 |
| **Saga** | 状态机编排正向+补偿事务 | 中 | 高 | 长流程业务 |
| **XA** | 基于数据库 XA 协议的 2PC | 低 | 低 | 强一致要求、短事务 |

**AT 模式核心流程**（最常用）：执行 SQL 前记录 beforeImage → 执行 SQL → 记录 afterImage → 生成 undo_log（和业务数据同一本地事务提交）→ 全局提交时异步删 undo_log / 全局回滚时用 undo_log 反向补偿。

### 9.2 RocketMQ 事务消息

RocketMQ 提供了**事务消息**机制，本质上是"本地消息表"思想的中间件级实现：

```
1. 发送 Half Message（消息暂不可消费）
2. 执行本地事务
3. 根据本地事务结果：
   - 成功 → 发送 Commit（消息可被消费）
   - 失败 → 发送 Rollback（删除消息）
4. 如果迟迟没有 Commit/Rollback → Broker 定时回查本地事务状态
```

相比本地消息表的优势：**不需要自己建消息表、不需要定时扫描，MQ 帮你做了**。

### 9.3 框架对比

| 维度 | Seata | RocketMQ 事务消息 |
|------|-------|-------------------|
| 定位 | 通用分布式事务框架 | 基于消息的最终一致性 |
| 一致性 | AT/XA 强一致；Saga 最终一致 | 最终一致 |
| 性能 | AT 模式有 undo_log 开销；TCC 性能最好 | 高（异步解耦） |
| 复杂度 | 引入 TC（事务协调器）组件 | 仅依赖 MQ |
| 适用场景 | 需要同步确认结果的事务 | 异步可接受、最终一致即可 |

---

## 本篇小结

| 你需要记住的 | 一句话 |
|-------------|--------|
| 分布式事务的本质 | 多个独立服务/数据库的操作如何保持一致 |
| 2PC | 强一致但同步阻塞，协调者单点问题 |
| 3PC | 2PC 的理论改进，工程中不用 |
| TCC | 业务层强一致、无锁高性能，但开发成本高 |
| Saga | 长事务拆成本地事务链+补偿链，最终一致 |
| 本地消息表 | 业务操作+写消息在同一本地事务，简单可靠 |
| 最大努力通知 | 跨公司弱一致场景，多次重试 |
| 选型原则 | 先看一致性要求，再看开发成本和运维复杂度 |
