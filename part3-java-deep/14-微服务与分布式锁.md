# 3.14 微服务治理与分布式锁：从单体到微服务后多出来的麻烦

> **一句话定位**：微服务不是银弹——你把一个单体拆成 20 个服务后，得到了独立部署的自由，却也换来了服务发现、配置管理、分布式锁、链路追踪等一堆"单体时代根本不存在"的新问题。本篇把这些麻烦逐一拆解，重点放在面试高频考点——**分布式锁**。

---

## 一、为什么拆微服务？——好处与代价

你在面试中一定会被问："为什么要用微服务？" 如果你只会说"高内聚低耦合"，那等于没说。真正的答案是一笔**收益与代价的账**。

### 1.1 拆了能得到什么

| 好处 | 具体解释 | 举例 |
|------|---------|------|
| **独立部署** | 改了订单服务，只部署订单，不碰用户服务 | 发版频率从一周一次变一天多次 |
| **技术栈自由** | 推荐系统用 Python，交易系统用 Java，各选最优 | 不用全公司绑死一门语言 |
| **团队自治** | 每个团队 own 一个服务，自己定迭代节奏 | 不用 20 个人合一个 master 分支 |
| **故障隔离** | 推荐服务 OOM 不影响支付 | 单体时代一个模块挂全站挂 |
| **弹性伸缩** | 热点服务单独扩容 | 秒杀只扩库存服务，不用整个单体扩 |

### 1.2 拆了要付什么代价

| 代价 | 痛点 | 单体时代为什么没这问题 |
|------|------|----------------------|
| **网络调用** | 本来一个方法调用，现在变成 HTTP/RPC，延迟 +、失败率 + | 单体里是进程内调用，纳秒级 |
| **数据一致性** | 每个服务有自己的数据库，跨服务事务很难搞 | 单体一个数据库，本地事务搞定 |
| **运维复杂度** | 20 个服务 × 3 个环境 = 60 套部署 | 单体就一个 jar 包 |
| **调试困难** | 一个请求经过 10 个服务，日志分散在 10 台机器 | 单体看一份日志就够 |
| **接口版本管理** | 服务 A 升级接口，服务 B 还在用旧版 | 单体里改了方法签名编译器直接报错 |

### 1.3 拆分原则

**按业务域拆（DDD），不要按技术层拆。**

```
❌ 错误拆法（按技术层）：
    controller-service → service-service → dao-service

✅ 正确拆法（按业务域）：
    order-service / payment-service / inventory-service / user-service
```

按技术层拆的结果是——改一个业务需求要同时改三个服务，比单体还累。按业务域拆，一个需求通常只改一个服务。

<details>
<summary><b>展开：面试追问——什么时候不该拆微服务？</b></summary>

- 团队就 3-5 个人，拆了没人维护
- 业务还在探索期，领域边界不清晰
- 团队没有 DevOps 基础设施（CI/CD、容器化、监控）
- 数据一致性要求极高且跨服务事务频繁

记住一句话：**微服务是组织架构的映射（康威定律），不是技术炫技。**

</details>

## 二、服务注册与发现

### 2.1 核心问题

你有一个 order-service，部署了 5 个实例，IP 和端口随时在变（容器重启、扩缩容）。payment-service 要调 order-service，它怎么知道该往哪个 IP 发请求？

**硬编码 IP 是不行的**——实例随时会增减。你需要一个"电话簿"，这就是**注册中心**。

### 2.2 注册中心流程

```
1. order-service 启动 → 向注册中心注册（IP:Port + 服务名）
2. order-service 每隔 N 秒发心跳（"我还活着"）
3. payment-service 启动 → 拉取 order-service 实例列表，本地缓存
4. payment-service 用负载均衡策略选一个实例调用
5. order-service 下线 → 主动注销 / 心跳超时被剔除
```

### 2.3 注册中心对比

| 特性 | Nacos | Eureka | Consul | ZooKeeper |
|------|-------|--------|--------|-----------|
| **一致性模型** | AP（默认）/ CP（可切换） | AP | CP | CP |
| **健康检查** | TCP/HTTP/MySQL 多种 | 客户端心跳 | 多种（含脚本） | 会话心跳 |
| **推/拉模型** | 推 + 拉结合 | 拉（30s 轮询） | 长轮询（阻塞查询） | Watch 推送 |
| **配置中心** | 自带 | 不支持 | 支持 KV | 勉强能用（不推荐） |
| **Spring Cloud 集成** | 原生支持 | 原生支持 | 需适配 | 需适配 |
| **社区活跃度（2024）** | 高（阿里维护） | 进入维护模式 | HashiCorp 维护 | Apache 维护 |
| **适用场景** | Java 微服务首选 | 老项目沿用 | 多语言/K8s | 强一致场景 |

<details>
<summary><b>展开：面试追问——注册中心挂了怎么办？</b></summary>

答：**本地缓存兜底。**

1. 调用方启动时会把服务列表缓存到本地内存/文件
2. 注册中心挂了 → 调用方用本地缓存继续路由（只是不会更新新实例）
3. 注册中心恢复后 → 增量同步最新列表

所以注册中心挂了不是"立刻全部不可用"，而是"无法感知实例变化"。这也是为什么注册中心通常选 AP 而非 CP——可用性 > 强一致性。

补充：Nacos 还支持**集群部署**（3 节点 Raft），单节点挂了自动选主，客户端无感切换。

</details>

## 三、配置中心

### 3.1 为什么不能把配置写死在 application.yml 里？

| 场景 | 写死配置的问题 | 配置中心怎么解决 |
|------|--------------|----------------|
| **动态修改** | 改个超时时间要重新打包部署 | 配置中心改完秒生效，不用重启 |
| **灰度发布** | 想让 10% 的实例用新配置试试 | 配置中心支持按实例/分组灰度推送 |
| **环境隔离** | dev/test/prod 配置混在一起容易出错 | 配置中心天然隔离 namespace/env |
| **敏感信息** | 数据库密码明文写在代码仓库里 | 配置中心支持加密存储、权限控制 |
| **多实例一致** | 5 个实例改配置要改 5 次 | 配置中心一次修改全部生效 |

### 3.2 主流配置中心对比

| 特性 | Nacos Config | Apollo | Spring Cloud Config |
|------|-------------|--------|-------------------|
| **推送方式** | 长轮询（30s） | 长轮询 + 推送 | Git Webhook + Bus |
| **灰度发布** | 支持 | 完善（按 IP/集群） | 不原生支持 |
| **版本回滚** | 支持 | 支持（带审计） | Git 天然支持 |
| **部署复杂度** | 低（和注册中心同一套） | 高（Portal+Admin+Config） | 低 |
| **国内使用率** | 高 | 高（携程开源） | 中等 |

### 3.3 长轮询推送原理（Nacos 为例）

```
1. 客户端发起 HTTP 请求：/listener?dataId=xxx&group=yyy&md5=当前配置的MD5
2. 服务端收到请求，对比 MD5：
   - 配置没变 → 挂起连接（Hold 住不返回），最长 30s
   - 配置变了 → 立即返回变更的 dataId
3. 客户端收到响应后：
   - 超时返回 → 配置没变，继续下一轮长轮询
   - 有变更 → 主动拉取最新配置 → 通知应用热更新
```

这比纯轮询省资源（没变化不返回），又比 WebSocket 简单（HTTP 无状态，天然兼容负载均衡）。

## 四、分布式锁（面试高频重点）

### 4.1 为什么需要分布式锁？

你在单机 Java 里用 `synchronized` 或 `ReentrantLock` 保护临界区，这没问题——因为所有线程都在**同一个 JVM 进程**里。

但微服务环境下，同一个服务部署了 3 个实例，跑在 3 台机器上。用户下单时要扣减库存：

```java
// 实例 A 和实例 B 同时执行这段代码
int stock = redis.get("stock");  // 都读到 stock=1
if (stock > 0) {
    redis.decr("stock");         // 都扣成功 → 超卖！
}
```

`synchronized` 只能锁住**本 JVM 内的线程**，管不了别的机器上的实例。你需要一把所有实例都能看到、都能争抢的锁——这就是**分布式锁**。

分布式锁的三个核心要求：
1. **互斥性**：同一时刻只有一个客户端持有锁
2. **无死锁**：持锁客户端崩溃后，锁能自动释放
3. **容错性**：锁服务部分节点挂了，锁依然可用

---

### 4.2 Redis 分布式锁

#### 加锁：SET NX EX（一条命令原子完成）

```bash
SET lock:order:1001 "uuid-of-instance-A" NX EX 30
# NX = 只在 key 不存在时才设置（互斥）
# EX 30 = 30 秒过期（防死锁）
# value 存唯一标识（释放时验证身份）
```

命令拆解：`lock:order:1001` 是 key，`uuid-of-instance-A` 是 value。key 的设计体现了加锁粒度——`lock:` 是命名空间前缀（区分不同业务用途），`order` 是业务域，`1001` 是订单 ID，加锁粒度是"订单级别"，订单 1002 用 `lock:order:1002` 互不影响。value 存的是实例的唯一标识（UUID），作用是释放锁时验证身份——只有 value 匹配的实例才能删掉这个 key，防止锁过期后被别的实例拿走，原持锁实例恢复后误删了新实例的锁。

为什么 key 不需要引号而 value 需要引号？这是 Redis 命令解析规则决定的。Redis 的命令格式是 `COMMAND arg1 arg2 ...`，用空格分隔参数。key 通常是单个连续的标识符（不含空格），不需要引号。value 如果包含空格或特殊字符，不引号的话 Redis 会把空格后的部分当成下一个参数，导致命令解析错误。`"uuid-of-instance-A"` 其实不含空格，加引号是防御性写法——如果 value 换成 `instance A`（带空格），不加引号就会被解析成两个参数。实际上 Redis 3.x+ 的 value 也不强制要求引号（不含空格时），但加引号是更安全的习惯。在代码中（如 Java 的 Jedis/Lettuce 客户端）调用时传的是字符串对象，不存在引号问题，引号只是 redis-cli 命令行交互时的需要。

为什么不能先 SETNX 再 EXPIRE？因为两条命令不是原子的——SETNX 成功后崩溃了，EXPIRE 没执行，锁永远不过期 → 死锁。

#### 释放锁：Lua 脚本判断是不是自己的锁

```lua
-- 释放锁的 Lua 脚本（Redis 保证原子执行）
-- KEYS[1] = lock:order:1001（锁的 key）
-- ARGV[1] = uuid-of-instance-A（当前实例的唯一标识，即加锁时存的 value）
-- 逻辑：先 GET 取出当前 value，跟自己的 UUID 比较，匹配才 DEL，不匹配就放弃
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

为什么不能直接 DEL？因为可能你的锁已经过期、被别人拿到了，你一个 DEL 把别人的锁删了 → 互斥性被破坏。value 存的是实例的唯一标识（UUID），作用是释放锁时验证身份——只有 value 匹配的实例才能删掉这个 key，防止实例 A 的锁过期后被实例 B 拿走，然后实例 A 恢复后误删了实例 B 的锁。如果不做验证直接 DEL，就会出现这种"误删别人锁"的问题。这个"先 GET 比较 value 再 DELETE"的两步操作必须用 Lua 脚本保证原子性——如果先 GET 再 DEL 是两条独立命令，中间可能发生锁过期切换，两条命令就不是原子的了，Lua 脚本让 Redis 把整个逻辑作为单条命令原子执行。

#### 问题：业务执行时间 > 锁过期时间

你设了 30 秒过期，但业务处理了 35 秒（GC 停顿、网络抖动等）。第 30 秒锁自动释放，别的实例拿到锁进来了 → 并发冲突。

**解法：Redisson 看门狗（Watchdog）续期机制**

```
1. 加锁成功后，Redisson 启动一个后台线程（看门狗）
2. 看门狗每隔 lockWatchdogTimeout/3（默认 10 秒）检查一次
3. 如果业务还没执行完（锁还被持有）→ 续期到 30 秒
4. 业务执行完主动释放锁 → 看门狗停止
5. 客户端宕机 → 看门狗也死了 → 不会续期 → 锁自动过期释放
```

#### 问题：Redis 主从切换时锁丢失

```
1. 客户端 A 在 master 上加锁成功
2. master 还没把这个 key 同步给 slave 就宕机了
3. slave 升级为新 master
4. 客户端 B 在新 master 上加锁成功（key 不存在）
5. A 和 B 同时持有锁 → 互斥性被破坏
```

**解法：RedLock 算法**

向 N 个（通常 5 个）独立的 Redis 实例分别加锁，只有在**大多数（N/2+1）**实例上加锁成功，才认为拿到了分布式锁。

<details>
<summary><b>展开：面试追问——RedLock 的争议？Martin Kleppmann 怎么说的？</b></summary>

Martin Kleppmann（《Designing Data-Intensive Applications》作者）在 2016 年写了一篇文章 *"How to do distributed locking"*，质疑 RedLock：

**Kleppmann 的核心论点**：

1. **GC 停顿问题**：客户端拿到锁后发生长时间 GC，锁过期了它还以为自己持有锁，继续操作共享资源
2. **时钟跳跃问题**：RedLock 依赖各节点的时钟大致同步，如果某个节点时钟跳跃，锁的过期时间计算就不准确
3. **他的替代方案**：用 **fencing token**（单调递增的令牌），每次加锁返回一个递增数字，存储层拒绝旧 token 的写入

**Antirez（Redis 作者）的回应**：

1. GC 停顿问题对所有分布式锁都存在，不是 RedLock 特有的
2. 现代 NTP 时钟偏差在毫秒级，不会造成实际问题
3. Fencing token 需要存储层支持，不是所有场景都能改造

**面试结论**：

- 如果你的场景对正确性要求极高（如金融扣款）→ 用 ZooKeeper 分布式锁 + fencing token
- 如果你的场景对性能要求高、容忍极低概率的不一致（如防重复提交）→ Redis 单节点锁就够了
- RedLock 处于中间地带，实际生产中用的不多（部署 5 个独立 Redis 成本太高）

</details>

### 4.3 ZooKeeper 分布式锁

#### 实现原理：临时顺序节点 + Watch

```
1. 客户端在 /locks/order-1001 下创建临时顺序节点：
   → /locks/order-1001/lock-0000000001
   → /locks/order-1001/lock-0000000002
   → /locks/order-1001/lock-0000000003

2. 客户端获取 /locks/order-1001 下所有子节点，排序

3. 判断自己创建的节点是不是序号最小的：
   - 是 → 拿到锁，执行业务
   - 不是 → Watch 前一个节点的删除事件（避免惊群）

4. 前一个节点被删除 → 收到通知 → 重新判断自己是不是最小的

5. 业务执行完 → 删除自己的临时节点 → 下一个人拿到锁
   客户端宕机 → 会话超时 → 临时节点自动删除 → 下一个人拿到锁
```

#### 为什么 Watch 前一个而不是 Watch 父节点？

如果所有等待者都 Watch 父节点，一个锁释放后所有人都被唤醒，但只有一个能拿到锁 → **惊群效应**，浪费资源。Watch 前一个节点 = 排队，一次只唤醒一个 → 公平锁。

#### 优缺点

| 维度 | 评价 |
|------|------|
| **无过期时间问题** | 临时节点随会话消亡，不需要手动设过期时间 |
| **强一致性** | ZooKeeper 是 CP 系统，不会出现 Redis 主从切换丢锁 |
| **可重入** | Curator 框架封装了可重入锁 |
| **性能差** | 每次加/释放锁都要 ZK 集群达成共识（Zab 协议），比 Redis 慢一个数量级 |
| **运维重** | ZK 集群本身就需要维护 |

### 4.4 MySQL 分布式锁

#### 方式一：SELECT ... FOR UPDATE

```sql
BEGIN;
SELECT * FROM distributed_lock WHERE lock_name = 'order-1001' FOR UPDATE;
-- 拿到行锁，执行业务逻辑
-- ...
COMMIT;  -- 释放锁
```

#### 方式二：基于唯一索引 INSERT

```sql
-- 加锁：插入成功 = 拿到锁
INSERT INTO distributed_lock (lock_name, owner, expire_time)
VALUES ('order-1001', 'instance-A', NOW() + INTERVAL 30 SECOND);

-- 释放锁：删除记录
DELETE FROM distributed_lock WHERE lock_name = 'order-1001' AND owner = 'instance-A';
```

#### 为什么一般不推荐？

| 问题 | 说明 |
|------|------|
| **性能差** | 数据库锁是重量级的，涉及磁盘 IO |
| **连接占用** | FOR UPDATE 期间一直占着数据库连接 |
| **单点问题** | 数据库本身成为瓶颈和单点 |
| **无自动过期** | 客户端宕机后，FOR UPDATE 的锁要等事务超时才释放 |
| **不可重入** | 需要自己实现 |

适用场景：并发量极低、已有 MySQL 不想引入新组件、对性能无要求的管理后台场景。

### 4.5 分布式锁对比总表

| 方案 | 性能 | 可靠性 | 实现复杂度 | 适用场景 |
|------|------|--------|-----------|---------|
| **Redis 单节点** | 极高（10 万+/s） | 中（主从切换可能丢锁） | 低 | 高并发、容忍极低概率不一致 |
| **RedLock** | 高 | 中高（有争议） | 中 | 需要较高可靠性但又要性能 |
| **ZooKeeper** | 中（万级/s） | 高（强一致） | 中（Curator 封装好了） | 对正确性要求极高 |
| **MySQL** | 低（千级/s） | 中 | 低 | 并发低、不想引入新组件 |
| **etcd** | 中高 | 高（Raft 共识） | 中 | K8s 生态/Go 技术栈 |

<details>
<summary><b>展开：面试万能回答模板——"你会怎么选分布式锁？"</b></summary>

```
1. 先看并发量：
   - 高并发（万级 QPS 以上）→ Redis
   - 中低并发 → ZooKeeper / etcd

2. 再看一致性要求：
   - 容忍极低概率不一致（如防重复提交）→ Redis 单节点
   - 必须强一致（如库存扣减、金融）→ ZooKeeper

3. 最后看技术栈：
   - 已有 Redis → 直接用 Redisson
   - 已有 ZooKeeper → 直接用 Curator
   - 什么都没有 → Redis 最简单

4. 加一句兜底：
   "实际项目中我们用的是 Redisson，看门狗续期 + Lua 脚本释放，
    cover 了 99.9% 的场景。极端情况用业务层幂等兜底。"
```

</details>

## 五、链路追踪

### 5.1 为什么需要链路追踪

单体应用出了 Bug，打开一份日志从头到尾看就行。微服务环境下：

```
用户下单 → API Gateway → Order Service → Inventory Service → Payment Service
                                       → Coupon Service
                                       → Notification Service
```

一个请求跨了 6 个服务，响应慢了，到底卡在哪一段？哪个服务抛了异常？没有链路追踪，你只能挨个服务翻日志。

### 5.2 核心概念

| 概念 | 解释 |
|------|------|
| **Trace** | 一次完整请求的调用链（类比：快递全程物流单） |
| **Span** | 一个服务内的一段处理（类比：某中转站停留记录） |
| **TraceId** | 全局唯一标识，贯穿整条链路（类比：快递单号） |
| **SpanId / ParentSpanId** | 当前 Span 和父 Span 的 ID，形成调用树 |

### 5.3 实现原理

```
1. 入口服务生成 TraceId，放入请求 Header
2. 下游服务取出 TraceId，创建自己的 Span，记录开始/结束时间
3. 调用再下游时，传递 TraceId + 当前 SpanId（作为 ParentSpanId）
4. 所有 Span 异步上报 → 追踪后端按 TraceId 聚合还原调用树
```

### 5.4 工具：SkyWalking / Zipkin / Jaeger

| 特性 | SkyWalking | Zipkin | Jaeger |
|------|-----------|--------|--------|
| **侵入性** | Java Agent（无侵入） | 需埋点 | 需埋点 |
| **性能开销** | 低（字节码增强） | 中 | 中 |
| **国内使用率** | 高（Apache 顶级项目） | 中 | 中 |

### 5.5 与日志系统联动

最佳实践：把 TraceId 注入到日志 MDC 中，每条日志都带上 TraceId。

```java
// Logback 配置
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%X{traceId}] [%thread] %-5level %logger - %msg%n</pattern>
```

这样出问题时：在追踪系统里找到慢的 Span → 拿到 TraceId → 去日志系统搜这个 TraceId → 看到详细报错。

## 六、服务网关

### 6.1 为什么需要网关

没有网关时，客户端要直接调 20 个微服务，每个服务都要自己实现鉴权、限流、日志。网关 = **统一入口**，把这些横切关注点集中处理。

| 能力 | 说明 |
|------|------|
| **路由** | /api/order/** → order-service，/api/user/** → user-service |
| **鉴权** | 统一校验 Token，不合法直接拒绝，不用每个服务都写一遍 |
| **限流** | 统一限流策略，保护后端服务不被打爆 |
| **灰度** | 根据 Header/Cookie 把部分流量路由到新版本 |
| **协议转换** | 外部 HTTP → 内部 gRPC |
| **日志/监控** | 统一记录请求日志和指标 |

### 6.2 Spring Cloud Gateway vs Nginx

| 维度 | Spring Cloud Gateway | Nginx |
|------|---------------------|-------|
| **定位** | 业务网关（L7） | 流量网关 / 反向代理 |
| **动态路由** | 原生集成注册中心 | 需 OpenResty + Lua |
| **性能** | 中高（受限于 JVM） | 极高（C 写的） |
| **适用** | Java 微服务体系 | 全公司统一入口/静态资源 |

实际架构常见两层：**Nginx（流量网关）→ Spring Cloud Gateway（业务网关）→ 微服务**。

### 6.3 过滤器链

Spring Cloud Gateway 的核心设计是 **Filter Chain**（Pre Filter → 路由 → Post Filter）。常见 Filter：鉴权（Pre，校验 JWT）、限流（Pre，令牌桶）、日志（Pre+Post）、响应改写（Post，统一格式包装）。

## 七、服务间通信

### 7.1 同步 vs 异步对比

| 维度 | 同步（Feign/gRPC/Dubbo） | 异步（消息队列，详见 [第 12 章](./12-消息队列.md)） |
|------|--------------------------|----------------------------------------------|
| **延迟** | 低（毫秒级） | 较高（取决于消费速度） |
| **吞吐** | 受调用链最慢节点限制 | 削峰填谷，吞吐高 |
| **耦合度** | 强耦合（需知道被调方地址） | 松耦合（通过 Topic 解耦） |
| **适用场景** | 查询类、需即时响应 | 通知类、可容忍延迟、削峰 |
| **失败处理** | 调用方直接感知异常 | 需要重试 + 死信队列 |

### 7.2 怎么选？

**需要立刻拿到结果的用同步，不需要的用异步。** 下单流程里扣库存必须同步（成功才能继续），发短信通知可以异步（晚几秒无所谓）。

| 主题 | 一句话记忆 |
|------|----------|
| 服务拆分 | 按业务域拆，不按技术层拆 |
| 注册发现 | 注册中心本质是一个「服务地址电话簿」 |
| 配置中心 | 让配置「活」起来，不用重启就能改 |
| 分布式锁 | Redis 快但有主从问题，ZK 可靠但慢，选型看场景 |
| 链路追踪 | TraceId 贯穿全链路，出问题一查便知 |
| 服务网关 | 统一入口，鉴权限流路由一把抓 |
| 服务通信 | 同步求快、异步求稳，按场景混用 |

## 八、Kubernetes 与容器编排：微服务的运行时基础设施

微服务拆分后需要部署到分布式环境——20 个服务 × 3 个实例 = 60 个进程需要管理。手动 SSH 部署不现实，Kubernetes（K8s）是事实标准的容器编排平台。

### 8.1 K8s 核心架构

K8s 是一个分布式系统，分 Control Plane（控制面）和 Data Plane（数据面，也叫 Worker Node）。控制面负责全局决策——API Server（唯一入口，所有操作经过它）、etcd（一致性 KV 存储，存集群状态，用 Raft 保证一致性，参见 3.10）、Scheduler（决定 Pod 调度到哪个节点）、Controller Manager（控制器集合，驱动实际状态向期望状态收敛）。数据面运行实际业务——Kubelet（与 API Server 通信，管理本节点 Pod 生命周期）、Kube-Proxy（维护网络规则，实现 Service 负载均衡）、Container Runtime（containerd/CRI-O 运行容器）。

Pod 是 K8s 最小调度单位，一个 Pod 可以包含多个容器（通常一个主容器 + sidecar 容器）。跟 Docker 的区别——Docker 管理单机容器，K8s 管理跨节点容器集群（调度、扩缩容、自愈、服务发现）。

### 8.2 调度框架（Scheduler Framework）

K8s 默认调度器的三阶段流水线：Filter（过滤不满足条件的节点——资源不足、GPU 不匹配、污点不容忍）、Score（对剩余节点打分——资源均衡、亲和性、反亲和性）、Bind（绑定 Pod 到得分最高的节点）。

自定义调度策略有三种方式。第一种是 Scheduler Framework 插件（K8s 1.19+）——在 Filter/Score/Bind 各阶段注册自定义插件，不影响默认调度逻辑。第二种是 PriorityClass——定义优先级等级，高优先级 Pod 可以抢占低优先级 Pod 的资源（被抢占的 Pod 会被安全终止并在其他节点重新调度）。第三种是自定义调度器——完全独立的调度器进程，跟默认调度器并存，通过 SchedulerName 字段区分哪些 Pod 用自定义调度器。

### 8.3 Job 批量调度

Deployment 适合长期运行的服务（Web API、微服务），Job 适合短生命周期的批处理任务（数据处理、仿真、模型训练）。Job 关键参数：completions（完成多少个 Pod 算 Job 成功）、parallelism（最大并发 Pod 数）、backoffLimit（失败重试次数）、ttlSecondsAfterFinished（完成后自动清理 Pod）。

CronJob 是定时执行的 Job——用 Cron 表达式定义调度周期。适合数据仓库的定时 ETL 任务、模型定时训练。

### 8.4 Operator 模式

Operator 是"用代码封装运维知识"的模式。核心思想：定义自定义资源（CRD，Custom Resource Definition），编写 Controller 监听 CRD 变化，自动执行运维操作。

Operator 的工作机制是 Informer + Reconcile Loop。Informer 监听 API Server 上 CRD 的变更（增删改），触发 Reconcile Loop——对比当前实际状态（Actual State）和期望状态（Desired State），执行操作使实际状态向期望状态收敛。比如定义 `SimulationTask` CRD，用户提交一个 `SimulationTask` 资源，Controller 监听到后自动创建 K8s Job、注入场景数据、拉起 Docker 镜像、收集结果。

### 8.5 弹性伸缩

HPA（Horizontal Pod Autoscaler）——根据指标水平扩缩容 Pod 数量。HPA Controller 每隔 15 秒（默认）从 Metrics Server 拉取各 Pod 的 CPU/内存指标，跟目标阈值比较，计算需要的 Pod 数量，修改 Deployment 的 replicas 字段。默认基于 CPU 利用率（如目标 60%），可配置自定义指标（如 Kafka 消费积压深度、HTTP QPS、GPU 利用率），需要部署 Prometheus Adapter 把 Prometheus 指标暴露给 K8s API。扩容过程通常 30-60 秒（拉镜像 + 启动应用）。缩容更慢——默认 5 分钟稳定窗口（stabilization window），即指标降下来后等 5 分钟确认持续低负载才缩，防止流量波动导致频繁扩缩容（抖动）。

VPA（Vertical Pod Autoscaler）——调整 Pod 的 CPU/内存 requests/limits。有三种模式：Recommend（只给建议不自动改，适合观察阶段）、Auto（自动改配置，但改时要重建 Pod——因为资源配置创建后不能动态修改）、Initial（只在 Pod 创建时应用推荐值，不修改已运行 Pod，适合不想中断的场景）。局限——改配置要重建 Pod 对有状态服务不友好，且 VPA 和 HPA 不能同时用同一个指标（会互相冲突）。

Cluster Autoscaler——节点级扩容。当 Pod 因资源不足处于 Pending 状态时，自动向云平台申请新节点加入集群。跟 HPA 配合使用——HPA 扩 Pod 数量，Cluster Autoscaler 扩节点数量。

选型实践——无状态服务用 HPA（扩 Pod 数量，扩容不中断），有状态服务用 VPA（调 Pod 配额，但需忍受重建中断），批处理任务用 Job 的 parallelism 参数控制并发数（不适用 HPA/VPA）。三者按服务类型分别使用，互不冲突。

### 8.6 K8s 与微服务的关系

K8s 解决了微服务架构中"部署和运维"层面的痛点。服务发现——K8s Service 自带 DNS 解析和负载均衡，不需要额外的注册中心（如 Eureka/Consul）。配置管理——ConfigMap + Secret 挂载为环境变量或文件，不需要额外的配置中心。弹性伸缩——HPA 自动扩缩容，不需要手动操作。故障自愈——Kubelet 定期健康检查，Pod 崩溃自动重启，节点故障自动重新调度。但 K8s 不解决"业务逻辑"层面的问题——分布式锁、分布式事务、链路追踪的业务逻辑仍需自己实现。

## 九、可观测性三支柱：Metrics / Logging / Tracing

第三章 3.5 的链路追踪是可观测性的一个维度。完整的可观测性体系需要三支柱协同——微服务架构下一个请求跨 10 个服务，只看单一维度无法定位问题。

### 9.1 Metrics（指标监控）

Metrics 回答"系统在发生什么"——量化的聚合数据。用 Prometheus + Grafana 技术栈。Prometheus 的数据模型是时序数据库——每个指标是 `metric_name{label1=val1, label2=val2} value timestamp`。Pull 模型——Prometheus 主动抓取 Exporter 暴露的 /metrics 端点，适合容器环境（每个 Pod 暴露指标端点）。

核心指标分四类（USE 方法论和 RED 方法论的结合）：Traffic（请求量/QPS/任务提交速率）、Latency（P50/P99 响应延迟/任务执行时长）、Errors（错误率/HTTP 5xx 比例/任务失败率）、Saturation（CPU/内存/GPU 利用率/队列积压深度）。

### 9.2 Logging（日志聚合）

Logging 回答"具体发生了什么"——离散的事件记录。用 ELK（Elasticsearch + Logstash + Kibana）或 Loki。结构化日志（JSON 格式）是核心实践——包含 trace_id（关联 Tracing）、task_id（关联业务实体）、service_name（服务名）、level（DEBUG/INFO/WARN/ERROR）。日志采集用 Fluent Bit 或 Filebeat，DaemonSet 方式在每个节点收集容器 stdout/stderr。

ELK vs Loki 的选型——ELK 全文检索能力强（基于 Elasticsearch 倒排索引，参见 A5），但资源开销大（每个节点都需要 Logstash + Elasticsearch）。Loki 是 Grafana 生态的轻量日志系统，只索引标签不索引内容（类似 Prometheus 的时序模型），存储成本低但检索能力弱于 ELK。高流量场景用 Loki 省钱，复杂检索场景用 ELK。

### 9.3 Tracing（分布式追踪）

Tracing 回答"一个请求经过了哪些服务、卡在哪里"——跨服务的调用链。用 Jaeger 或 OpenTelemetry。一个 Trace 由多个 Span 组成，每个 Span 记录服务名、操作名、开始时间、持续时间、tags（业务 ID、资源 ID）和上下文传播（TraceContext 通过 HTTP 头/消息队列 header 在服务间传递）。

Tracing 的核心价值是定位"慢在哪里"——看 Trace 的 Span 树，耗时最长的 Span 就是瓶颈。然后查该 Span 对应服务的 Metrics（确认是资源瓶颈还是代码问题）和 Logs（查具体的错误日志），三支柱协同。

### 9.4 三支柱关联

三支柱通过 trace_id 关联——Metrics 中的 label 可以包含 trace_id（通过 exemplar 技术），Logs 中包含 trace_id，Tracing 的 Span 中包含 trace_id。生产环境的排障流程：告警触发（Metrics 发现异常）→ 查看 Tracing 找到慢 Span → 查看该 Span 对应的 Logs 确认具体错误 → 修复问题。
