# 附录 A6：代码规范与设计原则——写出能活过三年的代码

> **一句话定位**：代码写给机器执行，但更是写给**三个月后的自己和接手你项目的同事**看的。本附录系统梳理 SOLID 设计原则和 Java 编码规范要点，帮你从"能跑"升级到"好维护"。

---

## 一、SOLID 原则——面向对象设计的五条军规

SOLID 是 Robert C. Martin（Uncle Bob）提出的五个面向对象设计原则，是 [3.17 设计模式](./17-设计模式.md) 的理论地基。

### 1.1 S — 单一职责原则（Single Responsibility Principle）

**一句话**：一个类应该只有一个引起它变化的原因。

```java
// ❌ 违反：UserService 既管业务逻辑又管发邮件
public class UserService {
    public void register(User user) { /* 注册逻辑 */ }
    public void sendEmail(User user, String content) { /* 发邮件 */ }
}

// ✅ 遵循：拆分职责
public class UserService {
    private final EmailService emailService;
    public void register(User user) {
        // 注册逻辑...
        emailService.send(user.getEmail(), "欢迎");
    }
}
public class EmailService {
    public void send(String to, String content) { /* 发邮件 */ }
}
```

### 1.2 O — 开闭原则（Open-Closed Principle）

**一句话**：对扩展开放，对修改关闭——加新功能不改老代码。

```java
// ❌ 违反：加新支付方式要改 if-else
public BigDecimal calculate(String type, BigDecimal amount) {
    if ("alipay".equals(type)) { return amount.multiply(new BigDecimal("0.98")); }
    else if ("wechat".equals(type)) { return amount.multiply(new BigDecimal("0.99")); }
    // 每加一种支付方式就要改这里...
}

// ✅ 遵循：策略模式
public interface PaymentStrategy { BigDecimal calculate(BigDecimal amount); }
// 新增支付方式只需加一个实现类，不改现有代码
```

### 1.3 L — 里氏替换原则（Liskov Substitution Principle）

**一句话**：子类必须能替代父类而不破坏程序正确性。

经典反例：正方形继承长方形。`Rectangle.setWidth(5)` 不影响高度，但 `Square.setWidth(5)` 同时改了高度——子类替换父类后行为变了，违反 LSP。

### 1.4 I — 接口隔离原则（Interface Segregation Principle）

**一句话**：接口要小而精，不要强迫实现者实现用不到的方法。

```java
// ❌ 违反：一个巨大的接口
public interface Animal {
    void fly();
    void swim();
    void run();
}
// Dog 被迫实现 fly()，只能抛 UnsupportedOperationException

// ✅ 遵循：拆分接口
public interface Flyable { void fly(); }
public interface Swimmable { void swim(); }
public interface Runnable { void run(); }
```

### 1.5 D — 依赖倒置原则（Dependency Inversion Principle）

**一句话**：高层模块不依赖低层模块，都依赖抽象。

这就是 Spring IOC 的理论基础（详见 [3.13 Spring 全家桶](./13-Spring全家桶.md)）。Service 不直接 `new MySQLDao()`，而是依赖 `Dao` 接口，由容器注入具体实现。换数据库只改配置，不改业务代码。

## 二、Java 编码规范精选——阿里巴巴规范要点

以下摘自《阿里巴巴 Java 开发手册》中最有价值的规则，按"最容易违反"排序。

### 2.1 命名规范

| 规则 | 正例 | 反例 |
|------|------|------|
| 类名 UpperCamelCase | `UserService` | `userService`、`User_Service` |
| 方法/变量 lowerCamelCase | `getUserById` | `GetUserById`、`get_user` |
| 常量全大写下划线分隔 | `MAX_RETRY_COUNT` | `maxRetryCount` |
| 布尔变量不加 is 前缀 | `boolean active` | `boolean isActive`（某些框架序列化会出问题） |
| 抽象类用 Abstract/Base 开头 | `AbstractHandler` | `Handler`（分不清是接口还是抽象类） |
| 异常类用 Exception 结尾 | `BusinessException` | `BusinessError` |
| 包名全小写 | `com.example.service` | `com.example.Service` |

### 2.2 异常处理规范

```java
// ❌ 吞掉异常
try { /* ... */ } catch (Exception e) { }  // 出了问题永远不知道

// ❌ 用 e.printStackTrace()
try { /* ... */ } catch (Exception e) { e.printStackTrace(); }  // 线上没人看控制台

// ✅ 用日志框架记录
try { /* ... */ } catch (Exception e) {
    log.error("处理订单失败, orderId={}", orderId, e);  // 带上下文 + 异常堆栈
    throw new BusinessException("订单处理失败", e);       // 包装后重新抛出
}
```

### 2.3 集合使用规范

```java
// ❌ 遍历时删除元素
for (String item : list) {
    if ("delete".equals(item)) list.remove(item);  // ConcurrentModificationException
}

// ✅ 用 Iterator 或 removeIf
list.removeIf("delete"::equals);

// ❌ Arrays.asList() 返回的 List 不能修改
List<String> list = Arrays.asList("a", "b");
list.add("c");  // UnsupportedOperationException

// ✅ 用 new ArrayList<> 包装
List<String> list = new ArrayList<>(Arrays.asList("a", "b"));
```

### 2.4 并发规范

```java
// ❌ 直接用 Executors 创建线程池
ExecutorService pool = Executors.newFixedThreadPool(10);
// FixedThreadPool 和 SingleThreadPool 的队列是 Integer.MAX_VALUE → OOM
// CachedThreadPool 的最大线程数是 Integer.MAX_VALUE → 创建过多线程

// ✅ 用 ThreadPoolExecutor 明确参数
ExecutorService pool = new ThreadPoolExecutor(
    5, 10, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),      // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 明确拒绝策略
);
```

### 2.5 日志规范

```java
// ❌ 字符串拼接（即使不输出也会执行拼接）
log.debug("Processing order: " + order.toString());

// ✅ 占位符（只在日志级别匹配时才拼接）
log.debug("Processing order: {}", order);

// 日志级别使用原则
log.error("...");   // 影响业务的错误，需要立即处理
log.warn("...");    // 可预期的异常分支，需要关注但不紧急
log.info("...");    // 关键业务流程节点（订单创建、支付成功等）
log.debug("...");   // 开发调试信息，线上不开启
```

## 三、代码坏味道速查

| 坏味道 | 症状 | 解药 |
|--------|------|------|
| 过长方法 | 方法超过 80 行 | 提取方法（Extract Method） |
| 过长参数列表 | 方法参数超过 5 个 | 封装为参数对象 / Builder |
| 重复代码 | 两处以上相同逻辑 | 提取公共方法 / 模板方法 |
| 过大的类 | 一个类超过 500 行 | 按职责拆分（SRP） |
| 魔法数字 | `if (status == 3)` | 用常量或枚举 `OrderStatus.PAID` |
| 深层嵌套 | 4 层以上 if/for 嵌套 | 提前返回（Guard Clause）/ 提取方法 |
| 上帝类 | 一个类什么都管 | 按领域拆分 |

## 四、面试深度剖析

### 考点 1：SOLID 原则

> **面试官**：「说说你对 SOLID 原则的理解，实际开发中怎么应用？」

挑 2-3 个结合实际项目说。比如开闭原则：我们用策略模式 + Spring 自动注入来处理不同类型的订单计算，新增类型只需加一个 `@Component`，不改主流程代码。依赖倒置：Service 层依赖 DAO 接口而非实现，单元测试时可以注入 Mock。

### 考点 2：代码规范

> **面试官**：「你们团队有什么代码规范？怎么落地的？」

参考阿里巴巴规范，用 Checkstyle/SpotBugs 做静态检查集成到 CI，Code Review 时重点关注命名、异常处理、集合使用、线程池配置。
