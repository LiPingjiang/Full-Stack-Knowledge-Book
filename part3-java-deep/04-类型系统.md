# 3.4 Java 类型系统：泛型、类型擦除与它们的坑

> 呼应 [1.3 从强类型到类型光谱](../part1-mindset-shift/03-从强类型到类型光谱.md)：Java 是「静态 + 强类型」的代表。
> 这一节深挖 Java 类型系统里最容易「会用但说不清」的部分——**泛型与类型擦除**，它是后面对比 TS、Rust 类型系统的基准。

---

## 一、泛型解决了什么：从 `Object` 到类型安全

回想没有泛型的年代（JDK 5 之前），集合里存的都是 `Object`，取出来要强转，编译器帮不了你：

```java
// 没有泛型的黑暗年代
List list = new ArrayList();
list.add("hello");
list.add(123);                      // 编译器不拦，啥都能塞
String s = (String) list.get(1);    // 运行时炸：ClassCastException！
```

泛型（Generics）的价值，就是**把「类型检查」从运行时提前到编译时**——这正是 [1.3](../part1-mindset-shift/03-从强类型到类型光谱.md) 讲的静态类型的核心红利：

```java
// 有泛型
List<String> list = new ArrayList<>();
list.add("hello");
list.add(123);          // 编译期直接报错！根本通不过编译
String s = list.get(0); // 不用强转，编译器知道里面是 String
```

泛型让你能写出**类型参数化的通用代码**，同时保持类型安全：

```java
// 一个通用的"盒子"，T 是类型参数
class Box<T> {
    private T content;
    public void put(T item) { this.content = item; }
    public T get() { return content; }
}

Box<String> sb = new Box<>();   // T = String
Box<Integer> ib = new Box<>();  // T = Integer，复用同一份代码
```

## 二、类型擦除：Java 泛型的「真相」与坑

这是 Java 泛型最反直觉、也最该搞懂的一点：**Java 的泛型是「编译期的把戏」，运行时泛型信息会被「擦除」。** 这叫**类型擦除（Type Erasure）**。

编译器用泛型做完类型检查后，会把类型参数「擦掉」，换成 `Object`（或边界类型）。所以运行时，`Box<String>` 和 `Box<Integer>` **其实是同一个类 `Box`**：

```java
Box<String> sb = new Box<>();
Box<Integer> ib = new Box<>();

System.out.println(sb.getClass() == ib.getClass());  // true！运行时是同一个类
```

为什么这么设计？**历史包袱**：泛型是 JDK 5 才加的，为了**兼容老代码**（让带泛型和不带泛型的代码能共存），选择了「编译期检查、运行时擦除」的方案。

类型擦除带来一串「坑」，这些都是面试和实战的高频点：

```java
// 坑1：运行时拿不到泛型类型，不能 new 泛型对象
class Box<T> {
    T create() {
        return new T();   // 编译错误！运行时不知道 T 是什么，没法 new
    }
}

// 坑2：不能用 instanceof 判断泛型类型
if (list instanceof List<String>) {}   // 编译错误！运行时 <String> 已被擦除

// 坑3：泛型数组不能直接创建
T[] arr = new T[10];   // 编译错误

// 坑4：重载冲突——擦除后签名相同
void f(List<String> a) {}
void f(List<Integer> a) {}   // 编译错误！擦除后都是 f(List)，签名重复
```

<details>
<summary><b>展开：关于「擦除」最容易误解的三件事（擦成什么 / 是不是替换成真实类型 / 是不是泛型专属）</b></summary>

这三个问题是层层递进的认知陷阱，很多人「会用泛型」但在这三处栽跟头。

**误区一：擦除是「把 T 替换成你写的真实类型」吗？——不是。**

擦除**不是**把 `T` 替换成你 new 时写的 `String`/`Integer`，而是把 `T` 替换成它的**上界**：无边界就是 `Object`，`<T extends Number>` 就是 `Number`。你写的真实类型实参（`<String>`）只在**编译期**用于类型检查，检查完就丢，运行时一点不剩。编译器同时帮你做两件事：① 类型参数擦成上界；② 在取值处**自动插入强制转换**。

```java
// 你写的
List<String> list = new ArrayList<>();
list.add("hi");
String s = list.get(0);

// 编译后字节码等价于（<String> 没了，get 返回 Object，编译器补强转）
List list = new ArrayList();
list.add("hi");                  // 实际签名 add(Object)
String s = (String) list.get(0); // 这个 (String) 是编译器替你插的
```

> 大白话：泛型像**报关单**。过海关（编译期）时官员严格核对你申报的货物类型；核对完单子就撕了，货物装箱（运行时）只剩一个写着「Object」的通用箱子，没人知道里面本来报的是 String 还是 Integer。

**误区二：只有 `List<T>` 里的 `T` 会擦，`List<String>` 不会吧？——都擦，只是结果不同。**

擦除作用在「泛型类型实参」这个语法成分上，**无论实参是类型变量 `T` 还是具体类型 `String`，运行时都不保留**：

- 类定义里的 `class Box<T>` 的 `T` → 擦成上界 `Object`；
- 使用处的 `List<String>` 的 `<String>` → 整个尖括号信息丢掉，变成裸 `List`，取值处补 `(String)` 强转。

这正是 `new ArrayList<String>().getClass() == new ArrayList<Integer>().getClass()` 为 `true` 的原因——运行时它俩就是同一个 `ArrayList`。

**误区三：擦除是通用的底层机制，非泛型也会被擦吧？——要分两层看。**

- 按 JLS（Java 语言规范）的**正式术语「类型擦除（Type Erasure）」**：它专指「参数化类型 → 裸类型、类型变量 → 上界」这套规则，**输入只有泛型语法**（类型变量、类型实参、通配符）。从这个角度，**类型擦除是泛型专属，非泛型不涉及**。你代码里普通的 `String name`、`int age`、`Person p`、方法返回类型、字段类型，运行时全原样保留、反射全拿得到。
- 但若按朴素的「**源码里有、运行时没了**」来理解，被抹掉的东西**不止泛型**：局部变量名（默认编译后丢，除非加 `-g`）、`RetentionPolicy.SOURCE/CLASS` 的注解、被内联的编译期常量（`final int X = 10` 直接换成字面量）等，都是「源码有、运行时无」——**只是这些不叫「类型擦除」**。

关键区别：泛型擦除丢的是「**类型系统的一等成分**」，直接决定你不能 `new T()`、不能 `instanceof List<String>`、两个 `List<X>` 运行时同类；而局部变量名那类丢的只是**调试/元数据**，不影响类型语义。所以社区把「类型擦除」这个正式名词专门留给了泛型。

> **一句话收口**：**「类型擦除」这个特定机制 100% 为泛型服务、只擦泛型信息；但它不等于"只有泛型才会在编译后丢失"。** 而且它是 Java 为**兼容 JDK 5 之前老代码**主动选的路，不是底层必然——反例是 Rust 单态化、C# reified 泛型，运行时都完整保留泛型类型。

</details>

> **对比钩子**：这是个绝佳的对比点。[Rust](../part4-multilang-compare/04-Java到Rust.md) 的泛型走**单态化（monomorphization）**——编译器为每个具体类型生成一份专门的代码，运行时类型信息完整保留，没有擦除，因此也没有这些坑（代价是编译产物更大、编译更慢）。[TypeScript](../part4-multilang-compare/02-Java到JS-TS.md) 更极端：它的类型**完全在编译期擦除**，编译成 JS 后一点类型信息都不剩（[1.3](../part1-mindset-shift/03-从强类型到类型光谱.md) 已埋过这个伏笔）。三种语言对「泛型/类型信息在运行时保留多少」给出了三种答案。

## 三、通配符与边界：协变、逆变的 PECS 原则

泛型还有个让人头疼的地方——通配符 `?` 和边界。看这个直觉陷阱：

```java
List<Integer> ints = new ArrayList<>();
List<Number> nums = ints;   // 编译错误！List<Integer> 不是 List<Number> 的子类
```

虽然 `Integer` 是 `Number` 的子类，但 `List<Integer>` **不是** `List<Number>` 的子类。这叫**泛型的不变性（invariance）**，是为了类型安全（否则你能往 `List<Number>` 里塞 `Double`，破坏原本的 `List<Integer>`）。

需要灵活性时，用通配符，记住 **PECS 原则（Producer Extends, Consumer Super）**：

```java
// 生产者用 extends：我只从里面"读取"数据（它生产给我）
void readFrom(List<? extends Number> producer) {
    Number n = producer.get(0);   // 能读，读出来当 Number 用
    // producer.add(...);          // 不能写！
}

// 消费者用 super：我只往里面"写入"数据（它消费我给的）
void writeTo(List<? super Integer> consumer) {
    consumer.add(123);            // 能写 Integer
    // Integer x = consumer.get(0); // 读出来只能当 Object
}
```

口诀：**要读用 `extends`，要写用 `super`**。这个机制在 Java 标准库里随处可见（如 `Collections.copy`）。

<details>
<summary><b>展开：extends / super 通配符到底什么用法？为什么"读用 extends、写用 super"（讲透原理）</b></summary>

先分清：这里的 `extends`/`super` 是**通配符边界**（写在 `<? extends T>` / `<? super T>` 里），跟 `class Dog extends Animal` 那个继承的 `extends` **名字一样但不是一回事**。核心难点不是语法，而是「为什么这么规定」——一句话先记住：**编译器在"不知道确切类型"时，只放行那个无论实际类型是什么都绝对安全的方向。**

**`<? extends T>`——上界通配符：「是 T 或 T 的某个子类，但不知道具体是哪个」**

例如 `List<? extends Number>`，它实际可能是 `List<Integer>`、`List<Double>`、`List<Number>`……编译器只知道「装的是 Number 的某个子类型」，**不知道到底哪一个**。

- **能读**：`get()` 出来的元素无论实际是 Integer 还是 Double，**一定是 Number**（都是 Number 子类），可安全当 Number 用。✅
- **不能写**：想 `add()` 时编译器不知道它到底是 `List<Integer>` 还是 `List<Double>`——塞 Integer 怕它其实是 `List<Double>`，塞 Double 怕它其实是 `List<Integer>`，没法保证安全，于是**任何具体类型都不让写**（只能写 null）。❌

```java
List<? extends Number> list = new ArrayList<Integer>();
Number n = list.get(0);   // ✅ 取出来一定是 Number
list.add(1);              // ❌ 编译错误！不知道实际类型，不让写
```

**`<? super T>`——下界通配符：「是 T 或 T 的某个父类，但不知道具体是哪个」**

例如 `List<? super Integer>`，实际可能是 `List<Integer>`、`List<Number>`、`List<Object>`……编译器只知道「它一定装得下 Integer」。

- **能写**：`add()` 一个 Integer，无论它实际是 `List<Integer>`/`List<Number>`/`List<Object>`，Integer **都装得下**（是它们共同的子类），写 Integer 永远安全。✅
- **读受限**：`get()` 出来的元素可能是 Integer/Number/Object……编译器唯一能确定的是「它至少是 Object」，所以**读出来只能当 Object 用**，拿不到具体类型。⚠️

```java
List<? super Integer> list = new ArrayList<Number>();
list.add(1);              // ✅ Integer 一定装得下
Object o = list.get(0);   // ⚠️ 只能当 Object 读，拿不回 Integer
```

**为什么叫 PECS = Producer-Extends, Consumer-Super**

把集合当**生产者**（你从它**读**、它给你提供元素）→ 用 `extends`；当**消费者**（你往它**写**、它接收你给的元素）→ 用 `super`。标准库 `Collections.copy` 是最经典的例子——从 src 读、往 dest 写：

```java
public static <T> void copy(List<? super T> dest, List<? extends T> src) {
    //                       消费者(被写)用 super     生产者(被读)用 extends
}
// 于是这样能编译：把 Integer 列表拷进 Number 列表
copy(new ArrayList<Number>(), new ArrayList<Integer>());
```

| 通配符 | 含义 | 能读吗 | 能写吗 | 角色 |
|------|------|:---:|:---:|------|
| `<? extends T>` | T 或其子类，不知具体哪个 | ✅ 读出来当 T 用 | ❌ 不让写（除 null） | 生产者 Producer |
| `<? super T>` | T 或其父类，不知具体哪个 | ⚠️ 只能当 Object 读 | ✅ 能写 T 及其子类 | 消费者 Consumer |

> **一句话本质**：`extends` 让你「读得到具体类型、但写不进去」（只出不进），`super` 让你「写得进去、但读出来只能当 Object」（只进不出）。这不是 Java 故意刁难，而是它在「确切类型未知」时唯一能保证类型安全的取舍。

</details>

## 四、Java 类型系统是「名义类型」

呼应 [1.3](../part1-mindset-shift/03-从强类型到类型光谱.md) 讲的**名义类型 vs 结构类型**——Java 是**名义类型（nominal typing）**的典型：两个类型是否兼容，看它们的**名字/声明关系**（是否显式 `implements`/`extends`），而不是看「长得像不像」。

```java
interface Flyable { void fly(); }

class Bird {
    public void fly() { }   // 有 fly 方法，但没 implements Flyable
}

Flyable f = new Bird();   // 编译错误！Bird 没"声明"自己是 Flyable
                          // 哪怕它有一模一样的 fly() 方法也不行
```

必须显式声明 `class Bird implements Flyable` 才行。

> **对比钩子**：[TypeScript](../part4-multilang-compare/02-Java到JS-TS.md) 是**结构类型（structural typing）**的代表——「长得像就是兼容」，上面那个 `Bird` 在 TS 里只要结构匹配就能直接当 `Flyable` 用，不需要显式声明。这是从 Java 转 TS 时最容易「水土不服」的认知差异之一，[1.3](../part1-mindset-shift/03-从强类型到类型光谱.md) 已埋下伏笔，第四章 [Java 到 JS/TS](../part4-multilang-compare/02-Java到JS-TS.md) 会详细展开。

## 六、面试深度剖析：大厂高频考点

类型系统是大厂笔面试里「区分度极高」的一块——它考的不是 API，而是你**对运行时模型的理解深度**。下面把高频追问链一条条拆开。

### 考点 1：反射如何「突破」类型擦除？

> 反射机制本身（`Class` 对象由来、四大入口、`setAccessible` 原理、`invoke` 膨胀机制、性能与优化）在 [3.3 JVM 运行时 · 反射机制与原理](./03-JVM运行时.md#五反射机制与原理深挖) 有系统讲解，这里只聚焦它与「类型擦除」的关系。

**面试官**：「既然泛型运行时被擦除了，那我能不能往 `List<Integer>` 里塞个 String？」

答案是**能**，而且这正是验证「擦除」的最佳实验：

```java
List<Integer> list = new ArrayList<>();
list.add(1);

// 通过反射拿到 add 方法（参数被擦除成 Object），绕过编译期检查
Method add = list.getClass().getMethod("add", Object.class);
add.invoke(list, "我是个 String");   // 成功塞进去！运行时根本不校验泛型

System.out.println(list.size());     // 2
// list.get(1) 当 Integer 用时才会 ClassCastException —— 取出点才暴露
```

> **要点提炼**：泛型检查只发生在**编译期**，编译后字节码里 `add` 的签名就是 `add(Object)`。反射直接调用字节码层面的方法，自然绕过了编译器这道关。这说明泛型是**编译期契约**，不是运行时屏障。
>
> **追问：那运行时就彻底拿不到泛型信息了吗？** 不绝对。**字段、方法参数、父类/接口上声明的泛型**会被保留在字节码的 Signature 属性里，可以通过 `getGenericSuperclass()` / `getGenericType()` 反射读出。这就是 Spring `ParameterizedTypeReference`、Gson `TypeToken` 用匿名子类「锚住」泛型的原理：

```java
// TypeToken 套路：用匿名内部类继承，把泛型固化到父类签名上
Type type = new TypeToken<List<String>>(){}.getType();  // 能拿到 List<String>
```

<details>
<summary><b>展开：既然反射能把 String 塞进 List&lt;Integer&gt;，那容器内存到底怎么存的？（容器存引用 + 扩容机制）</b></summary>

承接上面「反射真把 String 塞进去了」这个事实，很多人会困惑：`List<Integer>` 里塞个超长 String，内存怎么放得下、放进去是什么形式？关键认知一句话：**容器里存的从来不是对象本身，而是指向堆中对象的引用（指针）。**

**为什么容器不关心元素多大**

那个超长 String 在你 `add` 之前就已经躺在**堆**里了（`new`/字面量创建的），它占多大内存跟容器无关。`add` 这个动作**没有复制对象**，只是把它的**引用地址**记进容器。而一个引用的大小是固定的（开启指针压缩时 4 字节，未压缩 8 字节）——所以无论你指向的是一个装箱的 `Integer` 还是 1MB 的超长 `String`，**容器里占的「格子」大小完全一样**。这就是为什么类型/大小都不影响「塞得进去」。

```
ArrayList 内部（连续数组，每格存的是引用）：
elementData: [ ref→obj0 | ref→obj1 | ref→"超长String" | (空) | (空) ]
                                          │
                                          └──→ 堆里那个超长 String 对象（独立存放，容器只记地址）
```

**那「每加一个元素」到底是重开一块完整区域，还是尾部加指针？——取决于容器类型，两种猜想各对应一种：**

**① ArrayList——底层是连续数组 `Object[] elementData`**

数组是内存里**一整块连续区域**，每格存一个引用。

- 还有空位时：直接把新元素引用写进尾部空格、`size++`，**不重新分配**，O(1)。
- 满了（`size == 容量`）时：才触发**扩容**——新建一个更大数组（默认约 **1.5 倍**：`newCap = oldCap + (oldCap >> 1)`），把老数组里的引用**整体拷贝**过去，再写入新元素，O(n)。注意拷贝的是**引用**，那些 String/Integer 对象在堆里**原地不动、没被复制**。

> 所以 ArrayList **不是**「每加一个就重开」，而是**平时填空位（O(1)），填满了才整体搬一次家（O(n)）**，均摊下来仍是 O(1)。这也是为什么**预估容量用 `new ArrayList<>(初始容量)`** 能避免反复扩容拷贝——一个实战优化点。

**② LinkedList——这才是你说的「尾部加指针指向新元素」**

它是**双向链表**，每个元素包成一个 `Node`，节点里存：元素引用 + 前驱指针 + 后继指针。加元素就是 `new` 一个 Node 挂到尾部、改下指针，**不连续、不扩容、不搬家**；代价是每个节点多两个指针的内存开销，且随机访问要顺着指针走（O(n)）。

```
LinkedList：node0 ⇄ node1 ⇄ node2(新加) ，节点散落堆各处，靠指针串联
```

| | ArrayList | LinkedList |
|---|---|---|
| 内部结构 | 连续数组存引用 | 散落 Node + 指针串联 |
| 尾部加元素 | 填空位；满了整体扩容拷贝 | new 一个 Node 接到尾部 |
| 「重开整块」 | 仅扩容时 | 从不 |
| 「尾部加指针」 | 不是（填数组格子） | **正是** |
| 随机访问 | O(1)（按下标算地址） | O(n)（顺指针走） |
| 元素对象本身 | 都在堆里独立存放，容器只存引用 | 同左 |

> **一句话收口**：**Java 容器存的是引用不是对象，元素多大都只占一个引用格子（所以反射塞超长 String 也存得下）；ArrayList 用连续数组、满了整体扩容拷贝，LinkedList 用指针串联节点——你猜的「重开整块」和「尾部加指针」两种形式，恰好对应这两种容器。** 而塞进去的 String 直到 `get` 出来强转 `Integer` 时，那个编译器插的 `(Integer)` cast 才失败抛 `ClassCastException`。

</details>

### 考点 2：桥接方法（Bridge Method）是什么？

**面试官**：「子类重写父类的泛型方法，编译器偷偷帮你干了什么？」

```java
class Node<T> {
    public T get() { return null; }
    public void set(T t) { }
}
class IntNode extends Node<Integer> {
    @Override public Integer get() { return 1; }      // 你以为只重写了这个
    @Override public void set(Integer t) { }
}
```

擦除后父类签名是 `Object get()` / `void set(Object)`，而子类是 `Integer get()` / `void set(Integer)`，**签名对不上，多态会断**。编译器为此自动合成**桥接方法**：

```java
// 编译器替你生成的（字节码层面）
public Object get() { return get(); /* 调用真正的 Integer get() */ }
public void set(Object t) { set((Integer) t); }  // 这里有隐式强转
```

> **加分项**：桥接方法可通过 `Method.isBridge()` 判断。它的存在解释了一个诡异现象——`set(Object)` 桥接里有隐式强转，所以 `node.set("x")`（通过 `Node` 引用调用）会在运行时抛 `ClassCastException`，而不是在你重写的方法里。

### 考点 3：自动装箱拆箱的陷阱 + Integer 缓存池

**面试官**：「`Integer a = 127, b = 127; a == b` 是什么？换成 128 呢？」

```java
Integer a = 127, b = 127;
System.out.println(a == b);   // true

Integer c = 128, d = 128;
System.out.println(c == d);   // false ！

Integer e = new Integer(127), f = new Integer(127);
System.out.println(e == f);   // false（new 一定是新对象）
```

原因是 `Integer.valueOf()`（自动装箱调用它）对 **-128 ~ 127** 做了**缓存池**（`IntegerCache`），这个范围内返回同一个对象，超出则 `new`。`==` 比的是引用，所以 127 相等、128 不等。

> **陷阱清单**：
> - `==` 比较包装类型永远用 `.equals()`，别踩缓存边界坑。
> - 装箱拆箱有**性能开销**，循环里用 `Integer sum = 0; sum += i;` 会疯狂创建对象（应该用 `int`）。
> - 拆箱可能 **NPE**：`Integer x = map.get("k"); int y = x;`，若 key 不存在，`x` 为 null，拆箱直接空指针。
> - 缓存上界可通过 `-XX:AutoBoxCacheMax=N` 调大（Long/Short/Byte/Character 也有缓存，但 Float/Double 没有）。

### 考点 4：String 为什么不可变？常量池与 intern

**面试官**：「`String s = new String("abc")` 创建了几个对象？」

可能 **1 个或 2 个**：字面量 `"abc"` 若常量池已存在则复用（0 个新对象），否则在常量池创建 1 个；`new` 在堆上再创建 1 个。所以「**最多 2 个**」（堆 1 + 池 0 或 1）。

```java
String a = "abc";                 // 池中 1 个
String b = "abc";                 // 复用池中的，a == b 为 true
String c = new String("abc");     // 堆中新对象，a == c 为 false
String d = c.intern();            // intern 返回池中引用，a == d 为 true
```

**String 不可变（final class + final char[]/byte[]）的设计原因**：

> - **常量池可复用**：不可变才敢让多个引用共享同一对象，否则一处改了全乱套。
> - **线程安全**：天然不可变，无需同步即可在多线程共享。
> - **可作 HashMap 的 key**：hashCode 只算一次可缓存（`String` 内部缓存 `hash` 字段），且 key 不会被偷偷改掉。
> - **安全**：类加载路径、网络地址等用 String 传参，不可变防止校验后被篡改（TOCTOU）。

> **追问：`"a" + "b" + "c"` 和循环里 `s += i` 有何区别？** 前者编译期常量折叠成 `"abc"`；后者每次 `+=` 都 `new StringBuilder().append().toString()`，循环里会产生大量临时对象——循环拼接必须显式用 `StringBuilder`。

### 考点 5：equals 与 hashCode 的契约

**面试官**：「重写 equals 为什么必须重写 hashCode？」

因为 HashMap/HashSet 先用 `hashCode` 定位桶、再用 `equals` 比对。若只重写 `equals` 不重写 `hashCode`，两个「逻辑相等」的对象 hashCode 不同，会落到不同桶，`map.get` 找不到、`Set` 里出现重复：

```java
// 契约：equals 相等 → hashCode 必须相等；hashCode 相等 → equals 不一定相等（哈希冲突）
@Override public boolean equals(Object o) { /* 按字段比 */ }
@Override public int hashCode() { return Objects.hash(field1, field2); }  // 用同一批字段
```

> **陷阱**：
> - 用作 HashMap key 的对象，**hashCode 依赖的字段不能在放入后被修改**（否则永远找不回来）——这也是为什么常用不可变对象做 key。
> - `Objects.equals(a, b)` 可避免 NPE（两边都为 null 返回 true）。

> **常见对比题**：`==` 比引用（基本类型比值），`equals` 默认也是 `==`（Object 实现），重写后比逻辑相等。

<details>
<summary><b>展开：Java 的几种比较方式——哪些比引用、哪些比真实值？</b></summary>

这是初学者最容易混的一块。核心一句话：**`==` 的行为取决于两边的类型，其余方法（`equals`/`compareTo`…）的行为取决于有没有被重写。**

### 一、`==`：看两边是什么类型

| 两边类型 | `==` 比的是 | 例子 |
| --- | --- | --- |
| 基本类型（`int`/`double`/`char`/`boolean`…） | **值本身** | `int a=3,b=3; a==b` → `true` |
| 引用类型（任何对象，含 `Integer`/`String`/自定义类） | **引用地址**（是不是同一个对象） | `new String("a")==new String("a")` → `false` |
| 一基本 + 一包装（`int == Integer`） | 包装**自动拆箱**后**比值** | `Integer x=1000; int y=1000; x==y` → `true` |

> 基本类型在栈里直接存数值，根本没有「引用」可比，所以只能比值。
> 引用类型变量里存的是**堆地址**，`==` 比的就是这个地址，跟内容毫无关系。
>
> **隐藏坑**：`Integer x=1000, y=1000; x==y` 是 `false`（两个对象比地址），但只要有一边是 `int`，就会拆箱变成比值。包装类之间用 `==` 还要小心 **-128~127 缓存池**（见考点 3）。

### 二、`equals()`：默认比引用，重写后才比内容

```java
// Object 的默认实现——本质就是 ==
public boolean equals(Object obj) { return (this == obj); }
```

- **没重写**（如某些自定义类）：`equals` 跟 `==` 完全一样，**比引用**。
- **重写过**（`String`/`Integer`/集合类/你自己写的类）：才**比逻辑相等（内容）**。

这就是为什么字符串比较**永远用 `equals`**：`"abc".equals(s)` 比内容，`"abc" == s` 比地址（只有都来自常量池才碰巧相等）。

### 三、`Objects.equals(a, b)`：空安全版的 equals

```java
public static boolean equals(Object a, Object b) {
    return (a == b) || (a != null && a.equals(b));
}
```

先比引用（顺手处理「两个都为 null → `true`」），再 null 安全地调 `a.equals(b)`。本质仍是**比内容**，只是不怕 `NullPointerException`——推荐默认用它。

### 四、`compareTo()` / `Comparator.compare()`：比大小（排序）

这不是判等，而是**比内容的大小关系**，返回负/零/正用于排序。

> 注意它应与 `equals` **保持一致**（`compareTo==0` 最好等价于 `equals==true`），否则 `TreeMap`/`TreeSet` 会出诡异问题——它们**用 `compareTo` 判重**，不用 `equals`。

### 五、`hashCode()`：不是比较方法，但是 equals 的搭档

它产生一个 `int` 哈希值，配合 `equals` 在 `HashMap` 里「先定位桶、再精确比对」。契约：`equals` 相等 → `hashCode` 必相等。它本身不直接比较两个对象（详见考点 5）。

### 汇总对照

| 方式 | 比引用还是比值 | 关键说明 |
| --- | --- | --- |
| `==`（基本类型） | **比值** | 栈里直接是数值 |
| `==`（引用类型） | **比引用地址** | 是不是同一个对象 |
| `==`（基本+包装） | **比值** | 包装自动拆箱 |
| `equals()` 未重写 | **比引用** | 默认就是 `this == obj` |
| `equals()` 已重写 | **比内容** | `String`/`Integer`/自定义类 |
| `Objects.equals()` | **比内容**（空安全） | 推荐默认用 |
| `compareTo()`/`Comparator` | **比内容大小** | 排序用，非判等 |
| `hashCode()` | —（不是比较） | 给 `equals` 当哈希定位搭档 |

> **记忆链**：`==` 看类型（基本比值、引用比地址）→ `equals` 默认比地址、重写后比内容 → `Objects.equals` 是空安全的 equals → `compareTo` 比大小排序 → `hashCode` 给 equals 当搭档。

</details>

## 本章小结

- **泛型**把类型检查从运行时提前到编译期，是 Java「静态强类型」红利的集中体现（呼应 [1.3](../part1-mindset-shift/03-从强类型到类型光谱.md)）。
