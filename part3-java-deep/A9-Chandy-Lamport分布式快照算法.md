# A9 Chandy-Lamport 分布式快照算法

> **一句话定位**：Chandy-Lamport 算法解决的核心问题是——在一个持续运行、消息不断流动的分布式系统中，如何在**不暂停任何进程**的前提下，拍一张全局一致的"快照"。它是 Flink Checkpoint、分布式调试、全局状态检测的理论基础。

## 一、问题背景：为什么需要全局快照

分布式系统中，多个进程并发运行、通过消息通信。我们经常需要知道系统的"全局状态"，例如：

- **故障恢复**：系统崩溃后从某个一致的历史状态重启
- **死锁检测**：检查是否存在循环等待
- **垃圾回收**：检测分布式对象是否还有引用
- **调试诊断**：观察某一时刻的全局状态

**难点**：各节点没有共享时钟，无法在同一物理时刻"同时暂停"所有进程拍照。如果各自独立记录状态，可能出现**不一致快照**：

```
进程 P1 记录状态：已发送消息 M
进程 P2 记录状态：未收到消息 M
→ 快照中 M "凭空消失"，不一致！
```

Chandy-Lamport 算法（1985 年，K. Mani Chandy & Leslie Lamport）给出了一个优雅的解法。

## 二、算法前提条件

| 条件 | 说明 |
|------|------|
| **FIFO 信道** | 同一信道上的消息按发送顺序到达，不乱序 |
| **可靠传递** | 消息不丢失、不重复 |
| **强连通图** | 任意两个进程之间存在有向路径（保证 Marker 能传播到所有节点） |
| **进程不崩溃** | 快照过程中进程正常运行（容错恢复是上层应用的事） |

> **注意**：FIFO 是关键约束。正是因为 FIFO，Marker 之前发出的消息一定在 Marker 之前到达，这保证了信道状态记录的正确性。

---

## 三、核心概念：Marker 消息

Marker 是一种**特殊的控制消息**，混在普通数据消息流中传输：

- 不携带业务数据
- 作为"快照边界"的分隔符
- Marker 之前的消息属于"快照前"，之后的属于"快照后"

```
信道: ... [数据A] [数据B] [MARKER] [数据C] [数据D] ...
                           ↑
                      快照边界
```

## 四、算法步骤

### 4.1 发起者规则

任意一个进程 `Pi` 可以发起快照（通常是协调者）：

1. **记录自己的本地状态**（进程状态 `Si`）
2. 向**所有出边信道**发送一条 Marker 消息
3. 开始记录所有**入边信道**上收到的消息（用于记录信道状态）

### 4.2 接收者规则

进程 `Pj` 从信道 `Cij` 收到 Marker 时：

**情况一：Pj 尚未记录过自己的状态**
1. 记录自己的本地状态 `Sj`
2. 将信道 `Cij` 的状态记为**空**（因为 Marker 是该信道上第一个到达的"边界"，之前没有需要记录的消息）
3. 向所有出边信道发送 Marker
4. 开始记录其他所有入边信道上收到的消息

**情况二：Pj 已经记录过自己的状态**
1. 将信道 `Cij` 的状态记为：**从 Pj 记录状态之后、到收到这条 Marker 之前，从 `Cij` 收到的所有消息**
2. 停止记录信道 `Cij`

### 4.3 信道状态记录

信道状态 = 在快照边界期间"在途"的消息集合：

```
Pj 记录状态的时刻 ←─────────────────→ Pj 收到 Cij 上 Marker 的时刻
                    这段时间内从 Cij 收到的消息 = 信道 Cij 的状态
```

当所有进程都完成了本地状态记录，且所有信道状态都记录完毕，快照完成。

## 五、完整示例（三节点演示）

系统有三个进程 P1、P2、P3，信道均为 FIFO：

```
P1 ──m12──→ P2
P1 ──m13──→ P3
P2 ──m23──→ P3
P3 ──m31──→ P1
```

**初始状态**：
- P1 有 $100，P2 有 $200，P3 有 $300，总计 $600

**快照过程**：

```
Step 1: P1 发起快照
  - P1 记录自己状态：$100
  - P1 向 P2、P3 发送 Marker
  - P1 开始记录入边信道（来自 P3）的消息

Step 2: P2 收到来自 P1 的 Marker
  - P2 尚未记录状态 → 记录自己状态：$200
  - 信道 P1→P2 状态 = 空（Marker 是第一个边界）
  - P2 向 P3 发送 Marker
  - P2 开始记录入边信道（来自 P1 已完成，来自 P3 开始记录）

Step 3: P3 收到来自 P1 的 Marker
  - P3 尚未记录状态 → 记录自己状态：$300
  - 信道 P1→P3 状态 = 空
  - P3 向 P1 发送 Marker
  - P3 开始记录入边信道（来自 P2 开始记录）

Step 4: P3 收到来自 P2 的 Marker
  - P3 已记录状态 → 信道 P2→P3 状态 = 从 P3 记录状态后到收到此 Marker 前，P2→P3 上的消息
  - 假设这段时间 P2 发了一笔 $50 转账给 P3 → 信道 P2→P3 状态 = [$50]

Step 5: P1 收到来自 P3 的 Marker
  - P1 已记录状态 → 信道 P3→P1 状态 = 空（P3 记录状态后未向 P1 发消息）
```

**全局快照**：
- P1 状态：$100，P2 状态：$200，P3 状态：$300
- 信道 P2→P3：[$50]（在途消息）
- 其他信道：空

**验证一致性**：$100 + $200 + $300 + $50（在途）= $650？

> 实际上这里 P2 在记录状态（$200）之后又发出了 $50，所以 P2 的真实余额应该是 $150，但快照记录的是 $200。这正是"快照时刻"的含义——P2 的快照状态是发出转账**之前**的状态，而那笔 $50 被记录在信道状态里。恢复时 P3 会重新收到这笔 $50，总量守恒。

## 六、算法正确性分析

### 一致性（Consistency）

快照满足**因果一致性**：如果快照包含了"事件 e 的结果"，那么也一定包含了"导致 e 发生的所有前因"。

形式化：对于任意消息 M，
- 若快照包含"M 被接收"，则必然包含"M 被发送"
- 若快照不包含"M 被发送"，则必然不包含"M 被接收"

### 终止性（Termination）

由于信道可靠且强连通，Marker 最终会传播到所有进程，算法必然终止。

### 不干扰性（Non-intrusive）

算法不需要暂停任何进程，进程在快照过程中可以继续正常处理消息。

## 七、Flink 的改进：Barrier 机制

Flink 将 Chandy-Lamport 算法适配到流处理场景，做了以下改进：

| 维度 | 原始 Chandy-Lamport | Flink Barrier |
|------|---------------------|---------------|
| **Marker 注入者** | 任意进程可发起 | 仅 JobManager 统一注入（从 Source 开始） |
| **信道类型** | 点对点有向信道 | 数据流（有向无环图，DAG） |
| **多输入处理** | 每条入边独立记录 | **Barrier 对齐**：等所有输入流的 Barrier 都到达后再保存状态 |
| **信道状态** | 需要记录在途消息 | 对齐模式下不需要（对齐后信道为空）；非对齐模式下需要保存缓冲区 |
| **状态存储** | 由应用自定义 | State Backend（Memory / RocksDB）+ 外部存储（HDFS/S3） |
| **快照编号** | 无编号 | Checkpoint ID（单调递增） |

### Barrier 对齐 vs 非对齐

```
对齐 Checkpoint（默认）：
  Input1: ──[data]──[data]──[BARRIER]──(暂停)──
  Input2: ──[data]──────────────────[BARRIER]──
                                              ↓
                                        保存状态，继续处理

非对齐 Checkpoint（Flink 1.11+）：
  Input1: ──[data]──[data]──[BARRIER]──[data']──
  Input2: ──[data]──────────────────[BARRIER]──
                              ↓
                  立即保存状态 + 把 [data'] 也存入快照
```

**选择建议**：
- 正常情况用**对齐**（快照小，恢复快）
- 反压严重、对齐超时时用**非对齐**（快照大，但不受反压影响）

## 八、与其他快照方案对比

| 方案 | 原理 | 是否暂停 | 一致性 | 适用场景 |
|------|------|:--------:|:------:|----------|
| **Chandy-Lamport** | Marker 消息划分边界 | ❌ 不暂停 | 因果一致 | 通用分布式系统 |
| **Stop-the-World** | 暂停所有进程后统一拍照 | ✅ 暂停 | 强一致 | 简单系统、可接受停顿 |
| **Coordinated Checkpoint** | 协调者通知所有节点同时保存 | 部分暂停 | 强一致 | MPI 并行计算 |
| **Flink Barrier（对齐）** | Chandy-Lamport 变体 | ❌ 不暂停 | 因果一致 | 流处理 Exactly-Once |
| **Flink Barrier（非对齐）** | 不等对齐，缓冲区入快照 | ❌ 不暂停 | 因果一致 | 反压严重的流处理 |
| **Spark Checkpoint** | 全量写 RDD/状态到 HDFS | 部分暂停 | 最终一致 | 批处理/Structured Streaming |

## 九、应用场景

| 场景 | 说明 |
|------|------|
| **流处理容错**（Flink/Kafka Streams） | 定期快照，故障后从快照恢复，实现 Exactly-Once |
| **分布式死锁检测** | 快照全局等待图，检测是否有环 |
| **分布式垃圾回收** | 快照对象引用图，回收无引用对象 |
| **分布式调试** | 记录某时刻全局状态，离线分析 |
| **银行账户守恒检测** | 快照所有账户余额 + 在途转账，验证总量守恒 |
