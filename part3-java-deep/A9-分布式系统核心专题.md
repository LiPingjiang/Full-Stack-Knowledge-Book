## 一、Raft 共识算法

### 1.1 背景：为什么需要共识

当今数据中心通过横向扩展应对高并发，服务器随时可能上下线或故障。系统必须在正常操作期间处理节点变动，并在几秒内自动适应——对客户端来说，明显的中断是不可接受的。

**分布式共识**就是解决这个问题的核心手段：即使面对故障，多个服务器也能在共享状态上达成一致。

> **拜占庭将军类比（简化版）**：假设多位将军中没有叛军，信使可靠但可能被暗杀。解决方案是先选出一个"大将军"做决策，再通知其他人。Raft 的 Leader 选举本质上就是这个思路。

### 1.2 复制状态机

共识算法一般通过**复制状态机（Replicated State Machine）**实现：

```
Client → 共识模块 → 复制日志 → 状态机
```

- 每个 Server 存储一份包含命令序列的日志
- 状态机按顺序执行这些命令
- 因为每个日志包含相同命令且顺序相同，所以每个状态机得到相同输出

共识算法的工作就是**保持复制日志的一致性**。

### 1.3 基础概念

#### 三种角色

| 角色 | 职责 |
|------|------|
| **Leader** | 发起心跳、响应客户端、创建日志、同步日志 |
| **Candidate** | 选举过程中的临时角色，由 Follower 转化，发起投票 |
| **Follower** | 接受 Leader 心跳和日志同步，投票给 Candidate |

正常情况下只有一个 Leader，其余都是 Follower。

#### 任期（Term）

- Raft 将时间划分为任意长度的**任期**，用连续数字表示
- 每个任期开始都是一次选举
- 给定任期内最多一个 Leader
- 节点通信时交换 term 号；发现自己 term 过期则立即退回 Follower

#### 日志（Log）

- **Entry**：`<term, index, cmd>`，只有 Leader 可以创建
- **Log**：由 entry 构成的数组，每个 entry 有唯一 index
- Entry 先被 Leader 添加到自己的 log，再发起共识请求，获得多数同意后提交

### 1.4 Leader 选举

Raft 使用**心跳机制**触发选举：

1. Follower 在 `electionTimeout` 内未收到心跳 → 认为无可用 Leader
2. 自增 term，转为 Candidate，向所有节点发送 `RequestVote RPC`
3. 三种结果：
   - **赢得选举**：收到多数选票（N/2+1）→ 成为 Leader
   - **其他节点赢得选举**：收到 term ≥ 自己的心跳 → 退回 Follower
   - **无人胜出**：超时后重新选举

**随机化选举超时**：每个 Candidate 随机化一个新的超时时间（150~300ms），使各服务器分散开来，大多数情况下只有一个率先超时并赢得选举。

#### 投票规则

- 每个节点在一个 term 内只能投一票（先到先得）
- 只投给日志**至少和自己一样新**的 Candidate（保证安全性）

### 1.5 日志复制

Leader 选出后开始接受客户端请求：

```
1. Leader 收到请求 → 生成 entry <index, term, cmd> → 追加到本地日志
2. 向所有 Follower 广播 AppendEntries RPC
3. Follower 接受 → 追加到本地日志 → 返回成功
4. Leader 收到多数成功响应 → 提交（committed）→ 应用到状态机 → 返回客户端
```

#### 日志一致性保证

Raft 保证两个性质：
1. 两个日志中，相同 index + term 的 entry 一定有相同 cmd
2. 两个日志中，相同 index + term 的 entry，其前面的 entry 也一定相同

#### 日志冲突处理

Leader 崩溃可能导致日志不一致。Leader 通过**强制 Follower 复制自己的日志**来处理：

- Leader 为每个 Follower 维护 `nextIndex`（初始化为 Leader 最新 index + 1）
- AppendEntries 一致性检查失败 → Leader 递减 `nextIndex` 重试
- 最终找到一致点 → 删除 Follower 冲突日志 → 补齐缺失日志

### 1.6 安全性保证

#### 选举限制

- Candidate 的 `RequestVote` 携带最后一个 entry 信息
- 节点收到投票请求时，如果自己的日志更新则拒绝投票
- 判断新旧：先比 term（大的更新），term 相同比 index（长的更新）

#### 节点崩溃处理

| 场景 | 处理方式 |
|------|----------|
| Leader 崩溃 | 触发新一轮选举，选举期间集群不可用 |
| Follower/Candidate 崩溃 | 后续 RPC 失败则无限重试（幂等性保证安全） |

#### 时间与可用性

```
broadcastTime << electionTimeout << MTBF
```

- `broadcastTime`（0.5~20ms）：并发发送消息的平均响应时间
- `electionTimeout`（10~500ms）：选举超时时间
- `MTBF`（一两个月）：单台机器平均健康时间

安全性不依赖时间——错误的时钟最坏只影响可用性，不影响一致性。

### 1.7 日志压缩（Snapshot）

随着日志不断增长，需要通过**快照（Snapshot）**进行压缩：

- 将已提交的日志对应的状态机状态序列化为快照
- 丢弃快照之前的所有日志
- 新节点加入或落后太多时，Leader 通过 `InstallSnapshot RPC` 发送快照

```
┌─────────────────────────────────────────┐
│  Snapshot (state at index 5, term 3)    │
├─────────────────────────────────────────┤
│  Log: [6, term3] [7, term3] [8, term4]  │
└─────────────────────────────────────────┘
```

### 1.8 成员变更（Membership Change）

集群扩缩容时，直接切换配置可能导致同一 term 出现两个 Leader（脑裂）。

**单步成员变更（Single-Server Change）**：每次只增减一个节点，保证新旧配置的多数派必有交集，不会出现双主。

**联合共识（Joint Consensus）**：Raft 论文提出的两阶段方案，先进入 `C_old,new` 联合配置，再切换到 `C_new`。工程中 etcd 采用单步变更。

### 1.9 线性一致读（Linearizable Read）

默认情况下从 Leader 读可能读到过期数据（网络分区时旧 Leader 不知道自己已被替换）。

三种方案：

| 方案 | 原理 | 性能 |
|------|------|------|
| **Raft Log Read** | 读请求也走一遍日志复制 | 最慢，强一致 |
| **ReadIndex** | Leader 确认自己仍是 Leader（心跳确认）后读 | 一次 RTT |
| **Lease Read** | Leader 在租约期内直接读（依赖时钟） | 最快，弱依赖时钟 |

etcd 默认使用 ReadIndex 方案。

### 1.10 Raft 工程实践：etcd / Consul / TiKV

| 系统 | 语言 | 用途 | Raft 特点 |
|------|------|------|-----------|
| **etcd** | Go | K8s 配置存储、服务发现 | 标准 Raft，ReadIndex，单步成员变更 |
| **Consul** | Go | 服务网格、KV 存储 | HashiCorp Raft 库 |
| **TiKV** | Rust | 分布式 KV（TiDB 底层） | Multi-Raft，每个 Region 一个 Raft Group |
| **CockroachDB** | Go | 分布式 SQL | Multi-Raft |
| **RocketMQ DLedger** | Java | 消息队列日志复制 | 基于 Raft 的 CommitLog |

### 1.11 Raft 面试高频问题

1. **Raft 分为哪几个部分？** → Leader 选举、日志复制、安全性（+ 日志压缩、成员变更）
2. **任何节点都可以发起选举吗？** → 是的，任何 Follower 超时后都可以变为 Candidate
3. **投票的前提？** → Candidate 的日志至少和自己一样新，且本 term 未投过票
4. **网络分区下如何保证一致性？** → 少数派分区的 Leader 无法提交（得不到多数确认），多数派分区会选出新 Leader
5. **Raft vs Paxos？** → 等价但更易理解；Raft 强制 Leader，Paxos 允许多 Proposer
6. **时钟回拨会影响 Raft 吗？** → 不影响正确性，只可能影响可用性（选举超时异常）

## 二、API 网关

### 2.1 什么是 API 网关

微服务架构下，一个系统被拆分为多个服务，但安全认证、流量控制、日志、监控等功能是每个服务都需要的。**网关**作为统一入口，集中处理这些横切关注点。

```
Client → [API Gateway] → Service A
                       → Service B
                       → Service C
```

网关本质做两件事：**请求转发** + **请求过滤**。

### 2.2 网关核心功能

| 功能 | 说明 |
|------|------|
| 请求转发/路由 | 将请求转发到目标微服务 |
| 负载均衡 | 根据策略动态分配请求 |
| 安全认证 | 身份验证 + RBAC 授权 |
| 流量控制/限流 | 限制某时刻内的请求数 |
| 熔断降级 | 达到失败阈值后自动熔断，返回降级响应 |
| 响应缓存 | 缓存静态/低频更新数据的响应 |
| 响应聚合 | 合并多个服务的响应一并返回 |
| 灰度发布 | 将请求动态分流到不同服务版本 |
| 协议转换 | 整合 REST、gRPC、Dubbo 等不同协议 |
| 日志/监控/告警 | 记录行为日志，提供监控指标 |
| API 文档 | Swagger / OpenAPI 管理 |

### 2.3 主流网关对比

| 维度 | Nginx | Kong | APISIX | Spring Cloud Gateway | Zuul 2.x |
|------|-------|------|--------|----------------------|-----------|
| **技术栈** | C | Nginx + Lua | Nginx + Lua + etcd | Java (WebFlux/Reactor) | Java (Netty) |
| **性能** | ★★★★★ | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| **动态配置** | 需 reload | Admin API 热加载 | Admin API 热加载 | 配置文件/代码 | 配置文件 |
| **插件生态** | 需 Lua 扩展 | 100+ 官方插件 | 80+ 插件，支持多语言 | Spring 生态集成 | Netflix 生态 |
| **服务发现** | 需第三方 | Consul/etcd/K8s | etcd/Consul/Nacos/K8s | Eureka/Nacos/K8s | Eureka |
| **适用场景** | 边缘网关/静态资源 | 中大型多语言架构 | 云原生/高性能/大规模 | Spring Cloud 生态 | 遗留 Netflix 项目 |
| **开源协议** | BSD | Apache 2.0 | Apache 2.0 | Apache 2.0 | Apache 2.0 |
| **社区活跃度** | 极高 | 高 | 高（Apache 顶级项目） | 高 | 低（已停止维护） |

#### 补充：Shenyu（Apache）

- 基于 WebFlux 的响应式网关，Apache 顶级项目
- 通过插件扩展（限流、熔断、转发、重写、路由监控等）
- 支持 Dubbo、gRPC、Spring Cloud、Motan 等多种 RPC 框架

#### 补充：Higress（阿里）

- 基于 Envoy + Istio 的云原生网关
- 兼容 Nginx Ingress 注解，支持 Wasm 插件
- 适合 K8s 原生环境

### 2.4 双层网关架构

大型系统通常采用**双层网关**：

```
Internet → [Nginx/LB 集群 (流量网关)]
              ↓
         [业务网关 (Kong/APISIX/SCG)]
              ↓
         微服务集群
```

- **流量网关（边缘层）**：SSL 终结、全局限流、DDoS 防护、静态资源
- **业务网关（服务层）**：认证鉴权、路由、熔断、灰度、协议转换

### 2.5 选型建议

```
你的技术栈是 Spring Cloud？
  → Spring Cloud Gateway（生态无缝集成）

需要高性能 + 多语言 + 丰富插件？
  → APISIX（etcd 配置，毫秒级热加载）
  → Kong（PostgreSQL/Cassandra 配置，插件市场成熟）

纯 K8s 云原生？
  → Higress / APISIX Ingress Controller

已有 Nginx 基础设施？
  → OpenResty 自研 或 Kong/APISIX（底层都是 Nginx）
```

**性能参考**（8C32G 单 Worker）：

| 场景 | APISIX QPS | Kong QPS |
|------|-----------|----------|
| 不开启插件 | ~18,000 | ~8,000 |
| 开启限流+认证 | ~14,000 | ~5,000 |

## 三、分布式 ID

### 3.1 为什么需要分布式 ID

单机 MySQL 自增主键在分库分表后无法保证全局唯一。分布式系统需要一种方案为不同数据节点生成**全局唯一主键**。

### 3.2 核心需求

| 需求 | 说明 |
|------|------|
| **全局唯一** | 任何节点、任何时间生成的 ID 不重复 |
| **高性能** | 生成速度快，不成为系统瓶颈 |
| **高可用** | 服务可用性无限接近 100% |
| **趋势递增** | 有利于数据库 B+ 树索引写入性能 |
| **安全** | 不暴露业务量等敏感信息 |
| **独立部署** | 发号器服务与业务解耦 |

### 3.3 方案一览

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 128 位随机/时间/MAC 组合 | 本地生成，无网络开销 | 无序、过长、不适合做 DB 主键 |
| 数据库自增 | `AUTO_INCREMENT` | 简单、有序 | 单点、性能低、暴露业务量 |
| 数据库号段 | 批量获取一段 ID 缓存在内存 | 减少 DB 访问 | 仍有 DB 依赖 |
| Redis INCR | 原子自增 | 性能好、有序 | 需持久化、集群复杂度 |
| MongoDB ObjectId | 12 字节（时间+机器+进程+自增） | 有序、性能好 | 时钟问题可能重复 |
| **Snowflake** | 64 位（时间+机器+序列号） | 有序、高性能、灵活 | 时钟回拨问题 |
| **号段模式（双 Buffer）** | 预取号段 + 异步加载 | 稳定、容灾好 | 需要 DB |

### 3.4 Snowflake 雪花算法详解

Twitter 开源的分布式 ID 生成算法，生成 64 位 long 型 ID：

```
┌─1bit─┬──────41bit──────┬───10bit───┬───12bit───┐
│  0   │   时间戳(ms)     │  机器ID   │  序列号   │
└──────┴─────────────────┴───────────┴───────────┘
```

| 部分 | 位数 | 说明 |
|------|------|------|
| 符号位 | 1 | 始终为 0 |
| 时间戳 | 41 | 毫秒级，支持约 69 年 |
| 机器 ID | 10 | 5 位数据中心 + 5 位机器，最多 1024 节点 |
| 序列号 | 12 | 每毫秒最多 4096 个 ID |

**单机理论上限**：4096 × 1000 = **409.6 万 ID/秒**

#### 时钟回拨问题及解决

| 方案 | 原理 |
|------|------|
| 等待追上 | 回拨时间短（<几ms）则 sleep 等待 |
| 报错拒绝 | 回拨超过阈值直接抛异常 |
| 扩展位 | 预留几位作为时钟序列，回拨时递增 |
| 借用未来时间 | UidGenerator 的 RingBuffer 方案 |

#### Java 核心实现

```java
public synchronized long nextId() {
    long timestamp = System.currentTimeMillis();
    if (timestamp < lastTimestamp) {
        throw new RuntimeException("Clock moved backwards!");
    }
    if (timestamp == lastTimestamp) {
        sequence = (sequence + 1) & 0xFFF; // 4095
        if (sequence == 0) {
            timestamp = waitNextMillis(lastTimestamp);
        }
    } else {
        sequence = 0L;
    }
    lastTimestamp = timestamp;
    return ((timestamp - epoch) << 22)
         | (workerId << 12)
         | sequence;
}
```

### 3.5 号段模式详解

核心思想：**批量获取 ID 缓存在内存，用完再取下一批**。

```sql
CREATE TABLE sequence_id_generator (
  id INT NOT NULL,
  current_max_id BIGINT NOT NULL COMMENT '当前最大id',
  step INT NOT NULL COMMENT '号段长度',
  version INT NOT NULL COMMENT '乐观锁版本号',
  biz_type INT NOT NULL COMMENT '业务类型',
  PRIMARY KEY (id)
);
```

获取号段：
```sql
-- 1. 查询当前号段
SELECT current_max_id, step, version FROM sequence_id_generator WHERE biz_type = 101;

-- 2. CAS 更新（乐观锁）
UPDATE sequence_id_generator
SET current_max_id = current_max_id + step, version = version + 1
WHERE version = #{version} AND biz_type = 101;
```

获取成功后，应用在内存中使用 `[current_max_id, current_max_id + step)` 范围内的 ID。

#### 双 Buffer 优化

```
┌─────────────┐     ┌─────────────┐
│  Buffer A   │ ←── │  当前使用    │
│ [1, 1000]   │     └─────────────┘
├─────────────┤
│  Buffer B   │ ←── 异步预加载下一号段
│ [1001,2000] │
└─────────────┘
```

当 Buffer A 使用到一定比例（如 10%）时，异步加载 Buffer B，避免号段切换时阻塞。

### 3.6 开源框架：Leaf / UidGenerator / TinyId

#### 美团 Leaf

- 支持**号段模式**和 **Snowflake 模式**双模式
- 号段模式：双 Buffer + DB 多主
- Snowflake 模式：依赖 ZooKeeper 分配 workerId，解决时钟回拨
- QPS：号段模式 ~5w/s（4C8G），Snowflake 模式更高

#### 百度 UidGenerator

- 基于 Snowflake 改进，自定义各段位数
- 默认：28 位秒级时间戳 + 22 位 workerId + 13 位序列号
- **CachedUidGenerator**：预生成 ID 填入 RingBuffer，取 ID 无锁
- workerId 通过数据库自增分配，每次启动获取新 workerId
- 通过"借用未来时间"解决时钟回拨

#### 滴滴 TinyId

- 纯号段模式，双号段缓存 + 多 DB 支持
- 提供 **tinyid-client**（纯本地操作，无 HTTP 开销）
- 适合对有序性要求高、不需要时间信息的场景

### 3.7 方案对比总表

| 方案 | 全局唯一 | 有序 | 性能 | 可用性 | 安全 | 实现复杂度 |
|------|:--------:|:----:|:----:|:------:|:----:|:----------:|
| UUID | ✅ | ❌ | ★★★★★ | ★★★★★ | ❌ | 极低 |
| DB 自增 | ✅ | ✅ | ★★☆☆☆ | ★★☆☆☆ | ❌ | 低 |
| DB 号段 | ✅ | ✅ | ★★★★☆ | ★★★☆☆ | ❌ | 中 |
| Redis | ✅ | ✅ | ★★★★☆ | ★★★☆☆ | ❌ | 中 |
| Snowflake | ✅ | ✅ | ★★★★★ | ★★★★☆ | ✅ | 中 |
| Leaf（号段） | ✅ | ✅ | ★★★★☆ | ★★★★☆ | ❌ | 中 |
| Leaf（雪花） | ✅ | ✅ | ★★★★★ | ★★★★☆ | ✅ | 中高 |
| UidGenerator | ✅ | ✅ | ★★★★★ | ★★★★☆ | ✅ | 中高 |
| TinyId | ✅ | ✅ | ★★★★☆ | ★★★★☆ | ❌ | 中 |

### 3.8 选型建议

```
需要简单快速接入？
  → Leaf 号段模式（双 Buffer + 多 DB）

需要极致性能 + 趋势递增？
  → Snowflake 系（Leaf-Snowflake / UidGenerator）

需要严格递增 + 无时钟依赖？
  → 号段模式（TinyId / Leaf-Segment）

ID 需要包含业务含义？
  → 自定义 Snowflake 变体（在机器位中编码业务类型）

不想引入额外依赖？
  → 自实现 Snowflake（注意时钟回拨处理）
```
