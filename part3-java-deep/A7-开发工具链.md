## 一、Git 进阶——不只是 add / commit / push

### 1.1 分支策略

| 策略 | 核心思路 | 适用团队 |
|------|---------|---------|
| **Git Flow** | main + develop + feature/release/hotfix 五类分支 | 版本发布周期长的项目 |
| **GitHub Flow** | main + feature 分支 + PR，简单直接 | 持续部署的 Web 项目 |
| **Trunk-Based** | 只有 main，所有人在主干上开发，用 Feature Flag 控制 | 大型团队、CI/CD 成熟 |

### 1.2 高频进阶命令

| 场景 | 命令 | 说明 |
|------|------|------|
| 暂存工作区 | `git stash` / `git stash pop` | 临时切分支修 bug |
| 合并策略 | `git merge` vs `git rebase` | merge 保留分支历史，rebase 线性化历史 |
| 挑选提交 | `git cherry-pick <commit>` | 把某个 commit 单独移到当前分支 |
| 二分查 bug | `git bisect start/good/bad` | 用二分法定位引入 bug 的 commit |
| 修改最近提交 | `git commit --amend` | 修改 commit message 或补漏文件 |
| 回退提交 | `git revert <commit>` | 创建一个反向提交来撤销（不改历史） |
| 重置 | `git reset --soft/mixed/hard` | soft 保留暂存区，mixed 保留工作区，hard 全丢 |
| 查看文件历史 | `git log --follow -p <file>` | 追踪文件的变更历史（含重命名） |
| 子模块 | `git submodule add/update` | 管理项目依赖的外部仓库 |

### 1.3 rebase vs merge

```
merge（合并）：保留真实的分支历史，产生一个"合并提交"
    A---B---C feature
   /         \
  D---E---F---G main

rebase（变基）：把 feature 的提交"移植"到 main 最新位置，历史变成直线
  D---E---F---A'---B'---C' main
```

**经验规则**：自己的本地 feature 分支用 rebase 保持线性整洁；公共分支（main/develop）用 merge 保留历史。**永远不要 rebase 已经推送到远端的公共分支**。

<details>
<summary><b>展开：.gitignore 最佳实践</b></summary>

```gitignore
# IDE
.idea/

.vscode/

# 编译产物
target/
```

**原则**：编译产物、IDE 配置、敏感信息、操作系统文件都不提交。`git rm --cached <file>` 可以把已经跟踪的文件从版本控制中移除但保留本地文件。

## 二、Maven / Gradle——构建工具

### 2.1 Maven 生命周期

```
validate → compile → test → package → verify → install → deploy
```

每个阶段会执行它之前的所有阶段。`mvn package` 会依次执行 validate → compile → test → package。

### 2.2 Maven 依赖管理

**依赖范围（scope）**：

| scope | 编译 | 测试 | 运行 | 典型依赖 |
|-------|------|------|------|---------|
| `compile`（默认） | ✅ | ✅ | ✅ | Spring、MyBatis |
| `provided` | ✅ | ✅ | ❌ | Servlet API（容器提供） |
| `runtime` | ❌ | ✅ | ✅ | MySQL 驱动 |
| `test` | ❌ | ✅ | ❌ | JUnit、Mockito |

**依赖冲突排查**：

```bash
# 查看依赖树
mvn dependency:tree

# 查看某个依赖被谁引入
mvn dependency:tree -Dincludes=groupId:artifactId
```

Maven 的冲突解决策略：**最短路径优先**（离项目最近的版本胜出）→ **先声明优先**（路径相同时 pom 中先声明的胜出）。

**BOM（Bill of Materials）**：统一管理一组依赖的版本号，避免版本冲突。Spring Boot 的 `spring-boot-dependencies` 就是一个 BOM。

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

### 2.3 Maven vs Gradle

| 维度 | Maven | Gradle |
|------|-------|--------|
| 配置语言 | XML（冗长但明确） | Groovy/Kotlin DSL（简洁但学习曲线） |
| 构建速度 | 较慢（无增量编译） | 快（增量编译 + 构建缓存 + 守护进程） |
| 灵活性 | 约定优于配置，不够灵活 | 高度可定制 |
| 生态 | Java 后端主流 | Android 标准，Java 后端在增长 |
| 推荐 | 中小型 Java 项目、团队规范统一 | 大型项目、多模块、需要快速构建 |

## 三、CI/CD 概念与实践

### 3.1 核心概念

| 概念 | 全称 | 做什么 |
|------|------|--------|
| **CI** | Continuous Integration | 每次提交自动构建 + 跑测试，尽早发现问题 |
| **CD** | Continuous Delivery/Deployment | 自动打包部署到测试/生产环境 |

### 3.2 典型流水线

```
代码提交 → 触发 CI
  → 代码检查（Checkstyle / SpotBugs / SonarQube）
  → 编译
  → 单元测试
  → 集成测试
  → 打包（JAR / Docker Image）
  → 部署到测试环境
  → 自动化回归测试
  → 人工审批
  → 部署到生产环境
```

### 3.3 工具选型

| 工具 | 定位 | 特点 |
|------|------|------|
| **Jenkins** | 老牌 CI/CD 服务器 | 功能最全、插件生态丰富、但维护成本高 |
| **GitHub Actions** | GitHub 原生 CI/CD | 配置简单（YAML）、与 GitHub 深度集成 |
| **GitLab CI** | GitLab 原生 CI/CD | 私有部署友好 |

## 四、Linux 运维命令速查

### 4.1 文件与目录

| 命令 | 用途 | 常用参数 |
|------|------|---------|
| `ls` | 列出文件 | `-la`（详细 + 隐藏文件） |
| `find` | 查找文件 | `find /var/log -name "*.log" -mtime -7` |
| `grep` | 搜索文本 | `grep -rn "ERROR" /var/log/app/` |
| `tail` | 看文件尾部 | `tail -f app.log`（实时跟踪日志） |
| `wc` | 统计行数/字数 | `wc -l file.txt` |
| `du` | 目录大小 | `du -sh *`（当前目录各文件夹大小） |
| `df` | 磁盘使用率 | `df -h` |

### 4.2 进程管理

| 命令 | 用途 |
|------|------|
| `ps aux` | 查看所有进程 |
| `top` / `htop` | 实时查看 CPU/内存 |
| `kill -9 <pid>` | 强制杀进程 |
| `nohup java -jar app.jar &` | 后台运行 + 不随终端退出 |
| `jps` | 查看 Java 进程 |

### 4.3 网络相关

| 命令 | 用途 |
|------|------|
| `netstat -tlnp` / `ss -tlnp` | 查看端口监听 |
| `curl -v url` | 发 HTTP 请求（调试接口） |
| `ping` / `telnet host port` | 测试网络连通性 |
| `lsof -i:8080` | 查看占用 8080 端口的进程 |

### 4.4 线上排查组合拳

```bash
# CPU 100% 排查
top -Hp <pid>                     # 找到 CPU 最高的线程
printf "%x\n" <线程id>            # 转成 16 进制
jstack <pid> | grep <16进制tid>   # 在线程堆栈中定位

# 内存排查
jmap -heap <pid>                  # 查看堆内存使用
jmap -histo <pid> | head -20      # 查看对象数量排行
jmap -dump:format=b,file=dump.hprof <pid>  # 导出堆快照

# GC 排查
jstat -gcutil <pid> 1000 10       # 每秒打印 GC 统计，打 10 次
```
