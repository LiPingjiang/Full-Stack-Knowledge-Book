# 3.7 高可用架构（面试重点）

> **一句话定位**：高可用（High Availability, HA）不是「系统永远不出问题」，而是**故障一定会发生，但系统能限制爆炸半径、快速恢复，不让小故障变成全站事故**。面试考的核心就一件事——你有没有「故障视角」。

---

## 一、什么是高可用？——用 SLA 量化

SLA（Service Level Agreement）用「几个 9」描述可用性：

| 可用性 | 每年可停机 | 每月可停机 | 典型场景 |
|--------|-----------|-----------|---------|
| 99%（2 个 9） | 87.6 小时 | 7.3 小时 | 内部工具 |
| 99.9%（3 个 9） | 8.76 小时 | 43 分钟 | 普通电商 |
| 99.99%（4 个 9） | 52.6 分钟 | 4.3 分钟 | 支付/金融核心 |
| 99.999%（5 个 9） | 5.26 分钟 | 26 秒 | 电信级 |

> **面试问法**：「你们系统的可用性指标是多少？是怎么保证的？」——先报数字再展开手段，有数字才显得真实。

---

## 二、高可用的五个灵魂拷问

所有高可用手段都可以归到这五个问题下面（面试万能框架）：

1. **请求太多怎么办？** → 限流、排队、削峰
2. **下游变慢怎么办？** → 超时、重试、熔断、隔离
3. **核心服务挂了怎么办？** → 降级、冗余、故障转移
4. **重复请求怎么办？** → 幂等、防重、状态机
5. **上线前怎么证明扛得住？** → 压测、监控、故障演练

> 面试遇到「如何设计高可用系统」这种大题，直接按 1-5 展开即可，不会跑偏。

---

## 三、限流（Rate Limiting）

### 3.1 为什么限流？

系统容量有上限。不限流 → 突增流量把服务打穿 → 所有用户都不可用（比直接拒绝多余请求更糟）。

### 3.2 四种限流算法（面试高频对比）

| 算法 | 原理 | 优点 | 缺点 | 典型实现 |
|------|------|------|------|---------|
| **固定窗口** | 每个时间窗口（如 1s）维护计数器，超过阈值拒绝 | 简单 | **临界突刺**：窗口末尾+下个窗口开头的短时间内可能放行 2 倍流量 | 手写 AtomicInteger |
| **滑动窗口** | 把窗口拆成多个小格子，滑动统计总计数 | 平滑，解决突刺 | 内存存每个格子计数 | Sentinel 默认 |
| **漏桶** | 请求进桶，以恒定速率流出处理；桶满则拒绝 | 流量绝对平滑 | 无法应对突发合法流量（桶漏得太慢） | Nginx `limit_req` |
| **令牌桶** | 以恒定速率往桶中放令牌；请求取到令牌才放行，取不到就拒绝 | **允许突发**：桶里攒着令牌可以一次取多个 | 实现相对复杂 | Guava RateLimiter |

<details>
<summary><b>展开：面试追问「固定窗口的临界突刺是什么？」</b></summary>

假设限流规则是「每秒 100 次」。如果第 1 秒的最后 100ms 涌入 100 个请求（计数器还没到 100）、第 2 秒的前 100ms 又涌入 100 个请求（计数器刚重置），那么在 200ms 内系统实际放行了 200 个请求——是阈值的 2 倍。这就是「临界突刺」。

滑动窗口解决它的方式：不再整秒清零，而是把 1 秒拆成比如 10 个 100ms 的格子，每次统计当前时刻往前推 1 秒内的所有格子总和。这样边界处不会有 2 倍放行。

</details>

<details>
<summary><b>展开：令牌桶 vs 漏桶——面试必须说清的区别</b></summary>

最核心的区别是**对突发流量的态度**：

- **漏桶**：不管进来多少，处理速率恒定。突发流量只是在桶里排队或被丢弃——适合「强制匀速」的场景（如写数据库）。
- **令牌桶**：平时匀速攒令牌，突发时桶里有余量可以一次释放——适合「允许短时突发但长期不超标」的场景（如 API 网关）。

大白话：漏桶像匀速滴水的水龙头（不管你多急），令牌桶像有存款的钱包（攒够了可以突然花一大笔）。

</details>

### 3.3 单机限流 vs 分布式限流

| 维度 | 单机限流 | 分布式限流 |
|------|---------|-----------|
| 作用域 | 本机 QPS | 集群总 QPS |
| 实现 | Guava RateLimiter、Sentinel 本地模式 | Redis + Lua 脚本、Sentinel 集群模式、网关层 |
| 精确度 | 高（本机状态） | 有网络延迟，略有误差 |
| 适用场景 | 单体、本地保护 | 微服务、统一入口 |

## 四、熔断（Circuit Breaker）

### 4.1 为什么需要熔断？——防雪崩

服务 A 调服务 B，B 变慢了。如果 A 不断等待 B → A 的线程池被占满 → A 也变慢 → 依赖 A 的服务 C 也慢 → **级联雪崩**。

熔断器就是一个「自动保险丝」：**发现下游不行时，直接切断调用、快速失败，不让故障扩散。**

### 4.2 熔断器的三种状态

```
         失败率超阈值           探测请求成功
  CLOSED ──────────────→ OPEN ──────────────→ HALF_OPEN
    ↑                                            │
    └────────────── 探测全部成功 ─────────────────┘
                    （恢复）
```

| 状态 | 行为 |
|------|------|
| **Closed**（关闭/正常） | 放行所有请求，统计失败率 |
| **Open**（打开/熔断中） | 所有请求直接失败（快速失败），不真的调下游。经过一段冷却时间后进入 Half-Open |
| **Half-Open**（半开/探测） | 放行少量探测请求；如果成功 → 恢复 Closed；如果失败 → 回到 Open |

<details>
<summary><b>展开：面试追问「熔断器的判定规则有哪些指标？」</b></summary>

不是「一失败就熔断」，而是多个条件共同判定：

- **最小请求数**（如 20 次）：请求数不够时不触发，防误判
- **错误率阈值**（如 50%）：最近窗口内失败比例超标
- **慢调用比例**（如 RT > 1s 的请求占 60%）：下游虽然没报错但太慢
- **冷却时间**（如 10s）：Open 状态持续多久才尝试 Half-Open
- **探测数量**（如 5 次）：Half-Open 放多少请求试探

Sentinel 和 Resilience4j 的默认值不同，面试时能说出你项目里配的具体数字是加分项。

</details>

### 4.3 常见实现框架

| 框架 | 生态 | 特点 |
|------|------|------|
| **Sentinel**（阿里） | Spring Cloud Alibaba | 限流+熔断+降级一体，控制台可视化，规则可动态推送 |
| **Resilience4j** | Spring Boot / 通用 | 轻量、函数式 API、模块化（CircuitBreaker/RateLimiter/Retry 可单独用） |
| **Hystrix**（Netflix） | Spring Cloud Netflix | 已停止维护，但思想经典，面试仍会问 |

## 五、降级（Degradation）

### 5.1 降级 vs 熔断——面试必须区分

| | 降级 | 熔断 |
|--|------|------|
| **触发原因** | 自身系统压力大 / 主动策略 | 下游服务故障 / 被动触发 |
| **行为** | 关闭非核心功能，保核心 | 切断对故障下游的调用 |
| **比喻** | 自己丢车保帅 | 把坏掉的队友踢出局 |

### 5.2 降级策略

- **功能降级**：关闭评论、推荐、日志收集等非核心功能，只保登录+核心交易
- **数据降级**：返回缓存中的旧数据（兜底数据）而非实时查询
- **体验降级**：从个性化推荐退化为热门列表
- **写降级**：请求写入消息队列异步处理，而非同步落库

## 六、超时、重试与幂等（最容易出事故的三兄弟）

### 6.1 超时：所有远程调用必须设超时

> **铁律**：没有超时 = 线程可能无限等待 = 线程池耗尽 = 服务不可用。

设多长？典型策略：

- 读操作：P99 延迟的 2-3 倍（如 P99 是 200ms → 超时设 500ms）
- 写操作：略长（可能涉及落库），但一定要有上限
- 层层递减：网关 > 服务 A > 服务 B（越往下越短，否则上游超时了下游还在跑，浪费资源）

### 6.2 重试：不是万能药，用错会致命

| 可以重试的场景 | 不能重试的场景 |
|--------------|--------------|
| 网络抖动（超时、连接断开） | 业务逻辑错误（参数非法、余额不足） |
| 下游返回 503（过载） | 下游返回 400/422（明确拒绝） |
| 读操作（天然幂等） | 非幂等写操作（没做幂等设计的扣款） |

**重试三要素**（面试必答）：

1. **指数退避（Exponential Backoff）**：第 1 次等 100ms，第 2 次 200ms，第 3 次 400ms……避免同一时间所有客户端同时重试把下游打挂
2. **随机抖动（Jitter）**：退避时间加随机偏移，避免「惊群效应」（所有客户端退避到同一时刻再次同时重试）
3. **有限次数**：通常 2-3 次封顶，不无限重试

> **致命公式**：没有幂等的重试 + 没有退避的重试 = 重复扣款 + 下游被打挂 = 线上事故。

### 6.3 幂等：写操作可以安全重试的前提

**幂等**（Idempotent）= 同一个请求执行 1 次和执行 N 次，效果完全相同。

| 操作类型 | 天然幂等？ | 说明 |
|---------|-----------|------|
| 查询（GET） | ✅ | 读操作天然幂等 |
| 删除（幂等删除） | ✅ | 删一个不存在的东西也只是「没有变化」 |
| 更新为固定值（SET x=5） | ✅ | 无论执行几次结果一样 |
| 创建/新增（INSERT） | ❌ | 重复执行会多创建 |
| 增量更新（x=x+1） | ❌ | 每次执行结果不同 |

<details>
<summary><b>展开：保证幂等的常见方案（面试高频）</b></summary>

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **唯一请求 ID + 去重表** | 客户端生成全局唯一 requestId，服务端先查去重表，有就直接返回上次结果 | 通用，支付/下单 |
| **数据库唯一约束** | 利用 UNIQUE KEY 拒绝重复插入 | 防重复创建 |
| **乐观锁（version）** | UPDATE ... SET version=version+1 WHERE version=旧版本 | 防并发重复更新 |
| **状态机** | 只允许 A→B→C 单向流转，重复提交同一状态直接忽略 | 订单状态变更 |
| **Token 机制** | 先获取一次性 Token，提交时带上；服务端校验后删除 Token | 防页面重复提交 |

核心思想：**让服务端能识别出「这个请求我已经处理过了」，然后安全地返回上次结果而不是再执行一遍。**

</details>

## 七、冗余与故障转移

### 7.1 消除单点故障（SPOF）

凡是只有一份的组件都是单点。消除方式就是**冗余**——但冗余不是简单「多部署几台」：

| 层 | 单点风险 | 冗余手段 |
|---|---------|---------|
| 应用层 | 只有 1 台服务实例 | 多实例 + 负载均衡（Nginx/LB） |
| 数据库 | 只有 1 个 MySQL | 主从复制 + 自动主从切换（MHA/Orchestrator） |
| 缓存 | 只有 1 个 Redis | Redis Sentinel 或 Redis Cluster |
| 消息队列 | 只有 1 个 Broker | Kafka 多 Broker + 分区副本 |
| 网关/LB | 只有 1 个入口 | 双活网关 + DNS 轮询/VRRP |
| 机房 | 只有 1 个机房 | 同城双活 / 异地多活 |

### 7.2 负载均衡策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 轮询（Round Robin） | 依次分配 | 实例配置相同 |
| 加权轮询 | 按权重比例分配 | 实例配置不同 |
| 最少连接 | 分配给当前连接数最少的实例 | 长连接场景 |
| IP Hash | 同一 IP 固定到同一实例 | 需会话亲和（尽量避免） |
| 一致性 Hash | 节点增减时只迁移少量流量 | 缓存层 |

### 7.3 主从切换与故障转移

以 MySQL 为例：主库挂了 → Sentinel/MHA 检测到 → 提升从库为新主 → 应用切换连接。关键指标是**故障检测时间 + 切换时间**（目标：秒级）。

## 八、容灾与多活

| 模式 | 含义 | 复杂度 | RPO/RTO |
|------|------|--------|---------|
| 冷备 | 备份在另一机房存着，出事才拉起 | 低 | RPO 高（可能丢数据）、RTO 高（小时级） |
| 同城双活 | 同城两个机房同时服务，数据同步复制 | 中 | RPO≈0、RTO 秒级 |
| 异地多活 | 多城市机房同时服务，数据最终一致 | 极高 | RPO 可能有延迟、RTO 秒级 |

> **面试常见问法**：「你们的容灾方案是什么级别的？」「异地多活数据冲突怎么解决？」——能答出同城双活 + 数据同步延迟处理已经很不错了。

## 九、压测、监控与故障演练

### 9.1 压测

| 指标 | 含义 |
|------|------|
| QPS（Queries Per Second） | 每秒处理请求数 |
| RT（Response Time） | 响应时间（关注 P50/P99/P999） |
| 错误率 | 请求失败比例 |
| 资源使用率 | CPU/内存/网络/磁盘 IO |

**压测目的**：找到系统容量天花板、发现瓶颈、验证限流/熔断/降级是否生效。

### 9.2 监控三板斧

- **系统监控**：CPU、内存、磁盘、网络（Prometheus + Grafana）
- **应用监控**：QPS、RT、错误率、线程池、连接池（Metrics/Micrometer）
- **业务监控**：订单量、支付成功率、核心转化率（自定义大盘）

### 9.3 故障演练（Chaos Engineering）

> 主动注入故障（杀实例、断网络、加延迟），验证系统容错能力。代表工具：Netflix Chaos Monkey、阿里 ChaosBlade。

## 十、面试万能回答模板

遇到「如何设计一个高可用系统」这类大题，按七步走：

1. **明确 SLA 目标**：99.9% 还是 99.99%？成本完全不同
2. **识别核心链路**：哪些功能不能挂（下单/支付），哪些可以降级（推荐/评论）
3. **消除单点故障**：应用多实例、DB 主从、缓存集群、MQ 多副本
4. **控制入口流量**：网关限流 + 业务限流（令牌桶/滑动窗口）
5. **保护下游依赖**：超时 + 重试(退避+幂等) + 熔断 + 隔离
6. **保证请求安全**：幂等设计、防重机制
7. **建立观测与恢复能力**：监控告警 + 压测 + 预案 + 故障演练

---

## 本篇小结

- 高可用的核心不是「零故障」而是**故障可预期、可隔离、可快速恢复**。
- 限流四大算法（固定窗口/滑动窗口/漏桶/令牌桶）是高频对比题，重点记住临界突刺和突发流量的区别。
- 熔断三态（Closed→Open→Half-Open）防级联雪崩；降级是自己丢车保帅；两者触发原因不同。
- 超时/重试/幂等**必须一起设计**——无幂等的重试是事故根源。
- 冗余 ≠ 多部署几台；要配合健康检测、自动故障转移、数据同步才是真的高可用。
- 面试答「高可用设计」用七步模板：SLA → 核心链路 → 消除单点 → 限流 → 熔断超时重试 → 幂等 → 监控演练。
