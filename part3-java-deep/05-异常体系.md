# 3.5 Java 异常体系：受检 vs 非受检的设计哲学与争议

> 异常处理是后端工程师每天都在写的东西，但 Java 的「受检异常」设计是编程语言史上**最有争议的决定之一**。
> 这一节讲透它的哲学与争议，它是后面对比 Go 的 `error`、Rust 的 `Result`、JS 的 `try/catch` 的基准。

---

## 一、Java 异常的家族树

Java 所有异常都继承自 `Throwable`，分成两大分支：

```
                  Throwable
                 ╱         ╲
            Error          Exception
        (JVM级严重错误)    ╱          ╲
        OutOfMemoryError  RuntimeException   其他 Exception
        StackOverflow    (非受检/unchecked)  (受检/checked)
                          NullPointerException  IOException
                          IllegalArgument...    SQLException
```

三类，记住它们的定位：

| 类型 | 例子 | 是否强制处理 | 含义 |
|------|------|------------|------|
| **Error** | `OutOfMemoryError` | 不强制（也无力处理） | JVM 级严重错误，通常程序无法恢复 |
| **受检异常**（Checked） | `IOException`、`SQLException` | **编译器强制**处理 | 可预期、可恢复的外部问题 |
| **非受检异常**（Unchecked/Runtime） | `NullPointerException`、`IllegalArgumentException` | 不强制 | 通常是程序 bug |

---

## 二、受检异常：Java 独有的「编译器强制」

Java 最特别的设计是**受检异常**——如果一个方法可能抛出受检异常，**调用者必须显式处理（try-catch 或继续 throws）**，否则编译不通过。这是 Java 几乎独有的特性，主流语言里少见。

```java
// readFile 声明会抛 IOException（受检异常）
void readFile(String path) throws IOException {
    Files.readAllLines(Path.of(path));
}

// 调用者被编译器强制：要么 catch，要么继续 throws，否则编译不过
void caller() {
    readFile("a.txt");   // 编译错误！没处理 IOException
}

// 正确写法之一
void caller2() {
    try {
        readFile("a.txt");
    } catch (IOException e) {
        // 必须处理
    }
}
```

**设计初衷（理想很美好）**：受检异常想强制程序员**正视那些可预期的失败**（文件不存在、网络断开、SQL 出错），不让你「忘了处理」。这是一种「编译期就提醒你考虑错误路径」的善意设计。

---

## 三、争议：为什么后来的语言都不学受检异常

理想很美好，但受检异常在实践中暴露了严重问题，导致**几乎所有后起语言（C#、Kotlin、Go、Rust、Scala）都拒绝了它**。争议点：

**1. 样板代码泛滥**。受检异常迫使代码里塞满 try-catch，很多时候你根本不知道该怎么处理，只好「假装处理」：

```java
try {
    doSomething();
} catch (IOException e) {
    e.printStackTrace();   // 反模式！吞掉异常、假装处理，比不处理还糟
    // 或者
    throw new RuntimeException(e);   // 包装成非受检往上抛，绕过编译器
}
```

**2. 破坏抽象 / 接口污染**。一个底层方法加个受检异常，整条调用链的方法签名都得跟着改 `throws`，像传染病一样蔓延。

**3. 与函数式/Lambda 冲突**。Java 8 的 Stream、Lambda 几乎无法优雅地处理受检异常，写起来极其别扭：

```java
list.stream()
    .map(path -> readFile(path))   // 编译错误！Lambda 里抛受检异常无法直接处理
    .collect(toList());
```

正因为这些痛点，**Kotlin（同样跑在 JVM 上）直接取消了受检异常**，Spring 等现代框架也大量使用非受检异常（如 `DataAccessException`）来规避受检异常的传染性。

## 四、异常处理的实战最佳实践

抛开争议，落到日常，记住这些经过验证的准则：

```java
// ✅ 1. 捕获具体异常，不要一把抓 Exception/Throwable
try { ... }
catch (FileNotFoundException e) { ... }   // 具体
catch (IOException e) { ... }              // 而非笼统的 catch (Exception e)

// ✅ 2. 绝不"吞"异常（空 catch 块是大罪）
try { ... } catch (Exception e) { }   // ❌ 灾难：错误悄无声息地消失

// ✅ 3. 包装异常时保留原始 cause（保留调用栈，方便排查）
catch (SQLException e) {
    throw new ServiceException("查询用户失败", e);   // 传入 e 作为 cause
}

// ✅ 4. 用 try-with-resources 自动关资源（替代繁琐的 finally）
try (var conn = dataSource.getConnection();
     var stmt = conn.prepareStatement(sql)) {
    // conn 和 stmt 会自动 close，哪怕抛异常
}

// ✅ 5. 自定义业务异常时，优先继承 RuntimeException（避免受检传染）
class BusinessException extends RuntimeException { ... }
```

核心原则：**异常要么处理掉、要么如实抛出，绝不静默吞没；包装时一定保留原始 cause。** 一个吞掉的异常，可能让你排查线上问题时彻底抓瞎。

## 五、埋给第四章的「钩子」：错误处理的四种哲学

Java 的「异常 + 受检/非受检」只是错误处理的一种范式。受检异常的争议，恰恰引出了其他语言的不同选择。这是第四章 [错误处理对比](../part4-multilang-compare/06-错误处理对比.md) 的核心：

| 语言 | 错误处理范式 | 核心思想 |
|------|------------|---------|
| **Java** | 异常（throw/try-catch） + 受检/非受检 | 错误是「跳出正常流程」的事件 |
| **Go** | **多返回值 `error`** | 错误是「普通返回值」，必须显式检查 `if err != nil` |
| **Rust** | **`Result<T, E>` + `?` 运算符** | 错误是「类型系统的一部分」，编译器强制处理 |
| **JS/TS** | `try-catch` + Promise rejection | 类似 Java 但无受检，异步还有额外机制 |

> 这是个极有张力的对比：[Go](../part4-multilang-compare/03-Java到Go.md) 认为「异常是隐藏的控制流，不好」，干脆把错误降级成普通返回值，逼你显式检查（写起来啰嗦但清晰）。[Rust](../part4-multilang-compare/04-Java到Rust.md) 用 `Result` 把错误编进类型系统，并用 `?` 运算符让「显式传播」变得优雅——某种意义上是「受检异常」理念的成功重生（编译器强制，但不啰嗦）。**当你理解了 Java 受检异常「想做对却做砸了」的教训，就能看懂 Rust 的 `Result` 是怎么把它做对的。**

## 六、给后端大脑的速查表

| 概念 | 一句话本质 | 实战 |
|------|-----------|------|
| Error | JVM 级严重错误 | 一般不处理（处理不了） |
| 受检异常 | 编译器强制处理 | 可预期的外部失败（IO/SQL） |
| 非受检异常 | 不强制 | 多为程序 bug（NPE 等） |
| 受检异常争议 | 样板多、传染、与 Lambda 冲突 | 后来语言都不学它 |
| try-with-resources | 自动关资源 | 替代繁琐 finally |
| 保留 cause | 包装时传入原异常 | 不丢调用栈 |

---

## 本章小结

- Java 异常分 **Error**（JVM 严重错误）、**受检异常**（编译器强制处理，如 IOException）、**非受检异常**（不强制，多为 bug）。
- **受检异常**是 Java 几乎独有的设计，初衷是「强制正视可预期失败」，但因**样板泛滥、接口污染、与 Lambda 冲突**饱受争议，后来的语言（Kotlin/Go/Rust）都拒绝了它。
- 实战准则：捕获具体异常、绝不吞异常、包装保留 cause、用 try-with-resources 管资源、业务异常优先继承 RuntimeException。
- 错误处理有四种哲学：Java 异常、Go 返回值 error、Rust 的 Result、JS 的 try-catch——第四章 [错误处理对比](../part4-multilang-compare/06-错误处理对比.md) 会用同一个场景实测对比，你会看到 Rust 如何把「受检异常的理想」真正实现好。

至此第三章完结。你现在有了一把精确的「Java 尺子」——并发、内存模型、运行时、类型系统、错误处理五个维度都讲透了，每个维度都埋好了通往第四章的钩子。**带着这把尺子，我们进入第四章，开始真正的多语言对比。**
